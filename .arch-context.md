### `IntegrationRegistry.java` (full source — 172 lines)

- Discovers all `@IntegrationAdapter`-annotated beans at startup; fails fast on duplicate (domain, slug).
- `<T> T resolve(IntegrationDomain domain, Class<T> portInterface)` — looks up the active adapter for the **current tenant** via `RequestScopes.TENANT_ID`. Caches `(tenantSchema, domain)` → config in **Caffeine, 60s TTL, max 1000 entries**.
- Falls back to `domain.getDefaultSlug()` if no row, or row disabled, or row's slug has no registered adapter. Throws `IllegalStateException` if even the default has no adapter.
- `<T> T resolveBySlug(IntegrationDomain, String slug, Class<T>)` — for webhook controllers that route before tenant context exists. **SECURITY**: caller must allowlist the slug; raw-from-HTTP slugs are forbidden.
- `availableProviders(IntegrationDomain)` — lists registered slugs.
- `evict(tenantSchema, domain)` — invalidates cache on config change. `IntegrationService` calls this after every upsert/toggle.

### `IntegrationGuardService.java` (full source)

```java
@Service
public class IntegrationGuardService {
  private final OrgSettingsRepository orgSettingsRepository;

  public IntegrationGuardService(OrgSettingsRepository r) { this.orgSettingsRepository = r; }

  @Transactional(readOnly = true)
  public void requireEnabled(IntegrationDomain domain) {
    if (domain == IntegrationDomain.PAYMENT
        || domain == IntegrationDomain.EMAIL
        || domain == IntegrationDomain.KYC_VERIFICATION) {
      return; // Always available — core infrastructure
    }
    var settings = orgSettingsRepository.findForCurrentTenant();
    boolean enabled = settings.map(s -> switch (domain) {
      case ACCOUNTING       -> s.isAccountingEnabled();
      case AI               -> s.isAiEnabled();
      case DOCUMENT_SIGNING -> s.isDocumentSigningEnabled();
      // EMAIL/KYC/PAYMENT unreachable due to early return
      default -> true;
    }).orElse(false);
    if (!enabled) throw new IntegrationDisabledException(domain);
  }
}
```

Two-layer gating:
1. **Module-level**: `OrgSettings.accountingEnabled` flag — feature-module gating from Phase 62.
2. **Configuration-level**: `OrgIntegration.enabled` boolean on the row.

### `IntegrationService.java` (309 lines — orchestration patterns)

- **Upsert**: find-by-domain or new; `evictCache` afterward; emit `integration.configured` audit event with previous-provider details.
- **Toggle**: emit `integration.enabled` / `integration.disabled` audit event.
- **setApiKey**: validate non-blank, store in `SecretStore` keyed by `IntegrationKeys.apiKey(domain, slug)`, store last-6-chars suffix on entity, emit `integration.key_set` audit.
- **deleteApiKey**: delete from secret store, clear suffix, emit `integration.key_removed` audit.
- **testConnection**: dispatches per-domain via `IntegrationRegistry.resolve(domain, …Provider.class).testConnection()`; AI has bespoke logic. Emit `integration.connection_tested` audit including `errorMessage` if failed.
- **listAllIntegrations**: returns one DTO per `IntegrationDomain` value (synthesises `unconfigured(domain)` for missing rows).

### `EncryptedDatabaseSecretStore.java` (full source — 129 lines)

- Implementation: `OrgSecret` JPA entity in `org_secrets` table (per tenant schema).
- Encryption: **AES/GCM/NoPadding**, 256-bit key, 96-bit IV per record.
- Key sourced from `${integration.encryption-key}` Spring property — Base64-encoded 32 bytes. Validated at `@PostConstruct`; app refuses to start without it.
- Each record stores `secret_key`, `encrypted_value` (Base64), `iv` (Base64), `key_version` (currently always 1).
- `store()` upserts; `retrieve()` decrypts; `delete()` is single-row.

> Phase 71 should reuse `SecretStore` directly. Multiple Xero secrets become multiple `OrgSecret` rows under different keys, e.g.:
> - `accounting:xero:client_id`
> - `accounting:xero:client_secret`
> - `accounting:xero:refresh_token`
> - `accounting:xero:access_token` (short-lived; consider not persisting, or persisting with TTL semantics).
> Either extend `IntegrationKeys` with helpers, or define Xero-local key constants colocated with the adapter.

## 5. Invoice + Payment Domain

### `Invoice.java` (484 lines, key shape)

Entity `invoices` table. Important columns for Xero sync:

- `id` (UUID), `customer_id` (UUID), `invoice_number` (String, ≤20 chars — null until APPROVED), `status` (`InvoiceStatus` enum: DRAFT, APPROVED, SENT, PAID, VOID), `currency` (3-char ISO), `issue_date`, `due_date`.
- Money: `subtotal`, `tax_amount`, `total` — all `BigDecimal(14,2)`.
- Customer denormalisation: `customer_name`, `customer_email`, `customer_address` (snapshotted at creation — invoice doesn't reflect later customer edits).
- Org denormalisation: `org_name`.
- People: `created_by` (UUID), `approved_by` (UUID, set on `approve()`).
- Tax: `tax_type` (enum), `applied_field_groups` (JSONB List<UUID>).
- Custom fields: `custom_fields` JSONB Map.
- Payment glue: `payment_session_id`, `payment_url`, `payment_destination` (default "OPERATING"), `payment_reference`, `paid_at`, `billing_run_id`.
- Period: `billing_period_start`, `billing_period_end`, `po_number`.

State transitions encoded as methods:
- `approve(invoiceNumber, approvedBy)` — DRAFT → APPROVED, sets `issueDate = today` if null.
- `applyDefaults(defaultDueDate, defaultPaymentTerms)` — fills in dueDate / paymentTerms on APPROVED rows.
- `markSent()` — APPROVED → SENT.
- `recordPayment(reference)` — SENT → PAID, sets `paidAt = now`.
- `reversePayment()` — PAID → SENT.
- `voidInvoice()` — APPROVED|SENT → VOID. `voidDraft()` — DRAFT → VOID.

## 10. Domain Event Listener Pattern

### `InvoiceEmailEventListener.java` (full source — 78 lines)

```java
@Component
public class InvoiceEmailEventListener {
  private static final Logger log = …;
  private final InvoiceRepository invoiceRepository;
  private final InvoiceEmailService invoiceEmailService;
  private final PdfRenderingService pdfRenderingService;
  private final DocumentTemplateRepository documentTemplateRepository;

  public InvoiceEmailEventListener(…) { … }

  @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
  public void onInvoiceSent(InvoiceSentEvent event) {
    if (event.tenantId() == null) {
      log.warn("InvoiceSentEvent has null tenantId, dropping: invoice={}", event.entityId());
      return;
    }
    RequestScopes.runForTenant(event.tenantId(), event.orgId(),
        () -> handleInvoiceSent(event));
  }

  private void handleInvoiceSent(InvoiceSentEvent event) {
    try {
      var invoice = invoiceRepository.findById(event.entityId()).orElse(null);
      if (invoice == null) { log.warn(…); return; }
      var templateOpt = documentTemplateRepository
          .findByPrimaryEntityTypeAndActiveTrueOrderBySortOrder(TemplateEntityType.INVOICE)
          .stream().findFirst();
      if (templateOpt.isEmpty()) { log.warn(…); return; }
      var pdfResult = pdfRenderingService.generatePdf(
          templateOpt.get().getId(), invoice.getId(), event.actorMemberId());
      invoiceEmailService.sendInvoiceEmail(invoice, pdfResult.pdfBytes());
    } catch (Exception e) {
      log.error("Failed to send invoice email for invoice={}", event.entityId(), e);
    }
  }
}
```

**Pattern critical for Phase 71**:
1. `@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)` — fires only after the DB transaction commits (so the invoice row is durable before we sync externally).
2. `RequestScopes.runForTenant(event.tenantId(), event.orgId(), () -> …)` — re-establishes the tenant ScopedValue inside the listener thread; **mandatory** because event handlers run outside the original request context.
3. Defensive null-tenantId check — drop event with warning rather than NPE.
4. Try/catch around the work — the AFTER_COMMIT phase means the transaction is gone; nothing to roll back. Failures must be logged, not propagated. (For Xero sync: needs an outbox table or retry mechanism since this single-attempt fire-and-log isn't durable enough for accounting.)

```java
@Component
public class AutomationScheduler {
  private static final Logger log = …;
  private static final long POLL_INTERVAL_MS = 900_000; // 15 minutes

  private final TenantScopedRunner tenantScopedRunner;
  private final ActionExecutionRepository actionExecutionRepository;
  …
  private final TransactionTemplate transactionTemplate;
  private final ObjectMapper objectMapper;

  @Scheduled(fixedDelay = POLL_INTERVAL_MS)
  public void pollDelayedActions() {
    log.debug("Automation scheduler started");
    int[] processed = {0};
    tenantScopedRunner.forEachTenant((tenantId, orgId) ->
        processed[0] += processTenant());
    if (processed[0] > 0) log.info("…{} delayed actions processed", processed[0]);
  }

  private int processTenant() {
    Integer processed = transactionTemplate.execute(tx -> {
      var dueActions = actionExecutionRepository.findDueScheduledForUpdate(
          ActionExecutionStatus.SCHEDULED.name(), Instant.now());
      // … FOR UPDATE row lock; iterate and execute each
    });
    return processed != null ? processed : 0;
  }
}
```

**Pattern**:
1. `@Scheduled(fixedDelay = …)` — Spring scheduled annotation; `fixedDelay` (not `cron`) so the next run is delayed by the previous completion + this duration.
2. **`TenantScopedRunner.forEachTenant((tenantId, orgId) -> …)`** — this is the canonical multi-tenant scheduler primitive. It iterates over every active tenant and binds `RequestScopes.TENANT_ID` per-tenant for the duration of the lambda. Phase 71's "retry pending Xero sync" worker must use this.
3. **`SELECT … FOR UPDATE`** to claim rows during processing.
4. `TransactionTemplate` for explicit transaction boundary inside the loop iteration.

## 13. Other Cross-Cutting Notes for the Architect

1. **No Testcontainers**, ever (per backend CLAUDE.md). Embedded Postgres + `InMemoryStorageService`. Phase 71 integration tests for the Xero adapter must mock the HTTP client — a hand-rolled `XeroClient` interface with a fake impl is the canonical path. The codebase uses Mockito + manual fakes; no WireMock dependency exists today (verify before suggesting).
2. **Controllers are thin HTTP adapters only** (per backend CLAUDE.md). One service call per endpoint, `ResponseEntity` wrapper, `@RequiresCapability` annotation, no repos/conditions/business-logic. `IntegrationController` is COMPLIANT and is the right reference. Phase 71's new Xero endpoints must follow the thin-adapter rule.
3. **Spring profiles**: `local` (Docker Compose Postgres), `dev` (Neon dev branch + AWS S3), `prod` (Neon main + AWS S3 + full obs), `test` (embedded Postgres). Production secrets via env vars (e.g. `XERO_CLIENT_ID`, `XERO_CLIENT_SECRET` would be platform-level if it's a single Xero app servicing all tenants). Consider whether each tenant brings its own Xero app or whether the platform owns one app — affects secret model.
4. **Schema-per-tenant**: all Phase 71 tenant data lands in `tenant_<hash>` schema. No `tenant_id` column. Migration goes in `db/migration/tenant/`. Per ADR-T001.
5. **No plan-tier gating**: per project memory and Phase 70 stub note ("PlanTier / PRO-tier gating per strategic decision"), Phase 71 must not gate features on plan tiers. The frontend `IntegrationCard` still has stale `tier === "PRO"` logic for AI; do not propagate.
6. **OutboxEvent / DomainEvent**: events implement a `DomainEvent` interface with `tenantId`, `orgId`, etc. Look at `event/InvoiceApprovedEvent.java`. Phase 71's Xero sync should listen on existing invoice events and not emit a new domain event unless a downstream consumer needs it.
7. **JSONB + sealed-interface pattern** (ADR-148, restated by ADR-270): use this for `OrgIntegration.config_json` and any new per-provider config records. Sealed `AccountingProviderConfig` interface with `XeroConfig` record (carries org-id mapping, default account codes, tax-code mapping, sync-on-approve flag, etc.) is the right shape.
8. **Caffeine 60s TTL on integration registry** — Xero adapter resolution will pay a 60s cache hit; config updates (changing default account code) will stale up to 60s. Document this in the phase architecture; tighten only if user-visible.

## 14. Summary — What the Architect Needs to Decide

1. **Reuse vs extend `AccountingProvider` port**: current 3-method contract (`syncInvoice`, `syncCustomer`, `testConnection`) probably insufficient for Xero. Plausibly add: `voidInvoice`, `recordPayment` (push payment events back to Xero), `syncTaxRate` (or one-off mapping config), `disconnect` (revoke OAuth).
2. **Secret model**: extend `IntegrationKeys` for multi-secret-per-provider, OR define Xero-local key constants. OAuth refresh-token rotation needs a writable secret store (it exists — `SecretStore.store()` overwrites).
3. **OAuth flow**: state machine + redirect URI + callback controller. New endpoints under `/api/integrations/accounting/xero/oauth/{start,callback}`. Whose-Xero-app: platform vs per-tenant?
4. **Outbox / retry**: `@TransactionalEventListener(AFTER_COMMIT)` is single-attempt. Xero needs an outbox table + the AutomationScheduler-style polling worker for retries. Suggest new `accounting_sync_outbox` (id, invoice_id, attempt_count, next_attempt_at, status, error_message, last_attempted_at, created_at) per tenant.
5. **Tax-code / account-code mapping**: a SA `legal-za` tenant has VAT rates locally; Xero has its own tax codes. New entity `xero_tax_rate_mapping` (local_tax_rate_id → xero_tax_type) per tenant, OR JSONB on the integration row. Recommend table for query-ability.
6. **Capabilities**: reuse `TEAM_OVERSIGHT` for connect/disconnect (precedent from `IntegrationController`); reuse `INVOICING` for sync-now actions.
7. **ADR slots**: ADR-272 onward. Likely candidates: ADR-272 (XeroAccountingProvider over hand-rolled REST: use Xero's published Java SDK or roll our own?); ADR-273 (tenant-scoped Xero secrets via SecretStore vs Vault); ADR-274 (outbox over fire-and-forget for accounting sync); ADR-275 (per-tenant account/tax mapping table vs JSONB); ADR-276 (OAuth flow scope: platform-app vs tenant-app); ADR-277 (sync triggers: AFTER_COMMIT events + manual + scheduled retry); ADR-278 (failure semantics + audit-event surface); ADR-279 (deferred — reserved).
8. **Migration slot**: tenant V121 (with possible V122 for capability seed if a new capability is introduced).
9. **Architecture doc filename**: `architecture/phase71-xero-accounting-integration.md` (or product-decided slug).
