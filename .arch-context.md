## 7. Representative Tenant-Aware Entity Pattern

### 7a. Proposal (the win-loop reuse target — full source pattern)

`backend/src/main/java/io/b2mash/b2b/b2bstrawman/proposal/Proposal.java`

Key pattern elements (a Deal entity should mirror these):
- `@Entity @Table(name = "proposals")` — **no `tenant_id` column.** Tenant isolation is pure
  schema-per-tenant (Hibernate `search_path`); a standard `JpaRepository.findById()` is tenant-safe.
- `@Id @GeneratedValue(strategy = GenerationType.UUID) private UUID id;`
- Status as `@Enumerated(EnumType.STRING) @Column(...length=20)` + a `Set<ProposalStatus> TERMINAL_STATUSES`.
- A human-friendly counter field (`proposal_number`, fed by `ProposalNumberService` + `ProposalCounter`).
- Foreign keys held as raw `UUID` columns (`customer_id`, `portal_contact_id`, `created_project_id`),
  **not** JPA `@ManyToOne` relations — the codebase deliberately uses UUID references across aggregates.
- Tiptap document content stored as `@JdbcTypeCode(SqlTypes.JSON) @Column(columnDefinition="jsonb") Map<String,Object> contentJson`.
- `@PrePersist`/`@PreUpdate` callbacks set `createdAt`/`updatedAt`.
- **Rich-domain lifecycle methods** that guard transitions and throw `InvalidStateException`:
  `markSent(portalContactId)`, `markAccepted()` (rejects expired), `markDeclined(reason)`,
  `markWithdrawn()`, `markExpired()`. Plus `requireStatus(Set<Status>, action)` private guard,
  `isEditable()`, `isTerminal()`, `isExpired()`, and **guarded setters** that call `requireEditable()`
  so only DRAFT proposals are mutable. Result references (`setCreatedProjectId`) are NOT guarded.

```
Lifecycle: DRAFT → SENT → ACCEPTED | DECLINED | EXPIRED   (only DRAFT is editable)
```

`ProposalStatus` enum (`proposal/ProposalStatus.java`): `DRAFT, SENT, ACCEPTED, DECLINED, EXPIRED`.

> The architect should mirror this rich-domain + UUID-reference + JSONB-content + counter pattern for
> a `Deal` entity (e.g. stages `LEAD → QUALIFIED → PROPOSAL → WON | LOST`, guarded transitions,
> `customer_id`/`created_proposal_id` UUID refs, optional `notes` JSONB).

### 7b. Customer (Deals link to Customer)

`backend/src/main/java/io/b2mash/b2b/b2bstrawman/customer/Customer.java`

`@Entity @Table(name="customers")`. Fields: `id (UUID)`, `name`, `email`, `phone`, `idNumber`,
`status` (String `"ACTIVE"`/`"ARCHIVED"`), `notes (TEXT)`, `createdBy`, `createdAt`, `updatedAt`,
`customFields (jsonb Map)`, `appliedFieldGroups (jsonb List<UUID>)`, `customerType (enum)`,
`lifecycleStatus (enum)` + `lifecycleStatusChangedAt/By`, `offboardedAt`, plus company/address fields
(`registrationNumber`, `addressLine1/2`, `city`, `stateProvince`, `postalCode`, `country`,
`taxNumber`, `contactName/Email/Phone`, `entityType`, `financialYearEnd`).

Rich-domain methods: `update(...)`, `archive()`/`unarchive()`,
`transitionLifecycleStatus(target, actorId)` (validates via the enum's `canTransitionTo`, throws
`InvalidStateException`), `setLifecycleStatus(status, changedBy)` (bypasses validation for system ops),
`anonymize(replacementName)` (POPIA — nulls all PII).

**`LifecycleStatus` enum** (`customer/LifecycleStatus.java`) — validated transition graph:
```
PROSPECT    → ONBOARDING
ONBOARDING  → ACTIVE | OFFBOARDING
ACTIVE      → DORMANT | OFFBOARDING
DORMANT     → ACTIVE | OFFBOARDING
OFFBOARDING → OFFBOARDED
OFFBOARDED  → ACTIVE | ANONYMIZED
ANONYMIZED  → (terminal)
```
New customers default to `PROSPECT`. Proposal acceptance auto-transitions PROSPECT → ONBOARDING.

**`CustomerType` enum** (`customer/CustomerType.java`): `INDIVIDUAL, COMPANY, TRUST` (default INDIVIDUAL).

> Phase 80 relevance: a Deal references a Customer by `customer_id (UUID)`. The win-loop (deal won →
> proposal/project) should respect the customer lifecycle guard (a PROSPECT customer auto-advances to
> ONBOARDING on acceptance — see §8).

### The acceptance → project orchestration (win-loop)

**Entry point:** `ProposalOrchestrationService.acceptProposal(UUID proposalId, UUID portalContactId) : OrchestrationResult` — `@Transactional`.

Steps (within one transaction):
1. Load proposal; reject if status != `SENT`; reject if `portalContactId` mismatch.
2. `proposal.markAccepted()` (rejects if expired) → save.
3. **`transitionCustomerLifecycle(customerId, actorId)`** — if customer is `PROSPECT`, advance to
   `ONBOARDING` (the lifecycle guard blocks creating a project under a PROSPECT customer).
4. **`createProject(proposal)`** → set `proposal.createdProjectId`, save.
5. `auditService.log(...)` with `eventType="proposal.accepted"`, `entityType="proposal"`,
   `details` including `proposal_number`, `customer_id`, `project_id` (project_id MUST be present —
   the matter Activity feed and portal Firm-actions trail scope on `details->>'project_id'`).
6. `assignTeamMembers(proposalId, projectId, actorId)`.
7. Create billing entities by fee model: `RETAINER → createRetainerAgreement`; `FIXED → createFixedFeeInvoices`; `HOURLY → none`.
8. Engagement-prerequisite check (if `projectTemplateId` set) → notify admins/owners if unmet.
9. `eventPublisher.publishEvent(new ProposalAcceptedEvent(...))` (fires AFTER_COMMIT).
10. On any `RuntimeException`: publish `ProposalOrchestrationFailedEvent` (fires AFTER_ROLLBACK), rethrow.

**`ProposalAcceptedEvent`** record fields: `proposalId, createdProjectId, proposalNumber,
customerName, projectName, List<UUID> teamMemberIds, creatorMemberId, tenantId, orgId`.

**`ProposalAcceptedEventHandler`** — `@TransactionalEventListener(phase = AFTER_COMMIT)` on
`onProposalAccepted(ProposalAcceptedEvent)`. Re-binds tenant scope via
`RequestScopes.runForTenant(tenantId, orgId, () -> {...})`, then: (1) notifies creator
(`PROPOSAL_ACCEPTED`), (2) notifies each team member (`PROJECT_ASSIGNED`), (3) dispatches all via
`NotificationDispatcher` (email + in-app), (4) `proposalPortalSyncService.updatePortalProposalStatus(proposalId, "ACCEPTED")`.
A second listener `onProposalOrchestrationFailed` runs `AFTER_ROLLBACK` and notifies the creator.

> **Win-loop reuse for Phase 80:** A `Deal` marked WON can either (a) reuse this exact pattern — emit a
> `DealWonEvent`, a `DealWonEventHandler` (AFTER_COMMIT, `runForTenant`) and a `DealOrchestrationService`
> that creates a Proposal (or directly a Project) — or (b) simply create/link a Proposal and let the
> existing proposal orchestration fire. The transactional-event + `runForTenant` + UUID-reference +
> audit-with-project_id pattern is the template to copy.

## 9. Cross-Cutting Registries a New Deal Entity Must Plug Into

> For each: key file path(s) + how a NEW entity type registers.

### (a) Custom Fields / Field Definitions
- Enum: `backend/.../fielddefinition/EntityType.java` — **current values: `PROJECT, TASK, CUSTOMER, INVOICE`** (verified against source).
- Entity: `fielddefinition/FieldGroup.java`, `FieldDefinition.java`, `FieldGroupMember.java`.
- Migration that widens the field-group applicable-entity check constraint:
  `db/migration/tenant/V126__add_field_group_applicable_entity_values.sql`.
- Customer wires it via two columns (see §7b): `custom_fields jsonb` (flat key→value) and
  `applied_field_groups jsonb (List<UUID>)` (which auto-apply groups were attached at create time).
- **To make Deal field-able:** add `DEAL` to `EntityType`; add a matching tenant migration (V130+) to
  extend the `field_group.applicable_entity` allowed-values constraint (mirror V126); give the `Deal`
  entity `custom_fields jsonb` + `applied_field_groups jsonb` columns; optionally ship a field-pack JSON
  (`resources/field-packs/*.json`, `"entityType":"DEAL"`) discovered by the field pack seeder.

### (b) Tags
- `tag/Tag.java`, `tag/EntityTag.java`, `tag/EntityTagService.java`.
- `EntityTag` uses a **free-form `entityType` String discriminator** + `entityId UUID` — there is **no
  validation enum**. **Zero registration needed:** Deals are taggable immediately by passing
  `entityType="DEAL"` to `EntityTagService`.

### (c) Saved Views
- `view/SavedView.java`, `view/SavedViewService.java`.
- `SavedView.entityType` is a **String column, not an enum** — untyped. **Zero registration needed:**
  pass `entityType="DEAL"` when creating/querying views.

### (d) Audit Metadata Registry (Phase 69 — severity/group)
- `audit/AuditEventTypeRegistry.java` (verified exists), plus `audit/AuditSeverity.java`,
  `audit/AuditEventGroup.java`, `audit/AuditEvent.java`.
- A Spring bean holding a hardcoded list of `AuditEventTypeMetadata(eventType, label, severity, group)`
  with **longest-dotted-prefix-wins resolution** (per ADR-261) — `"deal.won"` exact beats a `"deal.*"`
  catch-all.
- **To register Deal events:** add metadata rows for `deal.created`, `deal.stage_changed`, `deal.won`,
  `deal.lost`, etc. (choose `AuditSeverity` + `AuditEventGroup`). Audit logging itself is via
  `AuditEventBuilder.builder().eventType("deal.won").entityType("deal").entityId(id).details(...).build()`.

### (e) Activity Feed Event Registration
- `activity/ActivityMessageFormatter.java` (a `switch` on `eventType` producing human text),
  `activity/ActivityService.java`.
- **No separate registry** — the activity feed reads audit events; formatting is opt-in. **To surface
  Deal activity:** (1) emit the audit events from (d) with `entityType="DEAL"`; (2) add `case "deal.*"`
  branches to `ActivityMessageFormatter.formatMessage()`. Include `project_id` in `details` if the
  event should appear in matter/portal activity feeds (they scope on `details->>'project_id'`).

### (f) Capability System + Vertical Module Gating
- Annotation: `orgrole/RequiresCapability.java`. Enum of all capabilities: `orgrole/Capability.java`
  (verified). Resolution: `orgrole/CapabilityAuthorizationManager.java` /
  `CapabilityAuthorizationService.java`; capabilities are resolved from the member's `OrgRole` by
  `MemberFilter` and bound to `RequestScopes.CAPABILITIES` (see `backend/CLAUDE.md` filter chain).
- Module gating (a DIFFERENT axis): `verticals/VerticalModuleGuard.java`,
  `verticals/VerticalModuleRegistry.java`, `verticals/ModuleCategory.java`,
  `verticals/VerticalProfileRegistry.java` (all verified to exist). `OrgSettings.enabledModules`
  (a JSON list on the settings spine) lists which modules an org has switched on;
  `VerticalModuleGuard.requireModule("...")` enforces it.
  - **Capabilities = per-action permission** (does this role get to do X), resolved from `OrgRole`.
  - **Modules = is this whole feature area turned on for the org** (e.g. `trust_accounting`,
    `resource_planning`), driven by the vertical profile + Settings → Features toggles.
- **To add CRM gating:** add capability constants to `Capability.java` (e.g. `VIEW_DEALS`,
  `MANAGE_DEALS`, `CLOSE_DEALS`) and annotate Deal controllers `@RequiresCapability(...)`. If CRM should
  be an optional feature area, register a module (e.g. `pipeline_crm`) in `VerticalModuleRegistry`
  (id, label, category HORIZONTAL/VERTICAL, nav items), guard endpoints with
  `verticalModuleGuard.requireModule("pipeline_crm")`, and add it to the relevant
  `resources/vertical-profiles/*.json` `enabledModules` defaults.

### (g) Pack-Seeding (per-vertical defaults — e.g. default pipeline stages)
- `packs/PackInstaller.java`, `packs/PackInstall.java` (entity), `seeder/AbstractPackSeeder.java`,
  `seeder/FieldPackSeeder.java` (reference impl).
- Vertical profile manifests: `backend/src/main/resources/vertical-profiles/*.json` (e.g. `legal-za.json`)
  — declare `profileId`, `enabledModules`, `locale`/`currency`, and a `packs` map (field, template,
  compliance, report, clause, request, automation, rate, schedule, project-template).
  `OrgSettings.PackStatusSettings` (embeddable) tracks which packs were applied per type (idempotency).
  Seeding runs at tenant provisioning + reconciliation (`verticals/VerticalProfileReconciliationSeeder.java`).
- **To seed default pipeline stages per vertical:** create a `DealPipelinePackSeeder extends
  AbstractPackSeeder` reading `resources/deal-pipeline-packs/*.json` (each declaring `verticalProfile`

## 11. Dashboard Widget Pattern (for a new Pipeline-Summary widget)

### Page
`frontend/app/(app)/org/[slug]/dashboard/page.tsx` — async Server Component. Fetches multiple
data streams in parallel (`Promise.all`) and composes widgets into grids, e.g.
`<TeamUtilizationWidget slug={slug} />`. Uses a client `useProfile()` hook for conditional/profile-gated
rendering.

### Representative widget
`frontend/components/dashboard/team-utilization-widget.tsx` — `"use client"`, uses **SWR** to poll
(refresh 5 min / dedup 60 s), calls a server action `fetchTeamUtilizationTrend(...)` which hits
`GET /api/utilization/team?weekStart=…&weekEnd=…`. Fetch is gated on vertical profile
(`shouldFetch ? key : null`).

### Backend endpoint feeding it
- Controller: `backend/.../capacity/UtilizationController.java`
  ```java
  @GetMapping("/api/utilization/team")
  public ResponseEntity<TeamUtilizationResponse> getTeamUtilization(
      @RequestParam LocalDate weekStart, @RequestParam LocalDate weekEnd)
  ```
  Gated by **vertical module guard** (`moduleGuard.requireModule("resource_planning")`), not `@RequiresCapability`.
- Service: `backend/.../capacity/UtilizationService.java`.
- DTOs: `backend/.../capacity/dto/UtilizationDtos.java` — `TeamUtilizationResponse(List<MemberUtilizationSummary> members, TeamAverages teamAverages)` etc.

### Canonical company-dashboard pattern (closest sibling for a pipeline widget)
- Controller: `backend/.../dashboard/DashboardController.java` — e.g.
  ```java
  @GetMapping("/api/dashboard/kpis")
  public ResponseEntity<KpiResponse> getCompanyKpis(
      @RequestParam @DateTimeFormat(iso = ISO.DATE) LocalDate from,
      @RequestParam @DateTimeFormat(iso = ISO.DATE) LocalDate to)
  ```
  `KpiResponse(int activeProjectCount, double totalHoursLogged, Double billablePercent,
  int overdueTaskCount, Double averageMarginPercent, List<TrendPoint> trend, KpiValues previousPeriod)`.
- Service: `backend/.../dashboard/DashboardService.java`.

### Copy-this pattern for a `GET /api/dashboard/pipeline-summary` widget
1. **Backend controller** — add a one-liner endpoint in `dashboard/DashboardController.java` returning
   `ResponseEntity<PipelineSummaryResponse>`, delegating to a single `dashboardService.getPipelineSummary(...)` call.
2. **Backend DTO** — add `PipelineSummaryResponse` (+ nested `StageBreakdown`) as Java records under
   `dashboard/dto/`.
3. **Backend service** — `dashboardService.getPipelineSummary(...)` (`@Transactional(readOnly = true)`),
   tenant-scoped automatically via `RequestScopes`/schema search_path; aggregate deals by stage.
   Gate with `@RequiresCapability("VIEW_DEALS")` and/or `verticalModuleGuard.requireModule("pipeline_crm")`.
4. **Frontend server action** — add `fetchPipelineSummary(from, to)` (`"use server"`, error-safe → null)
   calling `GET /api/dashboard/pipeline-summary`.
5. **Frontend widget** — `components/dashboard/pipeline-summary-widget.tsx` (Server Component for
   page-load data, or `"use client"` + SWR for polling, mirroring TeamUtilizationWidget). Mirror DTO as
   a TS interface in `lib/dashboard-types.ts`.
6. **Compose** — add `<PipelineSummaryWidget … />` to the dashboard `page.tsx` grid, optionally
   profile/module-gated via `useProfile()`.
