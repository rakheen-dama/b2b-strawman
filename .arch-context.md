**File:** `/frontend/components/help-tip.tsx`

**Props Interface:**
```typescript
interface HelpTipProps {
  code: string;
}
```

**Behavior:**
- Renders a `CircleHelp` icon (lucide-react, 4px size)
- Shows popover on click with `.title` and `.body` content from i18n
- Styling: `inline-flex text-slate-400 hover:text-slate-600` (dark mode: slate-500/300)
- Uses `createMessages("help")` to resolve i18n keys
- Key format: `{code}.title` and `{code}.body`

**Example Usage:**
```tsx

// Looks up help.custom-field-types.title and help.custom-field-types.body
```

**I18N Namespace:** `help` (file: `/lib/messages/en/help.json`, 6.7 KB, pre-populated with ~50+ help topics)

## 6. i18n Message Catalog Structure

**System:** JSON-based flat file structure with dot-notation path resolution.

**Active Namespaces:**
1. `"empty-states"` → `lib/messages/en/empty-states.json` (4.5 KB)
2. `"help"` → `lib/messages/en/help.json` (6.7 KB)
3. `"errors"` → `lib/messages/en/errors.json` (1.3 KB)
4. `"getting-started"` → `lib/messages/en/getting-started.json` (1.4 KB)
5. `"common"` → `lib/messages/en/common.json` (427 B)

**Message Resolution Function:**
```typescript
export function createMessages(namespace: MessageNamespace): MessagesReturn {
  return {
    t: (code: string, interpolations?: Record<string, string>) => string
  };
}
```

**Path Resolution:**
- Keys use dot notation: `"path.to.key"`
- Function recursively traverses JSON object tree
- Returns `undefined` if key not found; logs warning in dev mode, returns code as fallback

**Interpolation Pattern:**
- Uses `{{variableName}}` syntax in message strings
- Pass `interpolations: { variableName: "value" }` to `t()` to replace

**i18n Directory:**
```
lib/messages/
├── index.ts                    # Main export + resolution logic
├── en/
│   ├── empty-states.json
│   ├── help.json
│   ├── errors.json
│   ├── getting-started.json
│   └── common.json
└── en-ZA-accounting/           # Vertical variant (accounting terminology)
```

**Regional Variants:**
- Currently: `en` (default) and `en-ZA-accounting` (South Africa accounting-specific terminology)
- Vertical switching via `ADR-185` (terminology-switching-approach)

**TODO:** Multi-locale support not yet implemented (see TODO comment in `lib/messages/index.ts`)

## 13. Key Development Conventions

### React Patterns
- Default to **React Server Components (RSC)** — add `"use client"` only when needed
- **Next.js 16:** Params are Promises — always `await` in page/layout
- **Never pass functions from Server → Client** — Next.js 16 enforces serialization boundary

### API & Data Fetching
- **Server Components:** Fetch on server, pass as props
- **Server Actions:** Mutations (form submissions)
- **Client-side:** SWR for dialogs, polling, lazy-loaded data
- **API client:** `lib/api.ts` attaches Bearer JWT to all requests

### Styling
- Tailwind v4 CSS-first config
- Use `cn()` utility to merge classes
- Never use olive/neutral/zinc — use **slate** scale
- Never use indigo — use **teal** for accents

### Forms
- Zod schemas in `lib/schemas/` (one file per domain)
- React Hook Form + `@hookform/resolvers/zod`
- Shadcn Form components for accessible field wrappers
- Validation errors display inline via `<FormMessage />`

### Testing
- Vitest + `@testing-library/react`
- Colocate tests next to source or use `__tests__/` directory
- Radix UI components leak DOM — add `afterEach(() => cleanup())` in test files

## 15. Multi-Tenancy Architecture

**Strategy:** Schema-per-tenant (ADR-T001)

**Isolation:**
- Global database: `public` schema (subscriptions, organizations, users)
- Tenant database: `schema_{tenant_uuid}` per organization (ADR-064)
- Row-level security: NOT used; schema isolation is primary boundary (ADR-012)

**Migration Strategy:**
- Global migrations: Run on `public` schema
- Tenant migrations: Run on each `schema_{tenant_uuid}` during provisioning (ADR-007)
- Flyway handles both via separate migration paths

**Tenant Identification:**
- JWT claim: `organization_id` (from Keycloak, ADR-138, ADR-139)
- Gateway BFF extracts claim and adds to request header (ADR-140, ADR-214)
- Backend resolves schema from header (ADR-089)

## 18. Quick Reference: Adding a New Feature

### To add a new sidebar item:
1. Add `NavItem` to appropriate `NavGroup` in `lib/nav-items.ts`
2. Set `requiredCapability` or `requiredModule` if gated
3. Create route at `app/(app)/org/[slug]/{feature}/page.tsx`

### To add help text:
1. Add key to `lib/messages/en/help.json`: `{ "feature-code": { "title": "...", "body": "..." } }`
2. Use `<HelpTip code="feature-code" />` in component
3. Help lookup is automatic via i18n resolution

### To add empty state messaging:
1. Add key to `lib/messages/en/empty-states.json`
2. Use `<EmptyState icon={Icon} title={t('...')} description={t('...')} />`
3. Or use `createMessages("empty-states")` directly in component

### To add a new database table:
1. **Global:** Create `backend/src/main/resources/db/migration/global/V{next}__*.sql`
2. **Tenant:** Create `backend/src/main/resources/db/migration/tenant/V{next}__*.sql`
3. Update this document with new VN

### To add a new ADR:
1. Create `adr/ADR-{next}__title-in-kebab-case.md`
2. Update this inventory with new ADR number
