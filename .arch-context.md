## 7. Existing Frontend Patterns (UX Phase Reference)

### 7a. Reusable Component Pattern -- EmptyState

File: `frontend/components/empty-state.tsx`

```tsx
interface EmptyStateProps {
  icon: LucideIcon;
  title: string;
  description: string;
  action?: React.ReactNode;
  actionLabel?: string;
  actionHref?: string;
}
```

Pattern notes:
- LucideIcon type for icon props
- Slate color scale throughout (slate-300/700, slate-900/100, slate-600/400)
- `font-display` class for headings (Sora font)
- Dark mode variants included
- Shadcn `Button` with `asChild` pattern for link buttons
- Simple, no `"use client"` -- works as Server Component

### 7b. Error Handling -- Current State

**Toast library**: `sonner` (not shadcn's useToast)
- Import: `import { toast } from "sonner";`
- `<Toaster />` rendered in `app/layout.tsx`
- Used in ~10 client components for mutation feedback

**API error handling** (`frontend/lib/api.ts`):
- Server-only module (`import "server-only"`)
- `getAuthToken()` from `@/lib/auth` for JWT
- Backend returns RFC 9457 `ProblemDetail` on errors
- `ProblemDetail` type imported from `@/lib/types`
- Auth mode dispatch: Keycloak BFF (cookie + CSRF) vs Clerk/mock (Bearer token)

**Try/catch usage locations**:
- `lib/webhook-handlers.ts` -- webhook processing
- `lib/view-actions.ts` -- server actions (saved views)
- `lib/portal-api.ts` -- portal API calls
- `hooks/use-prerequisite-check.ts` -- client-side data fetch
- `hooks/use-notification-polling.ts` -- polling with error swallowing

**NO centralized error handling exists**:
- No shared `handleApiError()` utility
- No `useApiMutation()` hook
- No error boundary components
- No retry logic on client side
- Error handling is ad-hoc per file (try/catch with console.error or toast)

### 7c. i18n / Message Catalog -- Current State

**No i18n infrastructure exists.**

- No message catalog files (JSON/TS)
- No i18n library (react-intl, next-intl, i18next, etc.)
- No `useTranslation` hooks
- `locale` / `Intl` hits are from `Intl.NumberFormat` / `Intl.DateTimeFormat` in formatting utilities
- All user-facing strings are **hardcoded inline** in component JSX

This phase would introduce the first message catalog infrastructure.

## Key Gaps for UX Quality Phase

1. **No i18n infrastructure** -- all user-facing strings are hardcoded inline. This phase introduces the first message catalog.
2. **No centralized error handling** -- ad-hoc try/catch + toast per file. Need shared error recovery patterns (`handleApiError()`, error boundaries, retry logic).
3. **No `onboarding_dismissed_at`** on OrgSettings -- need migration V66 + entity field + API endpoint + frontend check.
4. **Only 2 custom hooks** -- opportunity to add `useApiMutation` or similar for consistent loading/error/success states across mutations.
5. **EmptyState component exists** but is minimal -- may need enhancement for contextual help (help links, getting-started guides, feature explanations).
6. **Toast via sonner** is the established pattern -- continue using it for all mutation feedback.
7. **No error boundary components** -- React error boundaries not implemented anywhere.
8. **No loading skeleton components** -- no shared skeleton/shimmer components for loading states.
