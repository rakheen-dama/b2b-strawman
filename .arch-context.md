### Key Integration Patterns

#### IntegrationRegistry (BYOAK)
- Package: `integration/`
- Pattern: Per-domain, per-interface resolution (`IntegrationRegistry`)
- Implementations use `@IntegrationAdapter(domain = ACCOUNTING, slug = "xero")`
- Phase 71: `XeroAccountingProvider` implements both `AccountingProvider` + `AccountingPaymentSource`
- Phase 21: Original `AccountingProvider` push-only interface defined here
- Phase 25: `PaymentGateway` interface (same pattern)

#### Assistant System (Phase 52/70)
- **Chat Provider**: `LlmChatProvider` interface (streaming, tool use)
  - `AnthropicLlmChatProvider` implementation (Anthropic SDK)
  - `LlmChatProviderRegistry` for multi-provider support
- **Specialists**: `Specialist` interface + `SpecialistRegistry`
  - Three bundled specialists: Billing, Intake, Inbox
  - Classpath markdown definitions (prompts + tools)
- **Invocation**: `AiSpecialistInvocation` entity (single table, JSONB output)
  - `AiSpecialistInvocationService` handles approval/rejection/application
  - `AiLlmCall` traces streaming watermarks
- **Tools**: `ToolRegistry` + sealed `ToolDefinition` hierarchy
  - Read tools: `ListDocuments`, `ExtractTextFromDocument`, etc.
  - Proposal tools: `ProposeTimeEntryPolish`, `ProposeInvoiceLineGrouping`, etc.
  - Applier pattern: `OutputPayload` → entity mutation

## 7. Existing AI Patterns (Phase 52 / Phase 70)

### LlmChatProvider Interface
**Location**: `assistant/provider/LlmChatProvider.java`

```java
public interface LlmChatProvider {
  String providerId();
  void chat(ChatRequest request, Consumer<StreamEvent> eventConsumer);
  boolean validateKey(String apiKey, String model);
  List<ModelInfo> availableModels();
}
```

- Streaming-based (consumer callback, not async/Flux to avoid WebFlux dependency)
- Tool use support (Anthropic-format `tools` array on every message)
- Provider-agnostic for multi-LLM future (currently Anthropic only)
- **Implementation**: `AnthropicLlmChatProvider` (Phase 52)

### AiProvider Interface (One-shot Text)
**Location**: `integration/ai/AiProvider.java`

```java
public interface AiProvider {
  String providerId();
  AiTextResult generateText(AiTextRequest request);
  AiTextResult summarize(String content, int maxLength);
  List<String> suggestCategories(String content, List<String> existing);
  ConnectionTestResult testConnection();
}
```

- Separate from `LlmChatProvider` (different operational semantics)
- Port pattern: tenant-scoped, implementations registered in `IntegrationRegistry`
- **Implementation**: `NoOpAiProvider` (default, fallback)
- Future: Anthropic text generation adapter

### Specialist Pattern
**Location**: `assistant/specialist/Specialist.java` (interface) + `SpecialistRegistry` + classpath markdown

- Three bundled specialists: **Billing**, **Intake**, **Inbox** (Phase 70)
- Each specialist has:
  - Unique `specialistId` (lowercase, alphanumeric)
  - Markdown prompt file (system instructions, context-aware rules)
  - Tools list (subset of phase 52 registry + specialist-specific tools)
  - Launcher context (e.g., invoice ID, customer ID)
- No database storage — loaded from classpath (immutable per release)

### AiSpecialistInvocation (Approval Queue)
**Location**: `assistant/invocation/AiSpecialistInvocation.java` (entity, V120 migration)

- Single-table design per ADR-270
- JSONB columns: `proposed_output`, `applied_output`
- Sealed `OutputPayload` hierarchy (Jackson polymorphism)
- Statuses: `PENDING_APPROVAL`, `APPROVED`, `REJECTED`, `FAILED`, `AUTO_APPLIED` (Inbox comment-posting only, direct mode)
- Optimistic locking (`@Version`) protects concurrent approve/reject
- Related: `AiLlmCall` entity logs streaming watermarks per invocation

### Tool Registry
**Location**: `assistant/tool/`

- **Read tools** (`read/`): list documents, extract text, get field value, list tasks, etc.
- **Write tools** (`write/`): `ProposeTimeEntryPolish`, `ProposeInvoiceLineGrouping`, `ExtractedFieldsProposal`, etc.
- Tool schemas in Anthropic format (JSON Schema on `input_schema`)
- Result types are typed Java records (not strings)

### Key Design Constraints
- **Confirmation on write**: All reversible writes require approval (Phase 52, reused Phase 70)
- **BYOAK**: Each tenant's LLM key (Anthropic) configured via `OrgIntegration` (Phase 21, extended Phase 52)
- **No persistence of chat history**: Only `AiSpecialistInvocation` rows are persisted; multi-turn context is ephemeral
- **Inline first, chat panel fallback**: Phase 70 puts specialists on entity pages; Phase 52 chat is secondary (ADR-266)

## 9. Integration Pattern (BYOAK)

### OrgIntegration Entity
**Location**: `integration/OrgIntegration.java`

- **Domain**: `IntegrationDomain` enum (ACCOUNTING, AI, EMAIL, KYC_VERIFICATION, PAYMENT, DOCUMENT_SIGNING)
- **Provider slug**: (e.g., "xero", "smtp", "noop")
- **Config JSON**: JSONB column for provider-specific settings
- **Enabled**: boolean flag
- **Key suffix**: 6-char random suffix for secret namespacing

### IntegrationRegistry Pattern
**Location**: `integration/IntegrationRegistry.java` (or registry bean resolver)

- Resolves `(domain, interfaceType)` → implementation bean
- Implementations use `@IntegrationAdapter(domain = ACCOUNTING, slug = "xero")`
- One bean can implement multiple interfaces (`XeroAccountingProvider` implements `AccountingProvider` + `AccountingPaymentSource`, Phase 71)
- Registry cached (Caffeine, 60-second TTL)

### Phase 71 Accounting Pattern
- **Push side** (Phase 21, unchanged):
  - `AccountingProvider` interface: `syncInvoice(InvoiceSyncRequest)`, `syncCustomer(CustomerSyncRequest)`, `testConnection()`
  - Called per-event (domain event triggers push)
  - Push worker: `AccountingSyncService`
- **Pull side** (Phase 71 new):
  - `AccountingPaymentSource` interface: `getPaymentsModifiedSince(Instant)`
  - Called per-schedule (every 15 minutes)
  - Pull worker: `AccountingPaymentPollingService`
  - Implementation: `XeroAccountingProvider` (both interfaces)
  - No-op: `NoOpAccountingProvider` (both interfaces, empty-list return for pull)
- **Rationale** (ADR-279): Interface segregation; push ≠ pull semantically and operationally
- **OrgSecret**: Encrypted API key/token storage (per `OrgIntegration.keySuffix`)

## 14. Key Architectural Decisions (Immutable Patterns)

### Tenant Isolation (ADR-T001)
- **Schema-per-tenant**: `tenant_<12-hex-hash>` per Clerk org ID
- No row-level isolation, no `tenant_id` columns
- Pure schema boundary enforcement

### Scoped Values (ADR-T002)
- Use `ScopedValue` (Java 21+) for request-scoped context
- Bindings: `TENANT_ID`, `MEMBER_ID`, `ORG_ROLE`, `CAPABILITIES`
- Thread-safe, virtual-thread-safe, guaranteed cleanup

### Product-Layer Roles (ADR-T003)
- `OrgRole` entity (product layer), not Keycloak org roles
- Capabilities enum drives `@RequiresCapability` authorization
- Keycloak used for identity only (JWT validation)

### Gateway/BFF (ADR-T004)
- Next.js frontend talks to Spring Boot backend via REST API
- No direct database access from frontend
- API key authentication for `/internal/*` endpoints

### Magic Links (ADR-T005)
- Portal authentication via one-time magic links, not passwords
- `MagicLinkToken` entity, POPIA/GDPR compliant
- No customer accounts in primary app; portal contacts only

### Dual-Author Comments (ADR-T006)
- `Comment` entity has `authorDiscriminator` (STAFF | PORTAL_CONTACT)
- Single table, type discriminator, both author types on same entity

### Idempotent Provisioning (ADR-T007)
- Tenant provisioning: migration replayable, schema name deterministic
- Webhook handler uses processed-event dedup

### Tenant-Scoped Runner (ADR-T008)
- Scheduled tasks wrap in `ScopedValue.where(TENANT_ID, "tenant_X").run()`
- Prevents cross-tenant leakage in background jobs
