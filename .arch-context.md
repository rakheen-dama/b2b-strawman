### Request-context binding — file paths + the capability line

Filter chain order (from `backend/CLAUDE.md`):
1. `security/ApiKeyAuthFilter.java` — validates `X-API-KEY` for `/internal/*`.
2. `BearerTokenAuthenticationFilter` + `security/ClerkJwtAuthenticationConverter` — JWT identity only.
3. `multitenancy/TenantFilter.java` — binds `RequestScopes.TENANT_ID` (+ `ORG_ID`) from JWT `o.id`.
4. `member/MemberFilter.java` — binds `MEMBER_ID`, `ORG_ROLE`, **and `CAPABILITIES`**.
5. `TenantLoggingFilter` — MDC.

`RequestScopes` (`multitenancy/RequestScopes.java`) — all context is `ScopedValue` (Java 25), never
ThreadLocal. Scoped values: `TENANT_ID`, `MEMBER_ID`, `ORG_ROLE`, `ORG_ID`, `CUSTOMER_ID`,
`PORTAL_CONTACT_ID`, `CAPABILITIES`, `SHARD_ID`, `GROUPS`.

**The capability binding line — `member/MemberFilter.java:80`:**
```java
capabilities = orgRoleService.resolveCapabilities(info.memberId());  // line 70
...
carrier = carrier.where(RequestScopes.CAPABILITIES, capabilities);   // line 80
ScopedFilterChain.runScoped(carrier, filterChain, request, response);// line 82
```
- `MemberFilter.shouldNotFilter` skips `/internal/*`, `/actuator/*`, `/portal/*` — **so member/capability context is NOT bound for `/internal/*` or `/portal/*` paths.** Critical for the MCP server: if MCP mounts under `/internal/*` (API-key auth), it will have **no** `MEMBER_ID` / `ORG_ROLE` / `CAPABILITIES` bound, and any `ActorContext.fromRequestScopes()` or `@RequiresCapability` call will fail. The MCP layer must either (a) run under the JWT-authenticated path, or (b) explicitly bind member/role/tenant scope.

To bind tenant scope programmatically (the sanctioned API, ADR-T008):
```java
RequestScopes.runForTenant(tenantId, orgId, () -> { ... });
RequestScopes.callForTenant(tenantId, orgId, () -> ...);          // returns a value
RequestScopes.runForTenantWithMember(tenantId, orgId, memberId, () -> { ... });
```
There is no public helper to bind `CAPABILITIES` outside `MemberFilter` — capability binding would
need to be added if MCP runs outside the JWT chain.

```java
package io.b2mash.b2b.b2bstrawman.integration.ai.profile;

import jakarta.persistence.Column;
import jakarta.persistence.Entity;
import jakarta.persistence.GeneratedValue;
import jakarta.persistence.GenerationType;
import jakarta.persistence.Id;

  private Instant updatedAt;
```

**Pattern takeaways (convention for new tenant-scoped entities):**
- **No tenant annotation of any kind** — no `@FilterDef`, no `@Filter`, no `TenantAware` interface, no
  listener, no `tenant_id` column. Tenant isolation is 100% via Hibernate `search_path` (schema-per-tenant).
- `@Id` + `@GeneratedValue(strategy = GenerationType.UUID)`.
- Audit columns hand-rolled: `created_at`/`updated_at` via `@PrePersist`/`@PreUpdate`; `created_by`/`updated_by`
  as `UUID` set in constructor / mutation methods. (No shared `@MappedSuperclass` base — each entity repeats this.)
- JSON columns via `@JdbcTypeCode(SqlTypes.JSON)` + `columnDefinition = "jsonb"`.
- `protected` no-arg constructor for Hibernate; a public constructor taking the creator's `UUID`.
- Mutations are domain methods on the entity (`updateProfile`, `markColdStartCompleted`), not setters.
- No Lombok (banned). Plain getters.

### Entity — `integration/OrgIntegration.java` (full, abridged getters)

```java
  // enable() / disable() / setConfigJson(...) / setKeySuffix(...) / clearKeySuffix()

```
Repository: `integration/OrgIntegrationRepository.java` (has `findByDomain(IntegrationDomain)`).

### The integration-type registry / how a new domain is added

**`integration/IntegrationDomain.java`** — the enum of domains, each with a `defaultSlug`:
```java
public enum IntegrationDomain {
  ACCOUNTING("noop"),
  AI("noop"),
  DOCUMENT_SIGNING("noop"),
  EMAIL("smtp"),
  KYC_VERIFICATION("noop"),
  PAYMENT("noop");
  // getDefaultSlug() returns the adapter slug used when no integration configured
}
```
> To introduce MCP as an integration domain you'd add e.g. `MCP("noop")` here (note the `domain`
> column is `length = 30`). But MCP may instead be modelled as a **per-org enable/credential** rather
> than an adapter-backed domain — see "implication" below.

**`integration/IntegrationAdapter.java`** — the bean-marker annotation:
```java
@Target(ElementType.TYPE) @Retention(RetentionPolicy.RUNTIME)
public @interface IntegrationAdapter {
  IntegrationDomain domain();
  String slug();          // unique within domain, e.g. "xero", "noop"
}
```

**`integration/IntegrationRegistry.java`** — `@Component` that at startup scans all
`@IntegrationAdapter` beans and builds `Map<IntegrationDomain, Map<slug, adapterBean>>` (fails fast on
duplicate domain+slug). At request time `resolve(domain, PortInterface.class)`:
- requires `RequestScopes.TENANT_ID` bound,
- Caffeine-caches (`tenantSchema:DOMAIN`, 60s TTL) the tenant's `OrgIntegration` config,
- returns the configured-and-enabled adapter, else the domain's default-slug adapter,
- `resolveBySlug(domain, slug, port)` bypasses tenant config (for webhooks pre-tenant-context),
- `availableProviders(domain)` lists registered slugs, `evict(tenantSchema, domain)` on config change.

**Pattern to add a new integration domain:**
1. Add the enum constant + default slug to `IntegrationDomain`.
2. Implement a port interface + one or more `@IntegrationAdapter(domain=..., slug=...)` beans (plus a
   `noop`/default adapter so `resolve` never throws).
3. Persist per-tenant config as an `OrgIntegration` row (domain + providerSlug + enabled + configJson).
4. Resolve at runtime via `IntegrationRegistry.resolve(domain, Port.class)`.

> **MCP implication:** MCP is an *inbound* read-only data exposure (firm staff's own Claude pulls Kazi
> data), not an *outbound* adapter to a third-party SaaS. The adapter/registry pattern is built for
> outbound ports (Xero, SMTP, PSP). MCP likely does NOT fit the `IntegrationAdapter` shape and should
> not force a new `IntegrationDomain` constant unless there's a genuine adapter to resolve. A simpler
> model: an MCP-enable flag + scoped API credential per org, surfaced under
> `settings/integrations/mcp/`. Architect's call — flag this explicitly.

## 11. Spring MCP / Spring AI SDK Availability

Checked `backend/pom.xml` (grep `mcp` / `spring-ai` / `modelcontextprotocol`):

- **No Spring AI dependency.** No `spring-ai-*`, no `spring-ai-bom`.
- **No MCP server SDK.** No `org.springframework.ai:spring-ai-starter-mcp-server*`, no
  `io.modelcontextprotocol.sdk:mcp` / `mcp-spring-*`.
- **Spring Boot version: `4.0.2`** (`<version>4.0.2</version>` in the parent). Java 25, Maven.

**Implication / flag for the architect:**
- The Spring AI MCP server starter (`spring-ai-starter-mcp-server-webmvc` / `-webflux`) and the
  underlying `io.modelcontextprotocol.sdk` would need to be **added**. ⚠️ **Version-compatibility risk:**
  Spring AI's GA line historically targets Spring Boot 3.x; MCP-server support on **Spring Boot 4.0.2**
  must be verified (BOM alignment, autoconfigure package moves — see Boot-4 gotchas in
  `backend/CLAUDE.md`). Confirm the exact compatible Spring AI / MCP SDK version against current docs
  before committing to the starter; a hand-rolled MCP endpoint (JSON-RPC over HTTP/SSE) is the
  fallback if the starter is not Boot-4-ready.
- There IS existing AI infrastructure (NOT MCP): `assistant/`, `integration/ai/{execution,gate,profile}`,
  `AiLlmCall`, `AiSpecialistInvocation`, `AiExecution`, `AiExecutionGate`, `AiFirmProfile`. These are the
  AI-core surfaces (live-verified V0–V12 per project memory) the MCP layer is intended to reuse — but
  none of them is an MCP transport.

## Cross-cutting flags for the architect (summary)

1. **Service return types diverge from the requirements' DTO names in several places** — the biggest
   are `CustomerService` (returns entities; controller does DTO + tag + member-name enrichment),
   `ClientLedgerService.getTotalTrustBalance` (not `getTotalBalance`; takes only `trustAccountId`),
   `UnbilledTimeService.getUnbilledSummary` (period-rollup list, not per-entity),
   `AuditService.findEventsForCustomer` (raw `Stream<AuditEvent>`, not enriched),
   `DocumentService.PresignDownloadResult` (not `PresignedUrlResult`), and `getProject` →
   `ProjectWithRole` (wraps entity, not flat DTO). See §7 for each.
2. **Several response DTOs are controller-private records, not in shared `dto` packages**
   (`ProjectResponse` lives in `ProjectController`). MCP tools either reuse the service's native return
   type or duplicate/relocate these DTOs.
3. **`MemberFilter.shouldNotFilter` excludes `/internal/*` and `/portal/*`** — capability/member context
   is unbound there. If MCP mounts under `/internal/*`, `@RequiresCapability` and
   `ActorContext.fromRequestScopes()` will not work without explicit scope binding.
4. **No MCP-specific `Capability`** exists; add one if a dedicated gate is wanted.
5. **No Spring AI / MCP SDK** in `pom.xml`; Boot-4 compatibility of the MCP starter must be verified.
6. MCP is inbound read-exposure and likely does **not** fit the outbound `IntegrationAdapter` pattern.
