# Infrastructure & CI/CD Audit

**Date**: 2026-04-01
**Auditor**: Claude (automated)

---

## 1. Executive Summary

The infrastructure is **partially ready** -- structurally complete for a 2-service (frontend + backend) ECS Fargate deployment, but **stale by approximately 6 months** relative to the current application architecture. The Terraform modules, CI/CD workflows, secrets, and ECS task definitions were written when the product used Clerk for auth and had only two deployable services (frontend + backend). The product now uses Keycloak with an API Gateway (BFF pattern) and has 5 deployable services, but the infra only knows about 2. Additionally, all CI/CD workflows use long-lived IAM keys instead of OIDC, the Dockerfiles lack health checks, and there is no RDS/ElastiCache provisioning (the current architecture calls for Redis for gateway sessions). The Terraform modules themselves are well-structured and follow best practices -- the issue is coverage, not quality.

- **CIDR ranges**: Properly separated per environment (dev: 10.0.0.0/16, staging: 10.1.0.0/16, prod: 10.2.0.0/16).

#### Security Groups (`infra/modules/security-groups/main.tf`)
- **Resources**: 4 security groups (public ALB, frontend, internal ALB, backend).

#### ECR (`infra/modules/ecr/main.tf`)
- **Resources**: 2 ECR repositories (frontend, backend) with lifecycle policies and image scanning.
- **Quality**: Good. Lifecycle policies expire untagged images after 1 day and cap tagged images.

#### ECS (`infra/modules/ecs/main.tf`)

### Detailed Findings

#### CI (`ci.yml`)
- **Frontend job**: pnpm 10, Node 20, lint + format check + test + build. Uses `CLERK_SECRET_KEY` and `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY` secrets for build. **Clerk is removed** -- should use `NEXT_PUBLIC_AUTH_MODE=keycloak` + Keycloak-related build args.
- **Backend job**: Temurin JDK 25, spotless:check + test + build. Looks correct for current stack.
- **Missing**: No CI job for gateway or portal.
- **Missing**: No Playwright/E2E tests in CI.

#### Build & Push (`build-and-push.yml`)
- **Change detection**: Uses `dorny/paths-filter@v3` to skip unnecessary builds -- good pattern.
- **AWS auth**: `aws-access-key-id` + `aws-secret-access-key` (long-lived keys). Workflow declares `id-token: write` permission but **does not use OIDC** -- it uses static secrets.
- **Hardcoded**: `ENVIRONMENT: dev` at workflow level -- this only pushes to dev ECR repos.
- **Frontend build**: Passes `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY` as build arg. **Stale** -- should be `NEXT_PUBLIC_AUTH_MODE` + `NEXT_PUBLIC_GATEWAY_URL`.
- **Missing**: No build jobs for gateway or portal.
- **Missing**: No change detection for `gateway/` or `portal/` paths.

#### Deploy Dev (`deploy-dev.yml`)
- **Trigger**: `workflow_run` on Build & Push completion. Auto-deploys to dev.
- **Pattern**: Uses composite action `.github/actions/ecs-deploy` -- well-designed (fetch current task def, patch image, register new revision, update service, wait for stability).
- **Only deploys**: frontend + backend.

#### Deploy Staging/Prod (`deploy-staging.yml`, `deploy-prod.yml`)
- **Trigger**: `workflow_dispatch` with optional ref input. Staging has environment approval. Prod has type-confirmation + environment approval.
- **Rebuilds images**: Staging and prod rebuild images from source (don't promote from dev ECR). This means a different binary could be deployed to prod vs what was tested in dev.
- **Only deploys**: frontend + backend.

#### Rollback (`rollback.yml`)
- **Quality**: Excellent design. Shows current vs rollback image, waits for stability, has smoke tests. Environment + service are selectable.
- **Only handles**: frontend + backend.

#### Composite Action (`ecs-deploy/action.yml`)
- **Quality**: Well-designed. Fetches current task def, patches image via `jq`, strips metadata, registers new revision, updates service, waits for stability with error reporting.
- **No issues** with the action itself.

### Cross-Cutting Issues

1. **IAM Key Auth**: All workflows use `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` secrets. The workflows already declare `id-token: write` permission, so switching to OIDC is straightforward.
2. **Only 2 services**: Every workflow only knows about frontend and backend. Gateway and portal are completely absent.
3. **Clerk references**: CI and build workflows reference Clerk secrets that are no longer used by the application.
4. **No image promotion**: Staging and prod rebuild from source instead of promoting a tested dev image. This risks deploying untested code.
5. **No Terraform in CI/CD**: No workflow runs `terraform plan` on PRs or `terraform apply` on merge, despite the infra CLAUDE.md saying this is expected.

## 5. Application Config

### Backend Production Profile (`application-prod.yml`)
- **Datasource**: HikariCP with tuned pool settings (max 10 connections, 10s timeout, 28-min max-lifetime for Neon compatibility, leak detection at 60s). Prepared statement caching enabled.
- **Migration datasource**: Separate connection (max 2) for Flyway.
- **JPA**: DDL auto `none`, `show-sql` off, PostgreSQL dialect explicit.
- **Flyway**: `enabled: false` -- **PROBLEM**: This means Flyway migrations won't run on startup. The application relies on a separate migration step. This needs to be either enabled or documented as requiring a separate migration job.
- **Auth**: References `CLERK_ISSUER` and `CLERK_JWKS_URI` env vars. **Stale** -- the keycloak profile (`application-keycloak.yml`) overrides these with `JWT_ISSUER_URI` and `JWT_JWK_SET_URI`, but the prod profile itself still references Clerk variable names.

- **Portal**: JWT secret and magic link secret from env vars.
- **Logging**: INFO level, no structured format specified (base `application.yml` sets `ecs` format).
- **Missing**: No Redis/session configuration for gateway. No mail configuration (base `application.yml` has it). No `spring.threads.virtual.enabled` (base has it).

### Gateway Config (`gateway/application.yml`)
- **Session**: JDBC-backed (`spring.session.store-type: jdbc`), 8-hour timeout. Has a `TODO` comment about using Flyway for session schema.
- **OAuth2**: Keycloak client registration with `authorization_code` grant.
- **Routing**: Spring Cloud Gateway MVC, single route proxying `/api/**` to backend with `TokenRelay` filter.

## 6. Gaps for Production

### Critical (Must Fix Before First Deploy)

1. **Missing 3 services from infra**: Gateway, Portal, and Keycloak are not provisioned by Terraform. No ECR repos, no ECS task definitions, no services, no security groups, no target groups, no ALB routing.

2. **Auth provider mismatch**: All Terraform modules, secrets, and CI/CD workflows reference Clerk. The application now uses Keycloak. Secrets need replacement (3 Clerk secrets out, ~10 Keycloak/portal secrets in). ECS task definitions need new env vars. CI build args need updating.

3. **No Keycloak deployment**: Keycloak is a critical service (auth provider) with no provisioned infrastructure. Options: managed Keycloak (e.g., Cloud-IAM), ECS task with persistent storage, or EC2 instance. Needs RDS or shared Postgres for Keycloak's own database.

4. **No Redis for Gateway sessions**: Gateway uses JDBC sessions (Postgres). Multi-instance gateway deployments need either Redis-backed sessions or sticky sessions on the ALB. ElastiCache (Redis) is not provisioned.

5. **ALB routing architecture wrong**: Public ALB routes directly to frontend/backend. With the Gateway BFF pattern, the public ALB should route to the Gateway, which handles OAuth2 flows and proxies `/api/**` to backend with token relay.

6. **IAM key auth in CI/CD**: All workflows use long-lived `AWS_ACCESS_KEY_ID`/`AWS_SECRET_ACCESS_KEY`. Should use GitHub OIDC with `role-to-assume`. The `id-token: write` permission is already declared.

7. **No `HEALTHCHECK` in any Dockerfile**: ECS benefits from container-level health checks in addition to ALB target group checks.

8. **Flyway disabled in prod profile**: `spring.flyway.enabled: false`. Migrations must be handled some other way (init container, separate job, or manual). If this is intentional, it needs a migration workflow or ECS one-off task.

### Important (Should Fix Before Production)

9. **No image promotion pipeline**: Each environment rebuilds from source. Dev builds should be promoted to staging/prod to ensure the same binary is tested and deployed.

10. **No Terraform plan/apply in CI**: The infra CLAUDE.md describes this as expected behavior, but no workflow exists. Changes to `infra/` don't trigger any automation.

11. **ALB deletion protection**: `enable_deletion_protection = false` on both ALBs. Should be `true` for staging/prod.

12. **Monitoring gaps**: No CloudWatch alarms (5xx rate, latency, ECS CPU/memory, error log patterns). No SNS topics for alerts. No dashboards. Only log groups exist.

13. **No WAF**: No AWS WAF on the public ALB. Required for rate limiting, IP blocking, SQL injection protection.

14. **No RDS provisioning**: Database is Neon (external) which is fine, but there's no Terraform for Neon management. Gateway needs its own Postgres database for sessions (currently uses `SPRING_SESSION` tables).

15. **Cookie `secure: false` in gateway**: Must be `true` when behind HTTPS ALB.

16. **Portal Dockerfile missing `public/` copy**: If the portal has static assets in `public/`, they won't be served.

17. **Hardcoded JAR names in Dockerfiles**: `b2b-strawman-backend-0.0.1-SNAPSHOT.jar` and `b2b-strawman-gateway-0.0.1-SNAPSHOT.jar` will break on version bump.

### Nice to Have

18. **No Container Insights custom metrics**: Cluster has Container Insights enabled, but no custom application metrics (Micrometer/Prometheus exporter).

19. **No secrets rotation**: Secrets Manager secrets have no rotation configuration.

20. **No VPC flow logs**: Useful for security auditing.

21. **No backup strategy documented**: Neon handles Postgres backups, but S3 versioning alone isn't a backup strategy. No lifecycle rules for transitioning old versions to cheaper storage.

22. **No bastion/VPN**: No way to access private subnet resources for debugging. Consider AWS Systems Manager Session Manager or a bastion host.

23. **Build matrix improvement**: Frontend CI uses Node 20 but doesn't test against multiple versions. Backend CI should cache Maven dependencies better.

## 7. Recommendation

**Update in-place, not rewrite.**

The Terraform module structure is well-designed -- proper separation, parameterization, tagging, and least-privilege IAM. The CI/CD workflows follow good patterns (change detection, composite actions, rollback support, smoke tests). The problem is **scope and staleness**, not **quality**.

### Recommended approach:

1. **Extend modules** to support 5 services (add gateway, portal, keycloak ECR/ECS/SG/ALB entries).
2. **Replace Clerk references** with Keycloak/generic JWT references in secrets, ECS task definitions, and CI/CD workflows.
3. **Add new modules**: `elasticache` (Redis for gateway sessions), `rds` (if self-hosting Keycloak on ECS) or document the managed Keycloak approach.
4. **Switch CI/CD to OIDC**: Add `aws_iam_openid_connect_provider` to Terraform, create deployment roles, update workflows.
5. **Add Terraform CI workflow**: `terraform plan` on PR, `terraform apply` on merge.
6. **Add image promotion**: Build once in dev, promote image to staging/prod ECR.
7. **Add monitoring module**: CloudWatch alarms + SNS topics for critical metrics.
8. **Fix Dockerfiles**: Add `HEALTHCHECK`, use wildcard JAR paths, fix portal `public/` copy.
