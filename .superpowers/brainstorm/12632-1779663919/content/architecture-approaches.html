Key Architectural Decision: Inbound Email Processor

The forwarding address needs a service to receive emails. Two viable options given your AWS stack.

A

AWS SES Inbound (Recommended)

Already on AWS. SES receives email at your domain → stores in S3 → triggers SNS → backend webhook. No new vendor. Platform-level (not BYOAK). Generous free tier.

Pros

  • Same AWS account, no new vendor
  • Raw email storage in S3 (existing infra)
  • SNS→webhook is reliable
  • Free for receiving

Cons

  • SES Inbound only in us-east-1, us-west-2, eu-west-1
  • Parsing is your job (need email parser lib)
  • More infrastructure config (MX records, receipt rules)
B

SendGrid Inbound Parse

Platform-level SendGrid account (separate from tenant BYOAK). SendGrid receives email → parses headers/body/attachments → POSTs to your webhook as multipart form. Pre-parsed.

Pros

  • Pre-parsed (headers, body, attachments separated)
  • Existing SendGrid webhook pattern to follow
  • Simpler backend code

Cons

  • New vendor dependency (platform-level account)
  • SendGrid free tier: 100 emails/day inbound
  • Separate from tenant BYOAK accounts

Key Architectural Decision: Google Calendar Sync Strategy

How to keep Kazi and Google Calendar in sync bidirectionally.

A

Push Notifications + On-Demand Pull (Recommended)

Register a webhook with Google Calendar API (push notifications). Google notifies us when events change. We pull the delta. For Kazi→Google, push immediately on domain events.

Pros

  • Near-real-time sync
  • Low API quota usage
  • Google's recommended pattern

Cons

  • Webhook registration needs public HTTPS endpoint
  • Push channels expire (need renewal job)
  • Must handle out-of-order notifications
B

Periodic Polling (Incremental Sync)

Scheduled job polls Google Calendar API every N minutes using syncToken. Simpler but higher latency. No webhook registration needed.

Pros

  • No public webhook endpoint needed
  • Simpler error handling
  • Works behind firewalls

Cons

  • 5-15 min sync delay
  • Higher API quota consumption
  • Not real-time
