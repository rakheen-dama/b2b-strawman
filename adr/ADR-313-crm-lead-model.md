# ADR-313: CRM Lead Model — Deal Linked to Customer (Reusing the PROSPECT Lifecycle)

**Status**: Accepted

**Context**:

Phase 80 adds the missing top-of-funnel layer to Kazi's revenue engine. Today the funnel starts mid-stream: a `Customer` already exists, a firm member writes a `Proposal`, acceptance creates a `Project`, and the `Project` produces `Invoice`s. A "prospect" is nothing more than a `Customer` sitting in the existing `LifecycleStatus.PROSPECT` state — there is no opportunity record, no stage, no pipeline. We must decide how an *enquiry that has not yet been won* is represented before it is captured as a deal.

The decision is load-bearing for the whole phase and for every vertical fork. Kazi is multi-tenant schema-per-tenant (`Deal` and `PipelineStage` live in the per-tenant schema, isolation is pure `search_path`, no `tenant_id` columns), and the `crm` context must reuse the existing `Customer`/`Proposal`/`Project` chain rather than fork it. The shape of the lead/opportunity record determines whether we duplicate the entire `Customer` surface (identity, contacts, custom fields, tags, lifecycle, audit) for an as-yet-unwon enquiry, or reuse it. Because every fork (legal, consulting, accounting, agency) inherits this domain default-on, the model must be the simplest one that doesn't foreclose growth.

**Options Considered**:

1. **Deal-linked-to-Customer reusing the existing PROSPECT lifecycle (CHOSEN)** — A `Deal` (opportunity) **always** carries a non-null `customerId`. An inbound enquiry is captured as a single intake action (`DealIntakeService.intake`, `@Transactional`) that either attaches a `Deal` to an existing `Customer` or creates a new `PROSPECT` `Customer` **and** the open `Deal` atomically, reusing `CustomerService.createProspect` so validation is not duplicated.
   - Pros: Reuses the existing `LifecycleStatus.PROSPECT` state rather than inventing a parallel pre-customer concept; zero duplication of the `Customer` surface (contacts, custom fields, tags, audit, lifecycle all come for free); `Deal.customerId` is always non-null so every query and join is simple and total; winning a deal is a clean `PROSPECT → ONBOARDING` lifecycle nudge with no Lead-promotion ceremony; maximal fork portability — every fork already has `Customer`, so it inherits deals with no new identity model.
   - Cons: Every enquiry materialises a real `Customer` row, so high-volume / low-quality inbound (cold leads, spam) creates customer records that may never convert — the prospect list can accumulate noise that must be managed by lifecycle filtering or archival rather than by a separate "lead bin".
2. **Separate lightweight Lead entity with Lead→Customer conversion** — A distinct pre-customer `Lead` table (loose name/email/company + status) that is explicitly *converted* into a `Customer` (and then a deal/proposal) when it qualifies.
   - Pros: Keeps unqualified inbound out of the customer table, so the `Customer` list stays "real clients only"; matches the mental model of large-funnel CRMs (Salesforce-style Lead→Account conversion); a natural home for cheap, low-friction capture.
   - Cons: Duplicates much of the `Customer` surface (a Lead also wants contacts, custom fields, tags, notes, audit) — either reimplemented or left poorer than `Customer`; introduces a stateful conversion step with its own edge cases (duplicate-on-convert, partial data carry-over, what happens to the Lead's activity history); two code paths (Lead-mode and Customer-mode) ripple into every downstream feature; heavy v1 surface for a foundation phase whose mandate is reuse-not-rebuild.
3. **Dual-mode Deal (optional `customerId` OR loose inline contact fields)** — A single `Deal` table that either references a `Customer` or carries its own nullable contact columns (name/email/company) when no customer exists yet.
   - Pros: One table, no separate Lead entity; cheap capture (a deal with just a name) without forcing a customer row up front; defers the customer decision to win-time.
   - Cons: `customerId` becomes nullable, so every join, aggregation, and "customer's deals" query must branch on null; the loose contact fields drift out of sync with `Customer` and duplicate a thin slice of it; "promote the inline contact to a real Customer" is the Lead→Customer conversion problem in disguise, just hidden inside `Deal`; two semantically different rows share one table, which is the classic dual-mode trap the requirements explicitly reject.

**Decision**: Option 1 — a `Deal` always has a non-null `customerId`; an enquiry creates a `PROSPECT` `Customer` and an open `Deal` atomically in one transaction, reusing the existing customer-creation service.

**Rationale**:

1. **The PROSPECT lifecycle already *is* the lead concept.** `Customer.lifecycleStatus` already models `PROSPECT → ONBOARDING → ACTIVE → …`. A separate Lead entity would be a second, weaker representation of "not yet a client" sitting beside a state that already means exactly that. Reusing `PROSPECT` is the smaller, more honest model.
2. **Reuse over rebuild is the phase mandate, and the cost of a Lead is the whole Customer surface.** A lead realistically wants contacts, custom fields, tags, activity, and audit — all of which `Customer` provides today. A Lead entity would either reimplement them (large surface, drift risk) or be a poorer cousin. The non-null `customerId` lets `Deal` inherit all of that for free and keeps `Deal` itself tiny.
3. **A total, non-null link keeps everything simple and tenant-correct.** Because `customerId` is never null, the customer "Deals" tab, the per-customer aggregation, and the win-time `PROSPECT → ONBOARDING` nudge are all straightforward, total operations within the tenant schema — no null-branching, no dual code paths. The intake service reuses `CustomerService.createProspect` so there is one validation path, callable programmatically by the legal `intake-triage` AI seam (the request DTO is deliberately not UI-coupled).
4. **Fork portability.** Every vertical fork already has `Customer`. Inheriting deals therefore requires no new identity model in any fork — exactly the foundation-quality property this phase targets.

**Consequences**:
- Positive: No Lead↔Customer duplication, no conversion step, no dual-mode branching; `Deal.customerId` is always non-null so joins and aggregations are total and simple; the existing `PROSPECT` lifecycle, contacts, custom fields, tags, and audit are reused unchanged; the intake path is programmatically callable by the `intake-triage` seam.
- Positive: Winning a deal is a clean lifecycle nudge (`PROSPECT → ONBOARDING`, only when currently PROSPECT — never a downgrade), wired to the existing customer lifecycle guard (see [ADR-315](ADR-315-win-proposal-conversion-reuse.md)).
- Negative: Every enquiry materialises a `Customer` row, so high-volume low-quality inbound accumulates prospect records that must be managed via lifecycle filtering/archival rather than a disposable "lead bin" — an accepted tradeoff for a foundation phase with no inbound auto-capture in scope.
- Negative: Organisations that genuinely want a pre-customer triage bin (large-funnel marketing) are not served by v1; a future Lead layer remains possible but is explicitly deferred.
