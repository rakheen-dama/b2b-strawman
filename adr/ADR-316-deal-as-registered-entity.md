# ADR-316: Deal as a Registered Field-able / Taggable / Saved-View / Audited Entity

**Status**: Accepted

**Context**:

A `Deal` should behave like every other first-class Kazi entity: it should carry custom fields (and auto-apply field groups), be taggable, appear in saved views, write to the audit log, and surface in the activity feed. Kazi already has mature, reused cross-cutting subsystems for all of these — the field-definition / field-pack system (`EntityType` enum + `FieldPackSeeder`), the free-form tag and saved-view services, the `AuditEventTypeRegistry`, and the `ActivityMessageFormatter`. The phase mandate is reuse-not-rebuild, so the decision is whether `Deal` plugs into those existing registries or grows its own bespoke columns and code.

The relevant codebase facts are specific. Custom fields are gated by an `EntityType` enum (`fielddefinition/EntityType.java`, currently `PROJECT, TASK, CUSTOMER, INVOICE`) and `Customer` carries `custom_fields jsonb` + `applied_field_groups jsonb`. Tags (`EntityTagService`) and saved views (`SavedViewService`) use a **free-form `entityType` String discriminator** plus `entityId UUID` — they need *no* registration, just a new string value. Audit metadata uses `AuditEventTypeRegistry` with longest-dotted-prefix-wins semantics (per ADR-261). The activity feed reads audit events and formats them in `ActivityMessageFormatter` switch cases. All of this is per-tenant schema-per-tenant. Adding `DEAL` to the `EntityType` enum requires a field-group applicable-entity widening migration (V131), mirroring the V126 idempotent ALTER pattern.

**Options Considered**:

1. **Register `Deal` into the existing cross-cutting subsystems (CHOSEN)** — Add `DEAL` to `EntityType.java`; give `Deal` `custom_fields jsonb` + `applied_field_groups jsonb` mirroring `Customer`; tag and saved-view immediately via the free-form `entityType="DEAL"` discriminator (zero registration); add `deal.created` / `deal.stage_changed` / `deal.won` / `deal.lost` / `deal.reopened` rows to `AuditEventTypeRegistry` and matching `ActivityMessageFormatter` cases. Ship the V131 migration to widen the field-group applicable-entity guard to accept `"DEAL"`.
   - Pros: Reuses five mature subsystems wholesale — `Deal` becomes field-able/taggable/viewable/audited/activity-formatted with near-zero new code; tags and saved views cost *nothing* beyond passing a string (their discriminator is already free-form); custom fields follow the exact `Customer` shape so the field-pack seeder and UI components work unchanged; audit/activity follow the established registry conventions; one small, idempotent V131 widening migration; consistent with how every other entity in the codebase gets these capabilities.
   - Cons: Requires widening the `EntityType` enum and shipping the V131 applicable-entity migration across all tenant schemas; couples `Deal`'s field/audit behaviour to the shared registries' conventions (a feature, but it means `Deal` must follow them, e.g. dotted-prefix audit keys).
2. **Bespoke per-Deal columns / attributes** — Give `Deal` its own dedicated columns (or a deal-specific attributes table) for the handful of extra fields forks want, and a deal-specific tag/note/audit implementation.
   - Pros: No registry changes, no V131 migration; `Deal` is self-contained.
   - Cons: Reimplements custom fields, tags, saved views, audit, and activity that already exist — exactly the duplication the phase forbids; forks can't add ad-hoc deal attributes without a schema migration; deal tags/views/audit would diverge from the rest of the app's UI and behaviour; large, redundant surface for zero benefit; violates the reuse mandate.
3. **No custom-field / tag / saved-view support for Deal in v1** — Ship `Deal` with only its fixed columns; no custom fields, no tags, no saved views, audit only.
   - Pros: Smallest possible v1; no `EntityType` change, no V131 migration.
   - Cons: `Deal` becomes a second-class entity inconsistent with `Customer`/`Project`/`Task`/`Invoice`; forks immediately need a deal "source" taxonomy, deal tags, and saved board filters — features the existing subsystems would have given almost for free; saved views back the very list/board filter UI the frontend reuses, so dropping them would force a bespoke filter layer; false economy.

**Decision**: Option 1 — `Deal` registers into the existing field-definition, tag, saved-view, audit-metadata, and activity-formatter subsystems (adding `EntityType.DEAL` + the V131 widening migration) rather than growing bespoke implementations.

**Rationale**:

1. **Tags and saved views cost essentially nothing.** `EntityTag` and `SavedView` already key off a free-form `entityType` String + `entityId UUID`. There is *no registry to extend* — passing `entityType="DEAL"` makes deals taggable and saved-view-able immediately, and the deal list/board filter UI reuses the existing saved-views/tags components verbatim. Reimplementing these would be pure waste.
2. **Custom fields follow the `Customer` template.** `Deal` gets `custom_fields jsonb` + `applied_field_groups jsonb` exactly as `Customer` has them, so the `FieldPackSeeder`, the auto-apply field-group logic, and the custom-field UI all work unchanged once `DEAL` is in the `EntityType` enum. The `source` field is deliberately a plain string (not an enum) so forks extend their source taxonomy without a migration — and richer ad-hoc attributes go through custom fields, not new columns.
3. **The V131 widening is small and idempotent.** Because `EntityType` is enforced at the Java enum boundary, V131's real job is widening the field-group applicable-entity machinery to accept `"DEAL"`, mirroring V126's idempotent `ALTER TABLE … ADD COLUMN IF NOT EXISTS` shape (and, only if a named CHECK constraint exists in the tenant baseline, widening it to include `DEAL`). It runs safely across every tenant schema and can collapse to a no-op-safe guard or fold into V130 if no DB-level CHECK exists.
4. **Audit and activity follow the registry conventions.** Five `deal.*` event types register in `AuditEventTypeRegistry` (longest-dotted-prefix-wins, per ADR-261) under a CRM/SALES group, and matching `ActivityMessageFormatter` cases render them. Deals then appear in the global audit log and the deal-detail activity tab automatically, with no separate registry — the activity feed reads audit events.

**Consequences**:
- Positive: `Deal` becomes field-able, taggable, saved-view-able, audited, and activity-formatted with near-zero bespoke code; the deal list/board filter UI reuses the existing saved-views and tags components; custom fields and field packs work via the `Customer` template; audit/activity follow established conventions and surface deals everywhere other entities appear.
- Positive: Forks extend deal sources and ad-hoc attributes via the free-form `source` string and custom fields without schema migrations.
- Negative: Requires widening the `EntityType` enum and shipping the V131 applicable-entity migration to every tenant schema (small, idempotent, mirroring V126); `Deal`'s field/audit behaviour is bound to the shared registries' conventions (e.g. dotted-prefix audit keys, the CRM/SALES audit group must be added if none fits).
- Negative: Audit-event vocabulary and the activity formatter must be kept in sync (five new `deal.*` cases in both `AuditEventTypeRegistry` and `ActivityMessageFormatter`).
