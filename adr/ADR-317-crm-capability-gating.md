# ADR-317: CRM Capability Gating — Default-On Capability Family, Not a Vertical Module

**Status**: Accepted

**Context**:

Phase 80 must decide how access to the CRM/sales-pipeline feature is governed. Kazi has **two orthogonal gating axes**. **Capabilities** answer "does this role get to do X?" — they are resolved from `OrgRole` by the `MemberFilter` into the request's `CAPABILITIES`, and write endpoints are annotated `@RequiresCapability(...)`. **Modules** answer "is this whole feature area switched on for this org?" — gated by `VerticalModuleGuard` against `OrgSettings.enabledModules`, vertical-specific, and **default-off**, toggled in Settings → Features. `resource_planning` and `trust_accounting` are modules: vertical add-ons a fork enables deliberately. The founder decision for Phase 80 is explicit: CRM is **core practice management**, foundation-quality, and ships **default-on for every fork** — so it must use the capability axis, not the module axis.

The constraint is sharpened by the deliberate contrast with `resource_planning`. Ideation verified that `resource_planning` (the vertical-gated, default-off capacity module) is ~90% built and must not be touched or imitated for CRM's gating. CRM is the opposite case: every fork (legal, consulting, accounting, agency) should have a pipeline out of the box, with vertically-appropriate default stages seeded at provisioning via the existing pack-seeding infrastructure (`AbstractPackSeeder` + `vertical-profiles/*.json`). The granularity must match the existing style — e.g. the trust capabilities are a granular family, not one coarse switch.

**Options Considered**:

1. **Default-on granular capability family (CHOSEN)** — Add `VIEW_DEALS`, `MANAGE_DEALS`, `CLOSE_DEALS`, `MANAGE_PIPELINE` to `Capability.java`, granted to the standard roles in the default `OrgRole` capability sets (like `CUSTOMER_MANAGEMENT` / `PROJECT_MANAGEMENT`), and gate endpoints with `@RequiresCapability("…")`. **No** `VerticalModuleGuard` module, **no** `OrgSettings.enabledModules` flag for CRM. Default pipeline stages are vertical-seeded via a `DealPipelinePackSeeder extends AbstractPackSeeder` reading `resources/deal-pipeline-packs/*.json`, wired into each profile's `packs` map in `vertical-profiles/*.json`.
   - Pros: CRM is on for every fork out of the box (the foundation/fork-friendly goal) while access stays role-scoped on the permission axis; granularity matches existing families (trust, customer, project capabilities) so admins get familiar, fine-grained control (read vs manage vs close vs configure-pipeline); cleanly separates the *permission* axis (who may act) from the *availability* axis (is the feature on) — CRM is always available, roles differ; vertical defaults come from pack-seeding, not module flags; no Settings → Features toggle to forget to enable.
   - Cons: Four new capability constants must be added to the default capability sets of the standard roles (a one-time wiring across role definitions); a fork that genuinely wanted CRM *off* would have to remove the capabilities from every role rather than flip one module switch (an accepted tradeoff, since CRM is meant to be universal).
2. **Vertical-gated module like `resource_planning` (default-off)** — Register a `pipeline_crm` module in `VerticalModuleGuard` + `OrgSettings.enabledModules`, default-off, enabled per vertical/per org in Settings → Features.
   - Pros: Symmetric with `resource_planning`/`trust_accounting`; one switch turns the whole feature on/off per org; CRM stays hidden until a fork opts in.
   - Cons: Wrong axis — CRM is core practice management, not a vertical add-on; default-off contradicts the founder decision that every fork inherits a pipeline; forks would have to remember to enable it, and a fresh tenant would ship *without* a pipeline; conflates "is the feature on" with "may this role act", which the capability axis already handles better; imitates exactly the module pattern ideation said not to copy for CRM.
3. **Single coarse `CRM` capability** — One `CRM` capability gating all deal/pipeline endpoints.
   - Pros: Minimal — one constant, one annotation everywhere; trivial to grant.
   - Cons: No granularity — can't grant a member read-only board access without also letting them close deals or reconfigure pipeline stages; mismatched with the existing granular families (trust, customer, project) and the requirement that CRM "matches existing style"; a junior member who should only *view* the pipeline would get full close/configure rights or nothing; configuring stages (an admin act) collapses into the same grant as moving a card (an everyday act).

**Decision**: Option 1 — CRM ships as a default-ON granular capability family (`VIEW_DEALS`, `MANAGE_DEALS`, `CLOSE_DEALS`, `MANAGE_PIPELINE` in `Capability.java`, gated via `@RequiresCapability`), not behind a `VerticalModuleGuard` / `OrgSettings.enabledModules` module flag; default pipeline stages are vertical-seeded via `AbstractPackSeeder` + `vertical-profiles/*.json`.

**Rationale**:

1. **A capability is the permission axis; a module is the availability axis — and CRM's availability is "always".** The codebase deliberately separates "does this role get to do X?" (capabilities, resolved by `MemberFilter`) from "is this whole feature switched on for the org?" (`VerticalModuleGuard` + `OrgSettings.enabledModules`). CRM is core practice management that every fork should have on by default, so its availability is not in question — only *who may do what* is. That is precisely the capability axis, and precisely *not* the module axis.
2. **The explicit contrast is `resource_planning`.** `resource_planning` and `trust_accounting` are vertical-gated, default-off **modules** — add-ons a fork opts into. CRM is the inverse: a foundation capability every fork inherits. Registering a `pipeline_crm` module would put CRM on the wrong axis, ship fresh tenants without a pipeline, and copy the very pattern ideation flagged as not-for-CRM.
3. **Granularity matches existing families.** The four-capability family (view / manage / close / manage-pipeline) mirrors the granular trust and customer/project capability families, giving admins familiar fine-grained control: a member can be granted read-only board access (`VIEW_DEALS`) without `CLOSE_DEALS` or `MANAGE_PIPELINE`. A single coarse `CRM` capability would collapse everyday acts (moving a card) and admin acts (reconfiguring stages) into one grant.
4. **Vertical defaults come from pack-seeding, not a module flag.** Default stages are seeded per vertical via `DealPipelinePackSeeder extends AbstractPackSeeder` and the `deal_pipeline` key in each `vertical-profiles/*.json` (legal-za, consulting-za, accounting-za, default), idempotently at provisioning and via the reconciliation seeder — so each fork gets sensible stages with no hand-written per-tenant SQL and no module toggle (see [ADR-314](ADR-314-pipeline-stage-model.md)).

**Consequences**:
- Positive: Every fork inherits a working sales pipeline default-on, with role-scoped access on the permission axis; granular grants (view/manage/close/manage-pipeline) match the existing capability-family style; no Settings → Features toggle to forget; vertical default stages arrive via pack-seeding, not bespoke SQL or a module flag.
- Positive: Keeps the permission axis (who may act) cleanly separate from the availability axis (is the feature on) — CRM is always available; roles differ.
- Negative: Four capability constants must be wired into the standard roles' default capability sets (a one-time change across role definitions); a fork wanting CRM *off* must strip the capabilities from roles rather than flip one module switch — an accepted tradeoff because CRM is meant to be universal.
- Negative: Because there is no module flag, CRM cannot be hidden org-by-org from Settings → Features the way `resource_planning` can; suppressing it is a role/capability exercise instead.
- Related: [ADR-314](ADR-314-pipeline-stage-model.md) (the stages this capability configures, vertical-seeded via pack-seeding), [ADR-316](ADR-316-deal-as-registered-entity.md) (the registered deal operations these capabilities gate), the `resource_planning` module as the explicit default-off contrast case.
