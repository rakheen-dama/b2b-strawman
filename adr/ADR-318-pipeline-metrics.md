# ADR-318: Pipeline Metrics — Weighted Value, Win Rate & Currency Assumption

**Status**: Accepted

**Context**:

Phase 80's reporting surface must give a firm an honest, at-a-glance read on its pipeline: how much weighted value is in play, how well it converts, and what each stage holds — surfaced through a pipeline-summary endpoint (`GET /api/dashboard/pipeline-summary`) and a dashboard widget (mirroring `TeamUtilizationWidget`). We must define the metrics precisely, choose the v1 reporting mechanism, and settle the currency question — all consistently, so the widget and the endpoint never disagree.

The constraints are concrete. `Deal` carries `valueAmount` + `valueCurrency` (defaulted from `OrgSettings` at create time), a nullable per-deal `probabilityPct` override, and `wonAt`/`lostAt` timestamps. Stages carry `defaultProbabilityPct` and a `stageType` (`OPEN`/`WON`/`LOST`, see [ADR-314](ADR-314-pipeline-stage-model.md)). The codebase has a `ReportDefinition` system for full configurable reports, but the founder decision defers that for CRM v1 in favour of a purpose-built summary. Aggregation runs in the per-tenant schema and should be a single SQL query, not N+1 entity loads. Money is first-class but the firm-internal pipeline is assumed single-currency for v1 (mixed-currency forecasting is out of scope). The win-rate window must be explicit and shared so the widget and endpoint agree.

**Options Considered**:

1. **Purpose-built summary endpoint: weighted value + windowed win rate (CHOSEN)** — A `PipelineSummaryService.getSummary(...)` (`@Transactional(readOnly = true)`) returning per-stage breakdown (count, total value, weighted value), open weighted pipeline value, win rate over a date window, average deal size, and optional average days-to-close. `effectiveProbability = per-deal override ?? stage.defaultProbabilityPct`, clamped to 100 for WON and 0 for LOST; weighted value = `Σ(valueAmount × effectiveProbability / 100)` over **OPEN** deals; win rate = `won / (won + lost)` counting deals whose `wonAt`/`lostAt` fall inside the window (default **trailing 90 days** by close date). Single org currency assumed; values summed directly without FX. The probability fallback and WON=100/LOST=0 clamp are applied **in SQL** so it's one query.
   - Pros: Exactly the v1 surface the requirements ask for, no more; one SQL query per metric (no N+1); the widget and endpoint share the same window and definitions so they never disagree; honest forecast — open deals contribute their weighted value, WON deals leave the forecast (realised), LOST contribute 0; cheap to build and to read; the single-currency assumption keeps the SQL a plain `SUM` with no FX layer.
   - Cons: Bespoke endpoint shape rather than a reusable report definition, so new pipeline cuts (by source, by owner-over-time, custom windows beyond the few params) require code changes; the single-currency assumption silently mis-sums if a tenant ever mixes currencies (documented v1 limitation).
2. **Full `ReportDefinition`-based pipeline reports** — Model pipeline metrics as configurable `ReportDefinition`s with the existing reporting/export/scheduling machinery.
   - Pros: Reuses an existing system; users can define arbitrary cuts, columns, filters, and scheduled deliveries without code changes; consistent with other reports.
   - Cons: Large v1 surface — weighted value, effective-probability fallback, WON=100/LOST=0 clamping, and the win-rate window are bespoke aggregations that don't map cleanly onto a generic report's column/filter model; building the report-definition wiring, exports, and scheduling is far more than the widget needs; explicitly deferred by the founder decision; over-engineering for a v1 whose only consumers are one endpoint and one widget.
3. **Raw counts only (no weighting)** — Report deal counts and unweighted total value per stage; no effective probability, no weighted forecast, no win rate.
   - Pros: Trivial — plain `COUNT`/`SUM`, no probability logic, no window, no clamp.
   - Cons: Gives a dishonest picture — a stack of early-stage long-shots looks identical to a stack of near-certain deals; no forecast (the whole point of weighting); no win-rate visibility, so the firm can't see conversion; fails the requirement for weighted value + win rate as the v1 metric.

**Decision**: Option 1 — a purpose-built pipeline-summary endpoint computing weighted value = `Σ(valueAmount × effectiveProbability)` over OPEN deals and win rate = `won / (won + lost)` over a trailing-90-day window by `wonAt`/`lostAt`, assuming a single org currency for v1; `ReportDefinition`-based reports are deferred.

**Rationale**:

1. **Weighted value over OPEN deals is the honest forecast.** `effectiveProbability` resolves to the per-deal override if set, else the stage's `defaultProbabilityPct`, clamped to 100 for WON and 0 for LOST regardless of override (reusing the [ADR-314](ADR-314-pipeline-stage-model.md) probability source). Summing `valueAmount × effectiveProbability / 100` over **OPEN** deals only means an open deal at "Qualified/20%" contributes 20% of its value, a WON deal *leaves* the forecast and becomes realised revenue, and a LOST deal contributes 0 — so the forecast reflects live opportunity, not history. Computing this in SQL (`COALESCE(d.probability_pct, s.default_probability_pct)`) keeps aggregation a single query.
2. **A defined, shared window keeps win rate trustworthy.** Win rate = `won / (won + lost)` counting deals whose `wonAt`/`lostAt` fall in the window, default trailing 90 days by close date. The endpoint and the widget read the **same** window so they never disagree; the window is a parameter so callers can widen it. Average deal size and optional average days-to-close (`wonAt − createdAt` over won deals in window) round out the v1 picture without adding machinery.
3. **`ReportDefinition` is the wrong tool for v1.** The metrics are bespoke aggregations (effective-probability fallback, WON/LOST clamp, OPEN-only weighting, win-rate window) that don't map cleanly onto a generic report's column/filter model, and the only v1 consumers are one endpoint and one widget. A purpose-built summary is smaller, faster, and honest; full configurable reports/exports/scheduling are deferred per the founder decision, with the summary endpoint as the explicit v1 surface.
4. **Single currency is a deliberate, documented v1 assumption.** `valueCurrency` defaults from `OrgSettings`, and aggregation sums `valueAmount` directly without FX conversion. For the firm-internal pipeline of a single tenant this is correct; mixed-currency deals would mis-sum, so mixed-currency forecasting is explicitly out of scope and documented as a v1 limitation rather than silently half-handled.

**Consequences**:
- Positive: The summary endpoint and dashboard widget share one set of definitions and one window, so they never disagree; each metric is a single SQL query (no N+1); the forecast is honest (OPEN-weighted, WON realised, LOST zeroed); the build is small and fast.
- Positive: The per-deal probability override and stage default ([ADR-314](ADR-314-pipeline-stage-model.md)) flow straight into the weighted value with no extra modelling.
- Negative: A single org currency is assumed; a tenant that mixes currencies would get a meaningless raw sum — a documented v1 limitation, not silently handled.
- Negative: New pipeline cuts beyond the endpoint's parameters require code changes rather than user-defined report configuration; `ReportDefinition`-based reports, exports, scheduled delivery, and forecasting/quota are deferred to a future phase.
