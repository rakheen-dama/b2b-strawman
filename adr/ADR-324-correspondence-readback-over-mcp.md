# ADR-324: Correspondence Read-Back over MCP — Body-from-Kazi, Project-Access-Gated, Existing-Egress-Consent

**Status**: Accepted

**Context**:

Phase 81 made the Kazi MCP server able to **write** correspondence (`file_correspondence`, `attach_document`, `propose_task`, ADR-319/321/322): a firm's own Claude (BYOC) files inbound email into the right matter, attaches its documents, and proposes follow-up tasks for in-Kazi approval. It left a read/write asymmetry — **Claude can write correspondence into Kazi but cannot read it back over MCP.** The only correspondence read surface is the in-app REST "Correspondence" tab (Epic 586A), which is not an MCP tool. The motivating consumer is a `correspondence-digest` skill in `../claude-for-legal-sa` that answers "across my matters' inboxes, what needs a reply and what implies a deadline?" — and the deadlines and obligations live in the email **body**, not the subject line, so a useful digest must read the body.

The `correspondence/` bounded context already persists everything needed: the `Correspondence` entity holds `subject`, `bodyText`, `bodyHtml`, `fromAddress`, `toAddresses`, `ccAddresses`, `sentAt`, `receivedAt`, `threadKey`, `messageId`, `direction`, and linkage (`customerId`/`projectId`); `CorrespondenceService.listForProject`/`listForCustomer` already return `Page<CorrespondenceListResponse>` with view-access enforcement and a 50-row clamp. So the design question is **not** how to model correspondence — that exists — but how to expose a read of it over MCP: where the body comes from, how the read is gated, and what consent governs egressing the body. Three constraints frame it: a legal correspondence body is sensitive client PII (privileged material, deadlines); the in-app tab already enforces per-matter view-access; and the firm's existing read tools (`get_matter`, `get_client`) already egress comparable PII under a single read-egress consent.

**Options Considered**:

### A. Where the body comes from

1. **Read the body from Kazi via `get_correspondence` (CHOSEN)** — a new body-bearing read tool over the already-filed `Correspondence` row.
   - Pros: provider-neutral — works regardless of whether the firm uses Gmail, Outlook, or a pasted email; works for **archived/deleted** mail (Kazi holds the filed copy); identical egress posture to `get_matter`/`get_client` (one consent, one audit family); reuses the existing entity and `findById` (zero new storage); the body Claude reads is exactly the body Kazi filed (no drift).
   - Cons: requires Kazi to hold and egress the body (it already does — that is what `file_correspondence` persisted); the digest sees only correspondence that was filed (un-filed mail is invisible — acceptable: the digest is about *filed matter correspondence*).
2. **Re-hydrate the body from the firm's Gmail by `messageId`** — return only metadata over MCP; have Claude fetch the body from the Gmail connector.
   - Pros: Kazi never re-egresses the body; the body stays in the mail provider.
   - Cons: **provider-locks the digest to Gmail** (founder rejection) — Outlook/IMAP/pasted-email firms get no body; breaks for archived or deleted mail (Message-ID no longer resolvable); splits the read across two systems with two trust boundaries; the `messageId` is already in Kazi anyway. Rejected in the 2026-06-27 founder ideation precisely because it would make the digest Gmail-only.
3. **Metadata-only / no body at all** — ship `list_correspondence` and stop; never expose the body.
   - Pros: smallest surface; no body egress to reason about.
   - Cons: too thin — the digest's entire value (deadlines, obligations, "what needs a reply") lives in the body; a subject-only digest is not useful. Rejected.

### B. Gating model for the two read tools

1. **Project view-access, mirroring `get_matter`/`search_documents`/`get_document_url` (CHOSEN)** — `list_correspondence`'s matter path and `get_correspondence` resolve the correspondence's `projectId` and call `requireViewAccess`. `emitDenied(tool, "project-access", …)` is emitted **only** when the record **is found** with a `projectId` and `requireViewAccess` **refuses** — never for an absent or cross-tenant id, which is invisible to `findById` (`Optional.empty()` → `ResourceNotFoundException` → `not_found` with **no** `mcp.access.denied`, a lookup miss is not a policy denial). Both surface a non-leaking `not_found`. This mirrors `get_matter` exactly. Customer-only correspondence (no `projectId`) falls back to existence-in-tenant, matching `listForCustomer`/`search_documents` CUSTOMER scope.
   - Pros: **matches the in-app REST tab exactly** — an MCP user sees precisely the correspondence their Kazi role permits, no more; consistent with the sibling single-entity reads on related domains (matters, documents); reuses `CorrespondenceService.listForProject`'s existing `requireViewAccess`; correspondence is matter-scoped firm data of the same sensitivity as documents, so it should gate like documents.
   - Cons: the digest degrades for correspondence on matters the member cannot view (correct behaviour, but worth stating); two code paths (project vs customer-only) — minor, already the established `search_documents` shape.
2. **Org-wide `MCP_ACCESS` capability gate, like `resolve_matter_by_email`** — wrap both tools in `McpCapabilityGuard.gatedTool("MCP_ACCESS", …)`; any authenticated tenant member can read any correspondence.
   - Pros: simpler, org-uniform, one gate.
   - Cons: **less restrictive than the in-app UI** — a member could read correspondence (including the privileged body) on a matter they cannot see in the app; and the `MCP_ACCESS` front-door gate is not yet live (565B pending), so relying on it alone would leave the tools effectively ungated today. Wrong for body-bearing matter-scoped PII. Rejected.
3. **A new correspondence-specific read capability** — e.g. `MCP_READ_CORRESPONDENCE`.
   - Pros: granular opt-in for body read-back.
   - Cons: a new capability to provision, document, and test for no behavioural gain over project-access; correspondence is not more sensitive than documents (which use project-access, not a bespoke capability); over-engineering for a two-tool phase. Rejected.

### C. Consent posture for body egress

1. **Inherit the existing read-egress consent (CHOSEN)** — the body egresses under the same enablement + POPIA-egress-consent gate (`McpEnablementService.effectiveState()`) that already governs `get_matter`/`get_client`.
   - Pros: a correspondence body is the same POPIA category as the client PII `get_matter`/`get_client` already egress under that consent; one consent for the firm to reason about; no new flag to provision, surface in the UI, or migrate; revoking consent or disabling the connector turns body read-back off uniformly with every other read.
   - Cons: the firm cannot consent to matter/client reads but *withhold* correspondence-body reads (acceptable — same sensitivity class; a separate flag implies a distinction that does not exist).
2. **A new, stricter consent flag for bodies** — a dedicated "allow correspondence body egress" toggle.
   - Pros: lets a firm enable metadata reads but block body egress.
   - Cons: implies the body is a *new* POPIA category when it is the same class of client PII already egressed; new flag to build, default, migrate, and document; fragments the consent model. Rejected.
3. **No consent** — egress the body with only the enablement check.
   - Pros: none beyond the enablement gate already present.
   - Cons: egressing client PII to an external LLM with no POPIA consent is unacceptable. Rejected.

**Decision**: Add two MCP **read** tools on a new `CorrespondenceReadTools` bean. `list_correspondence(matterId | customerId, page)` returns the existing metadata projection as an `McpPage`, reusing `CorrespondenceService.listForProject`/`listForCustomer` verbatim (option **A1** for the read source, no new list path). On the customer path, `listForCustomer` does `customerRepository.findById(customerId).orElseThrow(ResourceNotFoundException)` before listing, so an unknown/cross-tenant `customerId` returns a non-leaking `not_found` (not a silent empty page, and no `mcp.access.denied`). `get_correspondence(id)` returns a single record **with body** via one net-new `CorrespondenceService.requireDetailById` method and a new `McpCorrespondenceDto` (the JPA entity never crosses the MCP boundary). Both are gated by **project view-access** (option **B1**) — `requireViewAccess(projectId)` when the correspondence is matter-scoped, existence-in-tenant when customer-only — surfacing a found-but-refused access denial as a non-leaking `not_found` + `mcp.access.denied(gate=project-access)`, while an absent/cross-tenant id surfaces as `not_found` with **no** `mcp.access.denied` (security-by-obscurity). This is **not** the org-wide `MCP_ACCESS` capability gate — and supersedes the requirements doc's `McpCapabilityGuard.gatedTool("MCP_ACCESS", …)` suggestion: the new read tools do **not** call `McpCapabilityGuard`, matching `get_matter`/`get_document_url`. Body egress rides the **existing** read-egress consent (option **C1**) — no new consent flag, no new gate, no new audit family. Both tools sit behind the enablement + POPIA-consent guard and emit the pre-existing `mcp.tool.invoked` read audit carrying **safe entity refs only** (correspondence id, matter/customer ids) — never subject, from-address, or body. **No new entity, table, migration, write tool, LLM, or Gmail integration.**

**Rationale**:

1. **Body from Kazi keeps the digest provider-neutral and durable.** Reading from the filed `Correspondence` row works for Gmail, Outlook, IMAP, and pasted email alike, and for archived/deleted mail — the firm's filed copy is the source of truth. Re-hydrating from Gmail would provider-lock the digest, which the founder rejected.
2. **Project-access matches the UI and the sibling reads.** Correspondence is matter-scoped firm data of the same sensitivity as documents; gating it like `get_matter`/`get_document_url` means an MCP user reads exactly what their Kazi role permits — no MCP-vs-UI asymmetry. An org-wide capability gate would be *looser* than the in-app tab and is not even live yet (565B).
3. **Inheriting read-egress consent is honest about the data class.** A correspondence body is the same POPIA category of client PII that `get_matter`/`get_client` already egress under one consent. A new flag would imply a distinction that does not exist and fragment the firm's consent model.
4. **Reuse over rebuild.** `list_correspondence` reuses the access-checked, clamped public `listForProject`/`listForCustomer` (**not** the lower-level package-private `listByProject`/`listByCustomer`, which bypass both the view-access check and the `McpPagination` page clamp — the requirements doc loosely names `listBy*`, but wiring those would skip the gate and the 50-row clamp); `get_correspondence` adds exactly one service method over the inherited `findById`. No new repository method, no new storage, no migration.
5. **Read capability, read audit, no PII in audit.** The tools emit only the pre-existing `mcp.tool.invoked` / `mcp.access.denied` (no new audit event type — `AuditEventTypeRegistry` stays at 36) with `entityRef`s only, mirroring `resolve_matter_by_email`'s "entityRefs carries ids only, never the email/name (POPIA)". The body is *returned to the LLM* by design but *never written to audit details*.

**Consequences**:
- Positive: closes the Phase 81 read/write asymmetry; the `correspondence-digest` skill can ground on filed correspondence including the body; provider-neutral and works for archived mail; one consent, one audit family, zero new POPIA category; no new entity/table/migration/write-path.
- Positive: an MCP user reads exactly the correspondence their Kazi role permits — identical to the in-app tab; tenant-isolation holds via search_path (a cross-tenant id returns `not_found`, never a body).
- Positive: the only registry-test edit is one line in `ALLOWED_TOOL_BEANS`; no audit-count bump; no regex/exemption changes (`list_`/`get_` already match the read-verb pattern).
- Negative: **the digest degrades for correspondence whose matter the member cannot view** — the member sees only what their Kazi role permits, exactly as with documents. This is correct behaviour, but it means a digest's completeness depends on the running member's matter access (a firm owner sees more than a member).
- Negative: a body-bearing read tool egresses privileged client PII to an external LLM — acceptable only because it inherits the firm's deliberate read-egress consent and the enablement gate; if the firm has not consented, both tools return `not_enabled` and egress nothing.
- Negative: customer-only correspondence (no `projectId`) is gated by existence-in-tenant rather than per-matter view-access — consistent with `listForCustomer`/`search_documents` CUSTOMER scope, but slightly looser than the matter path; the asymmetry is inherent to the data model (a correspondence with no matter has no matter to access-check).
