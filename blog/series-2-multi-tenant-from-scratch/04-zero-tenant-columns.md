# 83 Entities, Zero Tenant Columns: How Schema Isolation Simplifies Everything

*Part 4 of "Multi-Tenant from Scratch" — a series on building production multi-tenant architecture with PostgreSQL, Spring Boot 4, and Hibernate 7.*

---

DocTeams has 83 JPA entities. Projects, documents, customers, tasks, time entries, invoices, rates, budgets, comments, notifications, audit events, templates, checklists, automation rules, and more.

Not a single one has a `tenant_id` column.

This isn't a shortcut. It's the design. And it's the single best architectural decision I made in the entire project.

## What "Zero Tenant Columns" Means

In most multi-tenant tutorials, every entity looks like this:

```java
@Entity
@Table(name = "projects")
public class Project {
    @Id private UUID id;
    @Column(name = "tenant_id") private String tenantId;  // <-- this
    private String name;
    // ...
}
```

And every query looks like this:

```java
@Query("SELECT p FROM Project p WHERE p.tenantId = :tenantId AND p.name = :name")
List<Project> findByNameInTenant(@Param("tenantId") String tenantId, @Param("name") String name);
```

And every code review includes: "Did you add `WHERE tenant_id = ?` to this query?"

In DocTeams, entities look like this:

```java
@Entity
@Table(name = "projects")
public class Project {
    @Id
    @GeneratedValue(strategy = GenerationType.UUID)
    private UUID id;

    @Column(name = "name", nullable = false)
    private String name;

    @Column(name = "description", columnDefinition = "TEXT")
    private String description;

    @Enumerated(EnumType.STRING)
    @Column(name = "status", nullable = false)
    private ProjectStatus status;

    @Column(name = "created_by", nullable = false)
    private UUID createdBy;

    @Column(name = "created_at", nullable = false, updatable = false)
    private Instant createdAt;

    // ... business fields only. No tenant_id.
}
```

And queries look like this:

```java
public interface ProjectRepository extends JpaRepository<Project, UUID> {
    List<Project> findByStatus(ProjectStatus status);
    Optional<Project> findByName(String name);
}
```

Standard Spring Data JPA. No tenant parameter. `findById()` works correctly. `findAll()` returns only the current tenant's projects. `count()` counts only the current tenant's rows.

The magic happens one level below: when Hibernate opens a database connection, the connection provider sets `SET search_path TO tenant_a1b2c3d4e5f6`. Every SQL query on that connection — whether generated by JPQL, Spring Data method names, native SQL, or raw JDBC — automatically hits the right schema.

The isolation isn't in the application code. It's in the PostgreSQL connection.

## The Developer Experience Win

Here's what it's like to add a new entity to DocTeams:

```java
// 1. Create the entity — plain JPA, nothing special
@Entity
@Table(name = "retainers")
public class Retainer {
    @Id @GeneratedValue(strategy = GenerationType.UUID)
    private UUID id;
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "customer_id", nullable = false)
    private Customer customer;
    private BigDecimal amount;
    private String currency;
    // ...
}

// 2. Create the repository — standard JpaRepository
public interface RetainerRepository extends JpaRepository<Retainer, UUID> {
    List<Retainer> findByCustomerId(UUID customerId);
}

// 3. Create the migration — standard DDL
// V24__create_retainers.sql
CREATE TABLE IF NOT EXISTS retainers (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    customer_id UUID NOT NULL REFERENCES customers(id),
    amount NUMERIC(12,2) NOT NULL,
    currency VARCHAR(3) NOT NULL DEFAULT 'ZAR',
    created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now()
);
```

That's it. No `TenantAware` interface. No `@FilterDef`. No `tenant_id` column. No custom `findOneById()`. No RLS policy. The entity is a plain entity. The repository is a plain repository. The migration is a plain migration.

Compare this to what it looked like during the shared-schema period (Phase 2-13):

```java
// Entity — had to implement TenantAware + add annotations + tenant column
@Entity
@Table(name = "projects")
@FilterDef(name = "tenantFilter",
    parameters = @ParamDef(name = "tenantId", type = String.class))
@Filter(name = "tenantFilter", condition = "tenant_id = :tenantId")
public class Project implements TenantAware {
    @Column(name = "tenant_id")
    private String tenantId;
    // ...
}

// Repository — had to add custom findOneById because findById bypasses @Filter
@Query("SELECT p FROM Project p WHERE p.id = :id")
Optional<Project> findOneById(@Param("id") UUID id);

// Migration — had to add RLS policy
ALTER TABLE projects ENABLE ROW LEVEL SECURITY;
CREATE POLICY tenant_isolation ON projects
    USING (tenant_id = current_setting('app.current_tenant'));
```

Every new entity required 4 extra pieces: the interface, the annotations, the column, and the RLS policy. Over 83 entities, that's 332 pieces of tenant boilerplate that no longer exist.

## What About Relationships?

Foreign keys work naturally across entities within the same schema:

```java
@Entity
@Table(name = "time_entries")
public class TimeEntry {

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "task_id")
    private Task task;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "project_id", nullable = false)
    private Project project;

    @Column(name = "member_id", nullable = false)
    private UUID memberId;
}
```

`TimeEntry.task` references a `Task` in the same schema. The `REFERENCES tasks(id)` constraint in the migration ensures referential integrity. Because both tables exist in the same schema, the constraint works without qualification.

Cross-schema references (a time entry in Tenant A referencing a task in Tenant B) are structurally impossible. The `tasks` table in Tenant A's schema is a completely different table from the `tasks` table in Tenant B's schema. A foreign key from one to the other would require explicit schema qualification — which doesn't exist in the migration.

This is what I mean by "structural isolation." The impossibility of cross-tenant data access isn't enforced by a policy or a filter. It's a consequence of the physical schema layout.

## Native Queries: Still Safe

One of the biggest risks with Hibernate `@Filter` was that native SQL queries bypassed it. If you wrote:

```java
@Query(value = "SELECT sum(amount) FROM invoices WHERE status = 'PAID'",
       nativeQuery = true)
BigDecimal totalPaidRevenue();
```

The `@Filter` wouldn't apply. In a shared schema, this query would sum *all* tenants' invoices.

With schema-per-tenant, native queries are safe by default. `SET search_path TO tenant_abc123` applies to all SQL on the connection — JPQL, named queries, native SQL, stored procedures, everything. The query above hits `tenant_abc123.invoices` and only `tenant_abc123.invoices`.

This was especially important for the profitability reports (Phase 8), which use complex native queries with aggregations:

```java
@Query(value = """
    SELECT p.id, p.name,
           COALESCE(SUM(te.hours * te.billing_rate), 0) as revenue,
           COALESCE(SUM(te.hours * te.cost_rate), 0) as cost,
           COALESCE(SUM(te.hours * te.billing_rate), 0) -
           COALESCE(SUM(te.hours * te.cost_rate), 0) as margin
    FROM projects p
    LEFT JOIN time_entries te ON te.project_id = p.id
    WHERE te.billable = true
    GROUP BY p.id, p.name
    ORDER BY margin DESC
    """, nativeQuery = true)
List<ProjectProfitabilityProjection> getProjectProfitability();
```

No `WHERE tenant_id = ?` needed. The `search_path` handles it.

## The `findById()` Question

In the shared-schema era, `JpaRepository.findById()` was dangerous because it used `Session.get()`, which bypasses `@Filter`. We had to add `findOneById()` to every repository.

With schema-per-tenant, `findById()` is safe:

```java
// This is safe. search_path ensures we only see this tenant's data.
var project = projectRepository.findById(projectId)
    .orElseThrow(() -> new ResourceNotFoundException("Project", projectId));
```

`Session.get()` generates `SELECT * FROM projects WHERE id = ?`. The `projects` table it hits is `tenant_abc123.projects` because that's what `search_path` resolves to. Even if someone passes a UUID from another tenant, the query returns empty — the row physically doesn't exist in this schema.

This is the single biggest quality-of-life improvement from schema isolation. `findById()` — the most commonly used JPA method — just works. No special handling. No custom wrappers.

## When You Need Cross-Tenant Data

There are two cases where you need to query across schemas:

**Platform admin views.** A system administrator needs aggregate metrics: total tenants, total projects across all tenants, revenue rollup. These queries need to reach into multiple schemas.

I handle this with explicit schema iteration:

```java
public PlatformStats getPlatformStats() {
    var mappings = mappingRepository.findAll();
    int totalProjects = 0;
    for (var mapping : mappings) {
        totalProjects += ScopedValue.where(RequestScopes.TENANT_ID, mapping.getSchemaName())
            .call(() -> projectRepository.count());
    }
    return new PlatformStats(mappings.size(), totalProjects);
}
```

Not elegant. But platform admin views are infrequent (daily dashboards, not per-request). And the explicit iteration makes it obvious what's happening — there's no hidden cross-tenant access.

**Scheduled jobs.** A cron job that sends overdue invoice reminders needs to check every tenant's invoices. Same pattern: iterate over schemas, bind each one, run the query.

Both cases are rare and explicit. The default is single-tenant isolation. Cross-tenant access requires deliberate code that iterates schemas — you can't accidentally query across tenants.

## The Numbers After Phase 13

When I removed the shared-schema path:

- **27 entities** stripped of `@FilterDef`, `@Filter`, and `tenant_id` column (now 83 entities with zero tenant columns)
- **22 repositories** had custom `findOneById()` removed
- **17 migrations** had RLS policies removed
- **~500 lines** of infrastructure code deleted (`TenantAware`, `TenantAwareEntityListener`, `TenantFilterTransactionManager`, `TenantUpgradeService`)
- **7 test files** deleted (shared-schema-specific tests)
- **74 call sites** changed from `findOneById()` back to `findById()`

The remaining multitenancy infrastructure is:
- `RequestScopes.java` — 5 `ScopedValue` declarations (~60 lines)
- `TenantFilter.java` — JWT → schema resolution (~80 lines)
- `SchemaMultiTenantConnectionProvider.java` — `SET search_path` (~50 lines)
- `TenantIdentifierResolver.java` — Hibernate callback (~15 lines)
- `ScopedFilterChain.java` — Servlet bridge (~30 lines)

About 235 lines of infrastructure code. For 83 entities and full tenant isolation.
