## Anti-cheat reminders (specific to this session's gotchas)

These are lessons learned the hard way during this session:

- **Vercel/Next.js skill consultation is now a user-rule.** Mid-session the user said "Refer ui questions to NextJS or Vercel expert agents." Memory `feedback_ui_questions_use_vercel_experts.md` is the durable record. Used `vercel:nextjs` for OBS-704 v3 hydration question and `vercel:shadcn` for audit-03 dialog composition pattern. **Apply this rule for any UI / Next.js / React / hydration / Radix / shadcn question — don't reason from first principles.**
- **`prettier --check .` is a CI gate the local pre-merge marker doesn't cover.** PR #1262 first push failed CI on Frontend because two pre-existing files (`edit-customer-dialog.test.tsx`, `TrustBalanceCard.tsx`) had drifted out of prettier compliance after a version bump. The drift wasn't caused by the PR but blocked it because `format:check` runs whole-tree. Folded the 3-line cleanup into the same PR (option #1, user-authorised). Watch for future drift on whole-tree-scoped CI gates.
- **CodeRabbit reviews once per PR, not per commit.** Saw on #1262 + #1263. Stale inline-comment lines after a fix-up commit are normal — the comments don't update against new code. The reviews-array timestamp gives the "this is the only review" signal. CI status is the truth-source for "should I merge".
- **Pre-merge gate hook checks marker mtime + exit:0, NOT SHA.** So the marker doesn't strictly need to point at HEAD — but updating it for honesty is good practice and was the convention I followed.
- **Frontend dev server on port 3000 is Keycloak-auth-only — agents can't browser-verify there.** Use SSR snapshot tests as the regression guard pattern (PR #1262 introduced one for `CreateProposalDialog`). The mock-auth E2E stack would take 3-5 min to spin up for a single attribute-presence check; deferred in favour of the snapshot test.
- **`AlertDialogAction` auto-closes regardless of action result.** Pre-existing latent bug pattern. CodeRabbit caught this on #1263 — the inline AlertDialog had `onClick={() => handleDelete(...)}` on `AlertDialogAction`, which closes the dialog before the action result is known. Replace with controlled `<Button>` when the action needs to stay open on failure. Should have been added to the bug-classes catalogue but didn't reach 2 occurrences this session — track for future.
- **Dialog-owns-button pattern propagation is a same-bug-class cluster, not scope creep.** Quality Gate #7 explicitly authorises bundling 3+ dialogs with identical defect in one PR. PR #1263 bundled 6 dialogs across 4 files. Don't fragment mechanical refactors into one-PR-per-file when the audit names them as a cluster.
- **The audit's "5-min eyeball check" estimate is for assessment, not migration.** Audit-03 named 4 files and called it "5 min each" — but each file required a 10-15 min refactor (migrate dialog API + update call sites + handle the inline AlertDialog extraction). The full PR took ~1.5 hours including review feedback. Don't trust audit time estimates for migration work; they're for reading.
- **The `source` field on `Comment` interface is a pre-existing TS warning** in `__tests__/comments/edit-comment-dialog.test.tsx` (`makeComment` helper missing `source: "INTERNAL" | "PORTAL"`). Vitest passes (transpilation only); `tsc --noEmit -p tsconfig.json` flags it. Not from any of this session's PRs. Possible cleanup target if a future PR is touching that test.
- **Don't trust gh pr merge's apparent success.** Same caveat as the morning handoff. Always `git fetch origin main && git log origin/main` to verify the merge SHA actually landed.
- **The user said "Claude is on thin ice."** Don't be cute, don't bypass, don't over-claim PASS. If the verify command's output is ambiguous, re-run; don't infer green from the absence of red.
