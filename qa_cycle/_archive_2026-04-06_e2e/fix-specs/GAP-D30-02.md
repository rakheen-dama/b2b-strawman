# Fix Spec: GAP-D30-02 — "Add Line" to invoice fails with backend parse error

## Problem
Adding a manual line item to an invoice fails with "The request body could not be read or parsed" backend error. This blocks disbursement lines and fixed-fee invoicing. Fork-readiness blocker.

## Root Cause (hypothesis — needs backend log confirmation)
Two potential issues between the frontend `AddLineItemRequest` and backend `AddLineItemRequest` record:

**Issue 1: Unknown `lineType` field in Jackson 3 strict records**
The frontend type (`frontend/lib/types/invoice.ts:114-123`) includes `lineType?: string` and `tariffItemId?: string`. The tariff dialog sends `lineType: "TARIFF"`. However, the backend `AddLineItemRequest` Java record (`backend/src/main/java/io/b2mash/b2b/b2bstrawman/invoice/dto/AddLineItemRequest.java`) does NOT have a `lineType` field:
```java
public record AddLineItemRequest(
    UUID projectId, String description,
    @NotNull @Positive BigDecimal quantity,
    @PositiveOrZero BigDecimal unitPrice,
    @PositiveOrZero int sortOrder,
    UUID taxRateId, UUID tariffItemId) {}
```

In Spring Boot 4 (Jackson 3 / `tools.jackson`), record deserialization may fail on unknown properties by default since records use canonical constructors.

**Issue 2: `taxRateId` sent as string "none"**
The frontend tax rate select uses `"none"` as the value for no-tax option. The `submitAddLine` function sends `taxRateId: newLineTaxRateId !== "none" ? newLineTaxRateId : null`. This correctly converts to null. However, `newLineTaxRateId` starts as `defaultTaxRate?.id ?? "none"`. If there is a default tax rate, it sends the tax rate ID as a string, but the backend expects `UUID`. String-to-UUID deserialization should work, so this is likely not the issue.

**Most likely cause**: The `lineType` field causes Jackson 3 record deserialization failure.

## Fix

### Backend fix (preferred — accept `lineType` in request)
Add `lineType` field to backend `AddLineItemRequest`:

**`backend/src/main/java/io/b2mash/b2b/b2bstrawman/invoice/dto/AddLineItemRequest.java`**:
```java
public record AddLineItemRequest(
    UUID projectId,
    String description,
    @NotNull @Positive BigDecimal quantity,
    @PositiveOrZero BigDecimal unitPrice,
    @PositiveOrZero int sortOrder,
    UUID taxRateId,
    UUID tariffItemId,
    String lineType) {}  // Add lineType
```

Then update `InvoiceService.addLineItem()` to use `request.lineType()` when setting the line type on the InvoiceLine entity (currently hardcoded to MANUAL for non-time lines).

### Alternative: Frontend fix (strip unknown fields)
Remove `lineType` from the request payload in both `submitAddLine()` and `TariffLineDialog.handleSubmit()`. This is less desirable because it loses semantic information.
