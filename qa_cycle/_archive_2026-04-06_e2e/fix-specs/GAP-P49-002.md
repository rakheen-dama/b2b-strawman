# Fix Spec: GAP-P49-002 — Template variable key mismatch (`company_registration_number` vs `acct_company_registration_number`)

## Problem
The `engagement-letter-bookkeeping` template content uses the variable key `customer.customFields.company_registration_number` (line 55 of the template JSON). However, the field pack `accounting-za-customer.json` defines this field with slug `acct_company_registration_number` (line 14). When a customer's registration number is populated via the CustomFieldSection UI, the value is stored under the key `acct_company_registration_number` in the customer's `customFields` JSONB column. The template variable lookup fails to find it and the `LenientOGNLEvaluator` returns `________` (the placeholder for unresolved expressions).

## Root Cause (confirmed)
The template content references a bare slug (`company_registration_number`) while the field pack uses an `acct_`-prefixed slug (`acct_company_registration_number`). This mismatch was likely introduced when the field pack adopted a namespaced slug convention but the template content was not updated to match.

Note: The compliance pack `sa-fica-company/pack.json` (line 71) also uses the bare key `company_registration_number` for its own field definition. This is a separate field definition in the compliance context and does NOT conflict with the fix below. The compliance pack's field is scoped to FICA checklists, not to the customer custom field group.

The seed script (`compose/seed/lifecycle-test.sh`, line 225) correctly uses the `acct_company_registration_number` key when populating customer custom fields, confirming the stored key matches the field pack, not the template.

## Fix
1. In `backend/src/main/resources/template-packs/accounting-za/engagement-letter-bookkeeping.json`, line 55:
   - Change `"key": "customer.customFields.company_registration_number"` to `"key": "customer.customFields.acct_company_registration_number"`

This is option (b) from the gap report -- updating the template to match the field pack slug. Option (a) (renaming the field pack slug) would require a migration to re-key existing custom field values in the JSONB column, which is more invasive and unnecessary.
