# Fix Spec: GAP-P49-016 — Clauses not rendered in document body

## Problem

All 3 engagement letter templates (bookkeeping, tax-return, advisory) allow clause selection in Step 1 of the generation dialog, but NO clause content appears in the rendered HTML preview. The document body jumps from "Client Responsibilities" / "Acceptance" to the footer with no clause sections. Engagement letters are legally incomplete without clause content.

## Root Cause (confirmed)

The template content JSON files have **no `clauseBlock` nodes**. The `TiptapRenderer` correctly handles `clauseBlock` nodes (lines 133-164 of `TiptapRenderer.java`), the frontend correctly passes clause selections to the backend, and `ClauseResolver` correctly resolves them. But the rendered `clauseMap` is never referenced because the template JSON simply does not contain any `{ "type": "clauseBlock", "attrs": { "clauseId": "...", "slug": "..." } }` nodes.

Searched all 7 templates in `backend/src/main/resources/template-packs/accounting-za/` -- zero `clauseBlock` occurrences.

The clauses are resolved, passed to `TiptapRenderer.render()`, stored as snapshots on the `GeneratedDocument`, and even audited -- but never actually injected into the output because the template JSON has no insertion points for them.

## Fix

Add `clauseBlock` nodes to the 3 engagement letter template JSON files, positioned between "Acceptance" and the closing signature. Each `clauseBlock` node references a clause by `slug` (not by ID, since IDs are tenant-specific and generated at seeding time).

However, there is a complication: `clauseBlock` nodes in the current `TiptapRenderer` require a `clauseId` attribute (a UUID), not just a slug. Clause IDs are tenant-specific -- they're generated when the clause pack is seeded into each tenant schema. So the template JSON files (which are pack resources, not tenant-specific) **cannot hardcode clause UUIDs**.

### Proposed approach (two parts):

**Part A — TiptapRenderer enhancement** (~30 min):
In `TiptapRenderer.renderNode()`, for the `clauseBlock` case, add a fallback: if `clauseId` is null but `slug` is present, look up the clause by slug from the `clauses` map. This requires the `clauses` map to be indexable by slug. Change the parameter type or add a secondary index:

```java
// In renderNode clauseBlock case, after clauseId lookup fails:
if (clause == null && slug != null) {
    clause = clauses.values().stream()
        .filter(c -> slug.equals(c.getSlug()))
        .findFirst()
        .orElse(null);
}
```

**Part B — Template JSON updates** (~30 min):
Add `clauseBlock` nodes to each engagement letter template, after the "Acceptance" section and before the closing "Yours faithfully". Each node references clauses by slug only:

For `engagement-letter-bookkeeping.json`, insert between the "Acceptance" paragraph and "Yours faithfully":
```json
{ "type": "clauseBlock", "attrs": { "slug": "limitation-of-liability-accounting" } },
{ "type": "clauseBlock", "attrs": { "slug": "fee-escalation" } },
{ "type": "clauseBlock", "attrs": { "slug": "termination-accounting" } },
{ "type": "clauseBlock", "attrs": { "slug": "confidentiality-accounting" } },
{ "type": "clauseBlock", "attrs": { "slug": "document-retention-accounting" } },
{ "type": "clauseBlock", "attrs": { "slug": "third-party-reliance" } },
{ "type": "clauseBlock", "attrs": { "slug": "electronic-communication-consent" } }
```

Similar additions for the tax-return and advisory templates (with their respective clause slugs).

**Important**: Verify clause slugs match those defined in `backend/src/main/resources/clause-packs/accounting-za-clauses/pack.json`.
