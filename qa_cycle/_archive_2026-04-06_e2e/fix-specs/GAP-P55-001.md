# Fix Spec: GAP-P55-001 — Demo provisioning addMember fails in KC 26.5.0

## Problem
The `DemoProvisionService` calls `KeycloakAdminClient.addMember(orgId, userId)` which POSTs to
`/organizations/{orgId}/members` with body `{"id": userId}`. In Keycloak 26.5.0 this endpoint
returns 400 "User does not exist" for users that DO exist but were registered in the realm (not
already in the org). The KC 26 Organizations API changed: adding a user by ID requires the user
to already be "associated" via the invite flow.

## Root Cause (hypothesis)
- File: `backend/src/main/java/io/b2mash/b2b/b2bstrawman/security/keycloak/KeycloakAdminClient.java` line 106-115
- The `addMember()` method uses the direct membership endpoint `POST /organizations/{orgId}/members` with `{"id": userId}`.
- Keycloak 26.x Organizations API requires the `invite-existing-user` flow for users not already associated. The `inviteMember()` method (line 153) uses `POST /organizations/{orgId}/members/invite-user` which is the correct flow.

## Fix
This is a Keycloak API compatibility issue that affects only the demo provisioning path (not production onboarding which uses invitations).

**Option A (recommended, if invite-user works for existing users):**
1. In `DemoProvisionService.java` line 146, change `keycloakAdminClient.addMember(kcOrgId, userId)` to `keycloakAdminClient.inviteMember(kcOrgId, userEmail, role, null)`.
2. This requires passing the email to the demo provision flow instead of (or in addition to) the userId.

**Option B (fix the addMember method):**
1. In `KeycloakAdminClient.addMember()`, change the endpoint from `/organizations/{orgId}/members` to `/organizations/{orgId}/members/invite-existing-user` (if available in KC 26.5.0).
2. Or add a try-catch: attempt `addMember`, on 400 fall back to `inviteMember`.

## Triage
**WONT_FIX** for this bugfix cycle. The demo provisioning path was worked around during QA by using the invite flow manually. This requires Keycloak API research to determine the correct KC 26.5 endpoint, and may need changes to the demo seeding strategy. Not fixable in < 2 hours without risk.
