# Fix Spec: GAP-PE-001 + GAP-PE-008 — Portal branding uses "DocTeams" instead of firm name

## Problem
GAP-PE-001: The magic link email subject reads "Your portal access link from DocTeams" instead of "Your portal access link from Thornton & Associates". The `PortalEmailService` calls `emailContextBuilder.buildBaseContext()` which calls `resolveOrgName()`. That method looks up org name via `RequestScopes.getOrgIdOrNull()`, but the magic link request flows through the portal auth filter (not `TenantFilter`), so `ORG_ID` is never bound to the ScopedValue. Result: `getOrgIdOrNull()` returns null, and the fallback `"DocTeams"` is used.

GAP-PE-008: The portal root layout hardcodes `"Client Portal | DocTeams"` as the page title in the static metadata export. The branding API already returns the correct `orgName` but the layout does not use it because `metadata` is a static export (not dynamic).

## Root Cause (hypothesis)

**GAP-PE-001**: `EmailContextBuilder.resolveOrgName()` (line 80-89 of `backend/src/main/java/.../notification/template/EmailContextBuilder.java`) reads `RequestScopes.getOrgIdOrNull()`. During a portal magic link request, only `TENANT_ID` is bound (by `CustomerAuthFilter` or `PortalAuthFilter`), not `ORG_ID`. The `PortalEmailService.sendMagicLinkEmail()` is called from `MagicLinkService.generateToken()` which runs outside the request filter chain (the token persist is in a `TransactionTemplate`, then email is sent after commit). The tenant scope IS bound (line 68 of PortalEmailService checks it), but `ORG_ID` is not — so `resolveOrgName()` falls back to "DocTeams".

Alternative approach: The `PortalEmailService` already has access to the org information via the magic link auth flow. The `PortalAuthController.requestMagicLink()` resolves the org from the `orgId` request parameter. We can pass the org name down or resolve it directly in `PortalEmailService`.

**GAP-PE-008**: `portal/app/layout.tsx` lines 21-26 — static `metadata` export with hardcoded "DocTeams".

## Fix

### GAP-PE-001 — Email subject uses correct org name

**Option (simplest)**: Resolve org name directly in `PortalEmailService.sendMagicLinkEmail()` instead of relying on `emailContextBuilder.buildBaseContext()` for the org name. The service already has the tenant context bound. Add the org name resolution:

1. In `PortalEmailService.java`, inject `OrganizationRepository`.
2. After building the base context (line 81), override `orgName` by resolving from the organization table using the `orgId` derived from the portal contact's customer → org mapping. Since the service has `TENANT_ID` bound, it can look up the org via the `org_schema_mapping` table or by querying the global `organizations` table.
3. Simpler approach: Add an `orgId` parameter to `sendMagicLinkEmail()` and thread it from `PortalAuthController.requestMagicLink()` which already has the `orgId` from the request. Then use `organizationRepository.findByClerkOrgId(orgId)` to get the org name.

**Step-by-step:**
1. Add `OrganizationRepository` as a constructor parameter to `PortalEmailService`.
2. Add `String orgId` parameter to `sendMagicLinkEmail()`.
3. After line 85 (`String orgName = (String) context.get("orgName")`), add:
   ```java
   if ("DocTeams".equals(orgName) && orgId != null) {
       orgName = organizationRepository.findByClerkOrgId(orgId)
           .map(org -> org.getName())
           .orElse("DocTeams");
       context.put("orgName", orgName);
   }
   ```
4. Update the subject: `context.put("subject", "Your portal access link from " + orgName);`
5. Update callers: `MagicLinkService.generateToken()` → thread the orgId from the `PortalAuthController`.

### GAP-PE-008 — Portal page title uses firm name

1. In `portal/app/layout.tsx`, change the static `metadata` export to use a generic fallback:
   ```typescript
   export const metadata: Metadata = {
     title: {
       default: "Client Portal",
       template: "%s | Client Portal",
     },
   };
   ```
   The actual firm name is already shown in the header/footer via the branding hook. The browser tab just needs to not say "DocTeams".

2. Alternatively, for dynamic titles per-page, each page component can set `document.title` using the branding hook. But changing the static fallback to "Client Portal" is sufficient for this fix.
