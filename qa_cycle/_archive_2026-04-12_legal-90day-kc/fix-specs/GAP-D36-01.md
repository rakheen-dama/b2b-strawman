# Fix Spec: GAP-D36-01 — Null billing rate snapshot blocks invoicing for pre-existing time entries

## Problem

Time entries logged before billing rates were configured have null `billingRateSnapshot` and null `billingRateCurrency` values. The unbilled-items pipeline relies entirely on the snapshot for amount calculation and the currency column for filtering. As a result:

1. **Unbilled summary** (`getUnbilledSummary`) silently excludes null-currency entries via `AND te.billing_rate_currency = :currency`, so these entries don't appear in customer totals at all.
2. **Unbilled detail** (`getUnbilledTime`) returns null-rate entries but with `billableValue = null`. The frontend selection logic in `use-invoice-generation.ts:97` uses `entry.billingRateCurrency === currency` which evaluates to `false` for null, making checkboxes disabled.
3. **Billing run discovery** (`BillingRunSelectionService`) has the same `billing_rate_currency = :currency` filter, excluding null-rate entries from batch invoicing.

This blocks invoicing for all ~13.5h of pre-existing time entries across 3 matters in the demo.

## Root Cause

Three files, same pattern — null rate snapshot means no billable value, no currency match, and entry is invisible or unselectable:

- **`backend/.../invoice/UnbilledTimeService.java:78-86`** — `getUnbilledSummary()` SQL filters on `te.billing_rate_currency = :currency`, excluding null-currency rows.
- **`backend/.../invoice/UnbilledTimeService.java:203-214`** — `getUnbilledTime()` computes `billableValue = null` when `rate == null` (line 208-213), then frontend disables selection.
- **`backend/.../billingrun/BillingRunSelectionService.java:498,557`** — `discoverCustomers()` and `createEntrySelections()` SQL filters on `te.billing_rate_currency = :currency`, excluding null-currency rows.

The snapshot is set at log time in `RateSnapshotService.snapshotRates()` (line 35-36). If no rate exists at that moment, the snapshot stays null permanently. No fallback resolution happens at query time.

## Fix

**Strategy**: At query time in the unbilled-items fetcher, fall back to live rate card resolution for entries with null `billingRateSnapshot`. Do NOT backfill — that changes historical data.

### Step 1: Add `UnbilledTimeService` dependency on `BillingRateService`

File: `backend/.../invoice/UnbilledTimeService.java`

- Add `BillingRateService` and `TaskRepository` to the constructor.
- These are needed to resolve rates for null-snapshot entries.

### Step 2: Post-process null-rate entries in `getUnbilledTime()`

File: `backend/.../invoice/UnbilledTimeService.java`, method `getUnbilledTime()` (~line 198-237)

After the SQL query returns rows, for each entry where `rate == null`:

1. Look up the `projectId` via the task (already available from the SQL JOIN — add `t.id AS task_id` to the SELECT if needed, or use the existing project_id).
2. Call `billingRateService.resolveRate(memberId, projectId, entryDate)` to get the current rate.
3. If a rate is resolved, compute `billableValue` and set the currency from the resolved rate.
4. Mark the entry with a `rateSource` indicator (e.g., `"RESOLVED"` vs `"SNAPSHOT"`) so the UI can show a visual hint that this rate was resolved at query time, not snapshotted.

This requires extending the SQL SELECT to also return `te.member_id` (already present via JOIN on `m.id`) and ensuring `projectId` is available for each row (it already is).

### Step 3: Update `UnbilledTimeEntry` DTO to carry rate source

File: `backend/.../invoice/dto/UnbilledTimeEntry.java`

Add a `String rateSource` field to the record (nullable, values: `null`/`"SNAPSHOT"` for normal entries, `"RESOLVED"` for fallback-resolved entries). This is informational only — the frontend can optionally render a badge.

### Step 4: Fix `getUnbilledSummary()` SQL to include null-currency entries

File: `backend/.../invoice/UnbilledTimeService.java`, method `getUnbilledSummary()` (~line 60-106)

Change the time entries JOIN condition from:
```sql
AND te.billing_rate_currency = :currency
```
to:
```sql
AND (te.billing_rate_currency = :currency OR te.billing_rate_currency IS NULL)
```

For the SUM calculation, null snapshots will produce NULL which COALESCE handles. The resolved amounts will be added in a post-processing step (or the query can be restructured to compute amounts in Java after resolving rates).

**Simpler alternative for the summary**: Keep the SQL filter as-is for snapshotted entries, then run a second query for null-rate billable entries, resolve their rates in Java, and merge the totals. This avoids making the existing SQL more complex.

### Step 5: Fix `BillingRunSelectionService` SQL filters

File: `backend/.../billingrun/BillingRunSelectionService.java`

Apply the same pattern as Step 4 to:
- `discoverCustomers()` (~line 498): change `AND te.billing_rate_currency = :currency` to `AND (te.billing_rate_currency = :currency OR te.billing_rate_currency IS NULL)`.
- `createEntrySelections()` (~line 557): same change.

For billing run amount calculations in `recalculateItemTotals()` (~line 344-382), the code already uses `te.getBillableValue()` which returns null for null-rate entries. This needs to fall back to rate resolution as well (or the entries need their snapshots updated before totaling).

### Step 6: Frontend — handle `rateSource: "RESOLVED"` entries as selectable

File: `frontend/components/invoices/use-invoice-generation.ts`

- Line 97: Change `entry.billingRateCurrency === currency` to also accept entries where the backend resolved the rate (i.e., entries now have a non-null `billingRateCurrency` after the backend fix, so this should work automatically).
- Line 235: `nullRateEntries` filter (`e.billingRateSnapshot == null`) should now only match truly unresolvable entries (no rate card exists for that member/project at all).

File: `frontend/components/invoices/invoice-generation-dialog.tsx`

- Optionally show a subtle indicator (e.g., small "Rate resolved from rate card" tooltip) on entries where `rateSource === "RESOLVED"`.

### Step 7: Update `UnbilledTimeEntry` TypeScript type

File: `frontend/lib/types/invoice.ts` (or wherever `UnbilledTimeEntry` is defined)

Add optional `rateSource?: string` field.

## Scope

- **Backend**: 3 files modified (`UnbilledTimeService.java`, `BillingRunSelectionService.java`, `UnbilledTimeEntry.java`)
- **Frontend**: 3 files modified (`use-invoice-generation.ts`, `invoice-generation-dialog.tsx`, type definition)
- **Tests**: Add integration test in `UnbilledTimeIntegrationTest.java` — create time entry without rate, configure rate, verify unbilled response includes resolved rate
- **Migration**: None (no schema changes)

## Verification

1. Log a time entry with no billing rates configured → snapshot is null.
2. Configure billing rates for the member.
3. Fetch unbilled time for the customer → entry appears with resolved rate and correct billable value.
4. Entry checkbox is enabled and selectable in the invoice generation dialog.
5. Create a draft invoice including the resolved-rate entry → line item amount matches resolved rate.
6. Profitability page (GAP-D33-01) should now show revenue data for these entries.
7. Billing run preview includes the null-rate entries with resolved amounts.

## Effort

**M** (Medium) — 3-4 hours. Core change is ~50 lines of Java in `UnbilledTimeService` + SQL adjustments in `BillingRunSelectionService`. Frontend changes are minimal (type addition + optional UI hint). Main complexity is ensuring the rate resolution is efficient (batch resolve rather than N+1 queries).

- **GAP-D4-02** (MED): "Billing rate N/A" on time entry — original report of the same issue at entry time. This fix addresses the query-time impact; GAP-D4-02 is about the warning UX at log time (separate concern).
- **GAP-D36-02** (LOW): "$0.00" formatting for null-rate entries — once rates are resolved, the formatting issue disappears for resolvable entries.
