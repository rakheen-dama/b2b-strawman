# Fix Spec: GAP-D75-02 — Complete Matter returns 200 but silently fails

## Problem
The "Complete Matter" action on Sipho's litigation matter returns a 200 response but the project status remains "Active" after reload. Bob (Admin org role, Member project role on the project) attempted this twice. No console errors were observed. This is a MED severity gap as it blocks the Day 75 milestone demo scenario.

## Root Cause (confirmed)
The most likely cause is the **open tasks guardrail** in the backend.

File: `backend/src/main/java/io/b2mash/b2b/b2bstrawman/project/ProjectService.java`, lines 426-434.

```java
long openTaskCount = taskRepository.countByProjectIdAndStatusNotIn(
    projectId, List.of(TaskStatus.DONE, TaskStatus.CANCELLED));
if (openTaskCount > 0) {
    throw new InvalidStateException(
        "Cannot complete project",
        "Project has %d open task(s). Complete or cancel all tasks before completing the project."
            .formatted(openTaskCount));
}
```

Sipho's litigation matter was created from the Litigation template with 9 tasks. By Day 75, only 4 tasks were marked DONE (per Day 24-35 results), leaving 5 open tasks. The backend would throw `InvalidStateException` (HTTP 400).

**The "silent failure" behavior**: The frontend server action at `frontend/app/(app)/org/[slug]/projects/actions.ts`, line 160-168, catches the error and returns `{ success: false, error: message }`. The `CompleteProjectDialog` component (line 55-63) checks for the error message. IF the error message contains "unbilled", it shows the unbilled time confirmation flow and clears the error. Otherwise it shows the error text.

The QA reports "200 response" but this is likely the QA agent observing the server action's HTTP response (which IS 200 as a Next.js server action) rather than the backend API response (which would be 400). The server action wraps the error into a `{ success: false, error: ... }` response — always HTTP 200 from the Next.js perspective. The dialog SHOULD show the error message. The "silent" behavior suggests either:
1. The dialog closed unexpectedly (Radix dialog focus trap issue)
2. The error display was not visible (CSS issue)
3. The QA agent dismissed the dialog before noticing the error

**However**, the fix is still needed: the demo scenario expects to complete the matter at Day 75. The 5 remaining tasks need to be marked DONE or CANCELLED before completing.

## Fix
This is a **demo flow issue**, not a code bug. The backend correctly prevents completing a project with open tasks. Two approaches:

### Option A: Demo script fix (recommended)
Update the QA test plan to mark all remaining tasks as DONE or CANCELLED before attempting "Complete Matter" at Day 75. This is the correct workflow.

### Option B: Add explicit error messaging (defense in depth)
Improve the error display in the CompleteProjectDialog to make open-task blocking errors more visible:

File: `frontend/components/projects/complete-project-dialog.tsx`, line 55-63.

Currently the error is shown in a small `<p>` tag (line 113). Enhance it:
1. Add a pre-check that fetches open task count before showing the complete dialog
2. Show a clear warning with task count when there are open tasks
3. Optionally allow the user to see which tasks are still open

**Recommended**: Do both Option A (update the test plan to close tasks first) AND add a pre-check in the dialog that warns about open tasks.

### Minimal fix for the demo:
In the QA test plan, before Day 75 "Complete Matter":
1. Navigate to the matter's Action Items tab
2. Mark all remaining open tasks as DONE or CANCELLED
3. Then attempt "Complete Matter"
