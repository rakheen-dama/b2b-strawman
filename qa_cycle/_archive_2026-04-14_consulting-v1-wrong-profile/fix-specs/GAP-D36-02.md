# Fix Spec: GAP-D36-02 — Fetch Unbilled Time doesn't advance from step 1 to step 2

## Problem
In the invoice generation dialog, clicking "Fetch Unbilled Time" does not advance from step 1 to step 2. The QA report says the server action returns 200 but data doesn't populate. The dialog stays stuck on step 1 with no visible error message to the user. All Days 36-60 invoice/payment checkpoints are blocked by this.

## Root Cause (hypothesis)
This is a cascading failure from GAP-D36-01. The `fetchUnbilledTime` server action in `frontend/app/(app)/org/[slug]/customers/[id]/invoice-actions.ts` (lines 25-49) performs a capability gate BEFORE calling the backend:

```ts
const caps = await fetchMyCapabilities();
if (!caps.isAdmin && !caps.isOwner) {
    return { success: false, error: "Only admins and owners can view unbilled time." };
}
```

If `fetchMyCapabilities()` returns `{ isOwner: false, isAdmin: false }` due to empty capabilities (the same root cause as GAP-D36-01), then `fetchUnbilledTime` returns `{ success: false, error: "Only admins and owners..." }`.

In the hook `use-invoice-generation.ts` line 92-93:
```ts
if (result.success && result.data) {
    ...
    setStep(2);
} else {
    setError(result.error ?? "Failed to fetch unbilled time.");
}
```

The error IS set on the `else` branch (line 108), but the QA agent reports "no visible error message". This means either:

1. **The capabilities ARE resolving correctly** (isOwner=true), the backend API call succeeds (200), but the response data shape doesn't match `UnbilledTimeResponse` — so `result.data` is undefined/null, and the `result.success` check passes but `result.data` check fails, skipping `setStep(2)`. In this case, `result.error` is also undefined, so the error message defaults to "Failed to fetch unbilled time" — which the QA agent might not have noticed if it's displayed subtly.

2. **The capabilities resolution fails** and the error message IS displayed but the QA agent focused on the "no data populates" symptom rather than the error text.

**Most likely**: This is the same root cause as GAP-D36-01. If capabilities resolve to empty set, `isOwner` and `isAdmin` are both false, and the server action short-circuits before even calling the backend API. Fix GAP-D36-01 and this resolves automatically.

## Fix
1. **Fix GAP-D36-01 first** — this is the same root cause (capabilities not resolving).

2. **Improve error visibility in the dialog**: In `use-invoice-generation.ts` `handleFetchUnbilled`, ensure the error state is prominently displayed. Currently `setError()` is called but the error rendering in the dialog (line 656 of `create-customer-dialog.tsx` ... actually in `invoice-generation-dialog.tsx`) may not be visible in step 1. Check that the error paragraph renders when `step === 1`.

3. **Remove redundant capability gate in server action**: The backend endpoint `CustomerController.getUnbilledTime` already has `@RequiresCapability("CUSTOMER_MANAGEMENT")`. The frontend server action in `invoice-actions.ts` performs a SECOND capability check (`!caps.isAdmin && !caps.isOwner`) which duplicates the backend guard and introduces a point of failure. Remove the frontend capability check and let the backend handle authorization. This simplifies the flow and eliminates the data-not-populating failure mode.

   In `frontend/app/(app)/org/[slug]/customers/[id]/invoice-actions.ts`, change `fetchUnbilledTime`:
   ```ts
   export async function fetchUnbilledTime(customerId: string, from?: string, to?: string): Promise<UnbilledTimeResult> {
     try {
       const params = new URLSearchParams();
       if (from) params.set("from", from);
       if (to) params.set("to", to);
       const qs = params.toString();
       const url = `/api/customers/${customerId}/unbilled-time${qs ? `?${qs}` : ""}`;
       const data = await api.get<UnbilledTimeResponse>(url);
       return { success: true, data };
     } catch (error) {
       if (error instanceof ApiError) {
         return { success: false, error: error.message };
       }
       return { success: false, error: "Failed to fetch unbilled time." };
     }
   }
   ```
   Similarly, remove the capability gate in `createInvoiceDraft` (lines 82-85) — the backend `InvoiceController` already has `@RequiresCapability("INVOICING")`.
