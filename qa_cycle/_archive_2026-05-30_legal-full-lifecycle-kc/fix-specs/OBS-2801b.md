# OBS-2801b Fix Spec — Activate Customer button still no-op (checklist gate)

## Summary

The OBS-2801 fix (PR #1399) created `ClientHeaderCardWithLifecycle` to wire `TransitionConfirmDialog`, but the button remains a no-op because `lifecycleActionPrompt` in page.tsx requires `checklistProgress.total > 0`, which fails for Sipho (no onboarding checklist assigned).

## Gap Details

- **ID**: OBS-2801b
- **Severity**: HIGH (BLOCKER — same cascade as OBS-2801)
- **Day**: 28
- **Symptom**: "Activate Customer" button visible but click does nothing. No dialog, no network request, no error.
- **Root cause**: `targetLifecycleStatus` is `null` because `lifecycleActionPrompt` is `null`.

## Root Cause (Detailed)

### Code Flow

1. `getSmartPrimaryAction()` in `client-header-card.tsx` (line 38): Returns `{ label: "Activate Customer" }` for ANY `ONBOARDING` lifecycle status. **No checklist gate.** Button always renders.

2. `lifecycleActionPrompt` in `page.tsx` (lines 453-467): Computes the target status ONLY when:
   ```
   customer.lifecycleStatus === "ONBOARDING" &&
   customerReadiness?.checklistProgress?.completed === customerReadiness?.checklistProgress?.total &&
   (customerReadiness?.checklistProgress?.total ?? 0) > 0
   ```
   The `total > 0` condition fails for Sipho because no onboarding checklist was assigned.

3. `targetLifecycleStatus={lifecycleActionPrompt?.targetStatus ?? null}` (line 519): Passes `null`.

4. `ClientHeaderCardWithLifecycle` (line 73): `onPrimaryAction={targetLifecycleStatus ? handlePrimaryAction : undefined}` passes `undefined`.

5. `ClientHeaderCard` (line 163): `onClick={onPrimaryAction}` — `undefined`, so click does nothing.

### Why Sipho has no checklist

Sipho was onboarded on Day 2 without a formal onboarding checklist being assigned. The KYC/FICA compliance adapter is not configured (per mandate). The "Client Readiness" card shows "Setup complete" but `checklistProgress` is `null` or has `total = 0`.

## Fix Options (Product decision required)

### Option A: Relax the checklist gate (RECOMMENDED)

In `page.tsx`, change the ONBOARDING -> ACTIVE prompt condition to allow activation when no checklist is assigned:

```typescript
// Current (too restrictive):
customer.lifecycleStatus === "ONBOARDING" &&
  customerReadiness?.checklistProgress?.completed ===
    customerReadiness?.checklistProgress?.total &&
  (customerReadiness?.checklistProgress?.total ?? 0) > 0

// Proposed (relax for no-checklist case):
customer.lifecycleStatus === "ONBOARDING" &&
  (customerReadiness?.checklistProgress === null ||
    customerReadiness?.checklistProgress === undefined ||
    (customerReadiness.checklistProgress.completed ===
      customerReadiness.checklistProgress.total &&
      customerReadiness.checklistProgress.total > 0))
```

This means: if there's no checklist at all, activation is allowed. If there IS a checklist, it must be complete.

**Effort**: XS (1 condition change in 1 file)

### Option B: Add LifecycleTransitionDropdown to page

Import and render `LifecycleTransitionDropdown` on the customer detail page, next to or instead of the smart action button. This provides the "Change Status" dropdown that the cycle 16 scenario mentions as "the canonical path."

**Effort**: S (add import + render in 1 file, handle RSC boundary)

### Option C: Align button visibility with prompt

Make `getSmartPrimaryAction()` not return "Activate Customer" when `targetLifecycleStatus` is null. This hides the button when the transition isn't wired, but means no visible activation path exists.

**Effort**: XS but changes UX (button disappears)
