# OBS-5004: Trust field group auto-attached to non-trust clients

## Summary

The "SA Accounting -- Trust Details" field group is auto-applied to ALL new customers regardless of entity type. When a PTY_LTD or CC client is created, both the "SA Accounting -- Client Details" group and the "SA Accounting -- Trust Details" group are attached. The trust group contains 3 required fields (Trust Registration Number, Trust Deed Date, Trust Type) that are irrelevant for non-trust entities, inflating the "Required Fields" count.

## Root Cause

The trust field pack (`field-packs/accounting-za-customer-trust.json`) has `"autoApply": true` on its group definition. During customer creation, `CustomerService.createCustomer()` calls `fieldGroupService.resolveAutoApplyGroupIds(EntityType.CUSTOMER)` which returns ALL active CUSTOMER field groups where `auto_apply = true`. There is no filtering by the customer's `entityType` value.

The individual field definitions inside the trust group DO have `visibilityCondition` entries that gate display on `acct_entity_type == "TRUST"`. The frontend correctly evaluates these conditions -- when entity type is not TRUST, the fields are hidden and the group section itself returns `null` (line 63 of `intake-fields-section.tsx`). **However**, the backend completeness scorer (`CustomerReadinessService.batchComputeCompleteness()`) does NOT evaluate visibility conditions -- it counts all required fields from all active field definitions regardless of conditions. This causes the "Required Fields: 2/5" display to inflate the denominator with trust-specific required fields for non-trust clients.

Additionally, the trust field group appears in the FIELD GROUPS section of the client detail page even though none of its fields are visible, requiring manual removal via the X button.

## Two-layered problem

1. **Group-level auto-apply is entity-type blind.** The `FieldGroup` entity has no concept of "only apply to customers with entityType = X". The existing `applicableWorkTypes` mechanism (GAP-L-37) is PROJECT-only and enforced at the entity-type guard level.
2. **Completeness scoring ignores visibility conditions.** Even if the group is auto-applied, the required field count should only include fields whose visibility conditions are met.

## Fix Options

### Option A (Recommended): Add `applicableEntityValues` to FieldGroup (structural fix)

Add a nullable `applicable_entity_values` JSONB column to `field_groups` table. When non-null, the auto-apply logic in `FieldGroupService.resolveAutoApplyGroupIds()` checks the customer's `entityType` against this list before including the group. This mirrors the `applicableWorkTypes` pattern for PROJECT groups.

**Changes:**
1. **Migration**: `V{next}__add_applicable_entity_values_to_field_groups.sql` -- adds `applicable_entity_values jsonb` column
2. **FieldGroup entity**: add `applicableEntityValues` field (`List<String>`)
3. **FieldPackGroup record**: add `applicableEntityValues` field
4. **FieldPackSeeder**: persist `applicableEntityValues` from pack JSON
5. **FieldGroupService.resolveAutoApplyGroupIds(EntityType)**: add overload that accepts `String customerEntityType`, filter groups whose `applicableEntityValues` is non-null/non-empty and does not contain the value
6. **CustomerService.createCustomer()**: pass `entityType` to the new overload
7. **accounting-za-customer-trust.json**: add `"applicableEntityValues": ["TRUST"]` to the group
8. **CustomerReadinessService.batchComputeCompleteness()**: filter out field definitions whose `visibilityCondition` evaluates to false for the customer (defense in depth)

**Effort**: M (1-2 hours). Touches migration, entity, seeder, two services. Safe -- null/empty list means "all values" (backward compatible).

### Option B (Minimal): Set autoApply to false on trust group

Change `accounting-za-customer-trust.json` to `"autoApply": false`. Trust clients would need the group manually applied. This is the cheapest fix but degrades the trust client UX -- the scenario requires trust-specific fields to appear automatically for TRUST entity type (Day 15).

**Not recommended** because it breaks the TRUST client scenario.

### Option C (Minimal frontend): Fix completeness scoring + hide empty groups on detail page

Only fix the symptoms: make completeness scoring evaluate visibility conditions, and hide empty field groups on the detail page. The trust group is still applied but its fields are correctly hidden and not counted. The group remains in `appliedFieldGroups` but is invisible.

**Partial fix** -- the group lingering in `appliedFieldGroups` is messy.

## Recommendation

**Option A** is the clean fix. It extends the existing `applicableWorkTypes` pattern to customer entity values, is backward compatible (null = all), and solves both the auto-apply and completeness problems at the source. Also fix completeness scoring as defense-in-depth.
