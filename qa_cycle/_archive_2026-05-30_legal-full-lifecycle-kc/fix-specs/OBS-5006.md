# Fix Spec: OBS-5006 — Invoice PDF preview fails

## Problem

On the approved invoice INV-0001 detail page, two preview mechanisms fail:

1. **Preview button** (`handlePreview`): Opens `/api/invoices/{id}/preview` in a new tab via `window.open`. The new tab shows "Unauthorized" (HTTP 401). The backend preview endpoint works (Thymeleaf HTML rendering) but the frontend API route handler at `frontend/app/api/invoices/[id]/preview/route.ts` fails to authenticate.

2. **Generate Document > SA Tax Invoice**: Opens a dialog with "Failed to generate preview" error (red text). The Download PDF and Save to Documents buttons are present, suggesting the PDF generation backend works but the in-browser preview rendering is broken.

Evidence: Day 38 checkpoint 38.1 in `qa_cycle/checkpoint-results/day-36-60.md`.

## Root Cause (confirmed)

### Issue 1: Preview route returns "Unauthorized" in new tab

The API route handler at `frontend/app/api/invoices/[id]/preview/route.ts` calls `fetchMyCapabilities()` which internally calls the gateway API to check the user's role/capabilities. In Keycloak BFF mode, this requires the SESSION cookie to be forwarded to the gateway.

When `window.open('/api/invoices/{id}/preview', '_blank')` opens the preview in a new browser tab, the browser sends a regular GET request to the Next.js server. The route handler reads cookies via `next/headers` `cookies()` store. However, the `fetchMyCapabilities()` function (defined in `lib/api/capabilities.ts`) uses `api.get()` which calls `getAuthFetchOptions("GET")` — this reads the SESSION cookie from the Next.js cookie store and forwards it to the gateway.

The issue is that `fetchMyCapabilities()` may fail because:
- The gateway session has expired or the SESSION cookie value has rotated since the last full page load
- The route handler's cookie reading context differs from the page's cookie context

When `fetchMyCapabilities()` throws with status 401 or 403, the route handler returns `NextResponse("Unauthorized", { status: 401 })` (line 23).

**Root fix:** The preview route handler should proxy the backend's HTML preview response using the same auth mechanism as other API calls, without requiring an extra capabilities check. The capabilities check is redundant here — the backend already has `@RequiresCapability("INVOICING")` on the preview endpoint. If the user's session is valid, the backend will serve the HTML; if not, it will return 401/403 which the route handler can forward.

### Issue 2: Generate Document preview fails

The "Generate Document" flow uses a different code path — it renders a document template preview via the template engine. The "Failed to generate preview" error likely comes from the template rendering service failing to find or render the template in-browser. This is a secondary issue that shares the same auth/session root cause. The template preview likely makes a fetch call to the backend that fails for the same auth reasons.

## Fix

### Part A: Simplify preview route handler — remove redundant capabilities check

In `frontend/app/api/invoices/[id]/preview/route.ts`, remove the `fetchMyCapabilities()` check and instead directly proxy the backend preview endpoint. The backend's `@RequiresCapability("INVOICING")` annotation handles authorization.

**Current code (simplified):**
```typescript
export async function GET(_request: Request, { params }) {
  // 1. fetchMyCapabilities() — can fail with 401, returns "Unauthorized"
  // 2. Check isAdmin/isOwner — returns "Forbidden" if not
  // 3. getAuthFetchOptions("GET") — read SESSION cookie
  // 4. fetch(backend/preview) — proxy the request
}
```

**Fixed code:**
```typescript
export async function GET(_request: Request, { params }) {
  const { id } = await params;
  if (!UUID_RE.test(id)) {
    return new NextResponse("Bad Request", { status: 400 });
  }

  let authOptions;
  try {
    authOptions = await getAuthFetchOptions("GET");
  } catch {
    return new NextResponse("Unauthorized", { status: 401 });
  }

  const response = await fetch(`${API_BASE}/api/invoices/${id}/preview`, {
    headers: authOptions.headers,
    credentials: authOptions.credentials,
  });

  if (!response.ok) {
    const text = await response.text().catch(() => response.statusText);
    return new NextResponse(text, { status: response.status });
  }

  const html = await response.text();
  return new NextResponse(html, {
    status: 200,
    headers: { "Content-Type": "text/html" },
  });
}
```

Key changes:
1. **Remove `fetchMyCapabilities()` call** — the backend already enforces `@RequiresCapability("INVOICING")`. The double-check is what fails because it makes an additional API call that may not carry the session correctly.
2. **Forward the backend's error response** — if the backend returns 401/403/404, forward it instead of swallowing it.

### Part B: Template preview (secondary — investigate separately if Part A doesn't resolve)

The "Generate Document" preview failure in the dialog may be a separate issue in the template rendering pipeline. If Part A resolves the preview button issue, this dialog preview should be investigated as a follow-on. The dialog may need to fetch the preview via a server action rather than a direct API call.
