# Fix Spec: OBS-503 — Portal-submitted documents stuck PENDING; firm-side Download fails

## Problem
All 3 FICA documents Sipho uploaded through the portal on Day 4 are stuck at `documents.status = PENDING` even though the S3 objects exist (verified via LocalStack `aws s3 ls`, 615–627 B each). When Bob clicks Download on the firm request detail page, the server action returns `{"success":false,"error":"Document has not been uploaded yet"}` for every item and no file downloads. Day 5 summary checkpoint "Three uploaded documents retrievable firm-side" FAILED. Evidence: `qa_cycle/checkpoint-results/day-05.md` checkpoint 5.3.

## Root Cause (confirmed)
The portal upload flow never finalizes the Document lifecycle:

1. Portal frontend (`portal/app/(authenticated)/requests/[id]/page.tsx`, `handleUpload` ~L132–170) does: `POST /portal/requests/{id}/items/{itemId}/upload` (init) → presigned S3 PUT → `POST /portal/requests/{id}/items/{itemId}/submit`. There is no confirm step and no portal-facing confirm endpoint exists.
2. `backend/src/main/java/io/b2mash/b2b/b2bstrawman/customerbackend/service/PortalInformationRequestService.java`:
   - `initiateUpload` (L86–180) creates the `Document` — constructor sets `status = PENDING` (`Document.java` L74/L99).
   - `submitItem` (L201–228) marks the RequestItem SUBMITTED but **never calls `document.confirmUpload()`** — the loaded `document` (L217–220) is only used for scope verification.
3. Firm-side `DocumentService.getPresignedDownloadUrl` (`document/DocumentService.java` L488–491) gates on `Status.UPLOADED` and throws `InvalidStateException("Document not uploaded", "Document has not been uploaded yet")` → the exact error QA captured.

The firm flow doesn't hit this because the firm UI calls `POST /api/documents/{id}/confirm` (`DocumentService.confirmUpload`, L373) after the S3 PUT.

### Secondary claim NOT confirmed
QA reported the failure is "fully silent — no toast". Code says otherwise: `frontend/components/information-requests/request-detail-client.tsx` `ItemDocumentDownloadButton.handleDownload` (L476–481) calls `toast.error(result.error ?? "Download failed.")` on `success:false`, and `<Toaster>` is mounted in `frontend/app/layout.tsx` L50. Dev must observe toast behaviour during repro; if the toast genuinely does not render, report it back as a separate finding — do NOT bundle a speculative frontend change into this PR.

## Fix
Backend-only, in `PortalInformationRequestService.submitItem` (the file-submission path only — `submitTextResponse` has no document):

1. After `verifyDocumentScope(document, request)` (L220) and alongside `item.submit(documentId)`, finalize the document:
   ```java
   if (document.getStatus() != Document.Status.UPLOADED) {
     document.confirmUpload();
     documentRepository.save(document);
   }
   ```
   The S3 PUT has already happened by the time the portal calls submit (frontend only submits after `put.ok`), and the firm-side `DocumentService.confirmUpload` performs no S3 existence check either — parity is intentional.
2. Do **NOT** publish `DocumentUploadedEvent` (would double-notify the firm — `RequestItemSubmittedEvent` already does that, L226) and do **NOT** publish `DocumentCreatedEvent` (would newly project these docs into the portal documents read model — a behaviour change outside this gap; prior cycle completed ALL_DAYS without it). Do **NOT** emit a `document.uploaded` audit event — the activity trail already has `portal.document.upload_initiated` + `portal.request_item.submitted` and Day 5 verified the trail is complete. Status flip only.

### Existing stuck rows (REQ-0001's 3 documents)
Per cycle mandate, data is disposable and backward compat is not a priority — no migration/backfill. The three Day 4 rows stay PENDING. QA re-verifies via a fresh round-trip (see Verification). Days 7+ and 46/60 use newly uploaded portal documents, which will confirm correctly post-fix.
