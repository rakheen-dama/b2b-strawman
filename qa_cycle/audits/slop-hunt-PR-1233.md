# Slop hunt — PR #1233: fix(OBS-703): dispatch portal email on ProposalSentEvent

**Batch**: A — notifications
**Reviewed**: 2026-05-01
**Verdict**: NEEDS-FOLLOW-UP

## PR description vs diff

Description is honest about scope: 3 files (handler, service wrapper, test). Diff matches. The author states this honors ADR-258 (PortalEmailNotificationChannel intentionally excludes proposal events) — verified: `PortalEmailNotificationChannel` does not handle `ProposalSentEvent`, and the new handler lives in `proposal/`. The test does drive the full pipeline (`POST /api/proposals/{id}/send` → AFTER_COMMIT listener → GreenMail), not just the body builder. Good shape.

The test plan claims `./mvnw test -Dtest=ProposalSendTest,ProposalSentEmailHandlerTest,ProposalPortalSync*Test` was run as the "no regressions" check. **This is exactly the targeted-test pattern the OBS-2102 → OBS-2108 cascade taught us is unsafe.** A new `@Component` listener subscribed to `ProposalSentEvent` could fire on any test that publishes that event class-wide, but only the targeted tests were run. Per the new gate (CLAUDE.md §5), full `./mvnw verify` is mandatory before merge for any production-behaviour change. This PR did not run full verify per its own description.

## Test scope check

- PR description **does not claim full `./mvnw verify`** — only targeted tests. Per the new merge-gate, this should not have shipped without the full run.
- The new test does drive the listener via the public `POST /api/proposals/{id}/send` endpoint (good — it exercises the actual listener firing, not just the email-builder). One test, one happy path.
- No negative-path test (what happens when GreenMail is down, when portalContactId is null, when the proposal has no expiresAt — the third was implicit in the test fixture but not asserted).

## Notes — listener-registration cross-reference

This is the canonical "new portal listener" PR in batch A. Other PRs in the batch should be evaluated against its pattern. Compared to it:

- **Tenant scoping**: uses copy-pasted `handleInTenantScope`. PortalEmailNotificationChannel uses its own copy with stricter null-handling (warn + skip). ProposalPortalSyncEventHandler uses another copy. **Drift confirmed.**
- **Dedup**: zero. PortalDocumentNotificationHandler has Caffeine 5-min dedup. EmailDeliveryLog records-after-the-fact, no pre-send check anywhere. **Drift confirmed.**
- **Phase**: AFTER_COMMIT (correct, matches PortalDocumentNotificationHandler + PortalEmailNotificationChannel + ProposalPortalSyncEventHandler). Consistent.
- **Error containment**: try/catch in the listener body, log.error, don't rethrow. Matches PortalDocumentNotificationHandler's onDocumentGenerated. Consistent.
- **Subject prefixing**: hard-codes `"%s: New proposal %s for your review"` template inline. Other handlers (PortalDocumentNotificationHandler, PortalEmailNotificationChannel) build subject in the Thymeleaf template via context keys. **Inconsistent.** The subject template should live in the .html template, not in the Java handler.
