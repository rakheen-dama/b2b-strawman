## Test scope check

- Targeted scope cited: `./mvnw test -Dtest='*StructuralPrerequisite*'` (23/23). **Did NOT cover** the broken test, which lives in package `compliance/` and is named `CustomerLifecyclePrerequisiteTest` ‚Äî does not match the glob.
- Surface area of the change: `StructuralPrerequisiteCheck.check(Customer, PrerequisiteContext, boolean)` ‚Äî a static utility imported by **at least 11 test classes** across `compliance/`, `prerequisite/`, `fielddefinition/`, `customer/`, `assistant/`, etc.
- Was the broken test in a different package? **Yes** ‚Äî `compliance/`, while the changed class is in `prerequisite/`. The glob `'*StructuralPrerequisite*'` was scoped to the changed-class package only.
- Was the regression preventable from the diff alone? **Yes**. Two signals were visible in the diff itself:
  1. `StructuralPrerequisiteCheckTest.lifecycleActivation_mirrorsInvoiceGenerationFields` was edited to switch from factory default (INDIVIDUAL) to COMPANY ‚Äî a clear flag that the factory default no longer satisfies the activation prerequisite. A careful reviewer should have asked "what other tests use the factory default and assert ONBOARDING after checklist completion?" ‚Äî that's a 30-second `grep -r "TestCustomerFactory.createCustomer\b" backend/src/test`.
  2. The Javadoc explicitly says "factory default is INDIVIDUAL, which now skips tax_number at activation" ‚Äî the dev agent **knew** the factory change risk; they fixed one test and missed the sibling.
- Per Quality Gate rule #5 ("A fix that changes production behaviour MUST run the full backend `./mvnw verify` before claiming green. The OBS-2102 ‚Üí OBS-2108 cascade happened because the dev only ran tests in the same package; the broken test was elsewhere."), the failure mode here is the reference example.
