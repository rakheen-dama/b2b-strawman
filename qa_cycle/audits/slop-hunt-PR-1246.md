# Slop hunt — PR #1246: fix(OBS-2106): closure-pack portal email dispatch

**Batch**: A — notifications
**Reviewed**: 2026-05-01
**Verdict**: NEEDS-FOLLOW-UP

## PR description vs diff

Description claims a 3-part fix: (1) DEBUG→INFO log uplift in `PortalDocumentNotificationHandler`, (2) follow-up `DocumentGeneratedEvent` with explicit `visibility=PORTAL` in `MatterClosureService.generateClosureLetterSafely`, (3) new `MatterClosureEmailIntegrationTest`. Diff matches exactly. The fix-spec was committed to the repo (`qa_cycle/fix-specs/OBS-2106.md`) — good discipline.

The author is honest about the race they're working around: "the AFTER_COMMIT listener observed the linked Document still at INTERNAL". They publish a second event with explicit details rather than fix the timing. This is pragmatic but doesn't eliminate the underlying race — it relies on the dedup cache to coalesce. If the dedup cache is empty (cold tenant, listener restart) AND the timing is unfortunate, you could still get one INTERNAL-rejection then one PORTAL-success, for a net of one email — the desired outcome — but the reasoning is fragile.

## Notes — listener-registration cross-reference

This PR doesn't add a listener — it modifies the trigger side (event publication). Cross-references:

- **Event publication pattern**: this PR adds a SECOND emitter for the same logical event. The codebase now has two emitters for closure-letter-portal-visible: the canonical one in `GeneratedDocumentService` and the compensating one in `MatterClosureService`. This is the kind of "two-paths-must-stay-in-sync" smell that gets re-broken in the next refactor.
- **Tenant context resolution**: uses `RequestScopes.requireTenantId()` / `requireOrgId()` (the `require*` variants throw on absence). Compare with PR #1233's `ProposalSentEmailHandler.handleInTenantScope` which silently skips on null. Different conventions in the same codebase.
- **Dedup mechanism**: relies on `PortalDocumentNotificationHandler`'s Caffeine dedup. PR #1233's `ProposalSentEmailHandler` has no dedup at all. **Confirmed listener-registration drift across batch A.**
- **Idempotency story**: PR #1246's "send twice → cache coalesces to one" model is explicitly documented. PR #1233's "send once → no dedup, retry would double-send" is undocumented. Need to pick one.
