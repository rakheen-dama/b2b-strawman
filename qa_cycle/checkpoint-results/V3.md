## The failure (UI)

```
AI response could not be parsed as valid matter intake output:

Array, Object or token 'null', 'true' or 'false')
at [Source: REDACTED ...; byte offset: #UNKNOWN]
```

The backtick (code 96) in the error is the leading ```` ` ```` of a ```` ```json ```` markdown fence — i.e. **Claude did respond** (live round-trip happened) but returned its JSON wrapped in a fenced code block, and the backend fed the raw fenced text to Jackson `ObjectMapper.readValue`, which rejects the leading backtick.

## STUB-vs-LIVE determination
- This was **NOT the stub.** `NoOpAiProvider` returns empty output → an empty/placeholder result, never a fenced-JSON parse error. The parse error can only arise from **real model output** containing a markdown fence. Provider slug is `anthropic`, key present, enabled (verified in V1).
- However, the live-Claude proof obligation (non-empty output + `costCents>0` + token usage) **cannot be satisfied as recorded**, because the failure rolls back persistence (see root cause). The live call is proven by the model's response text in the parse error and by the ~50–90 s latency; the **cost is silently lost** (incurred at Anthropic, never metered) — itself a defect.

## Root cause (pinned)
`backend/.../integration/ai/skill/intake/MatterIntakeSkill.java:220`
```java
output = objectMapper.readValue(outputContent, MatterIntakeOutput.class);  // raw, fenced text → JacksonException
```
No fence-stripping / JSON-extraction before deserialisation. Claude Sonnet 4.6 wraps structured output in a ```` ```json … ``` ```` fence by default.

**Transaction-rollback amplifier** — `AiSkillExecutionService.executeSkill` is `@Transactional` (line 82). Order:
1. save execution IN_PROGRESS (l.99) → 2. live LLM call (l.127, **succeeds**) → 3. `calculateCostCents` + markCompleted + **save** (l.153–155) → 4. `createGates(...)` (l.159) **throws** the parse `InvalidStateException`.
The `try/catch` at l.137–150 wraps **only** the LLM call, NOT `createGates`. So the parse exception propagates out of the `@Transactional` method and **rolls back the IN_PROGRESS + COMPLETED execution saves**. Result: `ai_executions`, `ai_llm_calls`, `ai_specialist_invocations`, `ai_execution_gates` are **all 0 rows** — the failure isn't even recorded as a FAILED execution, and the (real) cost is never metered.
