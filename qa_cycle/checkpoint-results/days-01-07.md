## GAP-C-04 watch

After 9.0h of time logged across 3 entries and 2 contributors, the TeamUtilizationWidget on `/org/zolani-creative/dashboard` **still shows "Unable to load utilization data."**. The server action returning 500 for the utilization endpoint does NOT self-heal with data.

**Updated GAP-C-04 diagnosis**: this is NOT a fresh-tenant empty-state bug. Even with time data present, the endpoint 500s. Likely root cause is the billable-rate requirement — the widget logic probably multiplies hours × billable rate and the rate-lookup is failing (because members have no rate assigned at the org level — see GAP-C-05). Widget should either (a) degrade gracefully to "hours only" when rates are missing, or (b) render a zero-state with an actionable "Assign billing rates to unlock utilization" CTA. Recommend severity upgrade from MED to HIGH-ish (still not a blocker for Day 1–7 but it breaks the Day 75 wow moment).

## Noteworthy Signals

- **Dashboard console errors**: each dashboard render logs one `500 Internal Server Error` for the utilization server action. Happens for all users (Bob, Carol). Confirms GAP-C-04 is a real server-side 500, not a UI copy issue.
- **Rate card warning in Log Time dialog**: "No rate card found for this combination. This time entry will have no billable rate. Set up rates in Project Settings > Rates." — appears on every log-time attempt. Cascade of GAP-C-05 (rate-pack UI has no Role column, so no rate is ever bound to a member) and GAP-C-06 (empty cost_rates). Time entries are saved but carry no billable amount; budget "Amount Consumed" stays at R0 even after 9h of billable time. Will break Day 30 Monthly Retainer Report (`retainer.hoursUsed` may work, but rate-based metrics won't), Day 34 Profitability (revenue / margin = 0), Day 45 unbilled-time rule (may still fire since it's age-based, not rate-based), and Day 60 SOW (clauses not rate-dependent). Day 36 first-invoice is the first HARD gate this cascades into — generating an invoice from 0-rate time entries will produce a zero-value invoice.
- **Template metadata not applied** (GAP-C-09 scope): affects custom-field defaults, budget defaults, and (at the customer level) conditional visibility. Suggests the template-application service reads the template rows but only applies the shallow fields (name, description, task list, project-lead placeholder) and skips the deep custom-field + budget metadata. Worth a focused investigation before Day 9+ (when the retainer templates' `retainer_tier=GROWTH/STARTER` auto-fill and retainer-period budget defaults are expected — all of which will likely also fail under this gap).
- **Carol's project-access block** was the only in-flow friction. Resolved by Bob adding Carol as project member. This is probably correct RBAC ("Members can only see projects they're on"), but the scenario step 2.7 ("Assign tasks across the team per template's `assigneeRole` defaults") should possibly imply auto-seeding the project's member list with everyone whose role matches an `assigneeRole` in the template. Flagging as ambiguity, not a gap.
- **Keycloak SSO session** does not clear via the in-app "Log Out" button — see GAP-C-10. QA workaround: admin-API user-logout + browser close.
