**Big ADR findings — 4 of 5 ADRs partially or not implemented**:
- **ADR-247** (disbursement-as-sibling-entity): backend uses generic `expenses` table with legal categories (FILING_FEE, TRAVEL, COURIER) — arguably the sibling entity — but the matter's **Disbursements tab has NO "Add/Log" button**, so the flow is inaccessible from UI. GAP-L-06 (MED).
- **ADR-248** (matter-closure-as-distinct-state-with-gates): `projects.status` enum is only `ACTIVE`/`COMPLETED`/`ARCHIVED` — **no distinct "CLOSED" state**. Completing a matter performs a one-way transition with an open-tasks gate (9 open tasks blocked Sipho's completion), but no trust-balance gate, no unbilled-time gate. GAP-L-07 (MED).
- **ADR-249** (retention-clock-starts-on-closure): `projects` table has no `retention_clock_started_at` column. `completed_at` is set but it's not described as a retention trigger. GAP-L-08 (MED, depends on L-07).
- **ADR-250** (statement-of-account template): **template category enum `chk_dt_category` has no STATEMENT_OF_ACCOUNT value**; fee note PDF uses the generic invoice template ("Invoice: INV-0002", no letterhead, no brand color, no trust balance, no WIP breakdown, no banking details). The legal-za template pack ships Client Trust Statement (report entity_type=CUSTOMER) but not a fee-note-specific Statement of Account. GAP-L-09 (MED).
- **ADR-251** (acceptance-eligible template manifest flag): no `acceptance_eligible` column on `document_templates`; no visual flag in Settings > Templates list. "Send for Acceptance" CTA appears on engagement-letter-generation result (observed Day 4), which suggests the flag is implied by category=ENGAGEMENT_LETTER rather than a per-template manifest. `acceptance_requests` table exists. GAP-L-10 (LOW).

**Also surfaced**:
- **1 HIGH blocker on Day 50** (RAF fee note): lifecycle guard rejects fee-note creation for Lerato (ONBOARDING status). This is correct guard behavior, but an explicit ACTIVE-only gate on billing was not foreshadowed by the scenario's "allow matter creation in ONBOARDING" (OBS-L-10 from days 1-30). Not filed as a gap — existing guard is legitimate. See below "Day 50".
- Trust payment dual control: correctly blocks same-member approval. Assigning APPROVE_TRUST_PAYMENT capability to a second admin requires setup that the `legal-za` vertical profile doesn't do automatically. Neither Bob (Admin) nor the other members have the capability — so a recorded payment is stuck AWAITING_APPROVAL without an operational setup step. OBS-L-13.

## ADR-Specific Findings

### ADR-247 — Disbursement as sibling entity (legal-disbursement-sibling-entity)
- **Finding**: Backend has `expenses` table with category enum including FILING_FEE, TRAVEL, COURIER, SOFTWARE, SUBCONTRACTOR, PRINTING, COMMUNICATION, OTHER. This is structurally a sibling to `time_entries` (both reference project_id, member_id, billable, invoice_id). **So backend data model satisfies ADR-247.**
- **UI gap**: Matter's Disbursements tab renders empty state "No expenses logged / Log disbursements like filing fees, travel, and courier costs" with **no Add / Log Disbursement button anywhere** on the page. Users cannot create a disbursement via UI. GAP-L-06 (MED).
- **Screenshot**: `days-31-60-adr247-disbursements-empty-no-add-button.png`

### ADR-248 — Matter closure as distinct state with gates
- **Finding**: `projects.status` enum is only `ACTIVE / COMPLETED / ARCHIVED`. **NO "CLOSED" state** distinct from COMPLETED. The "Complete Matter" button is a one-way ratchet.
- **Positive**: There IS one gate — "Project has 9 open task(s). Complete or cancel all tasks before completing the project." Clicking Complete Matter on Sipho's matter surfaces this alert and blocks the transition. ✓
- **Gaps**: No gate for (a) unbilled time, (b) non-zero trust balance, (c) pending fee notes, (d) open court dates. GAP-L-07 (MED).

### ADR-249 — Retention clock starts on closure
- **Finding**: `projects` table has `completed_at timestamp with time zone, completed_by uuid, archived_at, archived_by`. **No `retention_clock_started_at` or `retention_period_days` column**. No visible UI for retention.
- **Assessment**: If `completed_at` is intended to be the retention anchor, that's implicit. No explicit "retention" surface. GAP-L-08 (MED, depends on L-07).
- **`retention_policies` table exists** separately — I did not have time to inspect whether it's linked to project closure. Deferred to next cycle.

### ADR-250 — Statement of Account template
- **Finding**: 
  - `chk_dt_category` enum values are: ENGAGEMENT_LETTER, STATEMENT_OF_WORK, COVER_LETTER, PROJECT_SUMMARY, NDA, PROPOSAL, REPORT, COMPLIANCE, OTHER. **No STATEMENT_OF_ACCOUNT value.**
  - The template list shows: Engagement Letters (4), Cover Letters (2 — Invoice Cover + Section 35), Compliance (PAIA), Other (PoA, Notice of Motion, Founding Affidavit, Letter of Demand, Trust Receipt), Project Summary, Report (Client Trust Statement — customer-scoped).
  - No "Statement of Account" in the Document Templates list.
  - Fee note PDF uses a **generic invoice template** (text-only header, no logo, no brand color, no trust balance, no WIP breakdown, no disbursements section, no banking details, no VAT registration, no firm address).
- **Gap**: GAP-L-09 (MED). Implementing ADR-250 would require (a) adding `STATEMENT_OF_ACCOUNT` to the category enum, (b) shipping a template in the legal-za pack, (c) switching the fee-note render endpoint to use this template when category matches INVOICE entity type.

### ADR-251 — Acceptance-eligible template manifest flag
- **Finding**: 
  - `document_templates` table has **no `acceptance_eligible` column** and no equivalent boolean.
  - Settings > Templates list shows columns NAME / FORMAT / ENTITY TYPE / SOURCE / STATUS — no "Acceptance Eligible" column/badge.
  - However, `acceptance_requests` table exists (generated_document_id, portal_contact_id, status PENDING/SENT/VIEWED/accepted/revoked/expired, acceptance token, certificate_s3_key). So the backend can send for acceptance.
  - Prior Days 1-30 run captured that Engagement Letter generation showed a "Send for Acceptance" CTA (Day 4.3 — `engagement-letter-litigation-sipho-dlamini-...`). This suggests the "acceptance-eligible" gate is **implicit by category** (category == ENGAGEMENT_LETTER) rather than a per-template flag.
- **Gap**: GAP-L-10 (LOW). ADR-251 requires an explicit manifest flag. The implicit-by-category approach couples eligibility to template category, which is simpler but less flexible (e.g., a "Statement of Account" would need its own category to be acceptance-eligible).

## Observations (non-gap)

- **OBS-L-13** (LOW, Product): `legal-za` vertical profile doesn't seed a member with APPROVE_TRUST_PAYMENT capability other than the Owner. When the Owner records a trust payment, no other member can approve (verified: Bob Admin → "Insufficient permissions for this operation"). Real firms would assign this capability at setup. Suggestion: either grant APPROVE_TRUST_PAYMENT to Admin role by default in legal-za profile, or add an onboarding step "Assign Trust Approver".
- **OBS-L-14** (LOW, Dev): Fee note details panel labels the tax row as "(%)" rather than "VAT 15%" or the tax name. Easy fix in the totals component.
- **OBS-L-15** (LOW, Product): Approve → Sent is 2 clicks (Approve, then Send Fee Note), not 1. Scenario expects single-step. Current design is better (separates approval from delivery). Suggest updating scenario rather than UI.
- **OBS-L-16** (LOW, Product): Record Payment form has no Method picker (EFT / Credit Card / Cheque) — "Manual" provider is hard-coded. Real firms track payment method. Suggest a Method dropdown.
- **OBS-L-17** (LOW, Dev): Breadcrumb for fee note detail page shows "fee notes" (lowercase) — should be "Fee Notes" or "Invoices" with consistent case. Cosmetic.
- **OBS-L-18** (LOW, QA-infra): Radix Tabs panel activation is finicky under Playwright MCP — `.click()` on `[role="tab"]` sometimes doesn't switch data-state="active". Worked around with dispatching pointerdown+pointerup+click events, or setting URL query directly. Already noted in prior days.
- **OBS-L-19** (LOW, Product): Moroka Family Trust has a dummy email `info@moroka-trust.co.za` on the fee note "BILL TO" line. Thandi never entered this — it was auto-synthesized (likely derived from customer name). Works but could be surprising.
- **OBS-L-20** (LOW, Dev): PDF preview opens in new browser tab with URL `/api/invoices/:id/preview` (HTML, not PDF binary). Page title is "Invoice INV-0002", response type is HTML. Works for preview but if user hits Ctrl+P / Save As they get HTML file, not PDF. Consider adding "Download PDF" button that actually generates PDF binary.
