# Fix Spec: OBS-1001 — Record Deposit form requires raw UUID paste for Client and Matter

## Problem

The Record Deposit modal (Trust Accounting → Record Transaction → Record
Deposit) has Client and Matter fields rendered as raw text inputs with
placeholders `Client UUID` and `Matter UUID (optional)`. There is no
typeahead picker, no dropdown of clients, no matter list. A real user
who clicks **Record Transaction → Record Deposit** has no way to populate
these fields — clients do not know their internal UUIDs, and even firm
staff would need to leave the dialog, navigate to `/org/{slug}/customers`,
copy a UUID from the URL bar, return, and paste. This is an unusable
form.

QA evidence (Day 10 ck 10.4):

> "The Record Deposit form requires raw UUIDs for Client and Matter
> (placeholders read `Client UUID` / `Matter UUID (optional)`), not picker
> dropdowns. Functional — QA filled by pasting the known UUIDs from the
> status tracker — but real users will not have UUIDs to hand."

QA only succeeded because the agent had Sipho's customer UUID
(`a30bb16b-…`) and the matter UUID (`b7e319f7-…`) recorded in
`qa_cycle/status.md` from earlier days. A real lawyer would not.

This is a real bug under the cycle mandate. KYC + Payments are the only
exempt open gaps; this is a Trust Accounting UX defect on a daily-use
form (per LSSA Section 86 firms, every trust deposit goes through this
dialog).

## Root Cause (verified)

**File: `frontend/components/trust/RecordDepositDialog.tsx`**

- **Line 104-116** — `customerId` field is rendered as a raw `<Input
  placeholder="Client UUID" {...field} />`. No combobox, no Popover, no
  customer list passed in as a prop. The component has no awareness of
  the org's customer roster.
- **Line 118-130** — `projectId` field is rendered as a raw `<Input
  placeholder="Matter UUID (optional)" {...field} />`. Same gap.
- **Line 29-35** — `RecordDepositDialogProps` interface only takes
  `accountId`, `slug`, `open`, `onOpenChange`, `onSuccess` — there is no
  `customers` or `defaultCustomerId` prop and no plumbing for a server-
  side customer list.

**Invocation surfaces (3 entry points to the same dialog):**

1. `frontend/components/trust/transaction-actions.tsx:50` — Trust
   Accounting transactions page (org-level — no customer context known
   ahead of time).
2. `frontend/components/trust/TrustBalanceCard.tsx:180` — invoked from
   matter detail Trust tab (`projects/[id]/page.tsx:928` — both
   `customerId` and `projectId` are already known but **not passed** to
   the dialog).
3. `frontend/components/trust/TrustBalanceCard.tsx` — also rendered on
   client detail surface (verified by `customerId` prop on the card
   itself); same component instance, same gap.

**Sibling dialogs with the same defect** (file with the issue:line):

- `frontend/components/trust/RecordPaymentDialog.tsx:122-128` — same raw
  `Matter UUID (optional)` input.
- `frontend/components/trust/RecordRefundDialog.tsx:112-118` — same.

(Out-of-scope for this spec — fixing those sibling dialogs is recommended
follow-up but OBS-1001 was filed only on Record Deposit. List included
so the dev knows the pattern recurs.)

## Canonical pattern in this codebase

`frontend/components/proposals/create-proposal-dialog.tsx:245-323`
already implements the correct pattern for a client picker:

- Server passes a `customers: Array<{ id: string; name: string; email:
  string }>` prop (server-fetched once, no per-keystroke API call).
- Inside the dialog: `Popover` + `Command` + `CommandInput` (search) +
  `CommandList` + `CommandItem` per customer. Clicking sets
  `field.onChange(customer.id)`. The visible label is `customer.name`,
  the form value is `customer.id`.
- Supports a `defaultCustomerId` prop that locks the picker (used for
  matter-level CTAs where the customer is fixed).

This is a **client-side combobox over a pre-fetched list**, not a remote
typeahead. Existing imports already in the codebase:
`@/components/ui/popover`, `@/components/ui/command`,
`lucide-react/{ChevronsUpDown, Check}`, `@/lib/utils/cn`.

For matters: there is no single "list all matters in the org" frontend
helper today, but `GET /api/customers/{customerId}/projects` returns
`Project[]` (used in `frontend/app/(app)/org/[slug]/customers/[id]/actions.ts:28`
as `fetchCustomerProjects`). The Matter picker should be a **dependent
combobox** — it's empty/disabled until a Client is picked, then loads
that client's matters via SWR.

## Fix

**Three coordinated changes — all frontend, no backend, no migration.**

### Step 1 — Pass `customers` into the dialog at every invocation

Edit `frontend/components/trust/RecordDepositDialog.tsx` to extend
`RecordDepositDialogProps`:

```ts
interface RecordDepositDialogProps {
  accountId: string;
  slug: string;
  open: boolean;
  onOpenChange: (open: boolean) => void;
  onSuccess?: () => void;
  /** Pre-fetched customer list for the picker. Server component should
   *  fetch via GET /api/customers and pass through. */
  customers: Array<{ id: string; name: string; email: string }>;
  /** When provided, pre-selects + locks the customer picker. Used by
   *  matter-level / client-level CTAs where the customer is known. */
  defaultCustomerId?: string;
  /** When provided, pre-selects the matter picker. */
  defaultProjectId?: string;
}
```

### Step 2 — Replace both raw inputs with comboboxes

Inside `RecordDepositDialog.tsx`, replace lines 104-130:

**Customer field (lines 104-116)** — copy the Popover + Command pattern
from `create-proposal-dialog.tsx:245-323` verbatim, adapting:
- `name="customerId"` (already correct in the schema).
- Disable the trigger button when `defaultCustomerId` is set.
- Show selected `customer.name` (not UUID) in the trigger.
- `CommandItem.value` = `${customer.name} ${customer.email}` so search
  matches name OR email.
- Empty state: `<CommandEmpty>No clients found.</CommandEmpty>`.

**Matter field (lines 118-130)** — dependent combobox:
- `name="projectId"`.
- When no customer selected: render the trigger button disabled with
  text `"Select a client first"`.
- When customer selected: fetch matters via SWR using
  `fetchCustomerProjects(customerId)` from
  `frontend/app/(app)/org/[slug]/customers/[id]/actions.ts:24-29`.
  - Cache key: `conditionalKey(customerId, "matters-for-customer-${customerId}")`.
  - SWR pattern already in this codebase — see `frontend/lib/swr/fetcher.ts:40`.
- When user changes customer, reset `projectId` to `""` via
  `form.setValue("projectId", "", { shouldValidate: false })` in a
  `useEffect` watching `customerId`.
- `CommandItem.value` = `${matter.name} ${matter.referenceNumber ?? ""}`
  so search matches name OR matter reference.
- Display label: `matter.name` (the `referenceNumber` like `RAF-2026-001`
  is on `Project.referenceNumber` per
  `frontend/lib/types/project.ts:10`; show `referenceNumber — name`
  formatted in the CommandItem if present).
- Keep the field optional ("Matter (Optional)" label retained).
- Use the same `defaultProjectId` lock pattern as customer.

### Step 3 — Wire customer list into the three invocation surfaces

**3a. `frontend/components/trust/transaction-actions.tsx`** (org-level
trust transactions page):

This is currently a `"use client"` component. Two options:
- **Preferred**: convert to a server component wrapper that fetches
  customers, and pass `customers` down as a serializable prop. The
  `DropdownMenu`/dialog children stay `"use client"` but receive the
  list. Pattern: see `frontend/components/proposals/create-proposal-dialog.tsx`
  invocation in `frontend/app/(app)/org/[slug]/proposals/page.tsx`
  (server fetches customers, passes array).
- **Alt**: keep `"use client"` and use SWR with cache key
  `customers-${slug}` to fetch on first dropdown open. Either is
  acceptable; preferred is server-fetched for consistency with the
  proposals dialog.

The transactions page `frontend/app/(app)/org/[slug]/trust-accounting/transactions/page.tsx:183`
is already a server component (`async function ... await params`) — it
should fetch customers (via `api.get<Customer[]>("/api/customers")`,
mirroring `frontend/app/(app)/org/[slug]/customers/page.tsx:136`) and
pass the trimmed `{ id, name, email }[]` to `<TransactionActions
customers={...} ... />` which then forwards to each child dialog.

**3b. `frontend/components/trust/TrustBalanceCard.tsx`** (matter Trust
tab + client surface):

Already has `customerId` as a prop (line 26) and `projectId` (line 28).
- Pass `defaultCustomerId={customerId}` and (when present)
  `defaultProjectId={projectId}` down to `<RecordDepositDialog>` (line
  180), `<RecordPaymentDialog>` (line 187), and (eventually)
  `<RecordRefundDialog>`.
- The customers list still needs to be supplied — fetch via SWR on the
  card itself (cache key `customers-${slug}`) since this is already a
  client component, OR pull just the single customer the card is for
  and pass `customers={[{id: customerId, name: ledger.customerName,
  email: ""}]}` since the picker will be locked anyway. The latter
  avoids a redundant network call when the locked-picker path is the
  only one used here.

**3c. matter-level Record Deposit invocation
(`projects/[id]/page.tsx:927-934`)** — already passes `customerId` and
`projectId` to `<TrustBalanceCard>`, so 3b covers it transitively. No
direct edit needed here.

## Notes for the Dev

- Do NOT change the API contract — the form still POSTs `customerId` +
  `projectId` (UUIDs) on submit. Only the UI binding changes.
- The `recordDepositSchema` in `frontend/lib/schemas/trust.ts` should
  not need changes — `customerId` is already a UUID-shaped string and
  `projectId` is optional.
- Use `Popover modal={false}` (matches the proposals pattern) — Radix
  Popover-inside-Dialog needs this to avoid pointer-events lockout.
- The Matter combobox SWR fetcher (`fetchCustomerProjects`) is already
  a server action in `customers/[id]/actions.ts` — import it directly
  per the SWR-wraps-server-action pattern in
  `frontend/CLAUDE.md` ("Client Data Fetching (SWR)" section).
- If the dev adds the same picker to RecordPaymentDialog +
  RecordRefundDialog in the same PR (recommended — they share the
  same defect), call it out in the PR description so QA can verify
  all three at once. Otherwise file a follow-up obs.
