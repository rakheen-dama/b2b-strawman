## Why hard-coded ZA conventions

The straightforward `NumberFormat.getCurrencyInstance(Locale.of("en", "ZA"))`
returned `$50,000.00` under Maven Surefire on the dev box (the JDK's
locale-data provider does not ship full en-ZA on every distribution and
falls back to the default currency). Standalone JVM (`java FmtTest`)
returned `R50 000,00` correctly. To get a stable client-facing render
across environments, the formatter now builds the symbol +
`DecimalFormatSymbols` explicitly per ISO code instead of trusting
locale data. `VariableFormatter` has the same JDK-locale dependency
which is fine inside the firm-side variable substitution surface but
not appropriate for a portal-facing nudge.
