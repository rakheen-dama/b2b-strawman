# OBS-2104 — Bulk Billing wizard excludes Sipho even after activation: legal_disbursements not in expenses table + time entries with NULL currency

**Status**: SPEC_READY
**Severity**: HIGH (blocks Day 28 fee-note generation; cascade from OBS-2101 + LegalDisbursement modeling)
**Filed**: 2026-04-30, cycle 15 (during OBS-2102 verification, post-activation)
**Reporter**: QA agent
**Component**: `backend/src/main/java/io/b2mash/b2b/b2bstrawman/billingrun/BillingRunSelectionService.java` (`discoverCustomers` SQL, lines 465-527)

## Symptoms

After OBS-2102 fix merged and Sipho activated successfully (status=ACTIVE, lifecycleStatus=ACTIVE):
1. Bulk Billing wizard, Period 2026-04-01 to 2026-05-31 → step 2 "Select Customers": **"No customers with unbilled work found for this period."**
2. Sipho has on RAF-2026-001:
   - 4h billable time entries logged by Bob (2h 30m + 1h 30m on Day 21)
   - 1 sheriff disbursement R 1 250,00 (recoverable, billable, status=DRAFT approval / UNBILLED billing)
3. Backend logs confirm: `BillingRunSelectionService.Loaded preview for billing run abca60e3-... — 0 customers, total unbilled: 0`

## Root cause analysis

### Issue 1 — Time entries have NULL `billing_rate_currency` (OBS-2101 cascade)

Source: `RateSnapshotService.snapshotRates(...)` at `timeentry/RateSnapshotService.java:36`:
```java
billingRate.ifPresent(r -> entry.snapshotBillingRate(r.hourlyRate(), r.currency()));
```

If no rate card is found for the (member, project, date) tuple, `billingRate` is empty → `snapshotBillingRate` is never called → `billing_rate_currency` stays NULL on the entry.

`BillingRunSelectionService.discoverCustomers()` query line 498 filters:
```sql
LEFT JOIN time_entries te ON te.task_id = t.id
    AND te.billable = true
    AND te.invoice_id IS NULL
    AND te.billing_rate_currency = :currency  -- ← excludes NULL
    AND te.date >= :periodFrom
    AND te.date <= :periodTo
```

Equality comparison `= :currency` excludes NULL rows. Sipho's 4h time entries (saved with no rate card per the yellow warning banner) are dropped.

### Issue 2 — Legal disbursements live in a separate table

`LegalDisbursement` entity (`verticals/legal/disbursement/LegalDisbursement.java`) is mapped to `legal_disbursements` table. The Bulk Billing query joins `expenses e ON e.project_id = p.id` (line 501). Sipho's R 1,250 sheriff fee is in `legal_disbursements`, NOT `expenses`, so it is **never** considered by `discoverCustomers`.

Compounding effect: Sipho has 0 candidate `time_entries` (rate-currency NULL) AND 0 candidate `expenses` (disbursement is in a different table). The `HAVING` clause `COUNT(DISTINCT te.id) > 0 OR COUNT(DISTINCT e.id) > 0` evaluates false → Sipho excluded from `customer_discoveries`.

### Issue 3 — Approval gate (potential)

Even if `legal_disbursements` were joined, the disbursement is in `DRAFT` approval status (per UI: "Approval: Draft"). The SQL would need to gate on `approval_status = 'APPROVED'` to be sound, but that's a design decision, not a defect by itself.

## Suggested fix paths (for dev)

### Path A — relax NULL-currency on time entries (small)

Treat NULL `billing_rate_currency` as "matches the run currency" (i.e., the entry will render at R 0,00 on the fee note line). Change line 498 to:
```sql
AND (te.billing_rate_currency = :currency OR te.billing_rate_currency IS NULL)
```

Pair with a fee-note line renderer that shows R 0,00 with a "(no rate configured)" inline note. Acceptable for the current OBS-2101 WONT_FIX cascade.

### Path B — UNION legal_disbursements into the SQL (medium)

Extend `discoverCustomers` to LEFT JOIN both `expenses` AND `legal_disbursements`. Sum amounts from both. Filter `legal_disbursements` by `recoverable = true`, `invoice_id IS NULL`, `currency = :currency`, optionally `approval_status = 'APPROVED'`.

```sql
LEFT JOIN legal_disbursements ld ON ld.project_id = p.id
    AND ld.recoverable = true
    AND ld.invoice_id IS NULL
    AND ld.currency = :currency
    AND ld.date >= :periodFrom
    AND ld.date <= :periodTo
    AND ld.approval_status = 'APPROVED'
```

Update `unbilled_expense_count` and `unbilled_expense_amount` to sum across both `e` and `ld`.

### Path C — push disbursement approval before billing (consider)

Currently the disbursement is created in DRAFT. If Path B requires APPROVED, the QA scenario must include an approval step (Day 21.8b: submit + approve disbursement). That's a scenario amendment.
