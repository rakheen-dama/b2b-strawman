## Approach

Path A from spec (CTE rewrite). Path B (scalar correlated subqueries) was rejected — the planner has more freedom with the CTE shape and the structure parallels the existing `createEntrySelections` per-table queries.

The pre-fix flat query LEFT-JOINed five tables (`tasks`, `time_entries`, `expenses`, `legal_disbursements`) hanging off `projects p` and `GROUP BY c.id, c.name`. For a matter with N tasks and 1 disbursement, the disbursement row appeared N times in the join product, so `SUM(ld.amount + ld.vat_amount)` returned `tasks_count × actual`. `COUNT(DISTINCT)` masked the bug for the count columns.

The new query has three CTEs that each aggregate one source table per-customer in isolation, then a final `SELECT` that LEFT-JOINs each CTE onto a `customer_filter` CTE (which carries the `lifecycle_status='ACTIVE'` and optional `customerIds IN (...)` filter). With one row per customer per source, no multiplication is possible.

The existing 69 `*BillingRun*` tests continue to pass — semantics for the simple 1-task / 1-time-entry case are unchanged because each CTE returns the same per-customer aggregate the flat query produced when no row multiplication was occurring.

## Invariants Preserved

- **PR #1238 NULL-currency tolerance** — `time_agg` CTE keeps `(te.billing_rate_currency = :currency OR te.billing_rate_currency IS NULL)`. The OBS-2104 NULL-currency time-entry case continues to surface in step 2 with R 0,00 (the SUM via `COALESCE(billing_rate_snapshot, 0)` correctly reduces to zero).
- **Disbursement billing/approval lifecycle** — `disbursement_agg` CTE keeps `billing_status='UNBILLED' AND approval_status='APPROVED'`. Disbursements still must be approved before they appear in the wizard.
- **Disbursement amount formula** — `amount + COALESCE(vat_amount, 0)` (no markup column on `legal_disbursements`).
- **`customerIds` IN-filter** — moved into `customer_filter` CTE so it scopes the entire downstream join. The `:customerIds` parameter binding is set conditionally exactly as before.
- **HAVING-style filter** — replaced with `WHERE COALESCE(...count, 0) > 0 OR ...` against the joined CTEs in the final SELECT. Behaviour identical: customers with zero work in any source are excluded.

## Risks Considered

- **EXPLAIN at scale**: not run in this PR — the spec lists this as an acceptance criterion at scale (≤ 100 active customers). At test fixture scale the CTE pattern compiles cleanly. Recommend EXPLAIN ANALYZE on a populated dev database during QA verification (Day 28 retry) to confirm index scans on `time_entries.task_id`, `expenses.project_id`, `legal_disbursements.project_id` and an absence of sequential scans.
- **`customer_filter` join shape change**: the previous query joined `customers` first then driver-projected from there. The new query first builds `customer_filter` (just the customer list) then LEFT-JOINs three pre-aggregated CTEs onto it. Final `WHERE` filters out customers with no work in any source. Equivalent semantics; planner may choose different order.
- **Cherry-pick step 3 (OBS-2104c)**: independent issue — the missing Disbursements section in step 3 is unrelated to this fix and remains tracked separately.
