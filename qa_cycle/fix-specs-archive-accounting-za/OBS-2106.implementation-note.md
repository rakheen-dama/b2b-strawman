## What changed

### Part 1 — Diagnostic uplift (`PortalDocumentNotificationHandler.java`)
Promoted **7 skip-path log lines** from `DEBUG` → `INFO`:
1. `no OrgSettings for tenant=…`
2. `per-tenant allowlist empty (tenant=…)`
3. `template=… not in allowlist (tenant=…)`
4. `not portal-visible (template=…, doc=…)`
5. `missing orgId on event …`
6. `no portal contact resolved (project=…, template=…)`
7. `dedup hit (key=…, template=…)`

Added a new INFO log line at the top of `process(...)` so every entry into the listener is visible:
```
PortalDocumentNotificationHandler.process entered: tenant={}, template={}, project={}, generatedDoc={}
```

No gate semantics changed — only log levels.

### Part 2 — Structural fix (`MatterClosureService.java`)
- Constructor now accepts `MemberNameResolver` (added to the existing parameter list immediately before the `@Lazy MatterClosureService self` self-reference).
- New private helper `publishPortalReadyFollowUp(generatedDocId, projectId, actingMemberId, fileName)` publishes a `DocumentGeneratedEvent` with explicit `scope=PROJECT, visibility=PORTAL` in `details`, mirroring `StatementService.java:218-246`.
- Helper is invoked at the end of `generateClosureLetterSafely(...)` AFTER both PDF generation and the visibility flip succeed. The original canonical event from `GeneratedDocumentService.generateDocument` is untouched — both events fire, and the portal handler's 5-minute dedup keyed on `(tenant, customer, project)` coalesces them to a single email.
- Failure of the follow-up publish is swallowed (try/catch + WARN log) so it cannot affect the already-committed close.

### Part 3 — Test gap closed (`MatterClosureEmailIntegrationTest.java` — new)
Two end-to-end tests using `legal-za` tenant provisioning + GreenMail singleton on port 13025:
- `close_withClosureLetterAndSoa_dispatchesExactlyOnePortalEmail` — combined closure-pack happy path. Asserts `closureLetterDocumentId` AND `statementOfAccountDocumentId` both populated, and exactly ONE email lands in GreenMail (dedup invariant).
- `close_withClosureLetterOnly_dispatchesPortalEmail` — the Day 60 regression scenario. SoA suppressed, only the closure-letter generates, asserts the portal email still arrives.

Test pattern follows `PortalDocumentNotificationHandlerIntegrationTest` (GreenMail invariants + tenant ScopedValue) plus `MatterClosureServiceIntegrationTest` (the close-flow harness with `legal-za` provisioning, ACTIVE customer, and full capabilities for the override branch). Uses `greenMail.waitForIncomingEmail(5_000L, 1)` to wait for AFTER_COMMIT listener + SMTP delivery.
