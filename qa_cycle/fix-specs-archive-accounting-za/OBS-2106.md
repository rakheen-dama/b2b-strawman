# Fix Spec: OBS-2106 — closure-pack portal email never dispatched after matter closure

**Severity**: bug (HIGH — exit checkpoint E.13 silently broken)
**Surface**: Backend — `PortalDocumentNotificationHandler` + `MatterClosureService`
**Effort**: M (~4h)

## Problem
Day 60 / cycle 47 QA verified that closure execution generated both the closure-letter and SoA PDFs and persisted both with `visibility=PORTAL`, but **no portal-document-ready email landed in Mailpit**. The PortalContact existed and matched the customer; the per-tenant allowlist included `matter-closure-letter`. Yet the listener silently dropped the event.

The 4 gates inside `PortalDocumentNotificationHandler.process(...)` were all logged at DEBUG, so on-call had no visibility into which gate rejected the event without flipping the package logger and re-running the scenario.

## Root Cause (verified)
Two compounding issues:

1. **Diagnostic invisibility** — The 7 skip-path log lines + the implicit "process entered" trace were all at DEBUG. Production INFO logs showed nothing for the failed path.
2. **Race in the visibility fallback** — The canonical `GeneratedDocumentService.generateDocument` event publishes WITHOUT `scope`/`visibility` in `details`, expecting the listener to fall back to a DB read. But the visibility flip (`documentService.markSystemAutoShared`) happens AFTER the canonical event is queued, and although the AFTER_COMMIT listener runs only after the outer REQUIRES_NEW transaction commits, under load (or in the interleaving observed on Day 60) the listener observed the linked Document still at INTERNAL — and dropped the event silently.

The SoA path already publishes with explicit `scope=PROJECT, visibility=PORTAL` in `details` (StatementService.java:228-236), so SoA emails dispatch reliably. The closure-letter path does not.

## Fix (3 parts in one PR)

### Part 1 — Diagnostic uplift
Promote 7 skip-path log lines `DEBUG → INFO` in `PortalDocumentNotificationHandler.process(...)`, plus add a `process entered` INFO line at the top of the method. Future regressions are visible at standard log level — on-call sees exactly which gate rejected the event.

### Part 2 — Structural fix
After `MatterClosureService.generateClosureLetterSafely` finishes generating the letter AND completing the visibility flip, publish a follow-up `DocumentGeneratedEvent` with explicit `scope=PROJECT, visibility=PORTAL` in `details` — mirroring the SoA pattern in `StatementService`. Eliminates the timing-dependent DB fallback. The portal handler dedups on `(tenant, customer, project)` over a 5-minute window, so the duplicate event cannot produce a duplicate email — the first-arriving event wins.

### Part 3 — Test gap
New `MatterClosureEmailIntegrationTest` drives the full close flow end-to-end (close matter → both PDFs generated → portal email dispatched) and asserts GreenMail singleton receives the closure-pack email. Two tests:

- `close_withClosureLetterAndSoa_dispatchesExactlyOnePortalEmail` — combined-pack happy path + dedup invariant.
- `close_withClosureLetterOnly_dispatchesPortalEmail` — letter-only path (the exact Day 60 regression).

## Scope
- `backend/.../portal/PortalDocumentNotificationHandler.java` — log levels only, no gate-semantics changes.
- `backend/.../verticals/legal/closure/MatterClosureService.java` — adds `MemberNameResolver` dependency + new `publishPortalReadyFollowUp` helper invoked at the end of `generateClosureLetterSafely`.
- `backend/.../verticals/legal/closure/MatterClosureEmailIntegrationTest.java` — new file.
