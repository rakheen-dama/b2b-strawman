## Approach

Option A (V118 backfill migration) only. Option B (Java-side fallback in
`OrgSettings.getPortalNotificationDocTypes()`) deliberately not added —
reasoning documented in `OBS-2107.md`.

The migration is a single idempotent UPDATE against every tenant schema:

```sql
UPDATE org_settings
SET portal_notification_doc_types = '["matter-closure-letter", "statement-of-account"]'::jsonb
WHERE portal_notification_doc_types IS NULL
   OR portal_notification_doc_types = '[]'::jsonb;
```

Canonical default seed mirrors V117 verbatim. Any tenant with an
explicit non-empty allowlist (including a single-template list) is
preserved untouched.

## Risks / follow-ups

- The migration overwrites `'[]'::jsonb` rows. If a tenant has
  deliberately set `[]` to disable per-event sends entirely, this
  migration will re-enable them. We accept this trade-off because
  (a) no UI exposes this setting today, so an empty allowlist is
  always accidental, and (b) the existing weekly digest is the only
  alternative communication channel for closure events, and one-off
  sends are the desired default. If a tenant explicitly wants to opt
  out in future, the Settings UI work (not in scope here) would set the
  column to a sentinel like `["__disabled__"]` rather than `'[]'`.
- If the live column is found to be `NULL` (never written) on any
  tenant, the migration handles that case too (`IS NULL` branch).
