# OBS-4005: Activity event message shows literal "project" instead of engagement name

**Severity**: LOW (cosmetic)
**Decision**: SPEC_READY -- quick backend fix, < 30 min
**Day observed**: 12
**Scenario**: Thandi posts comment on Kgosi Holdings Year-End Pack engagement. Activity tab event reads `"commented on project 'project'"` instead of `"commented on project 'Kgosi Holdings -- FY2025/26 Year-End Pack'"`.

## Root Cause

Two call sites emit `comment.created` audit events without the project name in `entity_name`:

### 1. `CommentService.validateEntityBelongsToProject()` (line 388-389)

When `entityType` is `"PROJECT"`, the method returns the hardcoded literal `"project"`:

```java
if ("PROJECT".equals(entityType)) {
    return "project";  // BUG: should return the actual project name
}
```

This value is stored as `entity_name` in the audit event details (line 146), so the activity feed renders `"commented on project 'project'"`.

### 2. `PortalCommentService.createPortalComment()` (line 66-90)

The audit event details map omits `entity_name` entirely for portal comments. The `ActivityMessageFormatter.getParentName()` falls back to `getParentType()`, which lowercases `entity_type` to `"project"`.

### Formatter path (correct, no change needed)

`ActivityMessageFormatter.getParentName()` correctly reads `details.get("entity_name")` and falls back to `getParentType()` only when it's missing. The formatter is not the problem -- the emitters are.

## Fix

### File 1: `backend/src/main/java/io/b2mash/b2b/b2bstrawman/comment/CommentService.java`

**Change**: Inject `ProjectRepository` and look up the project name when `entityType` is `"PROJECT"`.

Replace `validateEntityBelongsToProject`:

```java
private String validateEntityBelongsToProject(String entityType, UUID entityId, UUID projectId) {
    if ("PROJECT".equals(entityType)) {
        // Look up project name for the audit trail / activity feed
        return projectRepository.findById(projectId)
                .map(p -> p.getName())
                .orElse("project");
    }
    // ... rest unchanged
}
```

Add `ProjectRepository` as a constructor parameter (alongside existing `TaskRepository`, `DocumentRepository`).

### File 2: `backend/src/main/java/io/b2mash/b2b/b2bstrawman/customerbackend/service/PortalCommentService.java`

**Change**: Add `entity_name` to the audit details map. The project name is already available by looking it up from the project ID (inject `ProjectRepository`), or pass it from the caller.

Add to the audit details map (alongside the existing `entity_type`, `entity_id` keys):
```java
"entity_name", projectName,
```

Where `projectName` is resolved from `projectRepository.findById(projectId).map(Project::getName).orElse("project")`.

### File 3: `backend/src/test/java/io/b2mash/b2b/b2bstrawman/activity/ActivityMessageFormatterTest.java`

**Change**: Add a regression test that verifies `comment.created` on a PROJECT entity with a proper `entity_name` renders the project name (not the literal "project").

```java
@Test
void commentCreatedOnProjectWithEntityNameProducesCorrectMessage() {
    var event = createEvent("comment.created", "comment", Map.of(
        "entity_type", "PROJECT",
        "entity_name", "Kgosi Holdings -- FY2025/26 Year-End Pack",
        "body", "@Bob Need FS draft by day 30"));
    var item = formatter.format(event, actorMap(), emptyPortalContactMap());
    assertThat(item.message()).isEqualTo(
        "Alice commented on project \"Kgosi Holdings -- FY2025/26 Year-End Pack\"");
    assertThat(item.message()).doesNotContain("'project'");
}
```

## Verification

1. Run `./mvnw verify` -- full green
2. Existing `commentCreatedWithEntityNameProducesCorrectMessage` test continues to pass (TASK entity)
3. New test for PROJECT entity passes
4. Browser retest: post a comment on a project-level entity, verify activity tab shows the real project name

## Notes

- The `comment.updated` and `comment.deleted` audit events in `CommentService` also omit `entity_name` in their details (only include `entity_type` + `entity_id`). The `comment.updated` path has the same bug for the activity feed fallback. However, the observation only mentions `comment.created`, so limit the fix scope to the `createComment` path + `PortalCommentService`. The delete/update paths can be addressed separately if observed.
- The formatter's `getParentType()` correctly lowercases the entity type, which will display "project" for accounting verticals. This is acceptable since the terminology mapping ("project" -> "engagement") is a frontend concern, not a backend one.
