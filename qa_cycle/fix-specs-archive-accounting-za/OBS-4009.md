## Problem

The "Start Onboarding" lifecycle transition via the Change Status dropdown on the customer detail page does not execute. When a user clicks "Start Onboarding" from the dropdown, the TransitionConfirmDialog either never opens or opens and immediately closes, so `handleConfirm()` is never called. No backend API call is issued, and the customer remains in PROSPECT status. Observed on Mathole Engineering (ID: `29b90b29-9a51-4e73-9157-b2d3622ed29b`) across 3 separate attempts, all identical. Backend log has no transition entry. The same flow worked for 3 other clients (Sipho, Kgosi, Moroka) on earlier days.

## Root Cause

**Radix DropdownMenu + AlertDialog dismissal race.** The `LifecycleTransitionDropdown` component uses a Radix `DropdownMenu` to present transition options and a Radix `AlertDialog` (via `TransitionConfirmDialog`) for confirmation. The dropdown item uses `onClick` to set state (`setPendingTarget` + `setDialogOpen(true)`), but does not call `event.preventDefault()` on Radix's `onSelect` event.

When a `DropdownMenuItem` is clicked, Radix's default behavior is:
1. Fire the `onSelect` event (which closes the dropdown and its portal layer)
2. Fire the `onClick` handler
3. The dropdown's `DismissableLayer` unmounts

The problem is that when the `AlertDialog` opens (via state update from `onClick`), Radix's focus management and `DismissableLayer` cleanup from the closing dropdown can interfere with the new dialog. The dropdown's portal removal triggers a pointer-down-outside detection on the AlertDialog, causing `onOpenChange(false)` to fire on the dialog before the user can interact with it. The dialog either never renders visually or flashes open and immediately closes.

**Why it worked for other customers:** The race is timing-sensitive. On earlier QA days (Days 2, 4, 15), the page had less data or different rendering timing that allowed the dialog to win the race. By Day 32, the page for Mathole Engineering (rendered alongside 4 other customers' accumulated data in the same org) hits the race window consistently.

**Evidence in the codebase:** Three other components already handle this correctly using `onSelect={(e) => e.preventDefault()}`:
- `frontend/app/(app)/org/[slug]/settings/custom-fields/custom-fields-content.tsx` (lines 191, 201, 290, 300)
- `frontend/components/projects/project-members-panel.tsx` (line 223)
- `frontend/components/templates/TemplateActionsMenu.tsx` (line 103)

The `LifecycleTransitionDropdown` is the only dropdown-to-dialog pattern that is missing this guard.

## Fix

**Frontend only.** Single file change in `LifecycleTransitionDropdown.tsx`.

### Changes

**`frontend/components/compliance/LifecycleTransitionDropdown.tsx`**

1. Add `onSelect={(e) => e.preventDefault()}` to each `DropdownMenuItem` to prevent the dropdown from auto-closing and racing with the dialog:

```tsx
<DropdownMenuItem
  variant={isDestructive ? "destructive" : "default"}
  onSelect={(e) => e.preventDefault()}
  onClick={() => handleSelect(target)}
>
  {getTransitionLabel(currentStatus, target)}
</DropdownMenuItem>
```

2. Track the dropdown open state explicitly so we can close it when the dialog opens:

```tsx
const [dropdownOpen, setDropdownOpen] = useState(false);
```

3. Pass `open` and `onOpenChange` to `<DropdownMenu>`:

```tsx
<DropdownMenu open={dropdownOpen} onOpenChange={setDropdownOpen}>
```

4. In `handleSelect`, close the dropdown before opening the dialog:

```tsx
async function handleSelect(target: LifecycleStatus) {
  setPendingTarget(target);
  setDropdownOpen(false);  // Close dropdown first

  if (requiresPrerequisiteCheck(currentStatus, target)) {
    // ... prerequisite check flow (unchanged)
  } else {
    setDialogOpen(true);
  }
}
```

The combination of `onSelect={(e) => e.preventDefault()}` (prevents Radix's automatic close + portal teardown) and manual `setDropdownOpen(false)` (closes dropdown on our terms before opening the dialog) ensures the dismissal layers don't race.
