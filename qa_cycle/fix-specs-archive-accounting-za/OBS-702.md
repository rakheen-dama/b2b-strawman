# OBS-702 — Proposal expiry shows +1 day vs ISO submitted

**Severity**: nit (cosmetic; ISO date persisted correctly)
**Surface**: Frontend (proposal detail page)
**Scope**: Frontend-only, single-line change + 1 vitest guard
**Effort**: XS (~15 min implementation, ~30 min total with test)

## Root cause (verified)

The Create Proposal dialog encodes the user-picked YYYY-MM-DD date as `${expiresAt}T23:59:59Z`
(UTC end-of-day) before submission:

- `frontend/components/proposals/create-proposal-dialog.tsx:173`
  ```ts
  ...(values.expiresAt && {
    expiresAt: `${values.expiresAt}T23:59:59Z`,
  }),
  ```

Backend persists the ISO instant verbatim (verified via API response payload —
`2026-05-12T23:59:59Z` for the QA submission).

The proposal detail page renders the expiry through `formatDate`:

- `frontend/app/(app)/org/[slug]/proposals/[id]/page.tsx:183` calls `formatDate(proposal.expiresAt)`
- `frontend/lib/format.ts:3-9`:
  ```ts
  export function formatDate(date: string | Date): string {
    return new Date(date).toLocaleDateString("en-US", {
      month: "short",
      day: "numeric",
      year: "numeric",
    });
  }
  ```

`toLocaleDateString` without an explicit `timeZone` uses the browser's local zone. In SAST
(UTC+2), `2026-05-12T23:59:59Z` becomes `2026-05-13T01:59:59+02:00` → "May 13, 2026".
The user picked May 12; the display reads May 13. Off-by-one.

## Decision

**Fix at the dialog layer** — submit the date as a calendar date that round-trips
the user's picked day, regardless of viewer timezone. The `expiresAt` field on
`Proposal` is a wall-clock expiry, not a true UTC instant; sending `T23:59:59Z`
was an attempt to encode "end of day" but it crosses the date line in any zone
east of UTC.

The `formatLocalDate` helper at `frontend/lib/format.ts:101-108` already exists for
exactly this kind of YYYY-MM-DD calendar-date round-trip; the proposal detail
page should use it for `expiresAt` since the user only ever picks a date (no
time component is exposed in the form).

We also fix the dialog: encode `expiresAt` as the local end-of-day in the user's
zone (the same wall-clock day the user picked), so the backend stores the same
calendar date the user selected. This means submit `expiresAt` without forcing
`Z`-suffixed UTC — use `toISOString()` after constructing a Date from local
components.

## Fix

**Two coordinated single-line changes:**

### 1. Dialog — encode local end-of-day instead of forced-UTC

`frontend/components/proposals/create-proposal-dialog.tsx:172-174`

**Before:**
```ts
...(values.expiresAt && {
  expiresAt: `${values.expiresAt}T23:59:59Z`,
}),
```

**After:**
```ts
...(values.expiresAt && {
  // Encode as the LOCAL end-of-day of the user's picked calendar date.
  // The previous `T23:59:59Z` form crossed the date line in any zone east of
  // UTC, causing "expires May 12" inputs to render as "May 13" (OBS-702).
  expiresAt: (() => {
    const [y, m, d] = values.expiresAt.split("-").map(Number);
    return new Date(y, m - 1, d, 23, 59, 59).toISOString();
  })(),
}),
```

### 2. Detail page — render with the calendar-date helper

`frontend/app/(app)/org/[slug]/proposals/[id]/page.tsx:183`

**Before:**
```tsx
{formatDate(proposal.expiresAt)}
```

**After:**
```tsx
{formatProposalExpiresAt(proposal.expiresAt)}
```

Add a small helper to `frontend/lib/format.ts` (next to `formatLocalDate`):

```ts
/**
 * Renders an ISO instant as a calendar date in the viewer's local zone.
 * Intended for proposal expiry / matter due-date fields where the underlying
 * value is a wall-clock end-of-day instant — calling toLocaleDateString with
 * the default zone correctly maps the local-end-of-day instant back to the
 * date the user picked.
 */
export function formatProposalExpiresAt(iso: string): string {
  return new Date(iso).toLocaleDateString("en-US", {
    month: "short",
    day: "numeric",
    year: "numeric",
  });
}
```

(The helper is functionally identical to `formatDate` today — the bug is the
*input* `T23:59:59Z`, not the formatter. The named helper signals intent and
gives a single insertion point if future formatting tweaks are needed for
expiry fields specifically.)

## Out of scope

- Other date fields on the proposal detail page (`createdAt`, `sentAt`, `acceptedAt`,
  `declinedAt`) are real UTC instants from server transitions — `formatDate`'s
  current behaviour is correct for those (users want to see the firm-local time).
  Only `expiresAt` is a user-picked calendar date.
- Other places that submit ISO dates from a `<Input type="date">` — those should
  be audited as a follow-up, but are out of scope for OBS-702 which is scoped to
  the proposal expiry surface QA flagged.
