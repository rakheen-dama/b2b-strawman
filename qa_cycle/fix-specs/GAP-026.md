# Fix Spec: GAP-026 — FICA/KYC checklist template not visible after seeding

## Problem
The Settings > Checklists page shows only 1 template ("Generic Client Onboarding" with 4 items). The expected "FICA/KYC — SA Accounting" checklist template with 9 items is not visible. This means customers transitioned to ONBOARDING only get the generic 4-item checklist, not the SA-specific FICA verification checklist. Blocks Day 1 compliance verification (checkpoint 1.2).

## Root Cause (confirmed)
The FICA checklist template IS being seeded by `CompliancePackSeeder` from `compliance-packs/fica-kyc-za/pack.json`. However, the seeder creates it as **inactive** (`active = false`) because the pack has `autoInstantiate: false`.

**File**: `backend/src/main/java/io/b2mash/b2b/b2bstrawman/compliance/CompliancePackSeeder.java`, lines 131-134:
```java
// FICA packs (autoInstantiate=false) should be inactive by default
if (!tpl.autoInstantiate()) {
    template.setActive(false);
}
```

The frontend checklist template listing endpoint (`GET /api/checklist-templates`) calls `ChecklistTemplateService.listActive(customerType)` which filters by `active = true`. Since the FICA template is inactive, it is hidden from:
- Settings > Checklists page
- The "Manually Add Checklist" dialog on customer detail
- Auto-instantiation during ONBOARDING transition

**File**: `backend/src/main/java/io/b2mash/b2b/b2bstrawman/checklist/ChecklistTemplateService.java`, line 43: `listActive` filters by `active = true`.

## Fix
Remove the logic that sets non-autoInstantiate templates as inactive. The `autoInstantiate` flag controls whether the checklist is automatically created when a customer transitions to ONBOARDING — it should NOT control whether the template is visible and available for manual instantiation.

### Step 1: Remove the inactive override in CompliancePackSeeder
**File**: `backend/src/main/java/io/b2mash/b2b/b2bstrawman/compliance/CompliancePackSeeder.java`

Delete lines 131-134:
```java
// DELETE THIS:
// FICA packs (autoInstantiate=false) should be inactive by default
if (!tpl.autoInstantiate()) {
    template.setActive(false);
}
```

### Step 2: Fix existing data for already-provisioned tenants
Since the FICA template was already created as inactive, we need a data fix. Add an activation step in the `applyPack` method or handle it in the pack reconciliation.

Option A (preferred): Modify `CompliancePackSeeder.isPackAlreadyApplied()` to also check if the template is active. If the pack was applied but the template is inactive, re-activate it.

Option B (simpler): After removing the inactive override, reset `compliancePackStatus` in `OrgSettings` for the FICA pack so it gets re-applied. This is destructive — it would create a duplicate template.

**Recommended approach**: Option A is safest. Add a reconciliation check:

In `CompliancePackSeeder`, override a lifecycle hook or add to `applyPack` logic:
```java
// After removing the inactive override, also fix existing inactive templates
var existing = checklistTemplateRepository.findBySlug(tpl.slug());
if (existing.isPresent() && !existing.get().isActive()) {
    existing.get().setActive(true);
    checklistTemplateRepository.save(existing.get());
    log.info("Reactivated compliance template {} for pack {}", tpl.slug(), pack.packId());
    return; // Already applied, just needed reactivation
}
```

But since the `isPackAlreadyApplied` check happens first and returns true, the `applyPack` method is never called for already-applied packs. So the fix needs to be at the reconciliation level.

**Best approach**: Add a post-seed reconciliation step in `CompliancePackSeeder.doSeedPacks()` (or override in AbstractPackSeeder) that activates any templates from applied packs that are currently inactive.

Actually, the simplest fix: Just remove the inactive override (Step 1) AND add a single line after the existing pack check in `doSeedPacks`:

**File**: `backend/src/main/java/io/b2mash/b2b/b2bstrawman/seeder/AbstractPackSeeder.java`

This is a framework-level concern. Instead, handle it in CompliancePackSeeder by adding a method that runs after all packs are processed:

**Simplest fix for existing data**: In `CompliancePackSeeder`, add a method to reactivate inactive pack templates, and call it at the end of seeding. OR, just write a SQL migration:

**File to create**: `backend/src/main/resources/db/migration/tenant/V71__activate_fica_checklist_template.sql`
```sql
UPDATE checklist_templates SET active = true WHERE pack_id = 'fica-kyc-za' AND active = false;
```

This is the cleanest approach for fixing existing data while the code fix prevents future occurrences.
