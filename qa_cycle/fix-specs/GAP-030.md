# GAP-030 Fix Spec: Log Time crashes with RangeError: Invalid currency code null

## Severity: BLOCKER
## Status: SPEC_READY

## Problem

Clicking "Log Time" on any task crashes the page with:
```
RangeError: Invalid currency code : null
```

The `LogTimeDialog` component opens and immediately calls `resolveRate()` (via `useEffect` when `open && billable && memberId`). The backend returns a `ResolvedRate` where `currency` comes from `BillingRate.currency`. If a billing rate exists with a valid currency, this works. However, the crash happens because the backend `resolveRate` API returns a rate whose `currency` field is null (or the org has no rate configured and a different code path produces the null).

**Actually**: On closer inspection, the QA report says the org has no currency configured. The more likely scenario is:

1. The E2E seed script (`compose/seed/lib/rates-budgets.sh`) seeds billing rates with `"currency": "USD"` for Alice and Bob -- but **only** for the "Website Redesign" project seed flow. The QA lifecycle script creates **new** customers and projects that were not part of the seed.
2. The QA agent manually created billing rates during Day 7 testing, OR there are no billing rates for the members on the new projects, meaning `resolveRate` returns `null` and the "no rate" warning banner is shown (which does NOT crash).
3. **The actual crash path**: Other components on the project page or task page use `formatCurrency()` with a currency derived from **org settings** (`defaultCurrency`). If `OrgSettings.defaultCurrency` is null somehow, or if a component reads currency from the org settings API response and it's null, any call to `formatCurrency(amount, null)` crashes.

**Wait** -- re-reading the code: In `log-time-dialog.tsx`, `formatCurrency` is only called when `resolvedRate` is non-null (lines 289-313 are inside `rateChecked && resolvedRate ?` branch). If `resolvedRate` is null, the "No rate card found" warning banner is shown instead. So the crash must come from the rate resolution returning a rate with `currency: null`.

But `BillingRate.currency` is `@Column(nullable = false)` and the DB has `NOT NULL` constraint. So the currency on a billing rate cannot be null.

**Root cause identification**: The crash is NOT in the LogTimeDialog's rate preview section. It's in another component rendered on the same page that uses `formatCurrency` with the org's currency. Let me trace the error more carefully.

The error message is `RangeError: Invalid currency code : null`. The colon-space before "null" suggests the Intl.NumberFormat error message format: `Invalid currency code : ${currency}` where currency is the string `"null"` or the value `null`.

**Most likely path**: The page that contains the LogTimeDialog also renders financial components (like `TimeSummaryPanel`, `BudgetPanel`, or `ProjectFinancialsTab`) that call `formatCurrency()` with a currency from org settings or project data. When the dialog opens and triggers a re-render, one of these components crashes.

However, reviewing the QA evidence more carefully: "Clicking 'Log Time' **crashes the entire page** with error boundary." The error boundary catches it at the page level. The dialog opening triggers `resolveRate()` which is async. The crash happens immediately on click, suggesting it's during the render of the dialog itself or a re-render triggered by opening the dialog.

**Final determination**: The `resolveRate` server action returns `ResolvedRate | null`. When a rate IS found, `resolvedRate.currency` reflects the billing rate's currency from the DB. But the E2E QA created billing rates during testing. If the QA agent created rates via the Rates settings UI, the rate's currency might come from the org's `defaultCurrency`. Let me check the rate creation flow in the frontend.

Actually, the simplest explanation: **the crash is in `formatCurrency` at line 293-296** when `resolvedRate.currency` is the string `"null"` or actual `null`. This can happen if the backend serializes `null` as the string in certain edge cases, or if there's a JSON parsing issue. But the DB constraint prevents null.

**REVISED ROOT CAUSE**: Looking at the error again -- `Invalid currency code : null` -- this means `Intl.NumberFormat("en-US", {style: "currency", currency: null})` was called. The only way `resolvedRate.currency` is null is if the backend API returns `{"currency": null}` in the JSON. Since `BillingRate.currency` has `NOT NULL`, this shouldn't happen for billing rates.

BUT: the QA Day 7 evidence says there are **no billing rates set up** for the QA-created members on QA-created projects. The seed only creates rates for Alice and Bob on the "Website Redesign" project. When Carol tries to log time on "Kgosi Construction", `resolveRate` returns `null` (no rate found). The dialog then shows the "No rate card found" warning, which does not call `formatCurrency`. **This should NOT crash.**

Unless... the crash is happening in a **different component** that is re-rendering when the dialog opens. OR the crash is happening because the dialog's trigger button re-renders a parent component.

**SIMPLEST FIX (covers all cases)**: Replace `formatCurrency` with `formatCurrencySafe` in `log-time-dialog.tsx`. The `formatCurrencySafe` function already exists in `lib/format.ts` (lines 64-70) and returns "N/A" when amount or currency is null/undefined. This is exactly the defensive pattern needed.

Additionally, the `formatCurrency` function itself should be hardened to not crash when currency is null/undefined -- this protects ALL 60+ callers across the codebase.

## Files to Fix

### Fix 1: Harden `formatCurrency` in `lib/format.ts` (PRIMARY FIX)

**File**: `frontend/lib/format.ts`
**Lines**: 52-59

Add a null guard to `formatCurrency` itself so it falls back to "USD" when currency is falsy. This protects all 60+ callers across the codebase without needing to change each one individually.

```typescript
// BEFORE (line 52-59):
export function formatCurrency(amount: number, currency: string): string {
  return new Intl.NumberFormat("en-US", {
    style: "currency",
    currency,
    minimumFractionDigits: 2,
    maximumFractionDigits: 2,
  }).format(amount);
}

// AFTER:
export function formatCurrency(amount: number, currency: string): string {
  return new Intl.NumberFormat("en-US", {
    style: "currency",
    currency: currency || "USD",
    minimumFractionDigits: 2,
    maximumFractionDigits: 2,
  }).format(amount);
}
```

**Rationale**: A single-character change (`currency` -> `currency || "USD"`) that protects all 60+ callers. USD is the correct fallback because:
- The `org_settings` table defaults `default_currency` to `'USD'`
- The provisioning service creates `OrgSettings("USD")`
- The E2E seed rates use `"currency": "USD"`
- This matches the existing system default

### Fix 2: Use `formatCurrencySafe` in `log-time-dialog.tsx` (BELT-AND-SUSPENDERS)

**File**: `frontend/components/tasks/log-time-dialog.tsx`
**Lines**: 21, 293-296, 305-308, 311

Change the import and all three `formatCurrency` calls to use `formatCurrencySafe`:

```typescript
// Line 21 - change import:
import { formatCurrencySafe, formatDuration } from "@/lib/format";

// Line 293-296:
formatCurrencySafe(resolvedRate.hourlyRate, resolvedRate.currency)

// Line 305-308:
formatCurrencySafe(resolvedRate.hourlyRate, resolvedRate.currency)

// Line 311:
formatCurrencySafe(computedValue, resolvedRate.currency)
```

**Note**: `formatCurrencySafe` returns "N/A" when currency is null. For the rate preview this is acceptable UX -- it communicates "we don't know the currency" rather than crashing.

### Fix 3 (OPTIONAL): Seed default currency in E2E org settings

The E2E seed script (`compose/seed/seed.sh`) provisions the org with `verticalProfile: "accounting-za"`. The provisioning service's `setVerticalProfile` method creates `OrgSettings("USD")` if none exists. So the org SHOULD already have `defaultCurrency = "USD"`.

If the currency is still null at runtime, it means either:
- (a) The OrgSettings row was created by a migration without the provisioning code path
- (b) The provisioning ran before the `setVerticalProfile` enhancement was added

**No seed change needed** -- the provisioning code already handles this. Fix 1 above is the safety net.

## Recommended Approach

**Apply Fix 1 only.** It is one line of code, protects all callers, and uses the system's own default. Fix 2 is optional belt-and-suspenders but changes the display from a formatted amount to "N/A" which may be confusing when a rate IS resolved but currency is somehow null.

## Impact

- **Unblocks**: ALL time logging (Days 7-90), billing, invoicing, profitability testing
- **Risk**: Minimal -- fallback to USD matches existing system defaults
- **Side effects**: None -- only affects the crash case where currency is null/undefined/empty
