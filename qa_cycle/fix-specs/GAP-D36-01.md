# Fix Spec: GAP-D36-01 — Invoice prerequisite check returns 403 "Module not enabled"

## Problem
When QA clicks "New Invoice" on the customer Invoices tab, the `checkPrerequisitesAction("INVOICE_GENERATION", "CUSTOMER", customerId)` call in `use-invoice-generation.ts` line 69 errors. The error message matches `ModuleNotEnabledException` text: "This feature is not enabled for your organization." The dialog opens anyway (catch block falls through on line 76-77) but the user sees console errors on every customer page load with `?tab=invoices`. Additionally, the `api.get('/api/invoices?customerId=${id}')` call on the customer detail page (line 156 of `page.tsx`) hits `InvoiceController` which uses `@RequiresCapability("INVOICING")` — this call is caught and silently returns an empty array.

## Root Cause (hypothesis)
**Primary**: The QA agent's identification of "Module not enabled" as the blocker is likely a misattribution. Invoicing is NOT a module in `VerticalModuleRegistry` — it is a core capability (`Capability.INVOICING`). The 403 errors visible in the console are coming from the `api.get('/api/invoices?customerId=${id}')` call which requires `@RequiresCapability("INVOICING")`.

The REAL issue is that the `MemberFilter` (line 66-76 of `MemberFilter.java`) resolves capabilities via `orgRoleService.resolveCapabilities(memberId)`. If this throws an exception (logged at line 70-75), capabilities fall back to `Collections.emptySet()` (line 75), which means ALL `@RequiresCapability` checks fail with 403.

The likely trigger: the Caffeine cache (`memberCache`) caches `MemberInfo` which only contains `memberId` and `orgRole` (line 100). Capabilities are resolved OUTSIDE the cache. If the capability resolution fails transiently (e.g., the org_role_id is null for the member, or the member's OrgRole entity lacks the expected capabilities), capabilities will be empty.

**To verify**: Check the backend logs for "Failed to resolve capabilities for member" warnings during the QA session. If present, investigate why `orgRoleService.resolveCapabilities()` is failing for Thandi's member ID.

**Secondary possibility**: The Keycloak session expired during testing (noted in the QA report), and the re-authentication created a new session that didn't properly resolve. The BFF session cookie might be stale.

## Fix
This is a diagnosis-first gap. The Dev agent should:

1. **Reproduce**: Start the Keycloak dev stack, authenticate as Thandi (owner), navigate to a customer's Invoices tab, and check browser console + backend logs for 403 errors.

2. **Check backend logs** for:
   - "Failed to resolve capabilities for member" (MemberFilter line 70)
   - Any SQL/Hibernate errors in capability resolution

3. **Verify member state in DB**: Query the tenant schema for Thandi's member record:
   ```sql
   SELECT m.id, m.org_role, m.org_role_id, r.name, r.slug
   FROM members m
   LEFT JOIN org_roles r ON m.org_role_id = r.id
   WHERE m.email = 'thandi@thornton-test.local';
   ```
   If `org_role_id` is NULL, the member was created before V67 migration ran or the `MemberFilter.lazyCreateMember()` didn't assign a role correctly.

4. **If org_role_id is NULL**: Fix the `MemberFilter.lazyCreateMember()` method (line 139+) to always assign the matching OrgRole entity when creating a new member. Currently line 151 sets `effectiveRole = Roles.ORG_MEMBER` but may not be assigning `orgRoleId`. Check that the invitation acceptance path (line 155-160) correctly sets both `org_role` (string) and `org_role_id` (FK).

5. **If capabilities are empty despite valid role**: Check `OrgRoleService.resolveCapabilities()` — the `OrgRole.getCapabilities()` set may be empty if the V67 migration didn't seed capabilities for this tenant's schema.

6. **Add error surfacing**: In `invoice-actions.ts` `fetchUnbilledTime`, if `!caps.isAdmin && !caps.isOwner`, log the actual caps response so QA can see the actual failure reason instead of a generic message.
