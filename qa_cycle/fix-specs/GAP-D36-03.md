# Fix Spec: GAP-D36-03 â€” Module guard 500 errors on customer invoices tab load

## Problem
Every page load of the customer detail page triggers a 500 error in the console from a server action wrapping a backend 403 `ModuleNotEnabledException`. The error is swallowed by the SWR error handler inside `TrustBalanceCard` and does not block any functionality, but it pollutes console logs and triggers unnecessary backend calls for tenants without `trust_accounting` enabled.

Severity: LOW (console noise only, no user-visible impact).

## Root Cause (confirmed)

The `TrustBalanceCard` component (`frontend/components/trust/TrustBalanceCard.tsx`) places a `ModuleGate` wrapper around its JSX return, but declares SWR hooks **above** the gate. React hooks execute unconditionally regardless of what the component returns, so:

1. `TrustBalanceCard` is rendered on the customer detail page (line 670 and line 804 of `frontend/app/(app)/org/[slug]/customers/[id]/page.tsx`)
2. SWR hook on line 56 sets `accountsCacheKey = "trust-accounts-${slug}"` (non-null, so fetch fires)
3. SWR calls `fetchTrustAccounts()` server action which calls `GET /api/trust-accounts`
4. Backend `TrustAccountService.listTrustAccounts()` calls `moduleGuard.requireModule("trust_accounting")`
5. For tenants without `trust_accounting` enabled (e.g., accounting-za profile), this throws `ModuleNotEnabledException` (HTTP 403)
6. Next.js wraps the server action error as a 500 in the console

The `ModuleGate` component correctly hides the rendered content, but cannot prevent the SWR fetch from firing because React hooks are unconditional.

## Fix

**Option A (recommended, ~30 min)**: Move the module check into the SWR cache key computation inside `TrustBalanceCard`. Use `useOrgProfile().isModuleEnabled("trust_accounting")` to conditionally set the SWR cache key to `null` when the module is disabled. SWR with a `null` key does not fetch.

### Changes

**File: `frontend/components/trust/TrustBalanceCard.tsx`**

```diff
 export function TrustBalanceCard({ ... }: TrustBalanceCardProps) {
+  const { isModuleEnabled } = useOrgProfile();
+  const moduleEnabled = isModuleEnabled("trust_accounting");
+
   const [depositOpen, setDepositOpen] = useState(false);
   // ...

-  const accountsCacheKey = !trustAccountId ? `trust-accounts-${slug}` : null;
+  const accountsCacheKey = moduleEnabled && !trustAccountId ? `trust-accounts-${slug}` : null;
```

Since the component already imports `ModuleGate` which uses `useOrgProfile`, and the hook is already available in the component's context, this is a minimal change. When `moduleEnabled` is false, `accountsCacheKey` is `null`, SWR skips the fetch, `resolvedAccountId` stays undefined, `ledgerCacheKey` is null, and both SWR calls are paused. The `ModuleGate` in the return still renders null, so the visual behavior is unchanged.

**Option B (alternative)**: Move `TrustBalanceCard` rendering behind a server-side module check in the page.tsx, passing `enabledModules` from the org profile and conditionally rendering the component. This is heavier and requires more changes.
