# GAP-P48-001: Minimal Proposal Creation & Detail UI

## Decision: SPEC_READY (Option B — minimal viable flow)

## Rationale

The backend has **complete** proposal CRUD + lifecycle (create, update, delete, send, withdraw, milestones, team members). The frontend has a working list page with summary cards and attention list. QA needs exactly 6 steps (1.19-1.24):

1. Navigate to Proposals page
2. Click "New Proposal"
3. Fill: Title, Customer, Fee Model (Retainer), Amount (R5,500), Hours (10), Expiry (30 days)
4. Save as DRAFT — verify it appears in list
5. Open proposal, click "Send" — verify status changes to SENT
6. Check Mailpit for notification

This is **not** a full proposal editor with milestones, team management, content editing, and accept/decline flows. It is a create dialog + a detail page with a send button. Estimated effort: **1.5-2 hours**.

## Scope — What to Build

### 1. Create Proposal Dialog (`frontend/components/proposals/create-proposal-dialog.tsx`)

A `"use client"` dialog component triggered by a "New Proposal" button.

**Fields:**
- Title (text, required)
- Customer (select dropdown — fetch from `/api/customers?size=200`)
- Fee Model (select: Fixed Fee / Hourly / Retainer)
- Conditional fee fields based on fee model:
  - FIXED: Fixed Fee Amount (number), Currency (default from org settings or "ZAR")
  - HOURLY: Hourly Rate Note (text, optional)
  - RETAINER: Retainer Amount (number), Currency, Hours Included (number)
- Expiry Date (date picker, optional)

**On submit:** `POST /api/proposals` with the form data. On success, redirect to the new proposal's detail page or refresh the list.

**Validation (Zod schema in `lib/schemas/proposal.ts`):**
- title: required, max 200 chars
- customerId: required UUID
- feeModel: required, one of FIXED/HOURLY/RETAINER
- fixedFeeAmount: required and > 0 when feeModel is FIXED
- retainerAmount: required and > 0 when feeModel is RETAINER

**Notes:**
- DO NOT include milestones, team members, contentJson, or projectTemplateId in the create dialog. These are advanced features not needed for QA.
- The FeeModel backend enum values are `FIXED`, `HOURLY`, `RETAINER`. The frontend type file has `FIXED_FEE` which is WRONG — fix the type to match backend: `"FIXED" | "HOURLY" | "RETAINER"`.
- Portal contact is NOT required at creation time (only at send time).

### 2. Server Action (`frontend/app/(app)/org/[slug]/proposals/actions.ts`)

Create a server action file with:
- `createProposalAction(formData)` — calls `POST /api/proposals`
- `fetchProposalAction(id)` — calls `GET /api/proposals/{id}`
- `sendProposalAction(id, portalContactId)` — calls `POST /api/proposals/{id}/send`
- `fetchCustomersAction()` — calls `GET /api/customers?size=200` (for the dropdown)
- `fetchPortalContactsAction(customerId)` — calls `GET /api/customers/{customerId}/portal-contacts`

### 3. Proposal Detail Page (`frontend/app/(app)/org/[slug]/proposals/[id]/page.tsx`)

A server component that:
1. Fetches the proposal via `GET /api/proposals/{id}`
2. Displays: title, proposal number, status badge, customer name, fee model + amounts, dates
3. Renders a client component for actions

**Layout:**
- Header: proposal number + title, status badge (reuse `STATUS_BADGE` mapping from proposal-table)
- Info cards: Customer, Fee Model + amounts, Created date, Expiry date, Sent date
- Action bar (client component): "Send Proposal" button (visible when status is DRAFT)

### 4. Proposal Detail Actions (`frontend/components/proposals/proposal-detail-actions.tsx`)

A `"use client"` component that:
- Shows "Send Proposal" button when status is DRAFT
- On click: opens a confirmation dialog that asks to select a portal contact (fetched from `/api/customers/{customerId}/portal-contacts`)
- Calls `POST /api/proposals/{id}/send` with the selected portal contact ID
- On success: refreshes the page (router.refresh())
- Shows "Withdraw" button when status is SENT (calls `POST /api/proposals/{id}/withdraw`)

### 5. Add "New Proposal" Button to Proposals List Page

In `frontend/app/(app)/org/[slug]/proposals/page.tsx`:
- Add a "New Proposal" button next to the page heading
- The button opens the CreateProposalDialog

Since the page is a server component, the button + dialog wrapper needs to be a small client component (e.g., `ProposalPageActions` or inline the button logic in a client wrapper).

### 6. Fix FeeModel Type Mismatch

In `frontend/lib/types/proposal.ts`:
- Change `"FIXED_FEE"` to `"FIXED"` to match the backend enum

## Out of Scope (Do NOT Build)

- Proposal content editor (rich text / contentJson)
- Milestone management
- Team member management
- Edit proposal dialog/page
- Delete proposal
- Accept/decline flows (these are portal-side actions)
- Proposal PDF generation

## Testing

After implementation, QA should be able to:
1. Navigate to `/proposals` and see the "New Proposal" button
2. Create a proposal for Kgosi Construction with Retainer model, R5,500, 10 hours
3. See the proposal in the list as DRAFT
4. Click into the proposal detail page
5. Click "Send Proposal", select a portal contact, confirm
6. Verify status changes to SENT
7. Check Mailpit for the proposal notification email
