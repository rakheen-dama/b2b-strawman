# GAP-P48-016: Invoice send requires overrideWarnings for 70%+ complete profiles

**Severity**: minor
**Status**: WONT_FIX
**Estimated effort**: N/A

## Problem

When sending an invoice, the API returns 422 with "customer_required_fields 7/10 filled" even when the customer has all invoice-relevant fields filled. The QA agent had to use `overrideWarnings: true` to send the invoice.

## Analysis

The `SendInvoiceRequest` record has a single field: `boolean overrideWarnings`. The invoice send validation checks customer profile completeness (all required custom fields, not just invoice-related ones). When a customer has 7/10 required fields filled, the validation warns but does not block if `overrideWarnings: true` is passed.

This is **intentional design**:
- The validation is a soft warning, not a hard block
- It encourages firms to complete customer profiles before sending official documents
- The `overrideWarnings` flag exists specifically for this use case
- In a real accounting firm, sending an invoice to a customer with incomplete compliance data (e.g., missing tax number) is a valid thing to warn about

## Verdict: WONT_FIX

The behavior is correct and intentional. The frontend invoice send flow should pass `overrideWarnings: true` when the user explicitly clicks "Send" after seeing the warning. This is how it currently works on the invoice detail page.

The QA agent encountered this because they used the API directly without the override flag.
