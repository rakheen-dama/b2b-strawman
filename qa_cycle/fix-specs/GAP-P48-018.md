# GAP-P48-018: Project/Customer Profitability show "No data" despite time entries

**Severity**: minor
**Status**: SPEC_READY
**Estimated effort**: 30 min (investigation + fix)

## Problem

On the Profitability page, Team Utilization shows real data (Carol 11h, Bob 4h, Alice 0.5h) but Project Profitability and Customer Profitability sections show "No data for this period" despite 21.5h of time entries across 7 projects.

## Root Cause Analysis

The profitability page calls two APIs:
1. `/api/reports/utilization?from=...&to=...` — works (shows team data)
2. `/api/reports/profitability?from=...&to=...` — returns `{ projects: [] }`

The utilization query aggregates from `time_entries` directly.

The profitability query (`ReportService.getOrgProfitability()`) also queries `time_entries` but with an additional filter:

```sql
WHERE te.billing_rate_currency IS NOT NULL
```

The key difference: **profitability requires `billing_rate_currency` to be set on time entries**. If time entries were created without a billing rate (i.e., no rate card was configured for the member at the time of entry), then `billing_rate_currency` is NULL and these entries are excluded from the profitability query.

The QA checkpoint confirmed that billing rates were set for Alice (R1,500), Bob (R850), and Carol (R450). However, the question is **when** the rates were set relative to when the time entries were logged. If the rates were configured after time entries were created, the existing time entries would not have `billing_rate_snapshot` or `billing_rate_currency` set (these are snapshot values captured at time-entry creation time).

## Expected Behavior

Time entries should capture the billing rate at creation time. If a rate card exists for the member at the time a time entry is logged, `billing_rate_snapshot` and `billing_rate_currency` should be populated.

## Investigation Steps

1. Check if time entries in the E2E database have `billing_rate_currency` set:
   ```sql
   SELECT id, billing_rate_snapshot, billing_rate_currency, cost_rate_snapshot, cost_rate_currency
   FROM time_entries;
   ```
2. If all are NULL, the rates were not being resolved at entry creation time — check `TimeEntryService.create()` for rate resolution logic.
3. If some are set, the date range filter may be excluding them.

## Fix (if rate snapshot is missing)

### Option A: Backend rate resolution at creation time

**File: `backend/.../timeentry/TimeEntryService.java`**

Ensure `create()` resolves the billing rate and cost rate for the member at the time of entry creation:
```java
var billingRate = billingRateService.resolveRate(memberId, projectId, customerId);
if (billingRate != null) {
  entry.setBillingRateSnapshot(billingRate.rate());
  entry.setBillingRateCurrency(billingRate.currency());
}
```

### Option B: Backfill existing entries (E2E data fix)

If the rate resolution logic exists but wasn't triggered (e.g., rates added after time entries), add a one-time backfill in the E2E seed or lifecycle script.

### Option C: Date range issue

The profitability page defaults to "current month" (first of month to today):
```ts
const firstDay = new Date(now.getFullYear(), now.getMonth(), 1);
```

If time entries were logged with dates outside the current month, they'd be excluded. Check the time entry dates vs. the default date range. The frontend allows changing the date range via date inputs, so the QA agent could have tested with a wider range.

3. Project Profitability table should show projects with billable hours, revenue, cost, and margin
4. Customer Profitability should aggregate by customer
