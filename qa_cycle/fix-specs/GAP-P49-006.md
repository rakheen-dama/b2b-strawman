# Fix Spec: GAP-P49-006 — Project custom field pack `autoApply: false`

## Problem
The `accounting-za-project` field pack has `"autoApply": false` (line 10 of the JSON), which means project custom fields (engagement_type, tax_year, sars_submission_deadline, assigned_reviewer, complexity) are NOT automatically applied to new projects. Users must manually navigate to Settings > Custom Fields and apply the field pack. Without this, the CustomFieldSection on project detail pages shows "No custom fields configured." This blocks Track T0.7 (project custom field population) and cascades to T1.1.4 (engagement_type blank), T1.2.4 (tax_year blank), and T1.2.5 (sars_submission_deadline blank).

By contrast, the customer field pack (`accounting-za-customer.json`, line 10) has `"autoApply": true`, so customer fields appear automatically.

## Root Cause (confirmed)
The `accounting-za-project.json` field pack explicitly sets `autoApply: false` at line 10. This was likely intentional during initial development (project fields are less universally required than customer fields), but for the accounting-za vertical profile where every project IS an engagement, auto-applying these fields is the correct behavior.

- File: `backend/src/main/resources/field-packs/accounting-za-project.json`
- Line 10: `"autoApply": false`

## Fix
1. In `backend/src/main/resources/field-packs/accounting-za-project.json`, line 10:
   - Change `"autoApply": false` to `"autoApply": true`

## Scope
Backend (seed data only -- field pack JSON file)

Files to modify:
- `backend/src/main/resources/field-packs/accounting-za-project.json` (line 10)

Files to create: none
Migration needed: no (field packs are applied at seeding time; existing tenants would need a re-seed or manual application)

- Verify the CustomFieldSection shows the 5 engagement fields (engagement_type, tax_year, sars_submission_deadline, assigned_reviewer, complexity) without manual configuration
