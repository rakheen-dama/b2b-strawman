# Fix Spec: GAP-P49-013 — Most projects lack customer link in template context

## Problem

5 of 6 client projects render as "Internal Project" with no customer data in documents. Only "Monthly Bookkeeping -- Kgosi" has a working customer link. Engagement letters for affected projects produce: "Dear ,", "Client:", and blank registration numbers.

## Root Cause (confirmed)

**Dual-path customer link mismatch.** The `Project` entity has a `customerId` column (set during creation), AND there is a separate `CustomerProject` join table for many-to-many customer-project associations.

The seed script's `_ensure_project()` correctly passes `customerId` in the POST body. The `ProjectService.createProject()` correctly sets `project.setCustomerId(customerId)` (line 131 of `ProjectService.java`). **But it never creates a `CustomerProject` join record.**

The `ProjectContextBuilder` (line 68) uses `customerProjectRepository.findByProjectId(entityId)` -- which queries the `CustomerProject` join table, NOT `project.getCustomerId()`. Since no join record exists for the seed-created projects, the context builder finds no customer.

The one working project ("Monthly Bookkeeping -- Kgosi") likely has a join record because it was also linked from the customer side during seeding.

This is a data consistency bug: `ProjectService.createProject()` should create a `CustomerProject` record when `customerId` is provided, or `ProjectContextBuilder` should fall back to `project.getCustomerId()`.

## Fix

**Option A (recommended): Fix ProjectContextBuilder to fall back to project.customerId** (~20 min)

In `ProjectContextBuilder.buildContext()`, after checking `customerProjectRepository.findByProjectId()`, add a fallback that checks `project.getCustomerId()`:

```java
// customer.* (via CustomerProject join table -- null-safe)
var customerProjects = customerProjectRepository.findByProjectId(entityId);
UUID resolvedCustomerId = null;
if (!customerProjects.isEmpty()) {
    resolvedCustomerId = customerProjects.getFirst().getCustomerId();
} else if (project.getCustomerId() != null) {
    // Fallback: project was created with customerId but no join record
    resolvedCustomerId = project.getCustomerId();
}

if (resolvedCustomerId != null) {
    customerRepository.findById(resolvedCustomerId)
        .ifPresentOrElse(
            customer -> { /* build customer context */ },
            () -> context.put("customer", null));
} else {
    context.put("customer", null);
}
```

**Option B: Fix ProjectService.createProject() to also create CustomerProject record** (~30 min)

Add to `createProject()` after `project = repository.save(project)`:
```java
if (customerId != null) {
    customerProjectService.linkProject(customerId, project.getId(), createdBy);
}
```

This ensures data consistency for all future project creation. But it doesn't fix existing seed data without a reseed.

**Recommendation**: Do BOTH. Option A fixes the context builder for any projects that only have `customerId` set. Option B prevents future inconsistency. The combination is robust.

## Scope
