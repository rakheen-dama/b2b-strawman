# Fix Spec: GAP-P49-020 — Mailpit receives no emails in E2E stack

## Problem
The E2E backend fails to send emails via Mailpit SMTP. Backend logs show `"Authentication failed"` for every email send attempt. Information request emails (sent via `SmtpEmailProvider`) fail silently. Proposal emails have additional issues (notification preferences default disabled, recipientEmail null) but those are out of scope for this fix.

## Root Cause
`application.yml` (base config) sets:
```yaml
mail.smtp.auth: true
mail.smtp.starttls.enable: true
```

These are correct for production (SES/real SMTP), but Mailpit does not support authentication. The `application-e2e.yml` does NOT override these properties. Compare with `application-local.yml` which correctly sets both to `false`.

The Docker Compose (`compose/docker-compose.e2e.yml`) only sets `SPRING_MAIL_HOST=mailpit` and `SPRING_MAIL_PORT=1025` — it does not override the auth properties.

## Fix

### Option A (preferred): Add mail overrides to `application-e2e.yml`

File: `backend/src/main/resources/application-e2e.yml`

Add:
```yaml
spring:
  mail:
    properties:
      mail.smtp.auth: false
      mail.smtp.starttls.enable: false
```

This mirrors what `application-local.yml` already does (lines 2-7).

### Option B (alternative): Add env vars to Docker Compose

File: `compose/docker-compose.e2e.yml`

Add to backend service environment:
```yaml
SPRING_MAIL_PROPERTIES_MAIL_SMTP_AUTH: "false"
SPRING_MAIL_PROPERTIES_MAIL_SMTP_STARTTLS_ENABLE: "false"
```

**Use Option A** — it's the standard Spring Boot approach and matches the existing `application-local.yml` pattern.

## Scope
