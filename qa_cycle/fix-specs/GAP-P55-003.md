# Fix Spec: GAP-P55-003 — Gated module direct URL shows generic error

## Problem
When an accounting tenant navigates directly to `/court-calendar` or `/conflict-check`, the frontend shows a generic "Something went wrong" error page instead of a module-specific "not available" message. The backend API correctly returns 403 with descriptive messages, but the frontend error boundary catches it generically.

## Root Cause (hypothesis)
- Pages: `frontend/app/(app)/org/[slug]/court-calendar/page.tsx` and `frontend/app/(app)/org/[slug]/conflict-check/page.tsx`
- Both pages check `enabledModules` and call `notFound()` if the module is missing (line 22-24 in both).
- The `getOrgSettings()` call at the top may be succeeding but `enabledModules` may not include the module, causing `notFound()` to trigger a 404-style page (not a "module not available" message).
- Alternatively, the API call for data fetching (e.g., `fetchCourtDates()`) may hit the module guard first, returning a 403 that the error boundary catches as a generic error before the module check on line 22 executes.
- The real fix: the server action calls `fetchCourtDates()` / `fetchConflictChecks()` which hit the module guard API and throw a 403 `ApiError`. The try/catch on the page catches this silently, but the `notFound()` path on line 22-24 should already handle it. The issue is likely that the 403 from the data fetch propagates to the error boundary before the module check completes.

## Fix
The pages already do the module check. The issue is the order of operations — the concurrent `fetchCourtDates()` call (inside `Promise.all`) can throw a 403 that isn't caught. Ensure the module check happens BEFORE any data fetching.

1. In both page files, restructure to check modules first, then fetch data:
   ```tsx
   // Already correct — the module check is before data fetching
   // But the data fetching is in a try/catch that should handle 403
   ```
   
   Actually, looking at the code again, the structure IS correct: module check first, data fetch second. The issue is that for a non-legal tenant, `getOrgSettings()` returns settings where `enabledModules` does NOT include the legal module, so `notFound()` is called. The `notFound()` function from Next.js triggers the not-found boundary, which may show a generic page.

2. Create a custom `not-found.tsx` in the route group, or handle the module-not-enabled case with a dedicated UI instead of `notFound()`:
   ```tsx
   if (!enabledModules.includes("court_calendar")) {
     return (
       <div className="flex flex-col items-center justify-center py-20">
         <h2 className="text-xl font-semibold">Module Not Available</h2>
         <p className="mt-2 text-slate-500">
           The Court Calendar module is not enabled for your organization.
         </p>
       </div>
     );
   }
   ```

3. Apply the same pattern to `/conflict-check/page.tsx` and `/legal/tariffs/page.tsx`.
