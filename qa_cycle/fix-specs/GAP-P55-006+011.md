# Fix Spec: GAP-P55-006 + GAP-P55-011 — Court Date dialog and Conflict Check page crash (TypeError .map())

## Problem
Two frontend crashes with the same root cause pattern:
1. **GAP-P55-006**: Clicking "New Court Date" on the Court Calendar page crashes with `TypeError: Cannot read properties of undefined (reading 'map')` in a react-hook-form Controller render.
2. **GAP-P55-011**: The Conflict Check page (`/conflict-check`) crashes on load with the same TypeError.

Both backend APIs work correctly. The crashes are frontend-only.

## Root Cause (hypothesis)
Both crashes involve components that call server actions (`fetchProjects()`, `fetchCustomers()`) from `useEffect` on mount/open. The server actions call `api.get<PaginatedResponse<...>>("/api/projects?size=200")` and return `result.content`.

The most likely cause: the server action call succeeds but the gateway returns an unexpected response (e.g., HTML login redirect, empty body, or malformed JSON) that parses to `undefined` or an object without `.content`. When `result.content` evaluates to `undefined`, it gets passed to `setProjects(undefined)`, causing the next render's `projects.map()` to throw.

**Key evidence**: The `apiRequest()` function (in `frontend/lib/api/client.ts` line 128-133) returns `undefined as T` for empty bodies, and does `JSON.parse(text) as T` without validation. If the gateway returns a non-JSON response that doesn't trigger the error path, or if the session cookie is missing/expired during the server action call from a client component, the result could be `undefined`.

**For GAP-P55-006** (`CreateCourtDateDialog`):
- File: `frontend/components/legal/create-court-date-dialog.tsx` line 78-83
- `fetchProjects().then((all) => setProjects(all))` — if `all` is `undefined`, `projects` becomes `undefined`
- Line 154: `projects.map(...)` throws

**For GAP-P55-011** (`ConflictCheckForm`):
- File: `frontend/components/legal/conflict-check-form.tsx` line 64-73
- `Promise.all([fetchProjects(), fetchCustomers()]).then(([p, c]) => { setProjects(p); setCustomers(c); })` — if `p` or `c` is `undefined`
- Lines 193 (`customers.map(...)`) and 219 (`projects.map(...)`) throw

**Note**: The conflict check PAGE also calls `fetchConflictChecks()` at the server level (page.tsx line 30-35) which expects paginated response `{ content, page }`. If that call fails, `initialChecks` stays `[]` and `initialTotal` stays `0` — so the page itself should still render. The crash is most likely in the client form component.

## Fix
Add defensive defaults (`?? []`) on the server action return values AND in the state setters:

### GAP-P55-006 — CreateCourtDateDialog
1. In `frontend/components/legal/create-court-date-dialog.tsx`, change the `useEffect` callback (line 80-82):
   ```tsx
   fetchProjects()
     .then((all) => setProjects(all ?? []))
     .catch(() => setProjects([]));
   ```

### GAP-P55-011 — ConflictCheckForm
2. In `frontend/components/legal/conflict-check-form.tsx`, change the `useEffect` callback (line 64-73):
   ```tsx
   Promise.all([fetchProjects(), fetchCustomers()])
     .then(([p, c]) => {
       setProjects(p ?? []);
       setCustomers(c ?? []);
     })
     .catch(() => {
       setProjects([]);
       setCustomers([]);
     });
   ```

### Both server actions — fetchProjects()
3. In `frontend/app/(app)/org/[slug]/court-calendar/actions.ts` line 219-226 and `frontend/app/(app)/org/[slug]/conflict-check/actions.ts` line 114-130, add defensive fallback:
   ```tsx
   export async function fetchProjects(): Promise<{ id: string; name: string }[]> {
     const result = await api.get<PaginatedResponse<{ id: string; name: string }>>(
       "/api/projects?size=200"
     );
     return result?.content ?? [];
   }
   ```
   Similarly for `fetchCustomers()` in conflict-check/actions.ts.
