# Fix Spec: GAP-P55-007 — Postponement reason not shown in court date detail

## Problem
After postponing a court date, the detail panel does not show the postponement reason. The cancellation reason IS shown (under "Outcome" label) because `cancelCourtDate()` stores the reason in the `outcome` field. Postponement does NOT store the reason in any field that the frontend displays.

## Root Cause (confirmed)
- Backend: `CourtCalendarService.postponeCourtDate()` (line 241-270 in CourtCalendarService.java)
  - The postpone reason is only logged to the audit event (line 264-267): `"reason", request.reason() != null ? request.reason() : ""`
  - The `CourtDate` entity has NO `postponeReason` field — the reason is not stored on the entity at all
  - The reason is lost after the audit log write
- Entity: `CourtDate.java` — no `postponeReason` or `reason` column
- Frontend: `court-calendar-client.tsx` detail panel (line 281-361) — only shows `outcome` field, no postpone reason field

The reason was never persisted to the entity. The cancel flow stores the reason in `outcome`, but the postpone flow discards it.

## Fix
**Option A (minimal — reuse outcome field):**
Store the postpone reason in the existing `outcome` field (same as cancel does). This avoids a migration.

1. In `CourtCalendarService.postponeCourtDate()` (line 251-253), add:
   ```java
   courtDate.setScheduledDate(request.newDate());
   courtDate.setStatus("POSTPONED");
   if (request.reason() != null && !request.reason().isBlank()) {
       courtDate.setOutcome("Postponed: " + request.reason());
   }
   ```

2. No frontend change needed — the detail panel already displays `outcome`.

**Option B (cleaner — add dedicated field):**
1. Add a `reason` column to `court_dates` table via Flyway migration.
2. Add `reason` field to `CourtDate` entity.
3. Store postpone/cancel reason in `reason` field.
4. Add "Reason" field to the frontend detail panel.
5. This is more work but cleaner separation from outcome.

**Recommended: Option A** for the bugfix cycle.
