## Scope expansion

Spec was authored against `RecordDepositDialog.tsx` only; lines 60–67 of
the spec note that `RecordPaymentDialog.tsx:122-128` and
`RecordRefundDialog.tsx:112-118` carry the **identical** defect (raw
`<Input placeholder="Matter UUID (optional)"/>` and Client UUID
fields).

Per the QA cycle mandate ("Don't skip any bugs") the dev brief expanded
scope to fix **all three** dialogs in one PR: same bug class, identical
fix pattern, identical risk profile. The shared
`TrustEntityPickers.tsx` module makes the three call sites near-trivial
(~20 lines each) — fixing two and leaving the third broken would have
left Payment + Refund vulnerable in production.

## What landed

### New file

- `frontend/components/trust/TrustEntityPickers.tsx` — shared
  `TrustCustomerPicker` + `TrustMatterPicker` components. Both wrap a
  Radix Popover + Command combobox over a server-fetched list. The
  customer picker takes a `customers` prop. The matter picker is
  dependent: takes a `customerId` prop and fetches `Project[]` via SWR
  using the existing `fetchCustomerProjects` server action; trigger is
  disabled with text "Select a client first" until the customer is
  picked. Both honour the locked-picker pattern via
  `defaultCustomerId` / `defaultProjectId`.

### Modified files (frontend only)

- `frontend/components/trust/RecordDepositDialog.tsx` — replaced the
  two raw `<Input placeholder="UUID"/>` fields with the shared
  pickers; extended props to accept `customers`, `defaultCustomerId`,
  `defaultProjectId`.
- `frontend/components/trust/RecordPaymentDialog.tsx` — same
  treatment (sibling defect, same prop shape).
- `frontend/components/trust/RecordRefundDialog.tsx` — same
  treatment (sibling defect; also added `onSuccess` callback for
  parity with Deposit/Payment).
- `frontend/components/trust/transaction-actions.tsx` — accepts a
  `customers` prop and forwards it to the three dialogs; removed
  raw-input pass-through.
- `frontend/components/trust/TrustBalanceCard.tsx` — passes
  `defaultCustomerId={customerId}`, `defaultProjectId={projectId}`,
  and a single-customer locked-picker list
  `[{id: customerId, name: ledger.customerName, email: ""}]` to both
  Deposit and Payment dialogs. Avoids a redundant `/api/customers`
  fetch on the client + matter detail surfaces (the picker is locked
  there anyway).
- `frontend/app/(app)/org/[slug]/trust-accounting/transactions/page.tsx` —
  server-side fetches `Customer[]` from `/api/customers?status=ACTIVE`
  (gated on `canManageTrust` to avoid extra calls for read-only
  users), trims to `{id, name, email}`, and forwards to
  `<TransactionActions customers={...}>`. Non-fatal on fetch error —
  pickers fall back to a "No clients found." empty state.

### Test changes

- `frontend/__tests__/trust-transactions.test.tsx` — extended the
  existing `RecordDepositDialog` submit test to pass the new required
  `customers` prop and exercise the locked-picker submission path.
  Added two new OBS-1001 cases: (1) customer combobox renders with
  the placeholder "Select a client..." and the legacy raw UUID
  inputs are gone, (2) matter trigger is disabled with "Select a
  client first" text until a client is picked.
- `frontend/__tests__/{trust-coexistence,trust-tabs-settings,stub-pages}.test.tsx`
  — added a `vi.mock("@/app/(app)/org/[slug]/customers/[id]/actions",
  ...)` stub since `TrustBalanceCard` now indirectly imports that
  server-action file (via `TrustEntityPickers.tsx`). Without the
  stub, happy-dom tries to load `server-only` and the suite fails
  to set up.

## API contract

**Unchanged.** The form still POSTs `customerId` + `projectId` UUIDs.
Only the UI binding changes — the user picks a customer by name, the
form value remains the customer's id. Backend validators
(`recordDepositSchema` etc. in `frontend/lib/schemas/trust.ts`) were
not touched.

## Known limitations

- Driving the Radix Popover open via `userEvent.click` in
  happy-dom is unreliable (the popover doesn't render its
  `CommandList` portal in the test DOM). The deeper select-and-fill
  flow is therefore exercised via the `defaultCustomerId` lock path
  in tests, which goes through the same submit code without
  scripting the popover. Manual QA verification (Day 10 ck 10.4
  retry) covers the popover-open + select path end-to-end.
