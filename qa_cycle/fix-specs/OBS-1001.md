# Fix Spec: OBS-1001 — Trust dialog combobox triggers unresponsive (triple Slot chain)

## Problem

The Client and Matter combobox pickers in the Record Deposit, Record Payment, and Record Refund trust transaction dialogs do not respond to clicks, keyboard (Space/Enter), or force-click. Users cannot select a Client or Matter when recording trust transactions from the Trust Accounting > Transactions page.

The QA agent was only able to record the Day 10 deposit by forcing React state programmatically via browser evaluate -- real users have no such workaround.

## Root Cause (hypothesis)

The `TrustCustomerPicker` and `TrustMatterPicker` components in `frontend/components/trust/TrustEntityPickers.tsx` use a triple Slot/cloneElement composition chain inside each picker:

```
<PopoverTrigger asChild>          -- Radix Slot: cloneElement on its single child
  <FormControl>                   -- cloneElement on its single child (line 86-98 of form.tsx)
    <Button>                      -- the actual DOM element
```

This is the **OBS-2103 bug class** (documented in `frontend/CLAUDE.md` under "Dialog Trigger Composition"). When `PopoverTrigger asChild` uses Radix `Slot` to clone its child, it expects a DOM element or a component that forwards refs cleanly. But `FormControl` is itself a `cloneElement` wrapper -- it receives the child, clones it with aria attributes, and returns the clone. The double-clone chain (`Slot` cloning a `cloneElement` wrapper) inside a Radix Dialog causes the `PopoverTrigger`'s click handler to fail to wire up to the final `<Button>` DOM node.

**Affected files (lines with the broken pattern):**

- `frontend/components/trust/TrustEntityPickers.tsx:68-72` — `TrustCustomerPicker`: `<PopoverTrigger asChild><FormControl><Button>...</Button></FormControl></PopoverTrigger>`
- `frontend/components/trust/TrustEntityPickers.tsx:185-199` — `TrustMatterPicker`: same pattern

**Consuming dialogs (all affected):**

- `frontend/components/trust/RecordDepositDialog.tsx` (lines 127-132, 144-149)
- `frontend/components/trust/RecordPaymentDialog.tsx` (lines 125-130, 139-144)
- `frontend/components/trust/RecordRefundDialog.tsx` (lines 125-130, 139-144)

**Not affected:** `RecordTransferDialog.tsx` and `RecordFeeTransferDialog.tsx` -- these use `FormControl` with standard `<Input>` children (no Popover), so the double-clone issue does not apply.

## Fix

Remove `FormControl` from the `PopoverTrigger asChild` composition chain in both `TrustCustomerPicker` and `TrustMatterPicker`. Instead, apply the form field aria attributes directly to the `<Button>` and keep `PopoverTrigger asChild` wrapping a single `<Button>` child (no intermediate cloneElement wrapper).

### Change 1: `TrustCustomerPicker` (TrustEntityPickers.tsx, lines 56-119)

Replace the trigger block:

```tsx
// BEFORE (broken — triple Slot chain)
<PopoverTrigger asChild>
  <FormControl>
    <Button type="button" variant="outline" role="combobox" ...>
      ...
    </Button>
  </FormControl>
</PopoverTrigger>

// AFTER (fixed — single Slot, direct child is Button)
<PopoverTrigger asChild>
  <Button type="button" variant="outline" role="combobox" ...>
    ...
  </Button>
</PopoverTrigger>
```

The `FormControl` wrapper was adding `id`, `aria-describedby`, and `aria-invalid` from the `useFormField` hook. Since `TrustCustomerPicker` and `TrustMatterPicker` are used inside `<FormField>` render props (which provide `field` as a controller prop), the form error state and field identity are already managed by `react-hook-form`. The combobox pickers are not standard form inputs -- they manage their own open/select state. The `FormMessage` component below the picker still renders validation errors correctly without `FormControl` because it reads from the form context, not from `FormControl`'s injected aria attributes.

Remove the `FormControl` import from `TrustEntityPickers.tsx` (line 31).

### Change 2: `TrustMatterPicker` (TrustEntityPickers.tsx, lines 173-238)

Same change -- remove `<FormControl>` wrapper from the `<PopoverTrigger asChild>` chain.

### No consumer changes needed

The three consuming dialogs (`RecordDepositDialog`, `RecordPaymentDialog`, `RecordRefundDialog`) pass `<TrustCustomerPicker>` and `<TrustMatterPicker>` as children of `<FormField>` render props. The fix is entirely inside `TrustEntityPickers.tsx` -- no call-site changes required.

## Scope

- **1 file changed**: `frontend/components/trust/TrustEntityPickers.tsx`
  - Remove `FormControl` import (line 31)
  - Remove `<FormControl>` wrapper in `TrustCustomerPicker` trigger (lines 69-71, 81-82 -> unwrap)
  - Remove `<FormControl>` wrapper in `TrustMatterPicker` trigger (lines 185-187, 197-198 -> unwrap)
- **0 consumer changes**: RecordDepositDialog, RecordPaymentDialog, RecordRefundDialog unchanged
- **0 backend changes**
- **No new tests required**: The existing `trust-transactions.test.tsx` covers the dialog rendering. The structural fix removes a composition layer; the Button, Popover, and form validation behavior are unchanged.
