# Fix Spec: OBS-201 — Client wizard: duplicate ID Number + nested SA Legal accordions

## Problem
The Create Client dialog presents **two ID-related fields** to the user during a single wizard flow: a generic "ID Number" on Step 1 (placeholder `e.g. CUS-001`) and an SA-specific "ID / Passport Number" on Step 2 inside the "SA Legal — Client Details" group. The duplication is confusing — Day 2 QA had to fill both with the same SA-ID `8501015800088` to satisfy both the entity column (used by Conflict Check pre-population and the client-detail header) and the vertical-promoted intake field (used by document templates).

Compounding the problem, the Step 2 SA Legal group is **buried under two stacked accordions**: the outer "SA Legal — Client Details" accordion and an inner "Additional Information (4)" toggle. Both are collapsed by default, so the legal-vertical fields are completely invisible until the user finds and clicks both expanders.

Evidence: `qa_cycle/checkpoint-results/day-02.md` checkpoint 2.2 (PARTIAL) and screenshots `qa_cycle/evidence/day-02/2.2-create-client-dialog-step1.png`, `2.2-sa-legal-fields-expanded.png`.

## Root Cause (verified)

### 1. Duplicate "ID Number" field
- `frontend/components/customers/create-customer-dialog.tsx` lines 374-389 render a hardcoded `idNumber` input on Step 1 with placeholder `"e.g. CUS-001"` and label `"ID Number"`. This input writes to the `Customer.idNumber` entity column.
- `backend/src/main/resources/field-packs/legal-za-customer.json` line 14-20 defines `id_passport_number` as a separate intake field on the same conceptual entity. This writes to `customer.customFields.id_passport_number`.
- Both are user-visible during the same wizard. The Step 1 label gives no signal that this is an *internal client reference* (CUS-001 style) rather than the SA national ID; the SA-specific input lives one step away in a nested accordion. Users (and Day 2 QA) populate both with the SA-ID and assume one is redundant.
- Both downstream consumers require non-empty data: Conflict Check pre-fills from `customer.idNumber` (`app/(app)/org/[slug]/customers/[id]/page.tsx` lines 547-548), the customer-detail header renders `customer.idNumber` (line 515), and four legal-za document templates (Power of Attorney, Deed of Transfer, Offer to Purchase, Bond Cancellation Instruction) reference `customer.customFields.id_passport_number`. So we cannot simply delete one without restitching consumers.

### 2. SA Legal accordion nested twice
Verified in `frontend/components/customers/intake-fields-section.tsx`:
- `GroupSection` (line 50) defaults `isOpen` to **`hasRequiredFields`** (line 59). All four `legal-za-customer.json` fields are `required: false`, so the group accordion **stays collapsed by default**.
- *Inside* the group accordion, optional fields are hidden behind a second-level "Additional Information (N)" toggle (lines 96-131) that defaults to closed (`showOptional` initial state `false`, line 60). Required fields render directly under the open group; optional fields go into the inner toggle.
- When a group has **zero required fields** (the legal-za-customer case), the entire field set is hidden behind two collapsed accordions: outer group (closed because no required fields) → inner Additional Information toggle (closed by default).
- This is the "stacked under two accordions" the QA agent reported.

## Fix

Two-part fix, frontend-only. Each part is independent — no backend, schema, or field-pack changes needed.

### Part A — Drop the Step-1 generic "ID Number" input

In `frontend/components/customers/create-customer-dialog.tsx`:

1. **Remove** the `FormField name="idNumber"` block at lines 374-389 (the input plus its label).
2. **Remove** `"idNumber"` from the `form.trigger([...])` array in `handleNext` at line 133.
3. **Keep**: the `idNumber` default value (line 68), the `idNumber` field in the submit payload (line 193), and the SARS-tax-number autofill effect (lines 244-253) — they continue to work, just without a Step 1 input.
4. **Add** at the start of `handleSubmit`, after computing `customFields`: if `customFields.id_passport_number` is a non-empty string and `values.idNumber` is empty, set `values.idNumber = customFields.id_passport_number` before calling `createCustomer`. This preserves the Conflict Check pre-population and the client-detail header rendering — the SA-specific input becomes the single source of truth for both the entity column and the custom field.

The `Customer.idNumber` column remains supported by the schema (`lib/schemas/customer.ts`), the `createCustomer` action, and the Edit Customer dialog (`edit-customer-dialog.tsx` lines 207-218). Existing data is unaffected. Users who need a non-SA-vertical reference can still set `idNumber` post-create via Edit.

### Part B — Auto-expand groups with zero required fields, and flatten the optional toggle in that case

In `frontend/components/customers/intake-fields-section.tsx`:

1. Change the initial `isOpen` derivation (line 59) so a group with **any visible fields** opens by default, not only when it has required fields:
   ```ts
   const [isOpen, setIsOpen] = useState(visibleFields.length > 0);
   ```
2. When a group has **zero required fields**, render the optional fields directly under the open group instead of behind the "Additional Information (N)" toggle. Concretely: skip the optional-toggle wrapper (lines 96-131) when `requiredFields.length === 0` and instead render the optional fields with the same layout used for required fields (minus the red asterisk). Keep the "Additional Information (N)" toggle only when the group mixes required + optional, so required fields stay above-the-fold and optional ones can be tucked away.
3. Update the key-based remount on line 152 from `${group.id}-${hasVisibleRequired}` to `${group.id}-${visibleFields.length > 0}` so the auto-open state recomputes consistently when fields appear via visibility conditions.

This single change handles every legal-za / accounting-za / consulting-za pack that ships with all-optional fields (the common case) without forcing every pack author to mark fields required just to surface them.

Frontend only.

- No new tests required if existing coverage already exercises group rendering; if there are no such tests, a small Vitest case for `IntakeFieldsSection` covering both "all-optional" and "mixed required+optional" rendering is welcome but not blocking.
