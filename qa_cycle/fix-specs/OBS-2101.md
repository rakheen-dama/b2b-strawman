# OBS-2101 — LSSA tariff dropdown missing from time-entry dialog

**Disposition (Product, 2026-04-30): WONT_FIX (feature-gap, scenario amendment).**
Same treatment as OBS-701 (proposal Tiptap/template/fee-estimate) and OBS-202 (KYC unwired):
the feature does not exist in the product yet. The scenario over-specified
relative to what's built. Amend Day 21 Phase A + Day 28 to drive the actual
non-tariff time-entry + free-text fee-note flow.

## Triage Investigation (Product agent, 2026-04-30)

Searched the time-entry domain for any tariff binding — there is none:

- **Entity** `backend/.../timeentry/TimeEntry.java`: columns are `task_id`,
  `member_id`, `date`, `duration_minutes`, `billable`, `rate_cents`
  (deprecated), `billing_rate_snapshot`, `billing_rate_currency`,
  `cost_rate_snapshot`, `cost_rate_currency`, `invoice_id`, `description`,
  `created_at`, `updated_at`. **No `tariff_item_id` FK, no `tariff_units`,
  no `tariff_unit_type`** — nothing to bind to an LSSA item.
- **DTOs** `TimeEntryController.CreateTimeEntryRequest` /
  `UpdateTimeEntryRequest` accept only `date`, `durationMinutes`, `billable`,
  `rateCents`, `description`. No tariff/activity field.
- **Service** `TimeEntryService.createTimeEntry(...)` snapshots rates from
  `RateSnapshotService` (member default → project override → customer
  override). There is no code path that resolves a rate from an
  LSSA tariff item.
- **Frontend** `frontend/components/tasks/log-time-dialog.tsx` renders
  Duration / Date / Description / Billable + a `RetainerIndicator`. The rate
  preview comes from `resolveRate(memberId, projectId, date)` (member rate
  card resolution) — no tariff combobox, no fetch of `/api/tariff-items`.
- **Migrations** `V11__create_time_entries.sql` +
  `V18__add_time_entry_rate_snapshots.sql` define the columns above.
  `V84__add_tariff_invoice_line_type.sql` adds `'TARIFF'` to the
  `invoice_lines.line_type` check constraint — that's the **invoice-side**
  tariff hook (Epic 403A) which is independent of the time-entry side and
  not exercised by Day 28 Bulk Billing.
- **Tariff side** `verticals/legal/tariff/{TariffSchedule,TariffItem,…}.java`
  + `GET /api/tariff-schedules`, `/api/tariff-items` exist and the
  `/org/{slug}/legal/tariffs` page renders the seeded LSSA 2024/2025 schedule
  (19 items, 7 sections). The data side is built. **There is no link from
  this side back to time entries either** — no "Log time against this tariff"
  button on `tariff-browser-client.tsx`, no `/api/tariff-items/{id}/usages`,
  no service that creates a `TimeEntry` from a `TariffItem`.

**Conclusion**: This is a **feature gap**, not a bug. The product currently
has two disconnected surfaces — (a) a generic time-entry domain that
snapshots rates from member rate cards, and (b) a legal-vertical tariff
schedule browser. No code anywhere wires them together. The original
fix-spec ("M effort, 1 dialog change + 1 read endpoint + 1 schema field")
**under-estimated** the work — it ignored that an entire end-to-end
integration would need to be built (tariff-aware rate resolution in
`RateSnapshotService`, tariff-typed invoice line generation in
`InvoiceCreationService` to surface "LSSA tariff line items with activity
descriptors" on Day 28, plus per-tariff unit-type maths e.g. "per 15min" vs
"per page" vs "per day"). That is multi-epic L+ effort, not M.

Per the cycle mandate, the only acceptable open feature gaps are KYC and
Payments. **However**, OBS-701 set precedent for a third class:
*scenario-as-written demanded a feature not built in the product*. The
remedy is the same — amend the scenario to drive the flow that exists,
move the missing capability to a future phase backlog, and proceed.

## Original repro (preserved for context)

1. Login firm `:3000` as Bob (Admin) at `bob@mathebula-test.local` / `SecureP@ss2`.
2. Navigate to matter RAF-2026-001 (`/org/mathebula-partners/projects/b7e319f7-fd7e-4526-a8b3-b40b1f85b34b`).
3. Click **Tasks** tab → click **Log Time** on any task.

Observed dialog: Duration (h/m) / Date / Description (optional) / Billable
+ yellow banner *"No rate card found for this combination. This time entry
will have no billable rate. Set up rates in Project Settings > Rates."*

Expected (per Phase 55 spec + Day 21 scenario): `Tariff Activity` dropdown
+ rate auto-populate from selected LSSA item + tariff-typed line items on

yet.**

## Resolution

**WONT_FIX — feature-gap.** Scenario amended at two checkpoint blocks:

### Day 21 Phase A — drive the time-entry flow that exists

The scenario expectation "select tariff activity from LSSA dropdown, rate
auto-populates" is replaced with the actual non-tariff path:
1. Set a member rate card (or project-level override) for Bob via
   `/org/{slug}/settings/rates` ahead of logging time, **OR** acknowledge
   that no rate card produces an `R 0` non-billable entry and proceed.
2. Open per-task `Log Time` dialog → enter Duration / Date / Description.
3. Submit → time entry saved (with whatever rate `RateSnapshotService`
   resolves from member/project rate cards, or `null`).

The "amount calculated" check becomes "duration saved correctly; if a rate
card matched, `billing_rate_snapshot` populated; if not, the yellow banner
warned the user".

### Day 28 — drive the bulk-billing flow that exists

The scenario expectation "fee note renders with LSSA tariff line items
with activity descriptors" is replaced with the actual `TIME` line-type
behaviour:
1. Bulk Billing → New Billing Run → `By Client` Sipho.
2. Preview shows unbilled time entries (rendered as `TIME` line items,
   one per time entry, labelled by task title + duration + member rate
   snapshot if any) + the R 1 250 sheriff disbursement (`EXPENSE` line).
3. Generate Fee Notes → preview the draft fee note. Verify it renders
   matter ref RAF-2026-001 + each time entry as a line + the disbursement
   as a separate line + VAT 15% + ZAR total.
4. Approve & Send → Mailpit fee note email to Sipho.

The "tariff line items with activity descriptors" assertion is dropped
(replaced with the canonical `TIME` line-type rendering).

## Out-of-scope (future phase — NOT this cycle)

Building the tariff↔time-entry binding requires:
- `time_entry.tariff_item_id` FK + `tariff_units` + `tariff_unit_type`
  columns (additive Flyway migration on tenant schema).
- `RateSnapshotService` extension that resolves a rate from a tariff item
  when one is bound (currently only member/project/customer paths).
- Tariff-aware invoice line generation in `InvoiceCreationService` so
  Day 28 fee notes surface tariff descriptors instead of task titles.
- Per-tariff unit-type maths (per 15min / per hour / per page / per day).
- Frontend: `LogTimeDialog` adds a dependent activity combobox that
  loads from `/api/tariff-schedules/active?category=…&courtLevel=…` and
  populates rate + unit type on selection.
- Vertical gating to `legal-za`.

This is a multi-epic legal-vertical phase (estimated L+ effort across
backend service extensions, frontend dialog, schema, and one read
endpoint). Scoping is out-of-band; do NOT attempt as a one-off bugfix
during this cycle.
