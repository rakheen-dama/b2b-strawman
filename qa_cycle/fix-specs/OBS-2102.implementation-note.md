## Runtime root-cause investigation (Bug 1)

Per spec, the gate split was the user-visible-blocker fix. The QA-observed render gap (Edit paints for Moroka but not Sipho even though both satisfy the original gate) was hypothesised by the Product agent to be a hidden `EditCustomerDialog` `useForm({ defaultValues: buildDefaults(customer) })` Zod-resolver crash on Sipho-specific payload shape (entityType / financialYearEnd / legacy country). Splitting the fragment so Edit always renders eliminates the entire `EditCustomerDialog` hydration-quirk class as a Day 28 blocker ‚Äî even if the underlying init-crash still exists, the trigger `<Button>` paints first because it is now outside the `<EditCustomerDialog>` boundary's render scope (Radix `DialogTrigger asChild` only wraps the child once the dialog itself mounts; if the dialog crashes its trigger no longer disappears under the new structure because the trigger is the bare `<Button>`).

Browser-side reproduction was deferred ‚Äî agent-driven repro requires the Keycloak dev stack (firm `:3000`) to be running with Sipho's tenant data, and the spec's recommended fix is structural (gate change), not symptomatic. QA agent will re-verify on cycle 14 Day 28 retry: hard-reload Sipho's customer page ‚Üí confirm Edit paints ‚Üí click Edit ‚Üí confirm `EditCustomerDialog` opens (this is where any latent payload-shape Zod issue would surface). If the dialog itself still crashes, file a follow-up OBS for Dev to add resolver tolerance to `EditCustomerFormData` defaults.

## Bug 2 SARS-correctness invariant ‚Äî preserved

`INVOICE_SEND_ONLY_FIELDS` / `checkInvoiceSendOnly(Customer)` still hard-blocks invoice send when `tax_number` is missing on the customer regardless of `customerType`. This means an INDIVIDUAL customer can be activated without a SARS tax number (Day 28 fee-note generation can include them in `BillingRunSelectionService.discoverCustomers` because the `WHERE c.lifecycle_status = 'ACTIVE'` predicate now passes), but the moment the firm tries to **send** the invoice, the existing send-time check fires and surfaces the prerequisite violation. SARS issuance correctness is unchanged ‚Äî it just no longer leaks into activation semantics.
