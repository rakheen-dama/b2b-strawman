# OBS-2102 — Sipho activation blocked: Edit dialog button missing + tax_number prereq unsettable

**Filed**: 2026-04-30 (Cycle 14, Day 28 attempt)  
**Severity**: HIGH (blocks Day 28 fee note flow end-to-end)  
**Status**: OPEN

## Symptom

Day 28 (Bulk Billing for Sipho) blocked at the **first step**:

1. Bulk Billing run wizard step 2 ("Select Customers") shows
   *"No customers with unbilled work found for this period."*
2. Root cause = the eligible-customer SQL in
   `BillingRunSelectionService.discoverCustomers(...)` filters
   `WHERE c.lifecycle_status = 'ACTIVE'`. Sipho's lifecycle_status is
   `ONBOARDING`, so he is excluded even though he has 4h of unbilled
   billable time + a R 1,250 sheriff disbursement on RAF-2026-001.
3. Attempting to **Activate** Sipho (Customer detail → Change Status → Activate)
   surfaces a **prerequisite dialog**:
   *"Tax Number is required for Customer Activation. Fill the Tax Number
   field on the customer profile."* — `Check & Continue` button does not
   override.
4. Setting `tax_number` requires the **Edit Customer** dialog. **The Edit
   button is missing on Sipho's customer page** (it renders for Moroka but
   not Sipho — see comparison evidence). The customer profile inline-saves
   only the SA Legal field group (ID/Postal/Correspondence/Referred By —
   no tax_number field there).
5. SA Legal field group has no Tax Number field. The sidebar Document
   Templates section is read-only. The Onboarding tab does not surface
   tax_number.
6. There is **no UI path** to set Sipho's `tax_number`.

## Root cause hypothesis

Two distinct issues stacked:

### Issue A — Edit Customer button non-render (UI bug)

In `frontend/app/(app)/org/[slug]/customers/[id]/page.tsx` line 606:
```tsx
{customer.status === "ACTIVE" && customer.lifecycleStatus !== "ANONYMIZED" && (
  <>
    <EditCustomerDialog customer={customer} slug={slug}>
      <Button variant="outline" size="sm">
        <Pencil className="mr-1.5 size-4" />
        Edit
      </Button>
    </EditCustomerDialog>
    <ArchiveCustomerDialog ...>...Archive...</ArchiveCustomerDialog>
  </>
)}
```

For Sipho, `Archive` renders but `Edit` does not — even though both are
inside the same fragment. The DOM contains zero `>Edit<` text. RSC payload
includes the `EditCustomerDialog` chunk reference but the trigger button
never paints. Likely a Radix `DialogTrigger asChild` interaction with the
specific shape of Sipho's customer object, OR a hydration mismatch
silently dropping the trigger. Reproducible across Thandi (OWNER) sessions
and after hard reload. No client console errors, no server-side errors in
the frontend log, no React hydration warnings.

### Issue B — `tax_number` field hard-required for ACTIVE transition

`StructuralPrerequisiteCheck.LIFECYCLE_ACTIVATION_FIELDS` requires
`tax_number` for INDIVIDUAL customers with no escape hatch. South African
natural persons typically do **not** have a SARS tax number until SARS
registers them (children, low-income, etc.); the SA-ID number serves as
the de-facto identifier. Mandating `tax_number` for INDIVIDUAL ZA
customers is too strict.

The Day 2 onboarding flow in this QA cycle did not capture a tax_number
(scenario step 2 fields = SA-ID, name, email, phone, address — no tax).
So every customer onboarded via the canonical flow ends up un-activatable.

## Suggested fix scope

**Fix A (UI)** — Investigate why `EditCustomerDialog` trigger fails to
render for INDIVIDUAL customers in ONBOARDING status. Suspect:
- Radix `DialogTrigger asChild` swallowing on a malformed child
- A guard inside `EditCustomerDialog` that returns `null` on a specific
  customer shape (no such guard found in source — needs runtime trace)
- Server-component to client-component prop serialization issue (Customer
  shape varies by entityType)

**Fix B (data model)** — One of:
1. Drop `tax_number` from `LIFECYCLE_ACTIVATION_FIELDS` for INDIVIDUAL
   customers; keep for COMPANY/TRUST.
2. Accept `idNumber` (SA-ID) as a fallback for `tax_number` for
   INDIVIDUAL ZA customers.
3. Make Day 2 onboarding capture tax_number explicitly (with helper text
   "ID number for natural persons, SARS tax ref for companies").

## Workaround (if WONT_FIX) — scenario amendment

Day 2 onboarding scenario could mandate filling `tax_number` (= SA-ID for
INDIVIDUAL) and then immediately running the lifecycle transition
ONBOARDING → ACTIVE so that all downstream billing flows work. But this
still relies on **Issue A** (Edit dialog or set-on-create) so isn't a clean
amendment.
