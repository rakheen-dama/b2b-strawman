# OBS-2102 — Sipho activation blocked: Edit dialog button missing + tax_number prereq unsettable

**Filed**: 2026-04-30 (Cycle 14, Day 28 attempt)
**Severity**: HIGH (blocks Day 28 fee note flow end-to-end)
**Status**: SPEC_READY (refined 2026-04-30 by Product agent)

## Symptom

Day 28 (Bulk Billing for Sipho) blocked at the **first step**:

1. Bulk Billing run wizard step 2 ("Select Customers") shows
   *"No customers with unbilled work found for this period."*
2. Root cause = the eligible-customer SQL in
   `BillingRunSelectionService.discoverCustomers(...)` filters
   `WHERE c.lifecycle_status = 'ACTIVE'`. Sipho's lifecycle_status is
   `ONBOARDING`, so he is excluded even though he has 4h of unbilled
   billable time + a R 1,250 sheriff disbursement on RAF-2026-001.
3. Attempting to **Activate** Sipho (Customer detail → Change Status → Activate)
   surfaces a **prerequisite dialog**:
   *"Tax Number is required for Customer Activation. Fill the Tax Number
   field on the customer profile."* — `Check & Continue` button does not
   override.
4. Setting `tax_number` requires the **Edit Customer** dialog. **The Edit
   button is missing on Sipho's customer page** (it renders for Moroka but
   not Sipho — see comparison evidence).

## Triage findings (Product, 2026-04-30)

### Bug 1 — Edit Customer button missing: **the conditional gate is incomplete, not "broken"**

The current gate at `frontend/app/(app)/org/[slug]/customers/[id]/page.tsx:606`:

```tsx
{customer.status === "ACTIVE" && customer.lifecycleStatus !== "ANONYMIZED" && (
  <>
    <EditCustomerDialog customer={customer} slug={slug}>
      <Button …>Edit</Button>
    </EditCustomerDialog>
    <ArchiveCustomerDialog …>…Archive…</ArchiveCustomerDialog>
  </>
)}
```

`customer.status` is the basic record state (`ACTIVE | ARCHIVED`) and is
distinct from `customer.lifecycleStatus` (compliance lifecycle:
`PROSPECT | ONBOARDING | ACTIVE | …`). Both Sipho (lifecycle=ONBOARDING)
and Moroka (lifecycle=PROSPECT) have `customer.status === "ACTIVE"`, so
the conditional **should** evaluate true for both — yet only Moroka's
header paints the Edit button.

The likeliest explanation given the QA evidence (Archive renders, Edit
does not, no console errors, no hydration warnings) is one of:

- **Most likely** — a hydration/render exception inside
  `EditCustomerDialog`'s `useForm({ defaultValues: buildDefaults(customer) })`
  silently boundary-traps when the customer payload contains a value that
  fails the Zod resolver during initialisation. `buildDefaults` casts
  `customer.entityType` to `EditCustomerFormData["entityType"]` without
  validating against `ENTITY_TYPES`; `financialYearEnd` is forwarded
  raw to `z.iso.date()`. A Sipho-specific payload shape (e.g.
  `entityType: "INDIVIDUAL"` is in the enum but `financialYearEnd: ""`
  vs `null` may be bypass-OK; legacy `country` codes can also trip the
  `.length(2)` check — Sipho was Day 2 onboarded with country=`ZA` so
  this should be fine, but worth confirming).
- **Less likely** — Radix `DialogTrigger asChild` swallowing the trigger
  because of an unexpected child shape, but Archive uses the same
  pattern with its own dialog.

**Fix (recommended — addresses the user-visible blocker without depending on
the runtime root cause)**: relax the visibility gate so Edit follows the
same simple rule as the other admin actions in the row — *visible to any
admin/owner whenever the customer is not anonymised*. Tightening this
button to `status === "ACTIVE"` was historic and never necessary; an
ARCHIVED customer should still be editable for late corrections, and the
backend `updateCustomer` permissions are the source of truth.

```tsx
{customer.lifecycleStatus !== "ANONYMIZED" && (
  <EditCustomerDialog customer={customer} slug={slug}>
    <Button variant="outline" size="sm">
      <Pencil className="mr-1.5 size-4" />
      Edit
    </Button>
  </EditCustomerDialog>
)}
{customer.status === "ACTIVE" && customer.lifecycleStatus !== "ANONYMIZED" && (
  <ArchiveCustomerDialog …>…Archive…</ArchiveCustomerDialog>
)}
```

This change does NOT explain the runtime symptom (Sipho already passes
the *current* gate), but it **does** mean Edit always paints when the
admin row paints, eliminating the entire `EditCustomerDialog`
hydration-quirk class as a Day 28 blocker. **Dev MUST also reproduce the
runtime symptom locally**: hard-reload Sipho's customer page, confirm
Edit paints under the new gate, attach a screen capture to the
implementation note.

**File:** `frontend/app/(app)/org/[slug]/customers/[id]/page.tsx`
**Lines:** 606-628 (split the fragment as shown above)
**Effort:** S (10 min frontend edit + 1 vitest case asserting Edit renders for ONBOARDING customers).

### Bug 2 — `tax_number` activation prereq is too strict for INDIVIDUAL ZA customers

`backend/.../prerequisite/StructuralPrerequisiteCheck.java:63-68` defines:

```java
private static final List<FieldCheck> LIFECYCLE_ACTIVATION_FIELDS =
    List.of(
        new FieldCheck("address_line1", "Address Line 1", Customer::getAddressLine1),
        new FieldCheck("city", "City", Customer::getCity),
        new FieldCheck("country", "Country", Customer::getCountry),
        new FieldCheck("tax_number", "Tax Number", Customer::getTaxNumber));
```

These fields apply uniformly to every `Customer.customerType` value
(`INDIVIDUAL | COMPANY | TRUST`). For South African natural persons this
is wrong: SARS only issues tax numbers to registered taxpayers; minors,
low-income natural persons, and many newly-onboarded individuals do not
have one. The SA-ID number is the canonical national identifier for
INDIVIDUAL customers — already captured on the entity column
`Customer.idNumber` and surfaced in the SA Legal field group as
`id_passport_number`.

The doc-comment above the constant (lines 56-62) explicitly justifies
the requirement as "an ACTIVE customer is invoiceable and the tax-number
prerequisite is strictly enforced at send time" — **but invoice-send is
already gated separately** by `INVOICE_SEND_ONLY_FIELDS` /
`checkInvoiceSendOnly(...)` (line 75-76 + 165-182). So removing
`tax_number` from `LIFECYCLE_ACTIVATION_FIELDS` for INDIVIDUAL does NOT
weaken the SARS-correctness invariant on issued invoices: the customer
becomes activatable, time/disbursements can be billed, but the moment
the invoice is **sent** the `INVOICE_SEND_ONLY_FIELDS` check still hard-
blocks until `tax_number` is filled. That is exactly the GAP-L-62
hybrid-B+C semantic the comment refers to — it just wasn't applied
consistently to INDIVIDUAL activation.

For COMPANY/TRUST the requirement is correct as-is (real businesses do
have SARS tax numbers and we want activation to surface that gap early).

**Fix (recommended)**: make the `tax_number` check **conditional on
customerType** at activation time. Keep the structural list pure but
filter it during evaluation.

**File:** `backend/src/main/java/io/b2mash/b2b/b2bstrawman/prerequisite/StructuralPrerequisiteCheck.java`

**Concrete change**: inside `check(Customer, PrerequisiteContext, boolean)` (lines 109-139), add a per-field skip rule mirroring the existing `portalContactIdentitySatisfied` pattern:

```java
for (FieldCheck field : requiredFields) {
  if (portalContactIdentitySatisfied
      && context == PrerequisiteContext.PROPOSAL_SEND
      && (field.fieldSlug().equals("contact_name")
          || field.fieldSlug().equals("contact_email"))) {
    continue;
  }
  // Activation: INDIVIDUAL customers do not need tax_number — SARS does
  // not issue tax numbers to every natural person, and SA-ID
  // (id_passport_number / Customer.idNumber) is the canonical
  // identifier. Tax number is still hard-enforced at invoice send via
  // checkInvoiceSendOnly(...). See OBS-2102 / GAP-L-62.
  if (context == PrerequisiteContext.LIFECYCLE_ACTIVATION
      && field.fieldSlug().equals("tax_number")
      && customer.getCustomerType() == CustomerType.INDIVIDUAL) {
    continue;
  }
  if (!hasValue(field, customer, customFields)) {
    violations.add(…);
  }
}
```

Update the constant's javadoc (lines 56-62) to reflect the new policy.
Add `import io.b2mash.b2b.b2bstrawman.customer.CustomerType;`.

**Tests** (in `StructuralPrerequisiteCheckTest`):
- INDIVIDUAL with no `tax_number` → `LIFECYCLE_ACTIVATION` returns 0 violations on the tax_number slug (still flags `address_line1` / `city` / `country` if missing).
- INDIVIDUAL with no `tax_number` → `INVOICE_SEND_ONLY` (`checkInvoiceSendOnly`) still returns the violation. Asserts the SARS-correctness invariant on send.
- COMPANY with no `tax_number` → `LIFECYCLE_ACTIVATION` still returns the violation.
- TRUST with no `tax_number` → `LIFECYCLE_ACTIVATION` still returns the violation.

**Effort:** S (~30 min — 5-line backend change + 4 unit-test cases + javadoc update). Backend-only.

Two-PR fix or one combined PR:

- **Bug 1 (frontend, S)**: split the line-606 conditional so Edit paints whenever the row paints (any non-anonymised customer). Verify on Sipho.
- **Bug 2 (backend, S)**: skip the `tax_number` check inside `StructuralPrerequisiteCheck.check(...)` when `context == LIFECYCLE_ACTIVATION && customer.getCustomerType() == INDIVIDUAL`. Keep `INVOICE_SEND_ONLY_FIELDS` enforcement intact.
