## Implementation pattern

Both dialogs adopt the same `cloneElement` shape:

```tsx
// OBS-2103: avoid Radix `Slot` (`asChild`) here. The Edit dialog renders
// adjacent to <ArchiveCustomerDialog/> on the customer detail page; under
// React 19 + radix-ui, two unkeyed sibling Slots cloning a <Button> child
// (Dialog vs AlertDialog) collapse during commit and only one survives in
// the DOM. Inject the open handler via React.cloneElement instead so the
// child renders as a plain <Button> with no Slot wrapper.
const trigger = isValidElement<{ onClick?: (event: MouseEvent<HTMLElement>) => void }>(children)
  ? cloneElement(children as ReactElement<{ onClick?: (event: MouseEvent<HTMLElement>) => void }>, {
      onClick: (event: MouseEvent<HTMLElement>) => {
        children.props.onClick?.(event);
        if (!event.defaultPrevented) {
          setOpen(true);
        }
      },
    })
  : children;

return (
  <Dialog open={open} onOpenChange={handleOpenChange}>
    {trigger}
    <DialogContent>...</DialogContent>
  </Dialog>
);
```

Key properties:

- **No Radix `Slot` wrapper** — the trigger button is rendered as a plain `<Button>`, eliminating the React-19/Radix Slot adjacency collision class entirely.
- **Existing `onClick` chained** — if a consumer ever passes a Button with its own `onClick`, that handler runs first; `event.preventDefault()` short-circuits the dialog open. Today's call sites in `customer/[id]/page.tsx` pass plain Buttons with no `onClick`, so behaviour is identical to the prior `asChild` path.
- **Dialog still controlled** — `Dialog`'s own state (Esc key, overlay click, programmatic close from form-submit) drives `onOpenChange`, so close behaviour is preserved.
- **Imports trimmed** — `DialogTrigger` and `AlertDialogTrigger` removed from the component imports because they are no longer used. ESLint flags unused imports, so leaving them stale would have failed lint.

## Why not the alternatives in the spec

The spec offered three implementation paths:

- **A. Consumer wraps trigger with explicit `onClick` setter prop.** Requires rewriting two call sites in `customer/[id]/page.tsx` and changes the public API of both dialog components. Bigger blast radius. **Rejected.**
- **B. Render trigger inside the dialog component via `triggerLabel`/`triggerIcon` props.** Loses caller-side button styling control (`variant`, `size`, custom classnames). Two existing call sites use distinct visual treatments (Edit = `variant="outline"`, Archive = `variant="ghost"` + red text). **Rejected.**
- **C. `React.cloneElement` injection (chosen)** — smallest diff, no consumer rewrites, preserves the existing `<Button>{...}>` pattern, removes the Slot collision class.

## Tests

### New regression test — `__tests__/customer-detail-action-row.test.tsx`

Renders both dialogs adjacent (mirrors the bug-triggering shape on the customer detail action row):

```tsx
<div className="flex items-center gap-2">
  <EditCustomerDialog customer={activeCustomer} slug="acme">
    <Button>Edit Action Row</Button>
  </EditCustomerDialog>
  <ArchiveCustomerDialog slug="acme" customerId={activeCustomer.id} customerName={activeCustomer.name}>
    <Button>Archive Action Row</Button>
  </ArchiveCustomerDialog>
</div>
```

Four cases:

1. `renders both Edit and Archive trigger buttons adjacently` — both buttons must be present in the DOM. The OBS-2103 bug dropped one of them; this guards against regression.
2. `opens the Edit dialog when the Edit trigger is clicked` — clicking Edit renders the "Edit Customer" dialog title + pre-populated form.
3. `opens the Archive dialog when the Archive trigger is clicked` — clicking Archive renders the "Archive Customer" confirmation copy.
4. `does not collide — clicking Edit does not open Archive` — confirms the triggers are independent (not connected to the same `open` state).

## What the fix does NOT touch

Per spec guidance, `DataExportDialog` and `AnonymizeCustomerDialog` retain their existing `asChild` patterns:

- `DataExportDialog` is a `Dialog` primitive adjacent to other `Dialog` siblings — same-primitive Slots do not collide.
- `AnonymizeCustomerDialog` is an `AlertDialog` primitive but is gated on `isOwner && !ANONYMIZED` — when it renders, it sits between the prior block and the Edit/Archive pair, but it is not adjacent to a different-primitive `asChild` sibling because no Dialog primitive renders between it and its peers. Confirmed manually: only the Edit (Dialog) ↔ Archive (AlertDialog) adjacency triggered the bug.

If a future change introduces another adjacent Dialog↔AlertDialog `asChild` pair, the same `cloneElement` pattern should be applied.
