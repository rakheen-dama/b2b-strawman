# OBS-2103 — Customer detail action row: Edit and Archive buttons mutually exclusive

**Status**: SPEC_READY
**Severity**: MEDIUM (cosmetic/UX, not a workflow blocker — backend updateCustomer/archiveCustomer perms remain authoritative)
**Filed**: 2026-04-30, cycle 15 (during OBS-2102 verification)
**Reporter**: QA agent
**Component**: `frontend/app/(app)/org/[slug]/customers/[id]/page.tsx` action row (lines 590–629)

## Symptoms

On the customer detail page for both ONBOARDING-and-now-ACTIVE Sipho (INDIVIDUAL) and PROSPECT Moroka (TRUST), only one of Edit / Archive renders — never both — even though their server-rendered RSC payloads include both button nodes.

| Customer | status | lifecycleStatus | customerType | Action row buttons rendered |
|----------|--------|-----------------|--------------|-----------------------------|
| Sipho Dlamini | ACTIVE | ACTIVE | INDIVIDUAL | Change Status, Run Conflict Check, Generate Document, Export Data, Anonymize, **Archive** (no Edit) |
| Moroka Family Trust | ACTIVE | PROSPECT | TRUST | Change Status, Run Conflict Check, Generate Document, Export Data, Anonymize, **Edit** (no Archive) |

Per the source (after OBS-2102 fix merged in `a50be2a0`), both buttons should render whenever `customer.status === "ACTIVE" && customer.lifecycleStatus !== "ANONYMIZED"`, which is true for both customers.

## Reproduction

1. Log in as Thandi (owner) on `:3000`.
2. Navigate to Sipho's customer page → expect Edit + Archive both visible. Observed: only Archive.
3. Navigate to Moroka's customer page → expect Edit + Archive both visible. Observed: only Edit.

## Diagnosis (preliminary)

- RSC payload contains **both** button JSX nodes for Sipho (verified via raw `__next_f.push` stream — `49c:[..."button"..."Edit"]` is defined).
- The Edit button node `$L49c` IS referenced as the `children` prop of `EditCustomerDialog` (`$L49b`) in the RSC tree.
- DOM after hydration only has 6 children in `#lifecycle-transition`, missing one of Edit/Archive.
- No console errors logged on hydrate.

The rendering pattern toggles between the two — strongly suggesting a Radix UI/React 19 reconciliation issue where two adjacent `Dialog`/`AlertDialog` triggers with `asChild` collide on hydration. May be related to:
- Both `EditCustomerDialog` (uses `Dialog` from `radix-ui`) and `ArchiveCustomerDialog` (uses `AlertDialog`) wrapping `<Button>` via `asChild` slots without unique React keys.
- Hydration mismatch dropping one slot.

## Repro probe (for dev)

```javascript
// In browser console on Sipho's page:
document.querySelector('#lifecycle-transition').children.length
// returns 6 — should be 7
```

```javascript
document.body.innerHTML.match(/49c:.*?Edit/)
// returns the RSC payload entry — confirms server stream emits the button
```

## Suggested fix paths (for dev)

1. **Add explicit keys**: wrap each conditional button block with `<React.Fragment key="edit">` / `key="archive"`.
2. **Avoid `asChild` slot collision**: render the trigger Button as a plain child rather than via Radix `asChild` Slot.
3. **Investigate Radix bundled package `radix-ui`**: confirm `radix-ui@<version>` Dialog/AlertDialog Slot mechanics under React 19 hydration.

## QA workaround

Activation flow does NOT need Edit (uses Change Status → Activate dropdown, which works). For QA cycle Day 28 onward, this gap does not block customer activation or Bulk Billing.
