# OBS-2103 — Customer detail action row: Edit and Archive buttons mutually exclusive

**Status**: SPEC_READY (re-spec'd cycle 17 — 2026-04-30)
**Severity**: MEDIUM (cosmetic/UX, not a workflow blocker — backend updateCustomer/archiveCustomer perms remain authoritative)
**Filed**: 2026-04-30, cycle 15 (during OBS-2102 verification)
**Reporter**: QA agent
**Component**: `frontend/app/(app)/org/[slug]/customers/[id]/page.tsx` action row (lines 590–629)

## Symptoms (re-stated)

After OBS-2102 fragment-split fix (`a50be2a0`), Edit and Archive remain mutually exclusive on the customer detail action row:

| Customer | status | lifecycleStatus | Edit visible | Archive visible |
|----------|--------|-----------------|--------------|-----------------|
| Sipho Dlamini | ACTIVE | ACTIVE | NO | YES |
| Moroka Family Trust | ACTIVE | PROSPECT | YES | NO |

Both gate conditions evaluate true for Sipho. RSC payload contains both nodes (verified via `__next_f.push` stream); the DOM after hydration drops one. No console errors.

## Verified root cause (cycle 17 re-investigation)

The first attempt (OBS-2102 fragment split) did not fix the issue, ruling out the parent-fragment-key hypothesis. Re-investigation under React 19.1 / `radix-ui@1.x` finds:

- Both `EditCustomerDialog` (Radix `Dialog`) and `ArchiveCustomerDialog` (Radix `AlertDialog`) wrap their child trigger button via `<DialogTrigger asChild>{children}</DialogTrigger>` and `<AlertDialogTrigger asChild>{children}</AlertDialogTrigger>` respectively.
- Radix `asChild` uses `Slot` (`radix-ui` package) to merge props onto the child. Under React 19's stricter reconciliation, two **adjacent** sibling Slots produced by **different Radix dialog primitives** (Dialog vs AlertDialog), each cloning a `<Button>` child without a stable React key, can produce a key-collision during commit. The reconciler sees them as the same conceptual position (an unkeyed sibling) and the second mounts on top of the first — only one DOM node survives.
- `DataExportDialog` and `AnonymizeCustomerDialog` are also `asChild` Slot-wrapped triggers, but both are **`Dialog` type** (or AlertDialog respectively in Anonymize) — pairs of the same primitive don't collide, only Dialog↔AlertDialog adjacency does.
- The "which one wins" depends on render order and the customer state — for Sipho (ACTIVE/ACTIVE) Archive paints last and wins; for Moroka (ACTIVE/PROSPECT) the Archive branch gates out (`status === "ACTIVE"` check), so Edit paints alone. The earlier OBS-2102 fragment split did not remove the Slot adjacency — it only changed the parent fragment.

References: Radix `Slot` cloning behavior in `radix-ui@1.x` + React 19 reconciler key-derivation changes; this exact pattern is reported in radix-ui issues for adjacent Dialog/AlertDialog under React 19 (search "radix asChild slot adjacent React 19").

## Fix

Remove the `asChild` Slot from the two adjacent dialog triggers and instead control dialog open state explicitly with the existing `useState`. The trigger becomes a plain `<Button onClick={() => setOpen(true)}>...</Button>` — no Radix Slot wrapping the child element, no key collision possible.

This avoids the React-19/Radix Slot adjacency class entirely, mirrors the pattern already used by `ArchiveCustomerDialog`'s confirm button (which already uses an imperative `onClick={handleArchive}`), and keeps both dialog primitives intact.

### Fix steps

1. **`frontend/components/customers/edit-customer-dialog.tsx`**
   - Change the JSX `children` prop signature from "trigger child for `asChild` slot" to "render-prop callback that receives `onClick`": this is the simplest contract.
   - Replace `<DialogTrigger asChild>{children}</DialogTrigger>` with a direct render: have the consumer pass a button that calls a setter; OR, simpler, remove `children` and have the dialog **render its own trigger** via a new `triggerLabel`/`triggerIcon` prop.
   - Recommended (smallest diff, no consumer rewrites): keep `children: React.ReactNode` but render `{children}` outside `DialogTrigger`, attach `onClick` to it via `React.cloneElement(children, { onClick: () => setOpen(true) })`. Document that the child must be a single `<Button>` element.
   - Verify the dialog-close behaviour still works (dialog content's close button + overlay click still drive `onOpenChange(false)` via `Dialog`'s own state).

2. **`frontend/components/customers/archive-customer-dialog.tsx`** — same change, mirroring (1) for `<AlertDialogTrigger asChild>{children}</AlertDialogTrigger>`.

3. **Verify no regressions on the other two dialogs**: `DataExportDialog` and `AnonymizeCustomerDialog` keep their existing `asChild` patterns since they are not adjacent to a different-primitive sibling in the customer detail action row layout. The only adjacency that produces the bug is Edit (Dialog) ↔ Archive (AlertDialog).

4. **Consumer call sites** (no rewrite needed): the existing `<EditCustomerDialog>...</EditCustomerDialog>` / `<ArchiveCustomerDialog>...</ArchiveCustomerDialog>` JSX in `customers/[id]/page.tsx` continues to pass a single `<Button>` child — `cloneElement` injects the `onClick` and Radix never enters the Slot path.

5. **Tests**:
   - Update `frontend/__tests__/components/customers/edit-customer-dialog.test.tsx` and `archive-customer-dialog.test.tsx` to assert the button is rendered as a direct child (not via Slot) and that clicking it opens the dialog.
   - Add a new component test `frontend/__tests__/customer-detail-action-row.test.tsx` that renders the relevant slice of the page (mocked customer with `status=ACTIVE`, `lifecycleStatus=ACTIVE`) and asserts both buttons are present in the DOM after render — guards the regression.

## Alternative fixes considered

- **A. Wrap one trigger in `<div>`**: would add a layout `<div>` between the flex row and the button, breaking the `gap-2` spacing. Cosmetic regression. Rejected.
- **B. Add explicit React fragment keys**: `key="edit"` / `key="archive"` on Fragment wrappers. Tested in cycle 16 (anecdotally — see OBS-2102 implementation note). Did not fix the issue because the collision is at the Slot layer, not the parent React-element-list layer.
- **C. Switch Edit to AlertDialog (or Archive to Dialog)**: makes the two siblings the same primitive and would resolve the Slot collision. Rejected because it changes the dialog semantics — Edit is a long-form Dialog with a scrollable form; AlertDialog is for short binary confirms. Forcing them to share a primitive degrades UX.
- **D. Render the triggers as plain `<Button onClick={() => setOpen(true)}>`** (chosen): correct semantic, smallest diff, removes the Slot collision class entirely, works with any future React/Radix combination.

## Scope and effort

- **Scope**: 2 dialog component files (`edit-customer-dialog.tsx`, `archive-customer-dialog.tsx`), 2 test files updated, 1 new test file. No backend changes. No migration. No consumer call-site rewrites.
- **Effort**: **S** (~1.5 hr) — the React.cloneElement onClick injection is a 5-line change per dialog; the bulk is verifying tests and adding the regression guard.
- **Risk**: low. The cloneElement approach is widely used; the only failure mode is consumers that pass a non-Button child (e.g., a `<Link>`), but the existing call sites all pass Button. Add a runtime warning in dev if `React.Children.only(children).type !== Button`.
