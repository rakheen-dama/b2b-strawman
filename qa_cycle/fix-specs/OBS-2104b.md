# OBS-2104b — Bulk Billing wizard step 2: expense total inflated by Cartesian-product row multiplication

**Status**: SPEC_READY
**Severity**: LOW (cosmetic — display only; fee notes generate at the correct R 1,250 because the post-discovery generation path reads disbursements directly, no Cartesian)
**Filed**: 2026-04-30, cycle 16 (during OBS-2104 verification)
**Reporter**: QA agent
**Component**: `backend/src/main/java/io/b2mash/b2b/b2bstrawman/billingrun/BillingRunSelectionService.java` `discoverCustomers()` — lines 482–531 (post-PR-1238 SQL)

## Symptoms

After OBS-2104 fix (`f2da4e65`), wizard step 2 lists Sipho with **Unbilled Expenses R 11 250,00** (and Total R 11 250,00). Expected: R 1 250,00 (one approved sheriff disbursement at R 1,250 + R 0 VAT). Generated fee note's Subtotal is correct at R 1 250,00 — only the wizard-step-2 preview total is wrong.

QA-observed math: 1 disbursement × 9 RAF tasks ≈ 9 × R 1,250 = R 11,250. (Slight discrepancy from the 10-row math below is consistent with empty-task GROUP BY collapsing or a 9-task fixture vs the 10 rows; the multiplier is empirically `tasks_count`.)

## Verified root cause

`discoverCustomers()` runs a single flat query that LEFT JOINs five tables hanging off `projects p`:

```
FROM customers c
JOIN customer_projects cp ON cp.customer_id = c.id
JOIN projects p ON cp.project_id = p.id
LEFT JOIN tasks t ON t.project_id = p.id           -- N task rows per project
LEFT JOIN time_entries te ON te.task_id = t.id     -- M time-entry rows per task
LEFT JOIN expenses e ON e.project_id = p.id        -- E expense rows per project
LEFT JOIN legal_disbursements ld ON ld.project_id = p.id   -- D disbursement rows per project
GROUP BY c.id, c.name
```

For Sipho's RAF matter: `tasks=9`, `time_entries=2` (both on one task), `expenses=0`, `legal_disbursements=1`. The cross product is `9 × max(2, 1) × 1 ≈ 9–18 rows` per customer before the GROUP BY collapses.

Inside the SUM:

```sql
SUM(CASE WHEN ld.id IS NOT NULL THEN ld.amount + COALESCE(ld.vat_amount, 0) ELSE 0 END)
```

This SUM fires once per output row of the join. Because `legal_disbursements ld` is joined to `projects p` (not gated by task or time entry), `ld.id` is non-NULL on every row of the join product where the disbursement matches the period filters. So the disbursement amount gets summed `tasks_count` (or `tasks × time_entries` in worse cases) times instead of once.

The same bug applies symmetrically to `expenses e` SUM (multiplied by `tasks_count` × `disbursements_count`), and to the time-entry SUM (multiplied by `disbursements_count` if disbursements exist on the same project).

Currently the COUNT aggregates use `COUNT(DISTINCT te.id)`, `COUNT(DISTINCT e.id)`, `COUNT(DISTINCT ld.id)` — those are correct. Only the SUMs are wrong.

## Fix

Replace each per-table SUM with a `SUM(DISTINCT ...)`-equivalent — Postgres doesn't support `SUM(DISTINCT)` over computed expressions cleanly, so the standard pattern is to **aggregate each table in a CTE before joining**, or use scalar correlated subqueries.

### Path A — CTEs (recommended, single SQL pass)

Rewrite `discoverCustomers()` to use CTEs:

```sql
WITH customer_filter AS (
  SELECT c.id AS customer_id, c.name AS customer_name
  FROM customers c
  WHERE c.lifecycle_status = 'ACTIVE'
    -- AND c.id IN :customerIds (when provided)
),
time_agg AS (
  SELECT cp.customer_id,
         COUNT(DISTINCT te.id) AS time_count,
         COALESCE(SUM((te.duration_minutes / 60.0) * COALESCE(te.billing_rate_snapshot, 0)), 0) AS time_amount
  FROM customer_projects cp
  JOIN tasks t ON t.project_id = cp.project_id
  JOIN time_entries te ON te.task_id = t.id
  WHERE te.billable = true
    AND te.invoice_id IS NULL
    AND (te.billing_rate_currency = :currency OR te.billing_rate_currency IS NULL)
    AND te.date BETWEEN :periodFrom AND :periodTo
  GROUP BY cp.customer_id
),
expense_agg AS (
  SELECT cp.customer_id,
         COUNT(DISTINCT e.id) AS expense_count,
         COALESCE(SUM(e.amount * (1 + COALESCE(e.markup_percent, 0) / 100.0)), 0) AS expense_amount
  FROM customer_projects cp
  JOIN expenses e ON e.project_id = cp.project_id
  WHERE e.billable = true
    AND e.invoice_id IS NULL
    AND e.currency = :currency
    AND e.date BETWEEN :periodFrom AND :periodTo
  GROUP BY cp.customer_id
),
disbursement_agg AS (
  SELECT cp.customer_id,
         COUNT(DISTINCT ld.id) AS disbursement_count,
         COALESCE(SUM(ld.amount + COALESCE(ld.vat_amount, 0)), 0) AS disbursement_amount
  FROM customer_projects cp
  JOIN legal_disbursements ld ON ld.project_id = cp.project_id
  WHERE ld.billing_status = 'UNBILLED'
    AND ld.approval_status = 'APPROVED'
    AND ld.incurred_date BETWEEN :periodFrom AND :periodTo
  GROUP BY cp.customer_id
)
SELECT cf.customer_id, cf.customer_name,
       COALESCE(ta.time_count, 0) AS unbilled_time_count,
       COALESCE(ta.time_amount, 0) AS unbilled_time_amount,
       COALESCE(ea.expense_count, 0) + COALESCE(da.disbursement_count, 0) AS unbilled_expense_count,
       COALESCE(ea.expense_amount, 0) + COALESCE(da.disbursement_amount, 0) AS unbilled_expense_amount
FROM customer_filter cf
LEFT JOIN time_agg ta ON ta.customer_id = cf.customer_id
LEFT JOIN expense_agg ea ON ea.customer_id = cf.customer_id
LEFT JOIN disbursement_agg da ON da.customer_id = cf.customer_id
WHERE COALESCE(ta.time_count, 0) > 0
   OR COALESCE(ea.expense_count, 0) > 0
   OR COALESCE(da.disbursement_count, 0) > 0
ORDER BY cf.customer_name
```

Each CTE aggregates per-table in isolation — no cross-table cardinality bleed. Sums are exact regardless of how many tasks/time entries a project has.

### Path B — scalar correlated subqueries (alternative)

Replace the four LEFT JOINs with four correlated `SELECT` subqueries in the projection:

```sql
SELECT c.id AS customer_id, c.name AS customer_name,
       (SELECT COUNT(DISTINCT te.id) FROM ...) AS unbilled_time_count,
       (SELECT COALESCE(SUM(...), 0) FROM ...) AS unbilled_time_amount,
       ...
FROM customers c
WHERE c.lifecycle_status = 'ACTIVE'
HAVING ...
```

Easier to read for one-table changes but slower at scale (Postgres planner may not optimise four uncorrelated subqueries as efficiently as the CTE approach).

**Choose Path A.**
