## What changed

### Backend (4 production files + 1 test file + 1 controller + 1 service facade)
- `billingrun/EntryType.java` — added `LEGAL_DISBURSEMENT` enum value. **No migration** because `BillingRunEntrySelection.entryType` is `@Enumerated(EnumType.STRING)` (verified in entity); existing rows continue using `TIME_ENTRY` / `EXPENSE`.
- `billingrun/BillingRunSelectionService.java`:
  - Constructor now takes `DisbursementRepository`.
  - `createEntrySelections()` extended with a third native-query block (gated on `includeExpenses`) selecting from `legal_disbursements` filtered by `billing_status='UNBILLED' AND approval_status='APPROVED' AND incurred_date BETWEEN :periodFrom AND :periodTo` (mirroring the OBS-2104b CTE so the same set surfaces in step 3 that contributed to the step-2 unbilled total). Persists each as `BillingRunEntrySelection` of type `LEGAL_DISBURSEMENT`.
  - New `getUnbilledDisbursements(billingRunId, billingRunItemId)` — mirror of `getUnbilledExpenses` but resolves IDs against `disbursementRepository`.
  - `recalculateItemTotals()` — disbursement amounts (`amount + COALESCE(vatAmount, 0)`) roll into the `unbilledExpenseAmount` aggregate; counts add into `unbilledExpenseCount` (matches the discoverCustomers projection where expense + disbursement collapse into a single column).
  - New package-private `resolveSelectedDisbursementIds(BillingRunItem)` for the generation service.
- `billingrun/dto/BillingRunDtos.java` — new `DisbursementResponse` record (id, projectId, customerId, incurredDate, description, category, amount, vatAmount, vatTreatment, supplierName, **billableAmount**) with `from(LegalDisbursement)` factory.
- `billingrun/BillingRunService.java` — facade method `getUnbilledDisbursements(...)` delegating to selection service (with the `bulk_billing` module guard).
- `billingrun/BillingRunController.java` — new `GET /api/billing-runs/{id}/items/{itemId}/unbilled-disbursements` endpoint. `INVOICING` capability.
- `billingrun/BillingRunGenerationService.java` — call `resolveSelectedDisbursementIds()` and pass the list as the 5th arg of `CreateInvoiceRequest` (`disbursementIds`). The invoice creation pipeline already supports this via slice 487A — generated draft fee notes now auto-attach cherry-picked disbursements, eliminating the step-4 "Add Disbursements" modal workaround as the primary path.

### Frontend (4 files + 1 test file)
- `lib/api/billing-runs.ts` — `EntryType` union extended; new `UnbilledDisbursement` interface; new `getUnbilledDisbursements(id, itemId)` API call.
- `app/(app)/org/[slug]/invoices/billing-runs/new/billing-run-actions.ts` — new `getUnbilledDisbursementsAction(billingRunId, itemId)` server-action wrapper with the same admin/owner capability gate.
- `components/billing-runs/cherry-pick-step.tsx`:
  - `CustomerData` shape gains `disbursements`, `includedDisbursementIds`.
  - `loadCustomerData()` Promise.all now has a third arm calling `getUnbilledDisbursementsAction`.
  - Default `includedDisbursementIds` to all disbursement IDs (parity with time/expense).
  - New `toggleDisbursementEntry(itemId, entryId)` mirroring `toggleExpenseEntry` — pushes `{ entryType: "LEGAL_DISBURSEMENT", entryId, included }` to the debounced flush queue.
  - `getSubtotal()` includes disbursements (`billableAmount` already covers any VAT component).
- `components/billing-runs/cherry-pick-customer-detail.tsx` — new Disbursements table block (Date / Description / Category / Supplier / Amount + Include checkbox), only renders when `data.disbursements.length > 0`. Empty-state message now requires all three lists empty.

### Tests
- **Backend** `BillingRunDisbursementSelectionTest.java` (2 tests):
  - `getUnbilledDisbursements_returnsApprovedDisbursement` — provisions a tenant with the legal vertical, seeds an approved R 1,250 sheriff fee, advances a billing run to PREVIEW, asserts the new endpoint returns the disbursement with the right shape (id, description, supplierName, amount=1250, billableAmount=1250).
  - `updateSelection_excludeDisbursement_recalculatesTotals` — toggles `entryType=LEGAL_DISBURSEMENT, included=false` via the existing `PUT .../selections` endpoint and asserts `unbilledExpenseCount` drops to 0 and `unbilledExpenseAmount` to 0.
- **Frontend** extended `cherry-pick-step.test.tsx` with `renders disbursements section and recalculates subtotal on toggle` — proves the Disbursements table renders with the expected fields, the Include checkbox is wired, and the subtotal recalculates from R 1 250,00 → R 0,00 on toggle.
