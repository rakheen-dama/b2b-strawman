# OBS-2104c — Bulk Billing wizard step 3 (Cherry-Pick) missing Disbursements section

**Status**: SPEC_READY
**Severity**: LOW (cosmetic — workflow not blocked because the post-generation editor in step 4 offers `Add Disbursements` modal as a workaround)
**Filed**: 2026-04-30, cycle 16 (during OBS-2104 verification)
**Reporter**: QA agent
**Component**:
- Frontend: `frontend/components/billing-runs/cherry-pick-customer-detail.tsx` (renders TimeEntries + Expenses tables)
- Frontend: `frontend/components/billing-runs/cherry-pick-step.tsx` (loads customer data via `getUnbilledExpensesAction`)
- Backend: `backend/src/main/java/io/b2mash/b2b/b2bstrawman/billingrun/BillingRunSelectionService.java` `getUnbilledExpenses()` (lines 173–189) + `createEntrySelections()` (lines 559–633)

## Symptoms

After OBS-2104 fix (`f2da4e65`), step 3 (Review & Cherry-Pick) shows:
- **Time Entries** section listing Sipho's 2 time entries (R 0 each — OBS-2101 NULL-rate cascade).
- **No Disbursements section** — Sipho's R 1,250 sheriff fee never appears as a cherry-pickable line.

Workaround currently used by QA: continue past step 3 with only time entries selected → generate draft INV-0001 → in step 4 editor click `Add Disbursements` → manually attach the sheriff fee → Subtotal correct at R 1,250.

The whole point of OBS-2104 was to surface disbursements in the wizard. Step 2 (discovery) now shows them in the Unbilled Expenses column, but step 3 (cherry-pick) silently drops them — the user sees R 1,250 in step 2 then no way to deselect/include the underlying disbursement in step 3.

## Verified root cause

Two coupled issues in the backend, surfacing as missing UI in the frontend:

### 1. Backend: `legal_disbursements` are never persisted as `BillingRunEntrySelection` rows

`BillingRunSelectionService.createEntrySelections()` (called when the billing run advances to PREVIEW) only persists two entry types:

```java
// time_entries
selections.add(new BillingRunEntrySelection(billingRunItemId, EntryType.TIME_ENTRY, entryId));
// expenses (when includeExpenses=true)
selections.add(new BillingRunEntrySelection(billingRunItemId, EntryType.EXPENSE, entryId));
```

There is **no** code path that selects rows from `legal_disbursements` and inserts them as a third entry type. Consequence: even though step-2 discovery now finds the disbursement and inflates the customer's expense total, no `BillingRunEntrySelection` exists for it. Step 3's `getUnbilledExpenses(billingRunId, billingRunItemId)` reads only `BillingRunEntrySelection` rows of type `EXPENSE` and returns `expenseRepository.findAllById(...)` — which, by design, queries the `expenses` table only and returns nothing for disbursements.

`backend/src/main/java/io/b2mash/b2b/b2bstrawman/billingrun/BillingRunSelectionService.java:599-628` — disbursement insert missing.
`backend/src/main/java/io/b2mash/b2b/b2bstrawman/billingrun/BillingRunSelectionService.java:174-189` — `getUnbilledExpenses()` reads only `EntryType.EXPENSE`, only `expenseRepository`.

### 2. The `EntryType` enum likely needs `LEGAL_DISBURSEMENT`

The `BillingRunEntrySelection` entity uses an `EntryType` enum with `TIME_ENTRY` and `EXPENSE`. To track per-disbursement inclusion, we need a third value. Options:
- **Path A**: Add `LEGAL_DISBURSEMENT` enum value. Pure additive change (with Flyway migration if the column is enum-typed in DB).
- **Path B**: Reuse `EXPENSE` and store disbursement IDs there. Conflates two tables behind one type, breaks the existing `expenseRepository.findAllById(...)` resolver. Not recommended.

Choose Path A.

## Fix

### Backend changes

1. **Add `LEGAL_DISBURSEMENT` to `EntryType` enum** (path: search for `EntryType.java` in `billingrun/`). If the column is mapped as `@Enumerated(EnumType.STRING)` (typical), no migration needed — existing rows use `TIME_ENTRY` / `EXPENSE`, new rows can use `LEGAL_DISBURSEMENT`. Verify column type.

2. **Extend `createEntrySelections()`** (after line 628) to insert disbursement selections, mirroring the existing expense block:

```java
String disbursementSql = """
    SELECT ld.id
    FROM legal_disbursements ld
    JOIN projects p ON ld.project_id = p.id
    JOIN customer_projects cp ON cp.project_id = p.id
    WHERE cp.customer_id = :customerId
      AND ld.billing_status = 'UNBILLED'
      AND ld.approval_status = 'APPROVED'
      AND ld.incurred_date >= :periodFrom
      AND ld.incurred_date <= :periodTo
    """;
List<UUID> disbursementIds = ((List<Tuple>) entityManager
    .createNativeQuery(disbursementSql, Tuple.class)
    .setParameter("customerId", customerId)
    .setParameter("periodFrom", periodFrom)
    .setParameter("periodTo", periodTo)
    .getResultList())
    .stream().map(t -> t.get("id", UUID.class)).toList();
for (UUID entryId : disbursementIds) {
    selections.add(new BillingRunEntrySelection(billingRunItemId, EntryType.LEGAL_DISBURSEMENT, entryId));
}
```

Gate this block on a feature predicate (e.g. legal vertical enabled, or simply existence of the table) — the legal-vertical convention is that this code path is only exercised when `legal_disbursements` rows exist for the project. The query is harmless (returns empty) for non-legal tenants.

3. **Add `getUnbilledDisbursements(billingRunId, billingRunItemId)`** service method that filters `BillingRunEntrySelection` for `EntryType.LEGAL_DISBURSEMENT` and resolves IDs against `legalDisbursementRepository`. Mirror the existing `getUnbilledExpenses()` shape. Add a new controller endpoint:

   ```java
   @GetMapping("/{id}/items/{itemId}/unbilled-disbursements")
   public ResponseEntity<List<LegalDisbursementResponse>> getUnbilledDisbursements(...)
   ```

4. **`updateEntrySelection()`** (line 192–241) already uses `EntryType` from the request; no change needed once the enum has a third value — frontend can post `{ entryType: "LEGAL_DISBURSEMENT", entryId, included }` and selections persist correctly.

5. **`recalculateItemTotals()`** — verify it sums disbursement amounts into `unbilled_expense_amount` (it should already, since OBS-2104 routed disbursements into the expense aggregate). If it currently re-derives totals from `BillingRunEntrySelection`, extend it to also resolve LEGAL_DISBURSEMENT IDs and sum their `amount + vat_amount`.

### Frontend changes

6. **`lib/api/billing-runs.ts`** — add `getUnbilledDisbursements(billingRunId, itemId)` API call returning `UnbilledDisbursement[]` (fields: `id`, `date`, `description`, `category`, `billableAmount`).

7. **`billing-run-actions.ts`** — add `getUnbilledDisbursementsAction` server action wrapper.

8. **`cherry-pick-step.tsx`**:
   - Extend `CustomerData` shape with `disbursements: UnbilledDisbursement[]`, `includedDisbursementIds: Set<string>`.
   - In `loadCustomerData()` add a third `Promise.all` arm: `getUnbilledDisbursementsAction(billingRunId, itemId)`.
   - Default `includedDisbursementIds` to all disbursement IDs (mirrors existing behavior for time/expense).
   - Add `toggleDisbursementEntry()` mirroring `toggleExpenseEntry()` — pushes a `{ entryType: "LEGAL_DISBURSEMENT", entryId, included }` selection to the debounced flush queue.
   - Update `getSubtotal()` to include disbursements: `disbursementTotal = data.disbursements.filter(includes).reduce(...)`.

9. **`cherry-pick-customer-detail.tsx`**:
   - Add a third table block "Disbursements" between Time Entries and Expenses (or after Expenses — match the step-2 column ordering: Expenses then Disbursements is fine since they share the column in step 2).
   - Columns: Include checkbox, Date, Description, Category, Amount.
   - Renders only when `data.disbursements.length > 0` (parallel to existing time/expense gates).

10. **Empty-state guard**: update the "No unbilled entries for this customer" condition to also check `disbursements.length === 0`.

### Tests

11. Backend integration test: create matter with 1 time entry + 1 approved disbursement, advance billing run to PREVIEW, GET `/billing-runs/{id}/items/{itemId}/unbilled-disbursements` → asserts the disbursement is returned. POST `/updateSelections` with `entryType=LEGAL_DISBURSEMENT, included=false` → asserts the recalculated totals drop the disbursement amount.

12. Frontend component test: render `CherryPickCustomerDetail` with `data.disbursements = [...]` → asserts a Disbursements section renders with the right rows + a working Include checkbox.

## Alternative considered (and rejected)

- **Conflate disbursements into the existing Expenses section** by merging `legal_disbursements` rows into the `getUnbilledExpenses()` response. Rejected: confuses the two domain concepts (legal disbursements have a separate approval lifecycle, vat handling, supplier metadata that don't map to `expenses`); makes the cherry-pick checkbox semantically wrong because excluding an "expense" should not affect the disbursement table.

## Acceptance criteria

- [ ] Step 3 (Cherry-Pick) renders a Disbursements section listing approved-and-unbilled disbursements for the selected period.
- [ ] User can deselect a disbursement via checkbox; subtotal updates; backend persists `BillingRunEntrySelection` with `included=false` for that disbursement.
- [ ] Generated draft fee note in step 4 includes the selected disbursements **without** requiring the `Add Disbursements` modal workaround.
- [ ] When no disbursements exist, the section is hidden (parallel to existing Time/Expenses behaviour).
- [ ] New backend integration test passes.
- [ ] New frontend component test passes.
