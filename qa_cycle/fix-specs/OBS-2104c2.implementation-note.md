## Background

OBS-2104c (PR #1241) added the Disbursements section to the Bulk Billing
wizard cherry-pick step. QA cycle 17 verified the section renders
correctly but flagged a sidecar React 19 dev-console warning that fires
on every step-3 expand:

```
Cannot update a component (Router) while rendering a different
component (CherryPickStep). … setState() in render … CherryPickStep @
chunks/0v0__next_dist_0hl3d8v._.js:3272
```

Cosmetic only — does not affect render — but violates the "frontend
must run clean" mandate.

## Root Cause

`toggleSection(itemId)` in
`frontend/components/billing-runs/cherry-pick-step.tsx`:

```tsx
function toggleSection(itemId: string) {
  setExpandedIds((prev) => {
    const next = new Set(prev);
    if (next.has(itemId)) {
      next.delete(itemId);
    } else {
      next.add(itemId);
      if (!customerData[itemId]?.isLoaded && !customerData[itemId]?.isLoading) {
        loadCustomerData(itemId);    // ← side effect inside a state updater
      }
    }
    return next;
  });
}
```

Two compounding issues:

1. **State updater function impurity** — the `setExpandedIds((prev) => …)`
   updater runs an async dispatch (`loadCustomerData`) as a side effect.
   Under React 19 strict mode the updater can run twice during render,
   firing two server-action dispatches.
2. **`loadCustomerData` synchronously calls `setCustomerData(...)`** at
   line 92 (the "isLoading: true" placeholder) AND fires `Promise.all` of
   three server actions (`getUnbilledTimeAction`, `getUnbilledExpensesAction`,
   `getUnbilledDisbursementsAction`). Server actions in Next.js update the
   App Router cache, which is the "Router" component that the warning
   complains about.

The sequence: render computing next state → updater runs → setState
fires on `customerData` AND server actions schedule Router updates →
React detects a cross-component update during the render phase → warning.

## Fix Applied

Pull the side-effect OUT of the updater. Resolve `isCurrentlyExpanded`
synchronously beforehand from the live `expandedIds` snapshot, keep the
updater pure, and issue `loadCustomerData(itemId)` after committing the
toggle.

### Single File Changed

`frontend/components/billing-runs/cherry-pick-step.tsx`:

```tsx
function toggleSection(itemId: string) {
  // OBS-2104c2: keep the setState updater pure — never trigger side
  // effects (loadCustomerData → setCustomerData → server-action dispatch)
  // from inside a state-updater function.
  const isCurrentlyExpanded = expandedIds.has(itemId);
  setExpandedIds((prev) => {
    const next = new Set(prev);
    if (isCurrentlyExpanded) {
      next.delete(itemId);
    } else {
      next.add(itemId);
    }
    return next;
  });
  if (
    !isCurrentlyExpanded &&
    !customerData[itemId]?.isLoaded &&
    !customerData[itemId]?.isLoading
  ) {
    loadCustomerData(itemId);
  }
}
```

### Why this is safe

- `expandedIds` is a `useState` snapshot from the current render — reading
  `expandedIds.has(itemId)` is identical to what the updater would read
  on its first invocation. Subsequent invocations of the updater under
  strict mode would have re-read from the same `prev`, so behaviour is
  unchanged.
- `loadCustomerData` is `useCallback`-stable per `[billingRunId]`, so the
  identity check is fine.
- The toggle and the load dispatch in different microtasks now — React
  commits the `setExpandedIds` update first, then `loadCustomerData`
  fires its own `setCustomerData` from a regular event handler context
  (no longer "during render of CherryPickStep").
