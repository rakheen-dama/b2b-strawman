# OBS-2106 — Closure-pack portal-document-ready email never dispatched

**Status**: SPEC_READY
**Severity**: Medium (notification gap, not a closure-execution blocker)
**Filed**: Day 60, cycle 20 (2026-04-30)
**Branch**: `bugfix_cycle_2026-04-30b`

## Problem

After Day 60 happy-path matter close, both PDF artefacts are generated and visible on the portal Documents tab, but the `portal-document-ready` email is never sent to Sipho. `PortalEmailService.sendDocumentReadyEmail` (template `portal-document-ready`) records zero invocations across the entire boot — magic-link, weekly-digest, and trust-activity emails all log at INFO and ARE delivered, so the SMTP / template machinery is healthy. Only `PortalDocumentNotificationHandler` is silent.

Evidence:
- Backend log filter `template=portal-document-ready` → 0 lines since boot.
- Backend log filter `PortalDocumentNotificationHandler` → 0 lines since boot.
- Backend log shows `PdfRenderingService` + `GeneratedDocumentService` + `StatementService` all fired at `19:31:42` for the close transaction (`qa_cycle/checkpoint-results/day-60.md`).
- Mailpit (`http://localhost:8025/api/v1/messages`) — most recent message at the closure timestamp is the Day 60 prep R 71,000 trust-activity at `19:21:36`; no closure-pack email at the `19:31:42` window.

## Verified static-read findings

All preconditions for the handler to fire are satisfied:

1. **Allowlist** (`OrgSettings.portalNotificationDocTypes`): default seed `["matter-closure-letter", "statement-of-account"]` per Flyway V117. Both event template slugs match.
2. **Closure-letter event** publishes from `GeneratedDocumentService.generateDocument` (`backend/.../template/GeneratedDocumentService.java:208`):
   - `templateName = templateDetail.slug()` → `"matter-closure-letter"` ✓ allowlisted
   - `details = Map.of("file_name", ..., "template_name", templateDetail.name())` — **NO `scope` key, NO `visibility` key**. Forces the handler's DB-fallback visibility check.
3. **SoA event** publishes from `StatementService.generate` (`backend/.../verticals/legal/statement/StatementService.java:217`):
   - `templateName = template.getSlug()` → `"statement-of-account"` ✓ allowlisted
   - `details = Map.of(..., "scope", "PROJECT", "visibility", "PORTAL")` — passes the fast-path visibility check.
4. **Linked Document visibility** for the closure letter: `MatterClosureService.generateClosureLetterSafely` calls `documentService.markSystemAutoShared(linkedDocumentId)` inside the same `REQUIRES_NEW` transaction that emits the event. By the time `AFTER_COMMIT` fires, the persisted `Document.visibility = "PORTAL"`.
5. **Project → customer link**: RAF-2026-001 was created on Day 3 via the deep-link `?customerId=Sipho-UUID`, which sets both `Project.customerId` AND a `customer_projects` join row (`ProjectService.createProject` line 232–246). `project.getCustomerId()` returns Sipho's customer id.
6. **PortalContact for Sipho** exists with `email = sipho.portal@example.com`, `role = GENERAL`, `status = ACTIVE` (auto-provisioned by `PortalContactAutoProvisioner` at Day 2 customer create — `c99db0e9-…`). Magic-link login from this contact has worked in every prior cycle (Days 4, 5, 11, 21, 28, 30, 45, 46, 61). `findPreferredByCustomerIdAndOrgId` orders PRIMARY → BILLING → GENERAL, so a single GENERAL contact resolves.

So all 4 production gates in `PortalDocumentNotificationHandler.process` (allowlist / portal-visible / portal-contact resolve / dedup-first-wins) should pass for at least the SoA event (whose details map carries explicit `scope=PROJECT, visibility=PORTAL`). Yet the email never fires.

## Hypothesis

The handler IS being invoked — `NotificationEventHandler.onDocumentGenerated` (the sibling listener) fires successfully (the activity feed shows "Statement of Account generated by Thandi 7 minutes ago" on Sipho's `/activity` page), and Spring's `@TransactionalEventListener(AFTER_COMMIT)` dispatches all matching listeners. The skip is at one of the four DEBUG-level `log.debug("Skipping portal-document-ready: ...")` gates in `PortalDocumentNotificationHandler.process` (lines 150 / 156 / 162 / 173 / 182 / 207 / 218). Without DEBUG-level logs we cannot pin which one — and the integration test `PortalDocumentNotificationHandlerIntegrationTest` covers the listener in isolation but does NOT exercise the closure flow end-to-end (it publishes synthetic events directly). The closure-flow regression is therefore invisible to existing tests.

The most likely candidate, given that the activity-feed listener works and the SoA event has explicit `scope=PROJECT, visibility=PORTAL`, is **gate (3) — portal-contact resolve returns null** OR **a thrown exception in the inner `transactionTemplate.execute` is being swallowed by the outer `catch` at line 125–130** (the catch logs at WARN, but only for the outer `process()` call — it does NOT cover exceptions inside `transactionTemplate.execute(...)` that are caught by Spring's `TransactionTemplate.execute` and rethrown as `TransactionException`s).

## Fix (two-part: diagnostic + structural)

### Part 1 — Promote the 7 skip-path log lines from DEBUG to INFO

File: `backend/src/main/java/io/b2mash/b2b/b2bstrawman/portal/PortalDocumentNotificationHandler.java`

Replace `log.debug(...)` with `log.info(...)` on lines 150, 156, 162, 172–176, 182, 206–209, 216–219. Add a single INFO entry on entry (`log.info("PortalDocumentNotificationHandler.process entered: tenant={}, template={}, generatedDoc={}", event.tenantId(), event.templateName(), event.generatedDocumentId());`) so the entry-point firing can be confirmed.

This is the standard diagnostic uplift for a "we know it skips, we don't know which gate" defect. INFO-level skip logs are a one-time elevation; once the gate is identified and fixed, they can revert to DEBUG (or stay INFO if the operations team finds them useful — closure-pack mail-send misses are user-visible incidents worth surfacing).

### Part 2 — Remove the structural ambiguity that lets this defect happen

The closure-letter event publishes with an empty-of-visibility details map, which forces a DB fallback that is provably correct only after the visibility flip has committed. This is the kind of timing-coupled correctness that integration tests rarely catch. Two structural fixes; pick whichever is least invasive:

**Option A (preferred — single-file backend change):** publish the event AFTER the visibility flip in `MatterClosureService.generateClosureLetterSafely`. Today the event is published deep inside `generateForProject → generateDocument → eventPublisher.publishEvent(...)` BEFORE `markSystemAutoShared` runs. Repurpose the publication so the event carries explicit `details.scope = "PROJECT", details.visibility = "PORTAL"` (matching the SoA pattern at `StatementService.java:228–236`).

Implementation: extend `GeneratedDocumentService.generateDocument` to accept an optional `Visibility` override parameter; when called from `generateForProject`, pass `null` (existing behaviour). Add a `generateForProjectWithVisibility(UUID projectId, String slug, UUID actor, String visibility)` helper that the closure path uses, which:
1. Generates the doc, saves it with `Document.Visibility.PORTAL` directly (skipping the post-publish flip).
2. Publishes the `DocumentGeneratedEvent` with `details = Map.of(..., "scope", "PROJECT", "visibility", "PORTAL")`.

Then `MatterClosureService.generateClosureLetterSafely` calls `generateForProjectWithVisibility(...)` and removes the `markSystemAutoShared` post-step. This makes the closure-letter path self-describing in the event payload — no DB fallback required, no race window.

**Option B (smaller surface, safer):** keep the existing flow, but flip the order in `MatterClosureService.generateClosureLetterSafely`:

```java
// before
var result = generatedDocumentService.generateForProject(...);
// ... save closure log ...
documentService.markSystemAutoShared(linkedDocumentId);  // visibility flipped
// REQUIRES_NEW commits → AFTER_COMMIT fires
```

The event is published inside `generateForProject` BEFORE `markSystemAutoShared`. If we move the visibility flip to happen first, the event still publishes after the flip is in the persistence context — but since both run in the same `REQUIRES_NEW` transaction, the order of method calls inside the transaction is irrelevant to AFTER_COMMIT consumers.

The actual structural fix is to **publish the event from a place that already knows the visibility is PORTAL**. Concretely: add a `linkedDocument` reference to the `DocumentGeneratedEvent.details` map at publish time, and have the handler call `linkedDocument.getVisibility()` directly (rather than re-reading via DB). Since the `Document` is already persisted by the time the event is published (the `documentRepository.save(document)` at `GeneratedDocumentService.java:691` runs before the event publish at line 208), we can include the `linkedDocId` AND the post-flip `visibility` snapshot in the event details.

But this still doesn't solve the timing — the closure-letter case flips visibility AFTER the event is published. For Option B, the cleanest implementation is:

```java
// MatterClosureService.generateClosureLetterSafely — re-publish a DocumentGeneratedEvent AFTER the
// visibility flip, so the AFTER_COMMIT path sees the explicit PORTAL marker.
if (linkedDocumentId != null) {
  documentService.markSystemAutoShared(linkedDocumentId);
  // NEW: re-emit a "post-flip" event tagged for portal notification consumers only.
  // The event is filtered by templateName allowlist and visibility, so duplicate dispatch is
  // harmless (NotificationEventHandler already deduped its own activity entry on the first event).
  eventPublisher.publishEvent(buildPostFlipEvent(letterDocId, linkedDocumentId, ...));
}
```

This is uglier than Option A. **Recommend Option A.**

### Part 3 — Strengthen the integration test

`PortalDocumentNotificationHandlerIntegrationTest` covers the listener in isolation. Add a new test class `MatterClosureEmailIntegrationTest` (or extend `MatterClosureServiceIntegrationTest` if it exists) that:

1. Provisions a tenant, member (owner), customer (with email + auto-provisioned PortalContact), project (with customerId).
2. Sets up trust balance R 0, no open tasks, no court dates, no info requests, no unbilled disbursements (mirrors Day 60 prep state — use `TestEntityHelper` + direct repository writes for setup state where APIs aren't available).
3. Calls `MatterClosureService.close(...)` with `generateClosureLetter=true, generateStatementOfAccount=true`.
4. Asserts:
   - `greenMail.getReceivedMessages()` contains exactly 1 message (dedup coalesces SoA + closure-letter into a single email).
   - Recipient is the auto-provisioned PortalContact's email.
   - Subject matches "Document ready: …".
5. Re-runs with `generateClosureLetter=false, generateStatementOfAccount=true` — asserts 1 message (SoA only path).

This test would have failed at the time OBS-2106 was first introduced and would prevent silent regressions in future closure-flow refactors.
