## Symptom

`PortalDocumentNotificationHandler.process` skips every closure-pack /
Statement-of-Account `DocumentGeneratedEvent` for tenant
`tenant_5039f2d497cf` (Mathebula & Partners) at gate #1 with the INFO log
line:

```
Skipping portal-document-ready: per-tenant allowlist empty (tenant=tenant_5039f2d497cf)
```

Surfaced by the diagnostic-uplift introduced in PR #1246 (which promoted
the seven skip-path log lines from DEBUG to INFO). Before #1246 the gate
was invisible at the default log level.

Functional impact:
- `PortalEmailService.sendDocumentReadyEmail` never invoked.
- Closure-letter and SoA `portal-document-ready` emails silently dropped
  for any tenant whose `org_settings` row pre-existed Flyway V117.
- Client must discover the SoA via portal `/activity` Firm-actions tab
  (works correctly) or direct nav to `/projects/[id]` Documents tab.
- SoA + closure-letter PDFs ARE attached and downloadable from both firm
  and portal sides; ledger correctness is unaffected.
- **Not a closure-execution blocker.**

## Root Cause

> **Misdiagnosis notice (added 2026-05-01 by slop-hunt audit)**: this section's
> "Postgres DEFAULTs only apply to new INSERTs" narrative is factually wrong for
> `NOT NULL DEFAULT` (PG 11+ atomically backfills existing rows at ALTER time). The
> actual root cause is in the Java entity, not the migration: `OrgSettings.java:245`
> initializes `portalNotificationDocTypes = new ArrayList<>()` and the 1-arg
> constructor doesn't override it, so Hibernate persists `[]` on every INSERT and
> overrides the V117 SQL DEFAULT. V118's backfill repairs existing rows but every
> newly provisioned tenant recreates the bug. The write-time invariant fix is
> tracked separately at `qa_cycle/fix-specs/OBS-2107-followup.md`. Original
> (incorrect) narrative preserved below for traceability.

Flyway tenant migration `V117__add_portal_notification_doc_types.sql`
introduced the column with a JSONB `DEFAULT`:

```sql
ALTER TABLE org_settings
    ADD COLUMN IF NOT EXISTS portal_notification_doc_types JSONB
        NOT NULL DEFAULT '["matter-closure-letter", "statement-of-account"]'::jsonb;
```

Postgres column DEFAULTs only apply to **new INSERTs**. Existing
`org_settings` rows (created before V117 ran) had the column added
populated with the DEFAULT value at migration time only if the migration
explicitly used a backfill — but `ADD COLUMN ... DEFAULT` does populate
existing rows since Postgres 11. However the row in question shows
`portal_notification_doc_types = '[]'::jsonb` (empty JSON array, not
the default), which suggests either: (a) the row was inserted before
V117 with an explicit `'[]'`, or (b) the row predates V117 and was
backfilled to `'[]'` by some earlier seed/migration that ran after V117
on this specific tenant.

Either way the symptom is the same: the live row has an empty allowlist
and the handler correctly skips every event.

## Fix

**Flyway tenant migration V118** — backfill UPDATE that runs against
every tenant schema at backend startup.

File: `backend/src/main/resources/db/migration/tenant/V118__backfill_portal_notification_doc_types.sql`

```sql
UPDATE org_settings
SET portal_notification_doc_types = '["matter-closure-letter", "statement-of-account"]'::jsonb
WHERE portal_notification_doc_types IS NULL
   OR portal_notification_doc_types = '[]'::jsonb;
```

Idempotent — re-running has no effect because rows that already contain
the default (or any non-empty list a tenant has explicitly configured)
are skipped. Tenant overrides are preserved.

## Why Option A (V118 backfill), not Option B (Java fallback)

Option B (returning a static default from `OrgSettings.getPortalNotificationDocTypes()`
when the persisted value is null/empty) was considered as a defence-in-depth
safety net. Rejected for now because:

1. Database self-consistency is preferable — `SELECT portal_notification_doc_types
   FROM org_settings` should return the truth.
2. A read-time fallback would mask future bugs (e.g. a buggy admin UI
   that wipes the column to `[]` would be silently corrected at read
   time and never surface).
3. The existing Java getter already returns an empty list (not null)
   when the column is null, so adding more lenience there changes the
   contract for callers that legitimately want to disable per-event
   sends entirely with `[]`.

If we observe future regressions where the column gets wiped to empty,
we can revisit Option B at that point.
