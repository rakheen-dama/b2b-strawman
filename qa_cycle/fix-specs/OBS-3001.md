# Fix Spec: OBS-3001 — Fee-note (invoice) PDF download returns 404; no GeneratedDocument ever persisted

## Problem
On the portal fee-note detail page (`/invoices/[id]`) the **Download PDF** button fails for every fee note. Clicking it triggers `GET /portal/invoices/{id}/download` → **HTTP 404**, and the portal shows a JS `alert("Download failed. Please try again.")`. The same 404 applies to any caller (firm or portal) that resolves an invoice PDF via the `generated_documents` table. Day 30 checkpoint 30.8 (and the "Receipt download works" summary checkpoint) FAILED. Evidence: `qa_cycle/checkpoint-results/day-30.md` checkpoint 30.8.

This is **not a regression** from the 2026-06 simplification roadmap — the prior 2026-05-30 cycle marked 30.8 PARTIAL but only verified the Download button was *present*; it never clicked it, so the 404 was latent. This cycle exercised the button and observed the failure.

## Root Cause (confirmed)
No `GeneratedDocument` row is ever persisted for an invoice, so the download lookup finds nothing and 404s.

1. Portal download endpoint `PortalInvoiceController.downloadInvoice` (`customerbackend/controller/PortalInvoiceController.java` L112–118) → `PortalReadModelService.getInvoiceDownloadUrl` (`customerbackend/service/PortalReadModelService.java` L163–179):
   ```java
   var docs = generatedDocumentRepository
       .findByPrimaryEntityTypeAndPrimaryEntityIdOrderByGeneratedAtDesc(TemplateEntityType.INVOICE, invoiceId);
   if (docs.isEmpty()) {
     throw new ResourceNotFoundException("GeneratedDocument", invoiceId);  // → 404
   }
   ```
2. The only place an invoice PDF is rendered is the **email-send** path: `InvoiceEmailEventListener.handleInvoiceSent` (`invoice/InvoiceEmailEventListener.java` L67–72) calls `pdfRenderingService.generatePdf(templateId, invoiceId, actorMemberId)` and passes the **bytes** straight to `invoiceEmailService.sendInvoiceEmail(invoice, pdfResult.pdfBytes())`.
3. `PdfRenderingService.generatePdf` (`template/PdfRenderingService.java` L73–97) is **ephemeral** — it returns a `PdfResult(bytes, fileName, html)` and does **not** upload to S3 or persist a `GeneratedDocument`.
4. `InvoiceEmailService.sendInvoiceEmail` (`invoice/InvoiceEmailService.java` L48+) only attaches the bytes to the email.

Confirmed in DB: `tenant_5039f2d497cf.generated_documents` is **empty tenant-wide** (0 rows) despite the Day-28 log line `Generated PDF: template=invoice-cover-letter, entity=6ac267af-…, size=1184bytes`. The render happened; the persist never did.

## Fix
Persist a `GeneratedDocument` for the invoice PDF on the send path, reusing the existing full pipeline so the same artefact is both emailed and downloadable. There is already a managed-TX orchestrator that renders → uploads to S3 → persists `GeneratedDocument`: `GeneratedDocumentService.generateDocument(...)` (`template/GeneratedDocumentService.java` L146+, S3 key `org/{tenantId}/generated/{fileName}`), plus `resolveDefaultInvoiceTemplate()` (L393).

**Preferred option (single source of bytes):** In `InvoiceEmailEventListener.handleInvoiceSent`, replace the bare `pdfRenderingService.generatePdf(...)` call with `generatedDocumentService.generateDocument(templateId, invoiceId, actorMemberId, …)` and attach the returned PDF bytes to the email. This guarantees the emailed PDF and the downloadable PDF are byte-identical (one render, one S3 object, one `GeneratedDocument` row), and the portal `/download` then resolves the presigned URL for it.

- Confirm `GenerationResult` exposes the rendered bytes (or fetch them back from `storageService.generateDownloadUrl` / the stored key) so the email attachment is unchanged.
- Do **NOT** create a linked portal-visible `Document` or publish `DocumentCreatedEvent` for the invoice cover letter — that would surface the firm cover-letter in the portal documents read model (a behaviour change outside this gap). The portal already reads the PDF via the dedicated `/portal/invoices/{id}/download` endpoint against `generated_documents`; that is the intended channel. Persist the `GeneratedDocument` only.
- Keep the existing graceful-degradation: if no invoice template exists, the listener already logs and skips (L60–65) — preserve that; a missing template must not break send.

**Architectural decision for the orchestrator/user:** whether to (a) route through `GeneratedDocumentService.generateDocument` (recommended — no duplicate render), or (b) keep the ephemeral email render and *additionally* persist via a thin `generatedDocumentService` call. Option (a) is cleaner and avoids two PDFs drifting. Flagging as an architectural call per Quality Gate #7.
