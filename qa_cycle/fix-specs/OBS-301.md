# Fix Spec: OBS-301 — Matter description maxLength mismatch (frontend 2000 vs backend 255) + no field-level error UI

## Problem

Creating a matter from a legal-za template fails on first submit when the
template ships a description longer than 255 chars (the RAF Litigation
template's prefilled description is ~273 chars). The backend rejects the
request with HTTP 400 and a `fieldErrors` array, but the dialog only shows
the generic top-level message **"1 field(s) have validation errors"** —
the user has no way to see which field is wrong or what the limit is.

This is a real bug. Mandate: only KYC and Payments are exempt; this is not.

## Root Cause (verified, not hypothesized)

1. **Backend constraint is too tight on the template-instantiation path only.**
   `backend/.../projecttemplate/dto/InstantiateTemplateRequest.java:11` declares:
   ```java
   @Size(max = 255) String description,
   ```
   By contrast, the regular Create/Update Matter DTOs allow 2000 chars:
   - `backend/.../project/ProjectController.java:358` — `CreateProjectRequest.description` is `@Size(max = 2000, ...)`.
   - `backend/.../project/ProjectController.java:372` — `UpdateProjectRequest.description` is `@Size(max = 2000, ...)`.

   The DB column is `TEXT` with no length limit
   (`backend/src/main/resources/db/migration/tenant/V1__create_projects.sql:5` →
   `description TEXT,` and the JPA entity at `Project.java:48` is
   `@Column(name = "description", columnDefinition = "TEXT")`).

   So the 255 cap on InstantiateTemplateRequest is an inconsistency — every
   other path through the same column allows 2000 — and it blocks any seeded
   template description over 255 chars. The legal-za field pack ships
   templates with descriptions over 255 (RAF Litigation prefill = ~273
   chars), so template-driven creation is broken out-of-the-box for legal-za.

2. **Frontend has no field-level error display.** The dialog at
   `frontend/components/templates/NewFromTemplateDialog.tsx`:
   - Sets `<Textarea maxLength={2000}>` (line 314), so the form lets the user
     type up to 2000.
   - On submit, calls `instantiateTemplateAction(...)` (line 172) which
     wraps `ApiError` and returns `{ success: false, error: error.message }`
     (`frontend/app/(app)/org/[slug]/settings/project-templates/actions.ts:189-194`).
   - The component only renders `error` as a single string at line 411:
     `{error && <p className="text-destructive text-sm">{error}</p>}`.

   The backend's `GlobalExceptionHandler.handleMethodArgumentNotValid`
   (lines 156-180) puts per-field errors into the ProblemDetail's
   `fieldErrors` property — so the data is on the wire — but the action
   throws away `error.detail` and only surfaces `error.message` (which is
   the generic `"1 field(s) have validation errors"` `detail` string).

   The `ApiError` class (`frontend/lib/api/client.ts:47-56`) already keeps
   the full ProblemDetail on `error.detail`, and the type definition in
   `frontend/lib/types/common.ts:30-37` allows any extra property via
   `[key: string]: unknown` — so threading `fieldErrors` through is purely
   plumbing.

## Fix

**Chosen option: Hybrid (Option A + field-level error display).**

Why Hybrid (not pure A or pure B):
- **Why raise backend (not lower frontend):** The seeded legal-za templates
  already ship descriptions >255 chars (RAF prefill = ~273). Truncating
  them in seed data would lose authored content and would silently break
  any vertical pack that has long template descriptions. The DB column is
  already `TEXT` — there is no storage reason to cap at 255. Every other
  path that writes `Project.description` allows 2000.
- **Why also surface field-level errors:** Even after the limit alignment
  any future limit change (or any other future field-level validation
  failure on this dialog) will continue to black-box-fail through the
  generic message. Surfacing per-field errors is a one-time small frontend
  change that prevents the entire class of "1 field(s) have validation
  errors with no detail" UX bug from recurring.

### Step-by-step changes

**Backend** (one file, one annotation):

1. Edit `backend/src/main/java/io/b2mash/b2b/b2bstrawman/projecttemplate/dto/InstantiateTemplateRequest.java`:
   - Change `@Size(max = 255) String description,` → `@Size(max = 2000) String description,`
   - This brings InstantiateTemplateRequest into line with
     CreateProjectRequest / UpdateProjectRequest (both already at 2000).
   - No other DTOs in `projecttemplate/dto/` constrain description; no
     change needed elsewhere in the package.

2. Add a backend integration test that POSTs to the instantiate endpoint
   with a 1500-char description and asserts 201 (or whatever the success
   status is). Place it next to the existing template-instantiation tests
   (find by grepping for `instantiateProjectTemplate` or
   `/api/project-templates/.*/instantiate`). If a test already covers
   this endpoint, just extend it with a long-description case.

**Frontend** (two changes in two files):

3. Edit `frontend/app/(app)/org/[slug]/settings/project-templates/actions.ts`,
   in `instantiateTemplateAction` (lines 170-197):
   - Extend the action's return type:
     ```ts
     fieldErrors?: Array<{ field: string; message: string }>;
     ```
   - In the `error.status === 400` branch (lines 187-191), pull
     `error.detail?.fieldErrors` (cast through
     `unknown as Array<{ field: string; message: string }> | undefined`)
     and return it on the result. Keep the existing `error` string as a
     fallback for older callers.

4. Edit `frontend/components/templates/NewFromTemplateDialog.tsx`:
   - Add state: `const [fieldErrors, setFieldErrors] = useState<Record<string, string>>({});`
   - In `doCreateProject` (around line 165), reset `fieldErrors` on entry
     and, on a non-success result that includes `fieldErrors`, populate
     the map (`{ description: "must be at most 2000 characters", ... }`).
   - For each form field whose key may appear in `fieldErrors` (at minimum
     `description`, `name`, `referenceNumber`, `workType`), render a small
     `<p className="text-destructive text-xs mt-1">` under the input
     showing `fieldErrors[fieldName]` when present. The Description block
     starts at line 302.
   - Keep the existing top-level `{error && ...}` block as a fallback for
     non-validation errors.

5. (Optional, defensive) — also lower the frontend Textarea `maxLength` to
   match the backend (2000 stays the same after the bump, so this
   particular edit is a no-op; documenting that no change is required).
