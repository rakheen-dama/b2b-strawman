# Fix Spec: OBS-4004 — Automations page not found in settings navigation

## Problem

Day 0 checkpoint 0.42 could not locate the Automations page in the settings navigation. The QA agent reported that Settings nav has "Features" and "Packs" but no explicit "Automations" link. The automation rules from `automation-accounting-za` pack ARE seeded into the tenant, but the UI to view/manage them is hidden.

## Root Cause (confirmed)

The Automations settings link is gated behind the `automation_builder` module:

1. `frontend/lib/nav-items.ts` (line 436): `requiredModule: "automation_builder"` on the Automations settings entry
2. `frontend/components/settings/settings-nav-groups.ts` (line 45): same `requiredModule: "automation_builder"` gate
3. `frontend/components/settings/settings-sidebar.tsx` (line 64): filters out items where `requiredModule` is not in the org's enabled modules
4. `frontend/app/(app)/org/[slug]/settings/automations/page.tsx` (line 19): server-side module gate `isModuleEnabledServer("automation_builder")` returns a `ModuleDisabledFallback` component

The `accounting-za` vertical profile at `backend/src/main/resources/vertical-profiles/accounting-za.json` has:
```json
"enabledModules": ["deadlines", "information_requests"]
```

The `automation_builder` module is NOT in this list. Meanwhile, the profile DOES include `"automation": ["automation-accounting-za"]` in its packs, which seeds 4 automation rules into the tenant. This is a configuration inconsistency: the rules are seeded but the UI to manage them is hidden.

For comparison, `consulting-za.json` includes `"automation_builder"` in its enabled modules.

## Fix

Add `"automation_builder"` to the `enabledModules` array in the accounting-za vertical profile.

**File**: `backend/src/main/resources/vertical-profiles/accounting-za.json`

**Change** (line 3):
```json
// Before:
"enabledModules": ["deadlines", "information_requests"],

// After:
"enabledModules": ["deadlines", "information_requests", "automation_builder"],
```

This is a seed-data-only change. The module list is read at tenant provisioning time and stored in the tenant's `org_settings` table. For this QA cycle (clean slate), re-provisioning the tenant will pick it up. No migration needed.
