# Fix Spec: OBS-4006 â€” Trust-specific custom fields not rendering

## Problem

The "SA Accounting -- Trust Details" field group renders its section header on the Moroka Family Trust client detail page, but all 6 fields are hidden. The fields have a `visibilityCondition` of `{ dependsOnSlug: "acct_entity_type", operator: "eq", value: "TRUST" }`. The entity type IS set to TRUST on this customer, but the frontend evaluates the condition against an empty lookup and hides all fields. Evidence: `qa_cycle/evidence/day-15/moroka-active-trust-fields-empty.png`, `qa_cycle/evidence/day-15/trust-details-section-empty.png`.

## Root Cause

**Field promotion gap.** The slug `acct_entity_type` was promoted from a JSONB custom field to a first-class structural column `Customer.entityType` in Phase 63 (Epic 463). The promoted slug is listed in:
- `backend/.../template/PromotedFieldSlugs.java` (line 45): `"acct_entity_type"` in the `CUSTOMER` set
- `frontend/lib/constants/promoted-field-slugs.ts` (line 27): `"acct_entity_type"` in `PROMOTED_CUSTOMER_SLUGS`

The backend's `CustomerResponse.from()` (`backend/.../customer/dto/CustomerDtos.java`, line 135) returns `customer.getCustomFields()` -- the raw JSONB map that no longer contains `acct_entity_type`. The promoted value is returned separately as `entityType` (line 152).

The template system already handles this via `CustomerContextBuilder.injectPromotedCustomerAliases()` (line 210), which injects `acct_entity_type` back into the customFields map for template rendering. But the REST API customer endpoint does NOT perform this injection.

On the frontend, `CustomFieldSection` (`frontend/components/field-definitions/CustomFieldSection.tsx`) initializes its values state from `customer.customFields ?? {}` (passed via the detail page at line 741). The `isFieldVisible()` function (line 179-202) looks up `condition.dependsOnSlug` ("acct_entity_type") in this map, gets `undefined`, and returns `false` -- hiding all 6 trust fields.

The same `isFieldVisible` logic in `intake-fields-section.tsx` (line 13-36) has the identical gap, but it only affects the create-customer dialog. Since the create dialog populates `acct_entity_type` as a live form value during entity type selection, the bug manifests only on the **detail page** where values come from the persisted `customFields` map.

## Fix

The fix is a **frontend-only** change. The `CustomFieldSection` component needs access to promoted field values to evaluate visibility conditions that reference promoted slugs. Two options:

### Option A: Pass promoted values into CustomFieldSection (Recommended -- minimal, no backend change)

1. **`frontend/components/field-definitions/CustomFieldSection.tsx`**
   - Add an optional `promotedFieldValues` prop of type `Record<string, unknown>` to `CustomFieldSectionProps`.
   - In the component body, merge promoted values into the working `values` map for visibility evaluation. The merge must use promoted values as the base (so they can be overridden by any actual customField entries):
     ```typescript
     // Merge promoted field values for visibility condition evaluation
     const effectiveValues = { ...promotedFieldValues, ...values };
     ```
   - Pass `effectiveValues` instead of `values` to all `isFieldVisible()` calls (lines 479, 541).
   - Do NOT use `effectiveValues` for field data editing/saving -- only for visibility evaluation. The save action should continue sending only `values` (the actual custom field map).

2. **`frontend/app/(app)/org/[slug]/customers/[id]/page.tsx`**
   - Build a promoted-field-values map from the customer entity and pass it to `CustomFieldSection`:
     ```typescript
     const promotedFieldValues: Record<string, unknown> = {};
     if (customer.entityType) promotedFieldValues["acct_entity_type"] = customer.entityType;
     if (customer.entityType) promotedFieldValues["client_type"] = customer.entityType;
     if (customer.taxNumber) promotedFieldValues["tax_number"] = customer.taxNumber;
     if (customer.taxNumber) promotedFieldValues["vat_number"] = customer.taxNumber;
     if (customer.registrationNumber) promotedFieldValues["acct_company_registration_number"] = customer.registrationNumber;
     if (customer.registrationNumber) promotedFieldValues["registration_number"] = customer.registrationNumber;
     // ... other promoted slugs as needed
     ```
   - Pass `promotedFieldValues={promotedFieldValues}` to `<CustomFieldSection>`.

**Why not a backend fix?** Injecting promoted slugs back into the `customFields` JSON in the REST response would blur the boundary between structural columns and dynamic fields. The template builder does it because templates reference `customFields.slug` by convention. The REST API should keep them separate -- the frontend is the right place to bridge the gap for visibility evaluation.
