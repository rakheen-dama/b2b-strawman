# Fix Spec: OBS-4007 — Budget Alert automation SEND_NOTIFICATION fails: no PROJECT_OWNER recipients

## Problem

The `accounting-za-budget-alert` automation rule (from `automation-templates/accounting-za.json`) fires on `BUDGET_THRESHOLD_REACHED` and targets `recipientType: "PROJECT_OWNER"`. When the Year-End Pack engagement hit 83% budget consumption, the automation executed but the SEND_NOTIFICATION action failed with "No recipients resolved for type: PROJECT_OWNER".

The root cause is twofold:
1. The Year-End Pack engagement has 0 members with the `lead` project role — members Carol and Bob were added via `addMember()` which assigns `Roles.PROJECT_MEMBER` ("member"), and no lead transfer was ever performed.
2. The `SendNotificationActionExecutor.resolveRecipients()` queries for `"LEAD"` (uppercase) but `Roles.PROJECT_LEAD` is `"lead"` (lowercase), so even if a lead existed, the query would return empty.

The direct `NotificationService.handleBudgetThreshold()` fallback succeeded because it uses a broader recipient set (project leads + org admins/owners), which is why Thandi still received the budget alert notification.

## Root Cause (hypothesis)

**Bug 1 — Case mismatch (code defect):**
- File: `backend/src/main/java/io/b2mash/b2b/b2bstrawman/automation/executor/SendNotificationActionExecutor.java`, line 103
- Code: `projectMemberRepository.findByProjectIdAndProjectRole(projectId, "LEAD")` — hardcoded uppercase `"LEAD"`
- But: `backend/src/main/java/io/b2mash/b2b/b2bstrawman/security/Roles.java`, line 18: `PROJECT_LEAD = "lead"` — lowercase
- The `ProjectMemberService.transferLead()` stores the role as `"lead"` (lowercase via `Roles.PROJECT_LEAD`), so the `"LEAD"` query will never match.

**Bug 2 — No fallback when project has no lead (design gap):**
- `SendNotificationActionExecutor` line 104: `yield leads.isEmpty() ? List.of() : ...` — returns empty list, causing immediate `ActionFailure`.
- The `NotificationService.handleBudgetThreshold()` (direct notification path) handles this correctly by falling back to org admins/owners (lines 507-511).
- The automation executor should mirror this fallback: if no project lead, fall back to org admins/owners (matching the behavior of the direct notification path).

## Fix

### Step 1: Fix case mismatch
In `SendNotificationActionExecutor.java`, change line 103 from:
```java
var leads = projectMemberRepository.findByProjectIdAndProjectRole(projectId, "LEAD");
```
to:
```java
var leads = projectMemberRepository.findByProjectIdAndProjectRole(projectId, Roles.PROJECT_LEAD);
```
Add import for `io.b2mash.b2b.b2bstrawman.security.Roles`.

### Step 2: Add fallback for PROJECT_OWNER when no lead exists
In `SendNotificationActionExecutor.java`, modify the `"PROJECT_OWNER"` case (lines 98-105) to fall back to org admins/owners when no project lead is found:
```java
case "PROJECT_OWNER" -> {
    UUID projectId = resolveProjectId(context);
    if (projectId == null) {
        yield List.of();
    }
    var leads = projectMemberRepository.findByProjectIdAndProjectRole(projectId, Roles.PROJECT_LEAD);
    if (!leads.isEmpty()) {
        yield List.of(leads.getFirst().getMemberId());
    }
    // Fallback: org admins/owners (mirrors NotificationService.handleBudgetThreshold)
    yield memberRepository.findByRoleSlugsIn(List.of("admin", "owner")).stream()
        .map(m -> m.getId())
        .toList();
}
```

### Step 3: Add regression test
Add a test to `AutomationActionExecutorTest.java` (or a new `SendNotificationActionExecutorTest.java`) covering:
1. `PROJECT_OWNER` resolves to lead when lead exists (using lowercase "lead" role)
2. `PROJECT_OWNER` falls back to org admins when no lead exists
