# Fix Spec: OBS-4010 — Portal does not respect vertical-profile terminology overrides

## Problem

The portal app (localhost:3002) displays hardcoded generic/legal-default terminology instead of the tenant's vertical-specific terms. For an accounting-za tenant:

1. **Sidebar**: Shows "Matters" instead of "Engagements" (the `portal.nav.matters` label resolves to the hardcoded `PORTAL_NAV_LABELS` map value without passing through `t()`).
2. **Projects list page heading**: Shows "Your Projects" instead of "Your Engagements" (hardcoded string in `projects/page.tsx`).
3. **Project detail breadcrumbs**: "Back to projects" instead of "Back to engagements" (hardcoded in `projects/[id]/page.tsx`).
4. **Project detail error state**: "Project not found" and "This project may have been removed..." use hardcoded "project".
5. **Project detail summary card**: "Project Summary" hardcoded.
6. **Empty state text**: "No projects yet", "No projects", "...share projects with you here" all hardcoded.
7. **Proposals page**: "Proposals" heading, "No proposals yet", "Past Proposals", "Proposal #", "Proposal list" aria-label, "Back to proposals" — all hardcoded. Should be "Engagement Letters" for accounting-za.
8. **Proposal detail**: "Back to proposals", "Accept Proposal", "Proposal Details", "Proposal has been declined/accepted/expired" — all hardcoded.
9. **Retainer page/detail**: "Retainer" label in sidebar and `feeModelLabel()` function — should translate per vertical.

The firm-side frontend (localhost:3000) correctly uses `t()` from `useTerminology()` for all these terms.

## Root Cause

The portal has a working `TerminologyProvider` + `useTerminology()` hook (identical to the frontend's implementation) and it IS wired into the authenticated layout (`app/(authenticated)/layout.tsx` line 38). The `terminology-map.ts` has correct mappings for all three verticals.

**The bug is that `t()` is only called in a few places (invoices pages, home page invoices card) but NOT in the sidebar nav labels, project pages, proposal pages, or other pages with translatable terms.**

Specifically in the sidebar (`portal-sidebar.tsx` line 70-72):
```tsx
{item.labelKey === "portal.nav.invoices"
  ? t(PORTAL_NAV_LABELS[item.labelKey])
  : (PORTAL_NAV_LABELS[item.labelKey] ?? item.labelKey)}
```

Only the invoices label is wrapped in `t()`. All other nav labels bypass translation entirely.

## Fix

### Part 1: Sidebar nav labels (portal-sidebar.tsx)

Remove the invoices-only special case. ALL nav labels should pass through `t()`:

**File**: `portal/components/portal-sidebar.tsx`, lines 69-73

Replace:
```tsx
<span className="truncate">
  {item.labelKey === "portal.nav.invoices"
    ? t(PORTAL_NAV_LABELS[item.labelKey])
    : (PORTAL_NAV_LABELS[item.labelKey] ?? item.labelKey)}
</span>
```

With:
```tsx
<span className="truncate">
  {t(PORTAL_NAV_LABELS[item.labelKey] ?? item.labelKey)}
</span>
```

This works because `t()` is an identity function for terms not in the map — so "Home", "Trust", "Deadlines" etc. pass through unchanged, while "Matters" maps to "Engagements" (accounting-za) or stays as "Matters" (legal-za), "Invoices" maps to "Fee Notes" (legal-za), "Proposals" maps to "Engagement Letters" (accounting-za), etc.

### Part 6: Terminology map additions

**File**: `portal/lib/terminology-map.ts`

Add missing entries needed by the portal for accounting-za and legal-za:

For **accounting-za**, add:
```
"Proposal #": "Engagement Letter #",
"Proposal list": "Engagement Letter list",
```

For **legal-za**, add:
```
"Proposal #": "Engagement Letter #",
"Proposal list": "Engagement Letter list",
```

These compound terms are needed because the portal constructs strings like `${t("Proposal")} #` which translates to "Engagement Letter #". Actually, the simple `t("Proposal")` --> "Engagement Letter" mapping already exists, so constructing the label with `{t("Proposal")} #` will produce "Engagement Letter #" naturally. No additional map entries are needed.
