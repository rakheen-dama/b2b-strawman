# Fix Spec: OBS-5001 — Engagements empty state uses "projects" terminology

## Problem

On the Engagements list page (accounting-za vertical), the empty state reads "No projects yet" and "Projects organise your work, documents, and time tracking. Create your first project to get started." The page title and sidebar correctly say "Engagements" (via `TerminologyHeading`), but the empty state body bypasses the terminology system. Additionally, the automation rule "New Project Welcome" from the `common.json` pack is seeded with hardcoded "Project" text rather than vertical-appropriate terminology.

Evidence: Day 0 checkpoint Phase H (0.47) in `qa_cycle/checkpoint-results/day-00.md`.

## Root Cause (confirmed)

**Sub-issue 1 — Frontend empty state text:**

The projects list page (`frontend/app/(app)/org/[slug]/projects/page.tsx`, lines 243-250) renders the empty state using `createMessages("empty-states")` which reads from `frontend/lib/messages/en/empty-states.json`. The messages at `projects.list.heading` / `projects.list.description` / `projects.list.descriptionMember` contain hardcoded "project(s)" strings without `{Projects}` / `{projects}` terminology placeholders.

Compare with the customers page (`frontend/app/(app)/org/[slug]/customers/page.tsx`, lines 286-298) which correctly wraps the title in `<TerminologyText template="No {customers} yet" />` and the description in `<TerminologyText template={t("customers.list.description")} />`.

The projects page passes raw message strings directly to `<EmptyState>` without wrapping them in `<TerminologyText>`.

Secondary instance: `frontend/components/dashboard/kpi-card-row.tsx` line 69 has hardcoded `emptyState: "No projects yet"` (though this string is less visible since it's the KPI card's zero-value state, not the main empty state).

**Sub-issue 2 — Automation rule name:**

The "New Project Welcome" automation rule is defined in `backend/src/main/resources/automation-templates/common.json` (line 84). The `common.json` pack has no `verticalProfile` field, so it applies to ALL tenants regardless of vertical. The `name` and `description` fields use "Project" / "project" hardcoded. The seeder (`AutomationTemplateSeeder.java`) inserts the name verbatim — there is no terminology substitution at seed time.

## Fix

### Sub-issue 1: Frontend empty state (MAIN FIX)

**Step 1.** In `frontend/lib/messages/en/empty-states.json`, update the `projects.list` messages to use `{Projects}` / `{projects}` / `{project}` terminology placeholders (curly brace format matching `TerminologyText`):

```json
"projects": {
  "list": {
    "heading": "No {projects} yet",
    "description": "{Projects} organise your work, documents, and time tracking. Create your first {project} to get started.",
    "descriptionMember": "You’re not on any {projects} yet.",
    "cta": "Create {project}"
  }
}
```

**Step 2.** In `frontend/app/(app)/org/[slug]/projects/page.tsx`, wrap the empty state title and description in `<TerminologyText>`, matching the pattern used by the customers page:

Change lines 243-250 from:
```tsx
<EmptyState
  icon={FolderOpen}
  title={t("projects.list.heading")}
  description={
    isAdmin ? t("projects.list.description") : t("projects.list.descriptionMember")
  }
  action={<CreateProjectDialog slug={slug} />}
  secondaryLink={{ label: "Read the guide", href: docsLink("/features/projects") }}
/>
```

To:
```tsx
<EmptyState
  icon={FolderOpen}
  title={<TerminologyText template={t("projects.list.heading")} />}
  description={
    <TerminologyText
      template={isAdmin ? t("projects.list.description") : t("projects.list.descriptionMember")}
    />
  }
  action={<CreateProjectDialog slug={slug} />}
  secondaryLink={{ label: "Read the guide", href: docsLink("/features/projects") }}
/>
```

**Step 3.** Add `TerminologyText` import to the projects page (it already imports `TerminologyHeading` from `@/components/terminology-heading` — add `TerminologyText` from `@/components/terminology-text`).

**Note:** `TerminologyText` is a `"use client"` component being rendered inside a Server Component (`ProjectsPage`). This is valid because it's passed as a `ReactNode` prop (JSX), not a function/component reference. The `EmptyState` component accepts `React.ReactNode` for both `title` and `description`.

**Step 4.** In `frontend/components/dashboard/kpi-card-row.tsx`, line 69, replace the hardcoded `emptyState: "No projects yet"` with `emptyState: t("No projects yet")` — or better, add `"No {projects} yet"` as a terminology-aware string. Since `kpi-card-row.tsx` is already a `"use client"` component that calls `useTerminology()` and uses `t()` for labels (line 63), simply change:

```tsx
emptyState: "No projects yet",
```
to:
```tsx
emptyState: `No ${t("projects")} yet`,
```

**Step 5.** Update tests in `frontend/app/(app)/org/[slug]/projects/projects-page.test.tsx` to match the new terminology-aware text. Since the tests don't wrap in a `TerminologyProvider`, the default identity translation returns the template key unchanged, so the rendered text will contain the `{projects}` placeholder tokens. Update assertions:

- Line 265: `"No projects yet"` stays as-is (default terminology returns "projects" for "projects")
- Line 266: `/not on any projects yet/` stays as-is
- Line 282: `/Projects organise your work/` stays as-is

Actually, since there's no `TerminologyProvider` in the tests, the `TerminologyText` component will use the default identity `t()` which returns the term as-is. So `{projects}` will be replaced by `projects` (the key itself), resulting in the same rendered text. The test assertions should not need changes.

**However**, verify by checking whether the test file wraps renders in `TerminologyProvider`. If not, `useTerminology()` falls back to identity function `(term) => term`, so `{projects}` becomes `projects` and the rendered text is identical. No test changes needed.

### Sub-issue 2: Automation rule name (BACKEND)

**Step 1.** In `backend/src/main/resources/automation-templates/common.json`, change the "New Project Welcome" rule (line 84-103) to use generic terminology that works across all verticals. Since the `common.json` pack has no verticalProfile and applies to all tenants, the name should be generic:

Change:
```json
"name": "New Project Welcome",
"description": "Sends a welcome notification to project members when a project becomes active",
```
To:
```json
"name": "New Engagement Welcome",
"description": "Sends a welcome notification to team members when an engagement becomes active",
```

Wait -- "Engagement" is accounting-za specific. The generic term is "Project." The issue is that there's no backend terminology system for automation rule display names.

**Revised approach:** Since this is the `common` pack (no vertical profile, applies to all), the name should stay generic ("Project"). The real fix for correct display is either:
- (a) Add an `accounting-za.json` override for this rule with name "New Engagement Welcome", OR
- (b) Apply frontend terminology translation when rendering automation rule names.

Option (a) is cleaner and matches the existing pack architecture. But the `common.json` pack already seeds the rule for all tenants, so the `accounting-za.json` pack would need to override the existing rule, which the seeder doesn't support (it creates new rules, doesn't update existing ones).

Option (b) is the correct long-term fix but requires frontend changes to the automations settings page.

**Pragmatic fix:** Change the `common.json` rule's display name/description to use generic but vertical-neutral language:

```json
"name": "Welcome Notification",
"description": "Sends a welcome notification to team members when work becomes active",
```

This avoids "Project" or "Engagement" entirely while remaining meaningful for all verticals.

**However**, the rule is already seeded in the database for the QA tenant. Changing the JSON won't update existing seeded rules. The automation rule name in the DB needs to be updated via the UI or API. For the fix to take effect on new tenants, update the JSON. For existing tenants, a data migration or manual update is needed.

Since the mandate says "No production data — backward data compatibility is NOT a priority", and the QA cycle will wipe and re-seed, changing the JSON template is sufficient.

**Step 1 (revised).** In `backend/src/main/resources/automation-templates/common.json`, update the rule at lines 83-103:

```json
{
  "slug": "new-project-welcome",
  "name": "Welcome Notification",
  "description": "Sends a welcome notification to team members when work becomes active",
  ...
  "actionConfig": {
    "recipientType": "PROJECT_MEMBERS",
    "title": "Welcome to {{project.name}}",
    "message": "{{project.name}} is now active. You can start logging time and creating tasks."
  }
}
```

Only the `name`, `description`, and `message` text change. The `slug`, `triggerType`, `triggerConfig`, `recipientType`, and `title` (which uses `{{project.name}}` as a template variable, not a display term) remain unchanged.
