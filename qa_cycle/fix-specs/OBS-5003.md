# OBS-5003 Fix Spec: FICA Checklist Entity-Type Filtering Not Working

## Gap Summary

FICA checklist instantiates all 11 items for every client regardless of entity type. For a SOLE_PROP (sole proprietor), 4 items with `applicableEntityTypes` restrictions (Company Registration, Beneficial Ownership Declaration, Letters of Authority, Trust Deed) are incorrectly included.

## Root Cause

**Field promotion regression.** `ChecklistInstanceService.resolveEntityType()` reads the entity type from `customer.getCustomFields().get("acct_entity_type")`. However, `acct_entity_type` was promoted from a JSONB custom field to a structural column (`Customer.entityType`) in Phase 63. The JSONB `custom_fields` map no longer contains `acct_entity_type` -- it lives in `Customer.getEntityType()`.

Result: `resolveEntityType()` always returns `null`, and `isItemApplicable()` with `entityType == null` returns `true` for all items (backwards-compatible fallback).

**Why the existing test passes:** `ChecklistEntityTypeFilterTest` manually injects `acct_entity_type` into `customFields` via `customFields.put("acct_entity_type", entityType)` (line 226), which is not how production data flows. In production, the frontend sends `entityType` as a structural field on the create/update customer DTO, and `CustomerService` sets `customer.setEntityType(entityType)` on the structural column. The custom fields map is never populated with this value.

## Fix

### 1. Update `resolveEntityType()` in `ChecklistInstanceService.java`

**File:** `backend/src/main/java/io/b2mash/b2b/b2bstrawman/checklist/ChecklistInstanceService.java`

Change `resolveEntityType()` to read from the structural column first, falling back to the legacy custom field for backwards compatibility:

```java
private String resolveEntityType(Customer customer, UUID customerId) {
    if (customer == null) {
        customer = customerRepository.findById(customerId)
                .orElseThrow(() -> new ResourceNotFoundException("Customer", customerId));
    }
    // Phase 63: acct_entity_type was promoted to a structural column
    if (customer.getEntityType() != null) {
        return customer.getEntityType();
    }
    // Backwards-compatible fallback for any pre-migration data
    var customFields = customer.getCustomFields();
    if (customFields == null) {
        return null;
    }
    Object value = customFields.get("acct_entity_type");
    return value != null ? value.toString() : null;
}
```

### 2. Fix `ChecklistEntityTypeFilterTest` to reflect production data flow

**File:** `backend/src/test/java/io/b2mash/b2b/b2bstrawman/checklist/ChecklistEntityTypeFilterTest.java`

Update `createCustomerWithEntityType()` to use the structural column instead of (or in addition to) the custom field:

```java
private Customer createCustomerWithEntityType(String entityType) {
    counter++;
    var customer = createActiveCustomer(
            "Test Customer " + counter,
            "etf_customer_" + counter + "_" + uuid8() + "@test.com",
            memberId);
    customer.setEntityType(entityType);
    return customerRepository.save(customer);
}
```

This tests the production path. Optionally add a separate test that verifies the legacy fallback (custom field path) still works for pre-migration data.

### 3. No pack JSON changes needed

The `fica-kyc-za/pack.json` already has correct `applicableEntityTypes` values:
- Items 3 (Company Registration): `["PTY_LTD", "CC", "NPC"]`
- Items 7 (Resolution/Mandate): `["PTY_LTD", "CC", "NPC", "TRUST"]` (optional)
- Items 8 (Beneficial Ownership): `["PTY_LTD", "CC", "NPC", "TRUST"]`
- Items 10 (Letters of Authority): `["TRUST"]`
- Items 11 (Trust Deed): `["TRUST"]`

For SOLE_PROP, once filtering works, the instantiated items will be:
1. Certified ID Copy (universal) -- required
2. Proof of Residence (universal) -- required
3. Tax Clearance Certificate (universal) -- required
4. Bank Confirmation Letter (universal) -- required
5. Proof of Business Address (universal) -- optional
6. Source of Funds Declaration (universal) -- optional

That's 4 required + 2 optional = 6 items (matching the scenario expectation of ~6).

### 4. No migration needed

The `applicable_entity_types` JSONB columns already exist on both `checklist_template_items` and `checklist_instance_items` (V72 migration). The seeder already populates them from the pack JSON. The only change is in the Java service method that reads the customer's entity type.
