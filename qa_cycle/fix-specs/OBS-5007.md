# Fix Spec: OBS-5007 — Send Invoice crashes the page

## Problem

On the invoice detail page for approved invoice INV-0001 (Kgosi Holdings bookkeeping, ID: 6bf52c2f), clicking "Send Invoice" causes the page to crash with the error boundary message "Something went wrong -- An unexpected error occurred while loading this page." The backend logs show no explicit error at the time of the crash. The invoice remains in "Approved" status after page refresh. This blocks the entire invoicing lifecycle (Days 38-60).

Evidence: Day 38 checkpoint 38.1b in `qa_cycle/checkpoint-results/day-36-60.md`.

## Root Cause (two layers)

### Layer 1: Backend throws InvoiceValidationFailedException (422) — customer missing tax_number

The backend `InvoiceTransitionService.send()` calls `invoiceValidationService.validateInvoiceSend(invoice)` which runs three CRITICAL checks:
1. `checkOrgBranding` — org name (PASSES: "Thornton & Associates")
2. `checkCustomerRequiredFields` — address etc. (PASSES: Kgosi has address fields)
3. `checkCustomerTaxNumber` — tax_number on the customer (FAILS: Kgosi was created with `vat_number` = "4123456789" but `tax_number` was never set)

When a CRITICAL check fails and `overrideWarnings` is false, the service throws `InvoiceValidationFailedException(validationChecks)` which produces a 422 response with `canOverride: true` and `validationChecks: [...]` in the ProblemDetail body.

### Layer 2: Frontend error handling catches 422 but the page still crashes

The `sendInvoice` server action in `frontend/app/(app)/org/[slug]/invoices/invoice-payment-actions.ts` catches this 422 and checks `error.status === 422 && error.detail?.canOverride`. If the check passes, it returns `{ success: false, canOverride: true, validationChecks }` to the client. The client hook `handleSend` then calls `setShowSendOverride(true)` which should render the `SendValidationOverride` component.

**The crash happens because `revalidatePath` is NOT called on the 422 path (correctly), but the `startTransition` wrapping `handleSend` propagates the server action rejection to React's error boundary when the action's internal error handling produces an unexpected throw.** The most likely specific cause:

The `sendInvoice` action's non-ApiError catch path calls `createMessages("errors").t(classified.messageCode)` — this is fine for normal errors. But if the request to the gateway itself fails (e.g., a transient network error or cookie/CSRF mismatch during the POST), the error is NOT an `ApiError` instance. The `classifyError` function sees no `status` property and returns `{ messageCode: "api.networkError" }`, which `createMessages("errors").t()` resolves correctly. This path should work.

**Alternative hypothesis (more likely):** The gateway at `:8443` returns a redirect (3xx) instead of forwarding the 422 from the backend. The `apiRequest` function in `lib/api/client.ts` line 97-99 catches 3xx responses and throws `ApiError(401, "Authentication session expired")`. This is NOT caught by the 422-specific check in the `sendInvoice` action — it falls through to `return { success: false, error: error.message }` which sets `error.message = "Authentication session expired"`. The client then shows this in the error banner. But if `revalidatePath` was called before the error (it shouldn't be — it's only called on success), or if the `startTransition` + Next.js internals handle the 401-redirect differently...

**Most pragmatic root cause:** The `POST /api/invoices/{id}/send` with an empty body (no `Content-Type: application/json` body, despite the header being set) may cause the Spring Cloud Gateway to handle it differently than the backend. The gateway might strip/modify the request in a way that causes the backend to return an unexpected error, OR the gateway's CSRF protection might reject the request differently when no body is present.

## Fix (two-part)

### Part A: Frontend resilience — catch ALL errors in sendInvoice action (IMMEDIATE)

Add a try-catch around the entire `sendInvoice` action body to prevent any unhandled throw from propagating to React's error boundary. This ensures the page never crashes on send — worst case, the user sees an error message.

**In `frontend/app/(app)/org/[slug]/invoices/invoice-payment-actions.ts`**, ensure the entire function body is wrapped:

```typescript
export async function sendInvoice(
  slug: string,
  invoiceId: string,
  customerId: string,
  overrideWarnings?: boolean
): Promise<InvoiceActionResult> {
  try {
    const caps = await fetchMyCapabilities();
    if (!caps.isAdmin && !caps.isOwner) {
      return { success: false, error: "Only admins and owners can send invoices." };
    }

    const body = overrideWarnings ? { overrideWarnings: true } : { overrideWarnings: false };
    const invoice = await api.post<InvoiceResponse>(`/api/invoices/${invoiceId}/send`, body);
    revalidateInvoicePaths(slug, invoiceId, customerId);
    return { success: true, invoice };
  } catch (error) {
    if (error instanceof ApiError) {
      if (error.status === 422 && error.detail?.canOverride) {
        return {
          success: false,
          error: error.message,
          canOverride: true,
          validationChecks: error.detail.validationChecks as ValidationCheck[] | undefined,
        };
      }
      return { success: false, error: error.message };
    }
    const classified = classifyError(error);
    return {
      success: false,
      error: createMessages("errors").t(classified.messageCode),
    };
  }
}
```

Key change: **Always send a body** (`{ overrideWarnings: false }` instead of `undefined`) to avoid the empty-body-with-Content-Type-application/json edge case. This ensures the gateway and backend always receive a well-formed JSON request body.

### Part B: Set tax_number for Kgosi to unblock the flow (SCENARIO NOTE)

The validation failure is correct product behavior — SARS requires tax numbers on issued invoices. However, the QA scenario step 4.4 (Kgosi creation) doesn't include a `tax_number` field. The Day 36 pre-generation checks passed (4/4 green) because `tax_number` is only a WARNING at draft time but CRITICAL at send time (GAP-L-62 design).

**Option B1 (scenario amendment):** Add `tax_number: "9876543210"` to the Kgosi creation step 4.4 in the scenario file. This is the correct fix — a real accounting firm would always have the tax number before invoicing.

**Option B2 (override flow):** If the frontend fix in Part A works correctly, the `SendValidationOverride` dialog should appear, allowing the user to "Send Anyway." This exercises the override path, which is also valid QA behavior. The scenario can note: "If tax number missing warning appears, override to proceed."

**Recommendation:** Apply B1 (add tax_number to scenario) so the happy path is clean, AND verify the override dialog works as a secondary check.

### Part C: Add error.tsx boundary for invoices route (DEFENSE IN DEPTH)

Create `frontend/app/(app)/org/[slug]/invoices/error.tsx` to provide a graceful error recovery UI instead of the generic "Something went wrong" page. This matches the pattern already used by `customers/` and `projects/`.
