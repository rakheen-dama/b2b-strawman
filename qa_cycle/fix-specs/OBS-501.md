# OBS-501 — Matter Requests "View request" link 404s

**Filed**: Day 5 (2026-04-30) by QA agent
**Severity**: bug (workaround exists — navigate via customer detail)
**Owner**: Dev
**Status**: SPEC_READY

## Problem

On the matter detail page, clicking the FICA "View request" link (rendered on the Overview tab via `<FicaStatusCard>`) emits the URL `/org/{slug}/requests/{id}` which returns Next.js **404**. The canonical firm-side info-request route is `/org/{slug}/information-requests/{id}` — confirmed by `find frontend/app -type d -name "information-requests"`. The Requests tab table (`<RequestList>`) already uses the correct route, but the FICA card does not.

QA evidence:
- Frontend dev-server log captured the failed GET:
  - `GET /org/mathebula-partners/requests/7f8f9422-e8ae-4966-976e-85f90199d6c2 404 in 42ms` (`.svc/logs/frontend.log:830`)
  - Followed by the working `GET /org/mathebula-partners/information-requests/7f8f9422-...` 200 from manual URL correction.
- Screenshots at `qa_cycle/evidence/day-05/day-05-matter-requests-tab.png` and `day-05-firm-request-detail-3-submitted.png`.

The QA report attributed this to the matter Requests tab table; the actual link generator is on the matter Overview tab's FICA Status Card, which renders alongside the Requests tab on the same matter detail surface. The FICA `requestId` in the screenshot evidence (`7f8f9422-e8ae-4966-976e-85f90199d6c2`) matches the FICA-pack info request used in the lifecycle (REQ-0001).

## Root Cause (verified)

`frontend/components/compliance/FicaStatusCard.tsx:88`:

```tsx
{ficaStatus.requestId && (
  <Link
    href={`/org/${slug}/requests/${ficaStatus.requestId}`}   // ← WRONG segment
    className="mt-2 inline-block text-xs text-teal-600 ..."
    data-testid="fica-request-link"
  >
    View request
  </Link>
)}
```

The href segment is `/requests/`, but the route at `frontend/app/(app)/org/[slug]/information-requests/[id]/page.tsx` requires `/information-requests/`. There is no `frontend/app/(app)/org/[slug]/requests/[id]/page.tsx` — only `compliance/requests` and `settings/data-protection/requests` exist as firm-side `/requests` paths, and neither matches the FICA info-request id.

For comparison, the matter Requests tab table renders correctly via `frontend/components/information-requests/request-list.tsx:68`:

```tsx
href={`/org/${slug}/information-requests/${request.id}`}   // ← canonical
```

## Fix

Single-file edit in `frontend/components/compliance/FicaStatusCard.tsx`:

**Line 88**, change:
```tsx
href={`/org/${slug}/requests/${ficaStatus.requestId}`}
```
to:
```tsx
href={`/org/${slug}/information-requests/${ficaStatus.requestId}`}
```

Optional regression guard (recommended): in `frontend/components/compliance/__tests__/FicaStatusCard.test.tsx`, add an assertion that the `fica-request-link` href contains `/information-requests/` and not the bare `/requests/` segment.

```tsx
it("links to the canonical /information-requests/{id} route (OBS-501)", () => {
  render(
    <FicaStatusCard
      ficaStatus={{
        status: "IN_PROGRESS",
        requestId: "req-123",
        lastVerifiedAt: null,
      }}
      slug="acme"
    />,
  );
  const link = screen.getByTestId("fica-request-link");
  expect(link).toHaveAttribute("href", "/org/acme/information-requests/req-123");
});
```
