# OBS-502 — Portal `/requests` index counter resets to "0/3 submitted" on COMPLETED envelopes

**Filed**: Day 5 (2026-04-30) by QA agent
**Severity**: bug (mandate: "all bugs must be fixed; only KYC + Payments exempt"; severity upgraded from nit because the literal misleads end-users on the canonical post-completion view)
**Owner**: Dev
**Status**: SPEC_READY

## Problem

After the firm accepts every item on a portal information-request envelope, the per-row counter on the portal `/requests` index reads **"0/3 submitted"** while the status badge correctly reads **"COMPLETED"**. The two pieces of UI signal disagree and the literal `0/3 submitted` reads as a regression to a portal contact who already uploaded all three documents.

Reproduction (verified end-to-end Day 5):
1. Portal: Sipho uploads + submits 3 of 3 items → counter renders "3/3 submitted", envelope status = `IN_PROGRESS`.
2. Firm: Bob clicks **Accept** on each of the 3 items → items move SUBMITTED → ACCEPTED.
3. Envelope auto-transitions to COMPLETED on the third Accept.
4. Portal `/requests` index now shows: status badge **COMPLETED**, counter **"0/3 submitted"**.

## Root Cause (verified)

Two coordinates:

1. **Backend counter semantics** — `backend/.../repository/PortalReadModelRepository.java:885`:
   ```java
   submitted_items = (SELECT COUNT(*) FROM portal.portal_request_items WHERE request_id = ? AND status = 'SUBMITTED')
   ```
   `submittedItems` is computed **strictly** as the count of items currently in `SUBMITTED` state, not as a monotonic running total. Once items advance to `ACCEPTED` (or `REJECTED`), the count drops back toward 0. The `acceptedItems` and `rejectedItems` columns are computed by the same projection and are already exposed on the wire (`PortalRequestView.java:6-21`, `PortalInformationRequestController.PortalRequestListResponse:140-151`).

2. **Portal frontend label** — `portal/app/(authenticated)/requests/page.tsx:95`:
   ```tsx
   <p className="mt-1 text-xs">
     {r.submittedItems}/{r.totalItems} submitted
   </p>
   ```
   The portal renders `submittedItems` directly, with the literal "submitted", regardless of envelope status. The same pattern repeats at:
   - `portal/app/(authenticated)/requests/[id]/page.tsx:101` — `{detail.submittedItems}/{detail.totalItems} submitted • status …`

The firm-side equivalent (`frontend/components/information-requests/request-list.tsx:91-94`) already drives its progress bar from `acceptedItems`, not `submittedItems`, so this is an isolated portal inconsistency.

## Fix — Option 2 (frontend-only)

The orchestrator's Option 1 (backend rename / re-semantic of `submittedItems`) would break the wire contract for any consumer that depends on the strict "currently SUBMITTED" meaning, would force a coordinated migration of three callsites (portal `/requests`, portal `/requests/[id]`, portal `/home` `pending-info-requests` filter at `home/page.tsx:71-72`), and would introduce a new derived column without changing what the portal shows. Option 2 keeps the existing semantics correct (each counter still means exactly what its name says) and uses fields that are already on the wire.

### Change 1 — `portal/app/(authenticated)/requests/page.tsx:93-96`

Render a label that reflects the envelope's current state. Treat COMPLETED as "all items accepted" and other states as "submitted progress":

```tsx
<div className="text-right text-sm text-slate-600">
  <p>{r.status}</p>
  <p className="mt-1 text-xs">
    {r.status === "COMPLETED"
      ? `${r.acceptedItems}/${r.totalItems} accepted`
      : `${r.submittedItems}/${r.totalItems} submitted`}
  </p>
</div>
```

### Change 2 — `portal/app/(authenticated)/requests/[id]/page.tsx:101`

Mirror the same conditional in the detail-page header:

```tsx
{detail.status === "COMPLETED"
  ? `${detail.acceptedItems}/${detail.totalItems} accepted • status `
  : `${detail.submittedItems}/${detail.totalItems} submitted • status `}
```

### Change 3 — Vitest coverage (regression guard)

Extend the existing `portal/__tests__/portal-requests.test.tsx` (or its detail-page sibling — both already exist per Day 4 evidence). Add two cases:

1. With a request payload that has `status: "COMPLETED"`, `submittedItems: 0`, `acceptedItems: 3`, `totalItems: 3`, assert the rendered counter contains the literal `3/3 accepted` and does **not** contain `0/3 submitted`.
2. With `status: "IN_PROGRESS"`, `submittedItems: 2`, `totalItems: 3`, assert it renders `2/3 submitted`.

  - `portal/app/(authenticated)/requests/page.tsx` — counter literal change.

  - `portal/__tests__/portal-requests.test.tsx` and `portal/__tests__/portal-request-detail.test.tsx` — guard tests (or whatever the existing portal vitest test file is named — verify before editing).

- **Migration**: No.
- **Data changes**: None.

## Why not Option 1 (backend)

Renaming `submittedItems` to `processedItems` (or making it the `SUBMITTED+ACCEPTED+REJECTED` sum) would:
- Force a coordinated change across the controller DTO (`PortalRequestListResponse`, `PortalRequestDetailResponse`), the read-model record (`PortalRequestView`), the JDBC projection (`PortalReadModelRepository.recalculatePortalRequestCounts` and the two `findRequests*` queries), and three portal frontend call sites.
- Break the meaning of the existing `submittedItems` column for any downstream consumer (analytics, audit), even though that field name is the natural name for "currently in SUBMITTED state".
- Not change the user-facing label, only the number it reports.
- Either require a DB migration to rename the column (heavyweight) or leave a stale name in storage (confusing).

The backend already exposes the data the UI needs (`acceptedItems`); the portal just isn't using it.
