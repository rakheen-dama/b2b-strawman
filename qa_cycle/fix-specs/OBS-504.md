# Fix Spec: OBS-504 — Activity feed misattributes info-request recipient to the sender

## Problem
Matter Activity feed renders "Information request REQ-0001 sent to **Bob Ndlovu**" — but REQ-0001 was sent to portal contact Sipho Dlamini; Bob is the sender. Pure display misattribution in the audit trail; lifecycle behaviour is correct. Evidence: `qa_cycle/checkpoint-results/day-05.md` checkpoint 5.5.

## Root Cause (confirmed)
- `backend/src/main/java/io/b2mash/b2b/b2bstrawman/activity/ActivityMessageFormatter.java` L101–103: `case "information_request.sent" -> "Information request %s sent to %s".formatted(getRequestNumber(details), getContactName(details, actorName))`.
- `getContactName` (L276–279) returns `details.get("contact_name")` if present, otherwise **falls back to the actor name** — i.e. the sender.
- `backend/src/main/java/io/b2mash/b2b/b2bstrawman/informationrequest/InformationRequestService.java` `send()` L359–363 builds `sentAuditDetails` with only `request_number` + `project_id` — `contact_name` is never written, so the fallback always fires.
- Pre-existing since PR #545 (formatter) — not a regression from PRs #1403–#1435.

## Fix
Backend, two small changes:

1. **Emit the recipient** — in `InformationRequestService.send()` (~L359), resolve the portal contact and add it to the audit details:
   ```java
   if (saved.getPortalContactId() != null) {
     portalContactRepository
         .findById(saved.getPortalContactId())
         .ifPresent(c -> sentAuditDetails.put("contact_name", c.getDisplayName()));
   }
   ```
   `PortalContactRepository` is already injected (field L58, constructor L72). Guard against a null/blank display name (skip the put — fallback then applies).
2. **Neutralize the fallback** — in `ActivityMessageFormatter` L103, change the fallback so a missing `contact_name` can never misattribute again: replace `getContactName(details, actorName)` with `getContactName(details, "the client contact")` (or equivalent neutral copy). This kills the bug class for any other emit site that forgets the detail; old REQ-0001 events on the QA tenant will render neutrally instead of wrongly (acceptable — data disposable).

Note: `resendNotification()` (L654–687) publishes only a notification event, no audit row — no change needed there.
