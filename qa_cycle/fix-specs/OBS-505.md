# Fix Spec: OBS-505 — Seeded AI-specialist automation templates can never execute (wrong specialist IDs + unresolvable contextRef)

## Problem
On the third Accept (REQ-0001 completion), the INFORMATION_REQUEST_COMPLETED automation rule "Extract fields from uploaded intake documents" (rule `0556d81d…`) failed: backend ERROR `Failed to execute INVOKE_AI_SPECIALIST action: 404 … No specialist found with id INTAKE` + full stack trace, action FAILED, and an `AUTOMATION_ACTION_FAILED` notification was pushed to all admins/owners. Evidence: `qa_cycle/checkpoint-results/day-05.md`. This will recur on **every** envelope completion, every invoice approval (Day 30+ billing), and every matter reactivation — a cascading failure that pollutes backend logs and the notification bell for the rest of the cycle.

## Root Cause (confirmed) — TWO defects, both required for the rules to ever execute

### Defect A — wrong specialist IDs (the observed 404)
- Seeded template packs reference display-style IDs that were never registered:
  - `backend/src/main/resources/automation-templates/ai-specialist-common.json` L17 `"BILLING"`, L41 `"INTAKE"`
  - `…/ai-specialist-legal-za.json` L20, L45 `"INBOX"`
  - `…/ai-specialist-consulting-za.json` L20, L45 `"INBOX"`
- Registered IDs: `intake-za` (`assistant/specialist/IntakeSpecialistConfig.java:14`), `billing-za` (`BillingSpecialistConfig.java:14`), `inbox-za` (`InboxSpecialistConfig.java:14`).
- `automation/executor/InvokeAiSpecialistActionExecutor.java:67` passes the raw config string to `SpecialistRegistry.requireById` → ResourceNotFoundException → caught at L124–127, logged ERROR with stack, ActionFailure.
- **Bug class extends to the frontend**: `frontend/components/automations/action-form.tsx` L392–394 hardcodes the same wrong values (`<SelectItem value="BILLING">` / `"INTAKE"` / `"INBOX"`) — every *user-created* INVOKE_AI_SPECIALIST rule is born broken too.
- Latent since PR #1300 (2026-05-06); first surfaced because this cycle checks backend logs per day and Day 5 is the first envelope completion.

### Defect B — `{{event.entityId}}` never resolves (would fail immediately after fixing A)
- All 6 template actions use `"contextRef": { "entityType": …, "entityId": "{{event.entityId}}" }`.
- `automation/AutomationContext.java` builds context sections `actor`, `rule`, plus trigger-specific keys — there is **no `event` section** (L44–77). For INFORMATION_REQUEST_COMPLETED it provides `request.id`, `request.customerId`, `customer.id` (L271–282); for INVOICE_STATUS_CHANGED `invoice.id` (L192–195); for PROJECT_STATUS_CHANGED `project.id` (L136–139).
- `automation/VariableResolver.java` L54–61 leaves unresolved placeholders **as-is**, so the executor receives the literal string `{{event.entityId}}` → `UUID.fromString` throws → `ActionFailure "Invalid entityId after variable resolution"` (`InvokeAiSpecialistActionExecutor.java` L94–101) → another AUTOMATION_ACTION_FAILED notification.
- Fixing only Defect A would hand QA a "fixed" rule that still fails on retest. Same bug class (templates shipped without end-to-end validation), same PR — explicitly authorized here per CLAUDE.md §7 same-bug-class exception.

## Fix

### 1. Template packs (Seed — fresh provisioning correct)
- `ai-specialist-common.json`:
  - `polish-invoice-on-send`: `"specialistId": "BILLING"` → `"billing-za"`; `entityId` `{{event.entityId}}` → `{{invoice.id}}`.
  - `extract-intake-fields`: `"specialistId": "INTAKE"` → `"intake-za"`; `entityId` `{{event.entityId}}` → `{{customer.id}}` (entityType stays `customer`).
- `ai-specialist-legal-za.json` + `ai-specialist-consulting-za.json` (identical structure):
  - Both actions: `"specialistId": "INBOX"` → `"inbox-za"`; `entityId` `{{event.entityId}}` → `{{project.id}}`.
  - `matter-reactivation-catchup` (PROJECT_STATUS_CHANGED) becomes fully functional. `weekly-matter-activity-summary` (SCHEDULED) still has no entity context until Epic 515C populates it (`AutomationContext.java` L69–73 TODO) — `{{project.id}}` is the correct target for when 515C lands; note this in the PR description, do not attempt to implement 515C.
- Bump `"version"` to `2` in all three packs (metadata hygiene; seeding is gated on packId only — `AutomationTemplateSeeder.isPackAlreadyApplied` — so no re-application side effects).

### 4. Existing QA tenant repair (legitimate means, no SQL)

## ADDENDUM (2026-06-13, orchestrator+user decision) — SCHEDULED fan-out (epic 515C core)

Review of PR #1437 surfaced that the ID/contextRef correction fully fixes the **4 EVENT-triggered** specialist actions (incl. the Day-5 envelope-completion blocker), but the **2 SCHEDULED** `weekly-matter-activity-summary` actions (legal-za + consulting-za) still cannot resolve their `{{project.id}}` contextRef at runtime: the scheduled path builds a context with only `actor`+`rule` (no entity), and fires each rule **once per tenant** with **no per-project fan-out** (confirmed by code-explorer).

**User decision (AskUserQuestion 2026-06-13): "Fully fix now"** — accept L-effort scope expansion + executor risk; implement scheduled per-active-project fan-out so the weekly per-matter summaries actually execute. Authorized scope expansion under CLAUDE.md §7 (orchestrator/user call).

### Confirmed execution path (code-explorer)
- Live path: `AutomationScheduler.pollScheduledTriggers()` (@Scheduled 60s, ShedLock) → `jobEnqueuer.fanOutToAllTenants("automation_poll_triggers")` → `JobWorker.processJob` binds TENANT_ID/ORG_ID/SHARD_ID (NOT MEMBER_ID) via `runForTenantOnShard` → `AutomationPollTriggersHandler.processScheduledTenant()` → `fireScheduledRule(rule, now)` (~L131–142). Fires each due rule ONCE with `{rule:{id,name,createdBy}}` context only. Conditions are NOT evaluated in the scheduled path.
- `AutomationContext.build()` (L40+) is only called in the EVENT path (`AutomationEventListener.evaluateRule`); SCHEDULED bypasses it. 515C TODO at L69–73.
- `VariableResolver` leaves unresolvable `{{x.y}}` as the literal string → `UUID.fromString` throws → `ActionFailure("Invalid entityId after variable resolution: …")` → AUTOMATION_ACTION_FAILED admin notification.
- `InvokeAiSpecialistActionExecutor` actor id: tries `actor.id` then falls back to `rule.createdBy` (present) — so actor binding is fine without MEMBER_ID.
- Possible duplicate scheduled logic in `AutomationScheduler` L164–179 — dev must confirm which is live and avoid fixing a dead path / leaving the live one broken.

### Fix shape (design)
1. **Templates** (already on fix branch): `entityType:"project"`, `entityId:"{{project.id}}"` for the 2 SCHEDULED actions. Keep.
2. **Per-project fan-out** in the live scheduled path (`AutomationPollTriggersHandler.fireScheduledRule`): if any of the rule's actions has a project-scoped contextRef (entityType=="project" or template contains `{{project.`), load active projects (`ProjectRepository.findByStatus(ACTIVE)` — add derived query), and for EACH active project: build a context with `actor`(rule.createdBy) + `rule` + `project`(id,name,status) sections, **evaluate the rule's conditions against that project context** (honors `project.status EQUALS ACTIVE` + any future conditions), and if they pass, `fireRule(rule, projectContext)`. Otherwise (no project ref, e.g. SEND_NOTIFICATION) fire once as today — no regression.
3. **Centralize** per-project scheduled context construction in `AutomationContext` (e.g. `buildScheduledProjectContext(rule, project)`) and resolve the 515C TODO, rather than inline map-building in the handler (elegance).
4. **Scale guard**: only query projects for due rules that actually need project context (bounded). Note the 60s tick + N rules × M projects cost in the PR.
5. **Wire `ConditionEvaluator`** into the scheduled per-project loop (it is currently bypassed for SCHEDULED).
6. **Frontend Issue 1 (in-scope, must fix)**: `action-form.tsx` ~L415 Context Reference input placeholder still reads `{{event.entityType}}:{{event.entityId}}` — change to a valid trigger-appropriate example (e.g. `{{project.id}}`) so the UI stops teaching the broken pattern.
7. **Guard test**: `AiSpecialistAutomationPackValidityTest` must STOP skipping SCHEDULED actions — assert their contextRefs resolve under the new scheduled-project context model.

### Tests (must add)
- Integration test in `AutomationSchedulerScheduledTriggerIntegrationTest`: seed 2 ACTIVE + 1 non-active project + a SCHEDULED INVOKE_AI_SPECIALIST rule; fire the cron tick; assert specialist runner invoked once per ACTIVE project with that project's id, NOT for the inactive one, and zero AUTOMATION_ACTION_FAILED.
- Regression: a SCHEDULED SEND_NOTIFICATION rule (no project ref) still fires exactly once (no fan-out).

### Riskiest part
Repository dependency + N×M query in the 60s scheduling tick; duplicate scheduled-fire logic (handler vs scheduler) — fix the live path, dedupe or fix both. Bundled into PR #1437 by explicit authorization (same gap, expanded).
