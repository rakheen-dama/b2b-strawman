# Fix Spec: OBS-6002 — Admin role lacks APPROVE_TRUST_PAYMENT capability

## Gap

Admin role users cannot approve trust payments. This creates a deadlock in the dual-approval flow: when the Owner records a trust payment, the system correctly prevents self-approval, but no other user in the firm can approve because `APPROVE_TRUST_PAYMENT` is Owner-only.

**Severity**: BLOCKER (cascading — blocks Day 60 trust balance zero-out, matter closure, SoA, and all subsequent days)

## Evidence

- Bob (Admin) clicks Approve on PAY/2026/001: **"Insufficient permissions for this operation"**
- Thandi (Owner/recorder) clicks Approve: **"The transaction recorder cannot be the sole approver. A different member with APPROVE_TRUST_PAYMENT capability must approve this transaction."**
- Trust Accounting Settings show "Single approval" mode (not dual) — but even single mode requires a different member from the recorder to approve.

## Root Cause (two layers)

### Layer 1: Java `Capability.OWNER_ONLY` set

**File**: `backend/src/main/java/io/b2mash/b2b/b2bstrawman/orgrole/Capability.java` (lines 57-62)

```java
public static final Set<String> OWNER_ONLY =
    Set.of(
        MANAGE_COMPLIANCE_DESTRUCTIVE.name(),
        APPROVE_TRUST_PAYMENT.name(),        // <-- THIS IS THE BUG
        OVERRIDE_MATTER_CLOSURE.name(),
        FINANCIAL_RECONCILE.name());
```

`OrgRoleService.resolveCapabilities()` (lines 71-75) strips all `OWNER_ONLY` capabilities from the admin role at runtime:

```java
if (role.isSystem() && "admin".equals(role.getSlug())) {
    var adminCaps = new HashSet<>(Capability.ALL_NAMES);
    adminCaps.removeAll(Capability.OWNER_ONLY);     // Removes APPROVE_TRUST_PAYMENT
    return Set.copyOf(adminCaps);
}
```

Even if the DB seed had the capability for admin, the Java code would strip it out.

### Layer 2: Flyway seed data (V85)

**File**: `backend/src/main/resources/db/migration/tenant/V85__create_trust_accounting_tables.sql` (lines 414-422)

`APPROVE_TRUST_PAYMENT` is only seeded for the `owner` slug, not `admin`. The admin role only gets `VIEW_TRUST` and `MANAGE_TRUST`.

## Fix

### Change 1: Remove `APPROVE_TRUST_PAYMENT` from `OWNER_ONLY` set

**File**: `backend/src/main/java/io/b2mash/b2b/b2bstrawman/orgrole/Capability.java`
**Lines**: 57-62

Remove `APPROVE_TRUST_PAYMENT.name()` from the `OWNER_ONLY` set. This allows the admin role's runtime capability resolution to include it.

Before:
```java
public static final Set<String> OWNER_ONLY =
    Set.of(
        MANAGE_COMPLIANCE_DESTRUCTIVE.name(),
        APPROVE_TRUST_PAYMENT.name(),
        OVERRIDE_MATTER_CLOSURE.name(),
        FINANCIAL_RECONCILE.name());
```

After:
```java
public static final Set<String> OWNER_ONLY =
    Set.of(
        MANAGE_COMPLIANCE_DESTRUCTIVE.name(),
        OVERRIDE_MATTER_CLOSURE.name(),
        FINANCIAL_RECONCILE.name());
```

**Rationale**: `APPROVE_TRUST_PAYMENT` is fundamentally different from the other OWNER_ONLY capabilities. The entire purpose of trust payment approval is dual-approval (Section 86 compliance) — which by definition requires a *different* team member to approve. If only the Owner can approve, and the Owner often records transactions, there is no second approver available. Admin users in a law firm (senior associates, practice managers) are the natural second approver.

### Change 2: New Flyway migration to seed `APPROVE_TRUST_PAYMENT` for Admin role

**File**: `backend/src/main/resources/db/migration/tenant/V125__grant_admin_approve_trust_payment.sql` (new)

```sql
-- OBS-6002: Grant APPROVE_TRUST_PAYMENT to Admin role for Section 86 dual-approval
-- Without this, only Owners can approve trust payments — creating a deadlock when
-- the Owner is also the recorder (self-approval is blocked by design).
INSERT INTO org_role_capabilities (org_role_id, capability)
SELECT id, 'APPROVE_TRUST_PAYMENT'
FROM org_roles
WHERE slug = 'admin'
  AND NOT EXISTS (
    SELECT 1 FROM org_role_capabilities
    WHERE org_role_id = org_roles.id AND capability = 'APPROVE_TRUST_PAYMENT'
  );
```

This is needed because existing tenant schemas were created with V85 which only seeded this capability for Owner. The Java code change (Change 1) ensures new admin-role resolution includes the capability, but the DB row is needed for custom roles that inherit from admin seed data, and for correctness of the `org_role_capabilities` table.

### Test Impact

1. **`CapabilityEndpointTest.java` line 158**: Asserts admin capabilities count = `ALL_NAMES.size() - OWNER_ONLY.size()`. Since we're removing one entry from `OWNER_ONLY`, the expected count increases by 1. **No code change needed** — the assertion is dynamic (`Capability.ALL_NAMES.size() - Capability.OWNER_ONLY.size()`), so it auto-adjusts.

2. **`TrustTransactionControllerTest.java`**: Tests use `ownerJwt` for approval. These still work since Owner retains the capability. Consider adding a new test where Admin approves (validates the fix), but this is optional — the service-level tests in `TrustTransactionServiceTest.java` already pass admin-role capabilities including `APPROVE_TRUST_PAYMENT` in their mock sets (lines 2821, 2832, 2843).
