## What changed

Frontend-only, four files touched, all matched the spec verbatim.

1. `frontend/components/proposals/create-proposal-dialog.tsx` — replaced
   ``expiresAt: `${values.expiresAt}T23:59:59Z` `` with an IIFE that splits
   the YYYY-MM-DD into year/month/day and constructs `new Date(y, m-1, d,
   23, 59, 59).toISOString()`. The resulting ISO instant is the LOCAL
   end-of-day in the user's zone — round-tripping through
   `toLocaleDateString` in any zone produces the same calendar date the
   user picked.

2. `frontend/lib/format.ts` — added `formatProposalExpiresAt(iso)`
   immediately below `formatLocalDate`. It is intentionally identical to
   `formatDate` today (en-US, month/day/year). The named helper signals
   intent at the read site and gives a single insertion point if the
   formatter ever needs to diverge for expiry fields specifically.

3. `frontend/app/(app)/org/[slug]/proposals/[id]/page.tsx` — added
   `formatProposalExpiresAt` to the existing `@/lib/format` import; swapped
   `formatDate(proposal.expiresAt)` → `formatProposalExpiresAt(proposal.expiresAt)`.
   Other fields on the page (`createdAt`, `sentAt`, `acceptedAt`,
   `declinedAt`) still use `formatDate` — they are real UTC instants from
   server-driven transitions, not user-picked calendar dates.

4. `frontend/components/proposals/__tests__/create-proposal-dialog.test.tsx`
   — added a new `describe` block with one test that fills `expiresAt =
   "2026-05-12"`, captures the `createProposalAction` payload, asserts
   `new Date(payload.expiresAt)` decodes to `getFullYear()=2026,
   getMonth()=4 (May), getDate()=12` AND that the literal `2026-05-12T23:59:59Z`
   form is never re-introduced. Belt-and-braces against future regressions.
