## Approach

Followed `qa_cycle/fix-specs/OBS-703.md` verbatim. Backend-only S fix; no schema/migration; no frontend touch.

1. **`ProposalSentEmailHandler`** — `@Component` with a single `@TransactionalEventListener(phase = AFTER_COMMIT)` for `ProposalSentEvent`. Mirrors the `handleInTenantScope` ScopedValue-bind pattern lifted directly from `ProposalPortalSyncEventHandler` in the same package — keeps proposal-side after-commit work co-located. Loads the proposal + portal contact, resolves org name + brand colour, builds the documented context (`contactName`, `orgName`, `proposalNumber`, `proposalTitle`, `portalUrl = portalBaseUrl + "/proposals/" + id`, `brandColor`, `expiresAtFormatted`, `subject`), and dispatches via the new `PortalEmailService.sendNewProposalEmail(...)`. All exceptions are caught and logged so a failed send never rolls back the originating `proposal.send` transaction.

2. **`PortalEmailService.sendNewProposalEmail`** — new public method + two constants (`NEW_PROPOSAL_TEMPLATE_NAME = "portal-new-proposal"`, `NEW_PROPOSAL_REFERENCE_TYPE = "PORTAL_NEW_PROPOSAL"`). Body is a thin one-liner delegating to the existing `sendPortalNotification(...)` private helper (template-renderer + rate-limiter + delivery-log already wired by Epic 498B — no new infrastructure).

3. **`ProposalSentEmailHandlerTest`** — `@SpringBootTest` integration test using `GreenMailTestSupport.getInstance()` JVM singleton on port 13025 (per backend CLAUDE.md — never start a new GreenMail). Drives the full production code path via `POST /api/proposals/{id}/send` rather than publishing the event manually, so the test covers controller → service → event publish → AFTER_COMMIT listener. Asserts: exactly one email arrives at the portal contact mailbox; recipient matches; subject contains the proposal number; body contains `View Proposal` CTA + `/proposals/{id}` link.

## ADR-258 compliance

`PortalEmailNotificationChannel` (Epic 498B per-event channel) explicitly excludes `ProposalSentEvent` per ADR-258 to avoid double-send. This fix does NOT touch the channel — it lives beside `ProposalPortalSyncEventHandler` in `proposal/` and dispatches independently. Firm-side `NotificationEventHandler.onProposalSent` (admin/owner audience) is also unchanged — different audience, different template, no conflict.
