## Decision notes

- Followed **Option 1** from `qa_cycle/fix-specs/OBS-704.md` — push the timestamp to the client via a hook. Keeps the timestamp local to the consumer; the page no longer has to know about hydration semantics. Pattern is already established in the codebase (`components/ui/relative-date.tsx`).
- Did **not** add a synthetic console-error capture test as suggested in the spec's verification section. The existing `ProposalTable` render tests in `__tests__/proposals-dashboard.test.tsx` already exercise the new prop shape; a hydration-mismatch console assertion needs Next.js SSR + hydrate (`renderToString` + `hydrateRoot`), which the current Vitest + happy-dom + RTL setup does not run. Adding that infrastructure is out-of-scope for an XS frontend fix and would be a larger separate change. Manual QA re-verification on the dev stack is the correct verification surface for the hydration-warning behaviour.
- Spec called out a likely Radix Dialog `useId()` cause; that turned out **not** to be the actual root cause — the hydration mismatch was upstream in the `now` prop flowing into descendant rendering. Spec already acknowledged this ("React anchors the diagnostic" at the dialog button); fix landed at the real source.
