# Fix Spec: OBS-704 — Hydration mismatch on `/proposals` index page

## Problem
The `/proposals` page (Server Component at `frontend/app/(app)/org/[slug]/proposals/page.tsx`) renders `CreateProposalDialog` (a `"use client"` component) with a `<Button>` child wrapped in Radix `<DialogTrigger asChild>`. Radix generates an `aria-controls` ID during SSR (e.g., `radix-_R_4clritrqiqbn5rknelb_`) that differs from the client-side hydration pass, causing a React hydration mismatch error in the console. The page renders correctly after client-side recovery, but the console error violates the "Frontend console clean" checkpoint (Day 7 checkpoint 7, scenario 7.10).

## Root Cause (hypothesis)
Radix `Dialog` generates internal IDs using `useId()` (or its own ID generation), which can produce different IDs on server vs client when the component tree includes conditional rendering or when the server-side render order differs from client hydration order. `CreateProposalDialog` at `frontend/components/proposals/create-proposal-dialog.tsx:204-205` uses the children-API pattern: `<Dialog><DialogTrigger asChild>{children}</DialogTrigger>...</Dialog>`. The `DialogTrigger` injects `aria-controls` with a Radix-generated ID into the child button. This is the exact pattern documented in the SSR snapshot test (`frontend/__tests__/components/dialog-family-ssr.snapshot.test.tsx:147-158`) as the "OBS-704 v3 sentinel".

The proposals page at `frontend/app/(app)/org/[slug]/proposals/page.tsx:87-92` passes a `<Button>` as children to `CreateProposalDialog`. The server renders the full `DialogTrigger` wrapper with an SSR-generated Radix ID, but on hydration the client generates a different ID.

## Fix
Convert `CreateProposalDialog` from the children-API pattern to the dialog-owns-button pattern (as documented in `frontend/CLAUDE.md` "Dialog Trigger Composition" section and precedented by `EditCustomerDialog` in PR #1242). This eliminates the `<DialogTrigger asChild>` from the SSR tree entirely, replacing it with a plain `<Button>` that opens the dialog via controlled state.

**Step-by-step:**

1. In `frontend/components/proposals/create-proposal-dialog.tsx`:
   - Remove the `children` prop from `CreateProposalDialogProps`
   - Add `triggerLabel?: React.ReactNode` prop (default: `<TerminologyText template="New {Proposal}" />`)
   - Add optional `triggerClassName?: string` prop
   - Remove the `<DialogTrigger asChild>{children}</DialogTrigger>` line (~line 205)
   - Add a plain `<Button>` before `<DialogContent>` that calls `setOpen(true)` on click (same pattern as `EditCustomerDialog`)
   - The dialog is already partially controlled (has `open`/`onOpenChange` props) — ensure `handleOpenChange` still works for both internal and controlled modes

2. In `frontend/app/(app)/org/[slug]/proposals/page.tsx`:
   - Remove the `<Button>` child passed to `CreateProposalDialog` (~lines 88-92)
   - Instead, pass trigger styling/label as props: `<CreateProposalDialog slug={slug} customers={customers} />`
   - The Plus icon + "New {Proposal}" text should be composed inside the dialog component's trigger button

3. In `frontend/components/projects/overflow-actions-menu.tsx`:
   - Check if this file also passes children to `CreateProposalDialog` and update if so

4. Update `frontend/__tests__/components/dialog-family-ssr.snapshot.test.tsx`:
   - Update the "CreateProposalDialog (children-API, OBS-704 v3 sentinel)" test to match the new dialog-owns-button pattern (no children, just props)
   - Update snapshot

5. Update `frontend/components/proposals/__tests__/create-proposal-dialog.test.tsx`:
   - Adjust any test that passes children to match the new prop-based trigger API
