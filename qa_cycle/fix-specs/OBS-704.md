# OBS-704 — Hydration mismatch on `/org/{slug}/proposals` index

**Severity**: bug (mandate: "frontend must run clean — no JS/Next errors in logs")
**Surface**: Frontend (proposals index page)
**Scope**: Frontend-only, single-line change + test
**Effort**: XS (~15 min implementation, ~30 min total with test)

## Root cause (verified)

`frontend/app/(app)/org/[slug]/proposals/page.tsx:112` renders:

```tsx
<ProposalTable proposals={proposals} slug={slug} now={new Date().getTime()} />
```

`new Date().getTime()` is called both on the server (during SSR) and on the client (during hydration). The two timestamps differ by network latency / cold-start time, so any descendant of `<ProposalTable>` that depends on `now` (e.g. expiry-countdown badges, "X days ago" rendering) computes a different output server-side vs client-side. React's reconciliation then logs a hydration mismatch.

Although the QA agent attributed the mismatch to `CreateProposalDialog → DialogTrigger asChild → Primitive.button.SlotClone`, that's just where React anchors the diagnostic — the actual divergence originates upstream in the `now` prop flowing into `ProposalTable`. Radix `DialogTrigger` itself is hydration-stable.

## Decision

**Compute `now` on the client only**, after mount, so SSR renders a stable placeholder and hydration matches server output.

Two acceptable options:
1. **Push `now` to the client** — replace the `now` prop with a `useNowMs()` hook inside `ProposalTable` that initialises to `Date.now()` only after mount via `useEffect` + state.
2. **Stable SSR sentinel** — render `now=0` (or a fixed timestamp) on the server, then update on the client via `useEffect`.

**Choose option 1** — keeps the timestamp local to the consumer, doesn't require the page to know about hydration. Pattern already used elsewhere in the frontend (see `frontend/hooks/` for similar mount-only hooks if one exists).

## Fix

### 1. Add a `useNowMs` hook

`frontend/hooks/use-now-ms.ts` (NEW):

```ts
"use client";

import { useEffect, useState } from "react";

/**
 * Returns the current time in ms, or 0 during SSR / pre-mount. Client-only —
 * avoids hydration mismatch when used as a "now" reference for relative-time
 * rendering (e.g. expiry countdowns).
 */
export function useNowMs(): number {
  const [now, setNow] = useState(0);
  useEffect(() => {
    setNow(Date.now());
  }, []);
  return now;
}
```

### 2. Update `ProposalTable` to call the hook internally

`frontend/components/proposals/proposal-table.tsx` (or wherever `ProposalTable` lives — find via grep):

- Remove `now: number` from the props.
- Call `const now = useNowMs();` at the top of the component.
- Use the same `now` value downstream as before.

### 3. Update the page to drop the `now` prop

`frontend/app/(app)/org/[slug]/proposals/page.tsx:112`:

**Before:**
```tsx
<ProposalTable proposals={proposals} slug={slug} now={new Date().getTime()} />
```

**After:**
```tsx
<ProposalTable proposals={proposals} slug={slug} />
```

## Out of scope

- Other pages that may pass `now={new Date()...}` as a prop. Should be audited as a follow-up but not part of this spec.
- Server-side time-sensitive logic (e.g. expiring proposals on the server) — those rely on `Instant.now()` server-side and are correct.
