# OBS-8801 — Client-safe firm milestones dropped from portal Activity "Firm actions" trail (missing `details.project_id` on emission)

- **Severity**: MEDIUM (non-blocking for the cycle — the portal `/activity` surface exists and renders client-safe; this is a completeness/narrative-coherence gap on specific milestone rows)
- **Found**: Day 88 (Activity-feed wow moment), cycle 32, branch `bugfix_cycle_2026-06-13`
- **Surfaces affected**: BOTH
  - Portal `/activity` → "Firm actions" tab (`PortalActivityService` FIRM/ALL via `AuditEventRepository.findActivityFirmForCustomer`)
  - Firm matter Activity tab (`ActivityService.getProjectActivity` via `AuditEventRepository.findByProjectId`)
- **Status**: SPEC_READY (pending Product confirmation that these milestones should appear — the allowlist comment already states they should)

## Symptom

On the portal `/activity` "Firm actions" tab for Sipho (RAF-2026-001), three client-facing milestones the firm performed are **absent**:
- **Proposal accepted** (Day 8) — `proposal.accepted`
- **Fee note sent** (Day 28) — `invoice.sent` (×2: INV-0001, INV-0002)
- **Matter closed** (Day 60) — `matter_closure.closed`

These are all explicitly on the client-safe allowlist `PortalActivityEventTypes.PORTAL_VISIBLE_FIRM_EVENT_TYPES`, with humanised labels intended for the client. They render correctly for `statement.generated` and one `document.generated`, but not these.

Scenario 88.4 expects "proposal accept (Day 8)" on the portal trail; 88.6 expects "every client-visible firm event has a matching client-side entry". Both are violated for these rows.

## Root cause

Both feed queries filter on the JSONB `details->>'project_id'`:

- `findActivityFirmForCustomer` (`AuditEventRepository.java:253-277`) requires
  `(ae.details->>'project_id') IS NOT NULL AND … ::uuid IN (SELECT project_id FROM customer_projects WHERE customer_id = :customerId)`.
- `findByProjectId` (`AuditEventRepository.java:56-77`) keys on `(details->>'project_id') = :projectId`.

The emit sites for these milestones write the audit row with `entity_type`/`entity_id` set to the proposal/invoice/project but **do NOT put `project_id` into `details`**. DB evidence (tenant `tenant_5039f2d497cf`):

```
event_type            | total | with_project_id
----------------------+-------+----------------
document.generated    |   2   |   1   (1 dropped)
statement.generated   |   1   |   1   (ok)
invoice.sent          |   2   |   0   (both dropped)
matter_closure.closed |   1   |   0   (dropped)
proposal.accepted     |   1   |   0   (dropped)
```

Because `details.project_id` is NULL on these rows, they are silently excluded from BOTH the portal client trail AND the firm matter Activity feed (the firm feed still surfaces them via the Settings → Audit Log entity filter, which keys on `entity_type+entity_id` — verified Day 85).

Note: `matter_closure.closed`, `project.created_from_template`, `project.updated` are stored as `entity_type='project'`, `entity_id=<projectId>` with NULL `details.project_id` — so "matter created" / "matter closed" also never appear in the firm matter Activity feed (88.1 lists them). This is the same emission-shape gap.

## Fix direction

At each emit site, add `project_id` to the audit `details` map so the activity feeds (which intentionally key on `details.project_id` for cross-entity matter scoping) include the row. Candidate sites:
- `ProposalService` / `ProposalOrchestrationService` — `proposal.accepted` (the created/linked project id is available post-orchestration).
- Invoice send path — `invoice.sent` (invoice → matter/project link).
- Matter closure path — `matter_closure.closed` and `project.created_from_template` / `project.updated` (the project id IS the entity_id; copy it into `details.project_id`).
- The one `document.generated` row missing `project_id`.

Alternative (broader): teach `findByProjectId` / `findActivityFirmForCustomer` to also match `entity_type='project' AND entity_id=:projectId` (mirrors `findMostRecentByProject` at `AuditEventRepository.java:93-101` which already does the OR). This is lower-touch for the project-self events but does not help proposal/invoice rows whose entity is not the project — those still need `details.project_id` populated. **Recommended: populate `details.project_id` at the emit sites** (consistent with the rest of the audit data plane and with the L-75c portal-contact fix that made `project_id` "load-bearing").

## Reproduce
