# Phase 70 — Specialist AI Assistants (In-Product Agents)

## System Context

Kazi has shipped two systems that Phase 70 will fuse:

- **Phase 37 — Workflow Automations v1** (PRs #555–#568) — a deterministic rule engine. Four entities (`AutomationRule`, `AutomationAction`, `AutomationExecution`, `ActionExecution`), a single `AutomationEventListener` bridging domain events to rule evaluation, six action executors (`CreateTask`, `SendNotification`, `SendEmail`, `UpdateStatus`, `CreateProject`, `AssignMember`), delayed-action scheduler, rule CRUD UI + template gallery + execution log (`frontend/app/(authenticated)/settings/automations/`).
- **Phase 52 — In-App AI Assistant (BYOAK)** (PRs #802–#812) — a conversational LLM surface. `LlmChatProvider` interface + Anthropic adapter (`backend/.../assistant/provider/`), `AssistantTool` interface + registry (`backend/.../assistant/tool/read/` + `write/`) with 14 read tools (ListProjects, GetProject, ListCustomers, GetCustomer, ListTasks, GetMyTasks, GetTimeSummary, GetUnbilledTime, GetProjectBudget, GetProfitability, ListInvoices, GetInvoice, SearchEntities, GetNavigationHelp) + 8 write tools (CreateProject, UpdateProject, CreateCustomer, UpdateCustomer, CreateTask, UpdateTask, LogTimeEntry, CreateInvoiceDraft). BYOAK via `SecretStore` (Phase 21). PRO-tier gated. Ephemeral sessions (no persistence). Capability-filtered tool visibility via `CapabilityAuthorizationService` (Phase 46). Chat panel UI at `frontend/components/assistant/` fired from the top-bar (Phase 44 command palette + `⌘K`).

These two systems have never spoken to each other. The assistant is a generic chat window that requires the user to go to it and formulate a prompt. The automation engine is deterministic and cannot invoke an LLM. Neither system shows a visible ROI moment on the demo path today.

**What the assistant is today**: a smart search box in a panel, pinned to `⌘K`. A user must (a) know the AI exists, (b) open it, (c) write a prompt, (d) judge the response. No inline placement. No specialisation. No proactive invocation. No integration with rules.

**What is missing for the "multiplier effect"**:

- **Inline placement** — an AI that lives where the work happens (on the invoice draft, on the intake dialog, on the matter Comments tab) rather than in a dedicated chat panel the user must remember to open.
- **Specialisation** — distinct assistants with narrowed tool sets and focused system prompts. A Billing specialist does not need `CreateCustomer`. An Intake specialist does not need `GetProfitability`. Narrower surface area = more predictable behaviour and clearer UX.
- **SA context** — the assistants today have no domain knowledge: no RSA ID number validation, no zero-rated disbursement VAT rule, no SA professional-register tone, no awareness of trust accounting Section 86 or POPIA. Kazi's vertical differentiation is SA specialisation; the AI layer should carry that differentiation too.
- **Automation-engine hook** — no way today to have a rule fire an LLM action. A rule can send an email; it cannot ask an LLM to *draft* the email. A rule can create a task; it cannot ask an LLM to *summarise* a week's activity into a task description.
- **Human-approval review queue for unattended AI output** — when AI output is generated by a rule (not a user chat prompt), there must be a reviewable, rejectable queue. "Reversible only + confirmation before write" (ADR from Phase 45) is non-negotiable; when the *invoker* is a rule rather than a user, the confirmation happens in a review queue, not in the chat.

**Founder decisions that constrain this phase** (2026-04-22 ideation, recorded in `.claude/ideas/phase70-ideation-2026-04-22.md`):

- **Inline primary, panel secondary** — specialists live as inline buttons on the specific pages where they serve; Phase 52's chat panel stays as the generalist fallback.
- **Partner + paralegal are the target users**, not clients. No portal-facing AI.
- **Small SA firms** where one person wears many hats. Specialists must produce *usable output* (polished text, extracted fields, drafted summaries), not conversation.
- **SA specialisation is the differentiator**, not "AI quality" vs Clio / LEAP / MyCase. The specialists must carry SA domain knowledge.
- **Three v1 specialists**: Billing, Intake, Inbox. Drafting + Compliance deferred.
- **Automation engine hook included** — `INVOKE_AI_SPECIALIST` action type.
- **MCP server omitted** — deferred; the founder's demo priority is the specialists themselves.
- **OCR approach**: text-first extraction; for image-only PDFs / scans, fall back to **Claude vision via the tenant's existing BYOAK key** — reuses the already-provisioned AI envelope, no new OCR vendor (Textract / Tesseract).
- **Inbox specialist supports both modes**: on-demand button (inline, user-triggered) and scheduled (via the automation hook — e.g. "every Monday 07:00 on every active matter, post a weekly summary").

## Objective

Turn Phase 52's single generic assistant into three SA-specialised inline in-product agents — Billing, Intake, Inbox — each placed where its drudgery lives. Extend the Phase 37 automation engine with a single `INVOKE_AI_SPECIALIST` action that lets any rule fire a specialist unattended, with output queued for human approval. Keep Phase 52's BYOAK cost model, confirmation-on-write guarantee, and capability-filtered tool resolution unchanged.

## Constraints & Assumptions

- **No new LLM provider abstraction.** Phase 52's `LlmChatProvider` + Anthropic adapter is unchanged. A specialist is a configured call into that provider — it is not a new provider.
- **Specialist = system prompt + tool subset + launcher metadata.** No specialist has its own database rows beyond its registry entry; there is no `Specialist` entity. The registry is Java in-code (`SpecialistRegistry`).
- **Confirmation-on-write unchanged.** Every write tool a specialist can call uses Phase 52's existing confirmation flow when invoked from chat. When invoked by an automation rule, the equivalent confirmation happens in the review queue (Section 5.3) — the rule itself cannot commit a write without a human approval step.
- **BYOAK cost model unchanged.** Each tenant provides its own Anthropic API key via Phase 21 `SecretStore`. Specialists (including Claude-vision OCR in Intake) cost the tenant the same per-call as chat does today. No platform-paid AI, no cost metering in this phase.
- **PRO tier gating unchanged.** Specialists are a PRO feature, same as the chat panel. Members on STARTER tenants see no specialist buttons.
- **Capability filtering unchanged.** Each specialist's tool set is resolved against the acting member's capabilities via `CapabilityAuthorizationService`. A member without `INVOICE_EDIT` who opens the Billing specialist sees it but cannot commit any write that requires that capability.
- **Inline placement is primary.** Each specialist launches from a button on its specific page. Phase 52's chat panel stays as the generalist fallback and **may** hand-off to a specialist ("for this, let me hand you over to the Billing Assistant") — not required in v1.
- **No chat-history persistence.** Specialist sessions are ephemeral, same as Phase 52.
- **No NL-to-rule builder.** Users do not describe automations in English; they build them in the Phase 37 rule wizard as today.
- **No fully agentic loops.** A specialist executes a bounded, user-initiated or rule-initiated task and returns. It does not independently decide to start new tasks.
- **No MCP server.** Deferred; Kazi's tool set stays in-app only.
- **Output review queue is one entity.** A new `AiSpecialistInvocation` table records every specialist run — invoked-by (MEMBER / AUTOMATION / SCHEDULED), input context, proposed output, review state, and the outcome applied. This is the single source of truth for "what has the AI been asked to do and what happened".
- **All three specialists speak SA English + ZAR + SA entity vocabulary.** System prompts explicitly constrain this.
- **OCR is Claude-vision-via-BYOAK only.** No new OCR vendor (Textract / Tesseract). Image-only PDFs are sent to Claude directly as part of the Intake specialist call; text-layer PDFs are parsed as text first. No new `OCRProvider` port in this phase.

## Section 1 — Specialist Framework

### 1.1 Registry

New Java type `Specialist` (record) + `SpecialistRegistry` (in-code, Spring-managed singleton), under `backend/.../assistant/specialist/`:

```java
public record Specialist(
    String id,                      // "BILLING" | "INTAKE" | "INBOX"
    String displayName,             // "Billing Assistant"
    String tagline,                 // short user-facing one-liner
    String systemPromptResource,    // classpath path to .md
    List<String> toolIds,           // subset of existing AssistantToolRegistry ids
    List<LauncherContext> launchers,// where in the UI this specialist can be launched
    boolean automationCapable       // can be invoked via INVOKE_AI_SPECIALIST
) {}

public record LauncherContext(
    String route,      // e.g. "/invoices/:id"
    String surface,    // e.g. "INVOICE_DRAFT_TOOLBAR"
    String ctaLabel    // e.g. "Polish time entries"
) {}
```

### 1.2 System Prompts

System prompts live as versioned markdown files under `backend/src/main/resources/assistant/specialists/`:

- `billing-za.md`
- `intake-za.md`
- `inbox-za.md`

Each file starts with YAML front-matter (`version`, `createdAt`, `specialist`) followed by the prompt body. Loaded at startup, cached. Reloadable via a dev-only endpoint (behind `local`/`dev` profile gate). The prompts **must** include:

- SA English tone + professional register + ZAR currency + SA date formatting (DD/MM/YYYY).
- The specialist's exact scope ("You are the Billing Assistant. You ONLY polish time-entry descriptions and suggest invoice line-item groupings. You do NOT create invoices, customers, or time entries.").
- Explicit refusal rules ("If asked to do X outside scope, politely hand off to the generalist.").
- SA-specific domain knowledge relevant to the specialist (see Sections 2, 3, 4).

### 1.3 Specialist API

Extends Phase 52's chat API with specialist-aware endpoints:

- `GET /api/assistant/specialists` — list specialists visible to the caller (filtered by PRO tier + capabilities + launcher matching). Returns `[{ id, displayName, tagline, ctaLabel, surface }]`.
- `POST /api/assistant/specialists/{specialistId}/sessions` — start a session. Request body: `{ contextRef: { entityType, entityId }, initialPrompt?: string }`. Returns a session handle that is then streamed against via the existing Phase 52 chat streaming endpoint (`POST /api/assistant/chat` with a `specialistId` added to the request). This means one streaming endpoint, two session-start patterns (generalist vs specialist).
- Phase 52's existing `/api/assistant/chat` gains an optional `specialistId` parameter. When present, the server resolves the specialist's system prompt + tool subset and injects both into the chat request.

### 1.4 Inline Launchers

New shared React component `frontend/components/assistant/specialist-launcher-button.tsx`:

```tsx
<SpecialistLauncherButton
  specialistId="BILLING"
  surface="INVOICE_DRAFT_TOOLBAR"
  contextRef={{ entityType: "invoice", entityId: invoice.id }}
  initialPrompt="Polish the time-entry descriptions on this invoice."
/>
```

Renders as an inline button with the specialist's `displayName` + icon + sparkle indicator. Clicking opens a **docked specialist panel** on the right side of the page (not a full-screen chat) with the session pre-seeded by `initialPrompt` and `contextRef`. Panel reuses most of Phase 52's chat component tree; a thin `<SpecialistPanel>` wraps the generic `<AssistantPanel>` with the specialist's branding.

The panel renders:
- Specialist name + tagline at top.
- Pre-seeded first message ("I'm here to polish the time entries on invoice #2025-042. Ready?").
- Existing Phase 52 streaming chat mechanics + confirmation cards for write tools.
- A "hand off to generalist" link at the bottom of the panel for out-of-scope questions.

### 1.5 Tests

- ~5 backend integration tests: registry wiring, tool-subset resolution against capabilities, PRO tier gate, system prompt loading + caching, specialist-scoped `chat` request.
- ~3 frontend tests: launcher button renders for authorised users only; specialist panel opens pre-seeded; hand-off to generalist preserves session context.

## Section 2 — Billing Assistant

### 2.1 Placement

- **Primary**: button on invoice draft page toolbar (`frontend/app/(authenticated)/invoices/[id]/page.tsx` when invoice is in `DRAFT` status). Label: **"Polish with AI"**. Surface id: `INVOICE_DRAFT_TOOLBAR`.
- **Secondary**: button on the Unbilled Time → Generate Invoice dialog (Phase 10 83B). Label: **"Suggest line-item grouping"**. Surface id: `UNBILLED_TIME_DIALOG`.

### 2.2 Scope

The Billing Assistant does two things:

1. **Polish time-entry descriptions** into professional client-facing text. Example: `"call w/ J re: transfer"` → `"Telephone consultation with client regarding property transfer documentation"`. Works on one or many time entries in a single call.
2. **Suggest line-item grouping** — given N time entries (often 20–50 on a billable matter), propose 4–7 grouped line items with aggregated hours and concise descriptions. Example: 47 entries of varying text → `{"Client consultations and correspondence (12.5h)", "Document drafting and review (18h)", "Court preparation (8h)", ...}`.

Both outputs are **proposals** — the user reviews a diff view (before / after) and approves, edits, or rejects per row.

### 2.3 Tool Subset

From Phase 52's existing registry:
- `GetInvoice` — read invoice + its lines
- `GetUnbilledTime` — read time entries
- `GetTimeSummary` — read aggregated time info for context
- `ListProjects`, `GetProject`, `ListCustomers`, `GetCustomer` — context lookup

**Two new tools** specific to Billing:
- `ProposeTimeEntryPolish(timeEntryIds: UUID[], polishedDescriptions: string[])` — proposes new descriptions. Writes to the `AiSpecialistInvocation` review queue, not directly to the entity. Human approval applies the changes.
- `ProposeInvoiceLineGrouping(invoiceId: UUID, groups: [{description, hours, sourceTimeEntryIds: UUID[]}])` — proposes a grouping. Review queue entry.

No `CreateInvoiceDraft`, no `UpdateTaskStatus`, no `CreateCustomer`. Strict subset.

### 2.4 SA Context in System Prompt

The `billing-za.md` prompt enforces:

- South African English tone (not US / UK idioms). "Telephone consultation" not "phone call". "Correspondence" not "emailing". "Attendance" in legal-za.
- ZAR currency always. Hours to one decimal place.
- Tariff-awareness for `legal-za`: when polishing legal time entries, prefer LSSA tariff-compatible phrasing where plausible (e.g. "Perusal" for reading correspondence, "Attendance upon" for meetings) — but do not invent tariff codes.
- VAT awareness: zero-rated disbursements (sheriff, deeds office, court fees) vs standard-rated services. The polish function does not touch VAT, but if asked about grouping disbursements it keeps them on separate lines.
- Professional register: avoid first-person ("I called the client") → passive or nominal ("Telephone attendance on the client").
- No invented details: if the source entry says `"call w/ J"`, the polish is `"Telephone attendance"` — not `"one-hour conference call with senior partner"`. Hallucinations are the failure mode to guard against.

### 2.5 Tests

- ~4 backend integration tests: polish tool records review-queue entry with before/after payload; grouping tool records review-queue entry with proposed line items; capability gate (member without `INVOICE_EDIT` cannot launch Billing); SA-context validation via a prompt-level assertion test (send a snippet, assert the response contains no forbidden idioms via a small checklist).
- ~2 frontend tests: launcher button appears only on DRAFT invoices; diff-review UI renders + accept/reject round-trips.

## Section 3 — Intake Assistant

### 3.1 Placement

- **Primary**: button on the new matter / customer create dialog (`frontend/components/customers/customer-create-dialog.tsx`). Label: **"Extract from uploaded documents"**. Surface id: `CUSTOMER_CREATE_DIALOG`.
- **Secondary**: button on the information-request response review page (Phase 34 `frontend/app/(authenticated)/requests/[id]/page.tsx`). Label: **"Extract client-supplied fields"**. Surface id: `INFO_REQUEST_REVIEW`.
- **Tertiary**: button on a customer detail page when the customer has missing prerequisite fields (per Phase 33 prerequisite infrastructure). Label: **"Fill in from uploads"**. Surface id: `CUSTOMER_DETAIL_PREREQ`.

### 3.2 Scope

The Intake Assistant reads uploaded documents attached to the context entity and extracts structured data into Kazi's fields. Two modes:

1. **Parse text-layer PDFs** — fast path, zero extra cost beyond the LLM call. Pre-extract text from the PDF via a server-side library (e.g. `pdfbox`), then pass the text + schema to Claude.
2. **Vision fallback for image-only PDFs / scans** — if text-layer extraction produces less than a threshold of text (builder tunes), send the PDF or rasterised images directly to Claude as vision inputs. Uses the tenant's existing BYOAK key. No new OCR vendor.

Output is a proposed field-set: e.g. `{ "firstName": "Thandiwe", "lastName": "Nkosi", "rsaIdNumber": "8503156123084", "address.line1": "42 Bree Street", "address.city": "Cape Town", "address.postalCode": "8001", "taxNumber": "4500123456", "entityType": "NATURAL_PERSON" }`.

User reviews the proposed values in a diff panel against the current form state; per-field accept / reject / edit.

### 3.3 Tool Subset

From Phase 52's existing registry:
- `GetCustomer` — read current customer state (if updating)
- `ListCustomers` — context (e.g. detect duplicate customer by name)

**Three new tools** specific to Intake:
- `ListDocumentsForContext(contextRef: {entityType, entityId}): DocumentRef[]` — lists documents attached to the customer / request / matter.
- `ExtractTextFromDocument(documentId: UUID): ExtractResult` — server-side text extraction from a PDF. Returns `{ text, characterCount, hasTextLayer }`. If `hasTextLayer=false` or `characterCount < threshold`, the caller switches to vision.
- `ProposeCustomerFieldExtraction(customerId?: UUID, proposedFields: Map<String, Any>)` — proposes field values. Writes to the review queue.

### 3.4 SA Context in System Prompt

The `intake-za.md` prompt enforces:

- **RSA ID number** — 13 digits, format `YYMMDDSSSSCAZ`. System prompt describes the structure and tells the model to validate internally (date-of-birth consistency, citizenship digit, Luhn checksum) — flag in the proposal if validation fails but the model found a number claiming to be an ID.
- **CIPC registration number** — formats like `2015/123456/07` (year / sequence / entity type).
- **SA VAT number** — 10 digits starting with `4`.
- **SA postal code** — 4 digits.
- **SA province list** (9 provinces, exact names).
- **Entity type recognition** — sole prop, (Pty) Ltd, CC, NPC, trust, inc, public company, partnership.
- **Matrimonial property regime** (for legal-za family matters) — "married in community of property", "married out of community without accrual", "married out of community with accrual" — recognised from marriage-certificate annexure references.
- **Trust documents** (legal-za) — extract trustee names, trust registration number (format `IT XXX/YYYY`), trust deed date.
- **POPIA awareness**: the specialist flags extracted fields that constitute "special personal information" under POPIA §26 (health, race, biometric) so Kazi's intake workflow can prompt for explicit consent.

### 3.5 Vision Fallback Flow

When `hasTextLayer=false` for an uploaded PDF:

1. Server rasterises PDF pages to JPEG at 150 DPI (or streams to Claude as PDF if the provider supports PDF vision directly — check at implementation time; Anthropic's API accepts PDFs natively, so rasterisation may be unnecessary).
2. Chat request includes the images / PDF as a vision content block alongside the extraction prompt.
3. Claude returns the extracted fields in the same shape as text-path extraction.
4. Review queue entry tracks which path was taken (`extractionPath: "TEXT"` | `"VISION"`) for diagnostics.

**Cost note**: vision requests are more expensive per call than text. The specialist prefers text where possible and falls back to vision only when needed. No cost guard-rail in this phase (BYOAK — tenant absorbs cost); surfaced in the gap report if problematic.

### 3.6 Tests

- ~5 backend integration tests: text-layer extraction path; vision fallback path (mock Anthropic response); review queue entry shape; RSA ID validation flag; capability gate (member without `CUSTOMER_EDIT` cannot launch Intake); POPIA §26 flag for a seeded health-mentioning document.
- ~3 frontend tests: per-field diff review accept / reject; empty-documents empty state; vision-fallback indicator visible when used.

## Section 4 — Inbox Assistant

### 4.1 Placement

- **Primary (on-demand)**: button on the matter / project detail page's **Activity** or **Comments** tab (Phase 6.5 65A). Label: **"Summarise recent activity"**. Surface id: `MATTER_ACTIVITY_TAB`.
- **Secondary (on-demand)**: button on the customer detail page. Label: **"Summarise customer activity"**. Surface id: `CUSTOMER_DETAIL`.
- **Tertiary (scheduled)**: via the `INVOKE_AI_SPECIALIST` automation action (Section 5) — e.g. a pre-seeded rule "every Monday 07:00, on every `ACTIVE` matter, invoke Inbox Assistant with lookback=7d".

### 4.2 Scope

The Inbox Assistant reads recent activity within a time window on a context entity (matter / project / customer) and produces a **short narrative summary** as a comment. Activity sources:

- Comments (Phase 6.5 59) — user-authored and system-authored.
- Domain events / activity feed (Phase 6.5 64) — status changes, document uploads, time entries, invoice activity.
- Information requests + responses (Phase 34) — what the client has sent / is still pending on.
- Trust transactions (Phase 60) [legal-za only].
- Deadline approaching flags (Phase 48 360A + Phase 51).

Output is a structured summary comment like:

```
**Week ending 2026-04-19**
• Client uploaded two documents (certified ID, proof of address) — awaiting paralegal review.
• Opposing counsel proposed 2026-04-25 for the interlocutory application; diary conflict with existing matter M-0037.
• Trust balance changed: R25,000 deposit received, allocated to disbursement reserve.
• Outstanding: response to settlement offer (original request 2026-04-12, 7 days overdue).
```

The summary is posted as a new comment on the matter with a visible **"Posted by Inbox Assistant"** tag. On-demand invocations ask for user approval before posting; scheduled (rule-driven) invocations post directly with the Inbox Assistant tag (review queue still records the invocation for audit, but does not block the post — this is the one exception to human-approval-before-write for AI output, justified because the output is a **comment, not a mutation of an existing entity**; still capability-gated and tenant-opt-in via the rule).

### 4.3 Tool Subset

From Phase 52's existing registry:
- `GetProject`, `GetCustomer` — context
- `ListTasks` — recent tasks
- `SearchEntities` — recent time entries / invoices / documents
- Existing Activity / Comments read (if not already a tool, add as new read tool).

**Two new tools** specific to Inbox:
- `GetMatterActivityWindow(entityType, entityId, from: Instant, to: Instant): ActivityBundle` — one-shot fetch of all activity in a window. Reduces N round-trips.
- `PostInboxSummary(entityType, entityId, summaryMarkdown: string, mode: "REVIEW" | "DIRECT")` — posts a comment, tagged as Inbox Assistant output. `REVIEW` mode queues for human approval; `DIRECT` mode posts immediately (used by scheduled rules that opt in).

### 4.4 SA Context in System Prompt

The `inbox-za.md` prompt enforces:

- Terminology match to the firm's `terminologyKey` (matter vs project, client vs customer, etc.) — read from tenant's OrgSettings at session start.
- **Privilege / confidentiality awareness** — summaries should not fabricate or leak privilege assertions. Specifically: no speculation about opposing-party intent, no legal opinion, no claim about what the firm "should" do. Just factual recap.
- **Matter-stage awareness** — different summary focus by lifecycle stage (PROSPECT vs ACTIVE vs CLOSING).
- **Professional register** — third-person factual ("Client uploaded...", "Opposing counsel proposed..."), not first-person or advisory.

### 4.5 Scheduled Mode Integration

The automation engine hook (Section 5) enables rules like:

- **Pre-seeded template: "Weekly matter summary"** — trigger = `SCHEDULED` (cron-like), condition = `project.status == ACTIVE`, action = `INVOKE_AI_SPECIALIST(specialistId=INBOX, lookback=P7D, mode=DIRECT)`. Seeded as an optional template that firms can enable.
- **Pre-seeded template: "Matter reactivated"** — trigger = `ProjectStatusChanged` where `newStatus=ACTIVE` and `previousStatus=ON_HOLD`, action = `INVOKE_AI_SPECIALIST(specialistId=INBOX, lookback=P30D, mode=DIRECT)` — post a catch-up summary.

The `SCHEDULED` trigger type is a small extension to Phase 37's trigger registry — cron-expression + last-run timestamp per rule. If Phase 37 doesn't already support pure-scheduled triggers (it supports delayed-follow-up actions, which is different), this extension lives in Epic E (Section 5).

### 4.6 Tests

- ~4 backend integration tests: activity-window fetch correctness across 6 source types; REVIEW vs DIRECT posting paths; scheduled trigger firing; capability gate.
- ~2 frontend tests: summary comment renders with "Posted by Inbox Assistant" tag; lookback-window picker on the on-demand button.

## Section 5 — Automation Engine Hook: `INVOKE_AI_SPECIALIST`

### 5.1 Action Executor

New action executor `InvokeAiSpecialistActionExecutor` under `backend/.../automation/executor/`. Registered with Phase 37's `AutomationActionExecutor` dispatcher. Action config (JSONB on `AutomationAction.config`):

```json
{
  "specialistId": "INBOX",
  "contextRef": { "entityType": "project", "entityId": "{{event.entityId}}" },
  "initialPrompt": "Summarise the last 7 days of activity on this matter.",
  "mode": "REVIEW" | "DIRECT",
  "timeoutSeconds": 60
}
```

Variables supported via Phase 37's existing variable resolver (`{{event.entityId}}`, `{{event.actorId}}`, etc.).

### 5.2 Execution Flow

1. Rule fires → `InvokeAiSpecialistActionExecutor.execute(context, config)`.
2. Executor resolves the specialist, builds a non-interactive chat session with `initialPrompt`, runs it to completion (including any tool calls the specialist decides to make).
3. Writes an `AiSpecialistInvocation` row with input context + proposed output + `invokedBy=AUTOMATION` + `automationActionExecutionId` FK + mode.
4. If `mode=REVIEW` → status `PENDING_APPROVAL`; review-queue UI picks it up; nothing else happens until human approval.
5. If `mode=DIRECT` (Inbox scheduled summaries only, and only for comment-posting specialist outputs) → status `AUTO_APPLIED`; output applied immediately; still visible in the review queue as an audit record.
6. Failures → status `FAILED`, error captured in `ActionExecution.errorMessage` per Phase 37 pattern.

### 5.3 `AiSpecialistInvocation` Entity + Review Queue

Migration V108+ (tenant schema). New table `ai_specialist_invocations`:

| Column | Type | Notes |
|---|---|---|
| `id` | UUID PK | |
| `specialistId` | VARCHAR(40) | `BILLING` / `INTAKE` / `INBOX` |
| `invokedBy` | VARCHAR(20) | `MEMBER` / `AUTOMATION` / `SCHEDULED` |
| `actorId` | UUID | Member id when `invokedBy=MEMBER`; the rule's actor for automation |
| `automationActionExecutionId` | UUID FK nullable | Link to Phase 37 `ActionExecution` when rule-invoked |
| `contextEntityType` | VARCHAR(50) | |
| `contextEntityId` | UUID | |
| `status` | VARCHAR(30) | `RUNNING` / `PENDING_APPROVAL` / `APPROVED` / `REJECTED` / `AUTO_APPLIED` / `FAILED` / `EXPIRED` |
| `proposedOutput` | JSONB | Specialist-specific payload (polished descriptions, extracted fields, summary markdown, etc.) |
| `appliedOutput` | JSONB | What was actually applied (may be edited by reviewer) |
| `createdAt` | TIMESTAMP | |
| `reviewedAt` | TIMESTAMP | |
| `reviewedById` | UUID | |
| `rejectReason` | TEXT | |

Indexes: `(status, createdAt)` for review-queue listing; `(contextEntityType, contextEntityId)` for entity-level audit; `(automationActionExecutionId)` for cross-reference.

### 5.4 REST API

- `GET /api/assistant/invocations?status=&specialistId=&from=&to=&page=&size=` — list (capability-gated, `TEAM_OVERSIGHT` or owner).
- `GET /api/assistant/invocations/{id}` — detail.
- `POST /api/assistant/invocations/{id}/approve` — approve (optionally with edited `appliedOutput`).
- `POST /api/assistant/invocations/{id}/reject` — reject with reason.
- `POST /api/assistant/invocations/{id}/retry` — retry a FAILED invocation.

An approval / rejection emits a domain event (`AiInvocationApproved` / `AiInvocationRejected`) that audit + notifications can subscribe to.

### 5.5 Review Queue UI

New page `frontend/app/(authenticated)/settings/automations/ai-queue/page.tsx` (lives under the existing Automations settings area from Phase 37). Filterable list + detail drawer with proposed-vs-current diff viewer per specialist output type. Badges on the Automations sidebar entry for pending count.

Per-entity detail pages also show a small "Pending AI suggestions" widget on matter / customer / invoice detail when any `PENDING_APPROVAL` invocation targets that entity — approve / reject inline.

### 5.6 Pre-Seeded Automation Templates

## Out of Scope

- **Drafting Assistant and Compliance Assistant.** Phase 71+ candidates.
- **MCP server.** Explicitly deferred 2026-04-22.
- **Dedicated OCR vendor** (AWS Textract, Tesseract, Google Document AI). Claude-vision via BYOAK is the only OCR path in this phase.
- **NL-to-rule builder** (English → automation rule config). Users build rules via the Phase 286 wizard as today.
- **Fully agentic loops** where the AI independently decides to start new bounded tasks. Every specialist invocation is user-initiated or rule-initiated.
- **Cost metering / rate limiting per tenant.** BYOAK — tenant absorbs cost. Observed in gap report, not gated.
- **Persistent chat / session history.** Specialist sessions are ephemeral; only `AiSpecialistInvocation` rows persist.
- **Portal-facing AI.** Specialists are firm-side only.
- **Multi-turn conversations inside the specialist panel beyond the one bounded task.** Once the specialist finishes its proposed output, the session ends; further questions go to the generalist via hand-off.
- **Cross-specialist chaining inside one session** ("Billing did this, now Inbox, now Drafting"). One specialist per session.
- **Fine-tuned / custom models.** Vanilla Claude via BYOAK.
- **Multi-provider routing** (OpenAI / local models). Anthropic only via the existing single-provider adapter.
- **Client-facing terminology translations** of AI output (e.g. auto-translating English to isiXhosa / Afrikaans). SA English only.
- **Automatic application of Billing / Intake outputs** (no DIRECT mode for them — only Inbox comment-posting gets DIRECT). Every Billing / Intake output requires human approval.
- **Self-learning from approvals / rejections.** The review queue records the outcome; no feedback loop into prompt tuning in this phase.

## ADR Topics

- **ADR-??? — Specialist = system prompt + tool subset + launcher metadata, not a new entity.** Records why: avoids migration churn, keeps specialists reloadable from classpath, lets us add specialists via code + markdown only. Documents the registry pattern.
- **ADR-??? — Inline launchers are primary; chat panel is secondary generalist fallback.** Records the founder's 2026-04-22 call. Rationale: drudgery removal only works when the AI is at the point of work, not at a ⌘K shortcut. Chat panel stays because some questions are genuinely generalist.
- **ADR-??? — Human approval is the default for AI output; Inbox comment-posting is the single DIRECT-mode exception.** Records why: (1) comment posting is low-stakes, reversible by deletion, and clearly attributed via the "Posted by Inbox Assistant" tag; (2) requiring approval on every scheduled weekly summary would create review-queue fatigue that dominates the drudgery cost of the automation. Every other mutation stays REVIEW-gated.
- **ADR-??? — OCR via Claude vision over tenant BYOAK key; no separate OCR vendor.** Records why: zero additional cost-centre for the platform, zero new dependency, tenant absorbs cost inside existing envelope, quality is competitive with dedicated OCR for intake document types. Trade-off: slower than Textract per page for very large batches; acceptable at phase-pilot volumes.
- **ADR-??? — SA-specialisation in system prompts (not in the LLM).** Records that Kazi does not fine-tune a model; SA context is carried in prompts (`billing-za.md`, `intake-za.md`, `inbox-za.md`). Rationale: fine-tuning costs + vendor lock-in + weak ROI at firm-pilot scale. Trade-off: prompt tokens cost per call; mitigated by Anthropic prompt-caching when used.
- **ADR-??? — `AiSpecialistInvocation` is one table, many output shapes (JSONB).** Records why a single review-queue table serves all specialists rather than three per-specialist tables. Trade-off: less type-safe at the DB layer; mitigated by sealed-class configs on the Java side (same pattern as Phase 37 `AutomationAction.config`).
- **ADR-??? — `SCHEDULED` trigger extension to Phase 37.** Records the small extension to the trigger registry for cron-like scheduled rule firing (distinct from Phase 37's delayed-follow-up action scheduler, which fires N hours after another event). Needed for the Inbox weekly-summary template.

## Style & Boundaries

- Follow all conventions in `frontend/CLAUDE.md` and `backend/CLAUDE.md`.
- Backend controllers stay one-line delegates (per `backend/CLAUDE.md` Controller Discipline). The new invocation controller + specialist controller conform.
- Capability gating: all new backend endpoints use `@RequiresCapability`. `AI_ASSISTANT_USE` already exists from Phase 52 — reuse. Review queue actions gate on `TEAM_OVERSIGHT` (reusing Phase 69's capability for consistency with audit surfaces).

- Code location: backend under `backend/.../assistant/specialist/` + `backend/.../assistant/invocation/` + `backend/.../automation/executor/InvokeAiSpecialistActionExecutor.java`. Frontend under `frontend/components/assistant/specialists/` + `frontend/app/(authenticated)/settings/automations/ai-queue/`.
- Migrations: one (V108) for `ai_specialist_invocations` table + indexes.
- Tests: follow `backend/CLAUDE.md` taxonomy. Use `TestcontainersConfiguration` (embedded Postgres). Anthropic API responses mocked via WireMock (same pattern as Phase 387B in Phase 52). Claude-vision responses mocked the same way.
- System prompt tests: a lightweight "prompt linter" asserting each specialist prompt contains required SA-context tokens ("ZAR", "SA English", RSA ID / CIPC / VAT format references per specialist) — protects against accidental deletion during prompt maintenance.
- i18n: specialist display names, tag lines, launcher labels, review queue copy all go through the Phase 43 message catalogue.
- Empty states: Phase 43 `<EmptyState>` for the review queue empty list.

- Multi-vertical coexistence: all three specialists work for `legal-za`, `accounting-za`, `consulting-za`, and unprofiled tenants. SA context in prompts applies across verticals; vertical-specific nuance (matter vs project terminology, tariff awareness) flows via session-time OrgSettings lookups.

- No changes to Phase 37 `AutomationAction.config` shape — the new action type uses the existing JSONB pattern with its own sealed-class config variant.
- No changes to Phase 21 `SecretStore` / BYOAK flow. The vision path uses the same key as the chat path; no separate vision provider configured.
- No changes to portal (`portal/`). Specialists are firm-side.
